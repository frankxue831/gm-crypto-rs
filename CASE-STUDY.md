# The Harness, Not the Builder — A Verification-First Method for Hard, Unforgiving Software

This is an honest account of building a pure-Rust SM2/SM3/SM4 and TLCP cryptography library under a verification system designed on the assumption that the builder couldn't be trusted to get it right. The builder happened to be an AI coding agent — which is why the discipline mattered more, not less: the agent did the construction (the implementation, the tests, the harnesses, the build loop) while a human owned scope, invariants, evidence standards, review pressure, and release authority. The method is the subject; the agent is one detail of how it was built. Crypto is a stress test here, not proof the method is generally safe; the piece is deliberate about what it does and does not show.

## The hook: a shipped leak, and the harness we widened to see it

The first release of this library, `v0.1.0`, leaked timing. The secret-key inversion ran in data-dependent time — `|tau| ≈ 0.70` measured directly on `ConstMontyForm::invert`, where anything above `0.20` is a finding ([SECURITY.md](SECURITY.md), "v0.1 vs. main"). That is a real side-channel in a real published crate. It happened because the dudect harness at the time had a structural blind spot: no target split the secret class the way that bug lived.

The fix was not a patch. The next cycle expanded the harness first — two direct invert diagnostics (`ct_fn_invert`, `ct_fp_invert`) for the secret-dependent inversion sites, plus `ct_sign_k_class` for the sign-level nonce-class path — and only then changed the dependency. On `main` the two invert diagnostics sit at `0.0071` and `0.0063`, `ct_sign_k_class` at `0.0708`, and the retained sign diagnostic `ct_sign` at `0.0044` ([SECURITY.md](SECURITY.md), post-upgrade table). The diagnosis came from the targets the agent was directed to build; the fix came from the `crypto-bigint` upgrade that followed. The lesson is the ordering: the verification system was widened to make the failure visible *before* it was closed, not narrated away.

Honesty cuts the other way too. In `v0.19` we built a self-calibrating CI gate — a fix-vs-fix noise-floor probe meant to relatively-gate two demoted invert targets against the runner's own measurement noise. We tested it on 100K samples and it was **falsified**: the probe stayed quiet at median `~0.0052` while the real class-split target spiked to `0.2606` on one CI leg — a ratio of `50`, a would-FAIL the probe could not see ([docs/v0.5-dudect-recalibration.md](docs/v0.5-dudect-recalibration.md), v0.19 resolution). The runner noise lives in the two-input class-split difference, not the operation duration a same-input probe measures. We reverted the gate to non-blocking telemetry — and **kept the falsified probes** as committed evidence of why the idea doesn't work, so nobody rebuilds it. A documented dead end is part of the trail.

### The narrow thesis

The AI agent was the **construction engine**. It wrote the SM2/SM3/SM4 and TLCP implementation code, wrote the dudect timing-leak targets, wrote the fuzz harnesses (30 targets, [fuzz/](fuzz/)), regenerated KAT vectors, and ran the build/clippy/test loop — boxed inside a verification system whose constraints the human set: `unsafe_code = "forbid"` on the core crate, a single-`Failed`-variant error invariant the agent is explicitly *not allowed to "improve"* ([SECURITY.md](SECURITY.md), "Failure-mode invariant"), per-cycle scope docs, and a constant-time gate calibrated against the CI runner's noise floor.

The **human owned everything load-bearing, and mostly before code existed**: scope (the TLCP arc was pre-registered in v1.5 — a standalone doc-only cycle committed before any TLCP code — and each cycle records its maintainer-signed forks in its own scope doc, e.g. [docs/v1.9-scope.md](docs/v1.9-scope.md); the squash-merge history proves the v1.5 ordering, not a strict scope-before-code for every later cycle), invariants (the failure-mode rule, the fixed-K retry loop, the forbid lint), evidence standards (which KAT source is authoritative — the [GM/T 0003.5 worked example](docs/v1.1-sm2kx-kat-sourcing.md), not a hand-rolled vector), review pressure (executed adversarial reviews that *compiled the risky slice in a worktree* and caught bugs a read-through misses — `v1.9` surfaced 6 must-fixes, two live at compile time, per [docs/v1.9-tlcp-ffi-plan.md](docs/v1.9-tlcp-ffi-plan.md)), and release authority (every `cargo publish` and SSH-signed tag is the maintainer's authenticated call). The pre-registration was real: a four-cycle TLCP map written in [docs/tlcp-decomposition.md](docs/tlcp-decomposition.md) §7 and walked exactly, v1.6 → v1.9.

The thesis is not "the agent built hard crypto." It is: **in this trail, an agent inside a verification harness the human designed produced cryptographic code that survived strong checks — because the harness, not the agent's judgment, is what catches the failures.**

### What we are not claiming

- **Not "agent-led."** This was expert-directed. The human had the domain knowledge to write the constraints, pick the authoritative oracle, and recognize a wrong KAT assumption. Strip the director and the receipts above become invisible failures.
- **Not "the agent is independently trustworthy."** It shipped the `v0.1` leak and built the `v0.19` gate that didn't work. In this trail it produced code that survived the checks — but the reliability belongs to the *harness that caught its misses*, not to the agent. A narrower claim than "trustworthy."
- **Authorship is attested, not audited.** That the *agent* (not the human) wrote a given line is the maintainer's attestation — git authorship is the human committer, and the session logs aren't public. The receipts prove the *artifacts and their measurable properties*, not who typed them.
- **Crypto review is not delegable.** The agent wrote the tests; the human judged whether they tested the right thing. The executed adversarial reviews ([v1.6](docs/v1.6-tlcp-key-schedule-plan.md) through [v1.9](docs/v1.9-tlcp-ffi-plan.md)) and multi-model scope consults exist *because* a human decided the agent's output needed an independent compile-and-check pass before merge.
- **No control case.** We did not run the same person building the same library without an agent. We cannot quantify the speedup or the quality delta — that is **not instrumented**. This is one trail, not a measured comparison.
- **This writeup shares the project's selection and narration bias.** It was assembled from the same artifacts the agent wrote, by the same parties. We linked the failures (`v0.1`, `v0.19`) precisely so the trail is checkable, but a self-authored case study is not an audit.
- **Generalizes only to domains with cheap, strong oracles.** Crypto is a **stress test, not a safety proof**: KAT vectors, constant-time measurements, fuzzers, and reference implementations make a wrong answer *hard to hide once an oracle covers it*. That is exactly what made the harness work — and exactly why the method does not obviously transfer to domains where correctness is fuzzy and the oracle is a human opinion.

## Why crypto — a stress test, not a proving ground

Crypto was chosen because it punishes the one failure mode AI agents are best at producing: **confident, plausible, wrong.** A function that compiles, passes a casual read, and returns bytes that *look* right — but diverges from the standard in the third block, or leaks the key through a timing side-channel. In most domains that ships. In crypto, cheap and strong oracles make it hard to hide — on any behavior an oracle actually covers. (The v0.1 leak hid precisely because no target covered it yet.)

A correctly-sourced KAT is hard to fake by accident. A known-answer test either reproduces the standard's byte vector or it doesn't — no partial credit, no "close enough" — and the catch is only as good as the vector, which is why the human (not the agent) owns which source is authoritative. When the agent assumed the GM/T 0003.5 SM2 key-exchange example used per-party email IDs (`ALICE123@YAHOO.COM` / `BILL456@YAHOO.COM`), every curve point still reproduced — and the shared key came out `3C136283…` instead of the standard's `6C893473…`. The KAT is what caught the wrong assumption; the diagnosis ([docs/v1.1-sm2kx-kat-sourcing.md](docs/v1.1-sm2kx-kat-sourcing.md)) is that both parties use the default ID `1234567812345678`. A plausible-but-wrong guess could not survive contact with the vector.

Timing is the second oracle, and it caught something already shipped. v0.1.0 published with a secret-dependent inversion: the dudect harness measured `|tau| ≈ 0.70` directly on `ConstMontyForm::invert`. The fix wasn't trust — it was a harness expansion (`ct_fn_invert` / `ct_fp_invert` / `ct_sign_k_class`) that diagnosed the leak and now sits near baseline (`ct_fn_invert` 0.0071, `ct_fp_invert` 0.0063, [SECURITY.md](SECURITY.md)). A leak you cannot see is a leak you cannot catch; the oracle made it visible.

National-standard conformance is unforgiving the same way — GM/T, GB/T, RFC 8998 specify exact bytes, and [docs/tlcp-decomposition.md](docs/tlcp-decomposition.md) tracks twelve numbered verification items (D-1…D-12) against the primary text, per cycle. And the target was genuinely novel: per the project's v1.5 survey (2026-06; [CLAUDE.md](CLAUDE.md)), *"No pure-Rust TLCP exists (only Tongsuo C bindings) — the toolkit is novel surface"* — none known to this project at that time. There was no pure-Rust implementation to copy; the toolkit was constructed against the standards and cross-checked against the gotlcp / GmSSL / OpenSSL oracles, which judged the result.

One caveat, stated plainly: **this is a stress test, not proof the method is generally safe.** Crypto is the right harness *because* its oracles are cheap and strong — KATs, timing measurements, byte-exact standards. Domains without that property (UI, business logic, anything graded by human taste) give you none of this enforcement for free. What crypto demonstrates is narrow and real: when you *can* build oracles that make wrong-but-plausible hard to hide, an AI agent can be driven hard against them. It does not demonstrate that the agent is safe where the oracles are weak.

**Steal this:** before pointing an agent at a domain, ask what your KAT is — the cheap, binary check, sourced from outside the agent, that fails loudly on plausible-but-wrong. If you can't name one, you're trusting output you can't audit.

## The operating loop

Treat the builder as a capable but unsupervised contractor: left alone it will build the wrong thing well, or "improve" a guardrail into a vulnerability. Seven disciplines kept it pointed at the right target. All are human-owned. None are about prompting better — they're about deciding *up front*, then refusing to relitigate.

### Step 1 — Scope narrowly, and pre-register the cuts

The unit of work was never "build TLCP." It was one cycle, scoped in a doc, with the next cycles named but deliberately deferred.

The receipt: [docs/tlcp-decomposition.md](docs/tlcp-decomposition.md) §7 pre-registered a four-cycle map as a standalone non-publishing cycle (v1.5), committed before any TLCP code existed — v1.6 key schedule (gaps G1+G3) → v1.7 record protection (G2) → v1.8 certificate-pair/chain verification (G4) → v1.9 one FFI cycle for the whole toolkit. The gap analysis in the same doc justified the cuts: roughly 90% of the crypto already shipped, so the work was two thin composition layers, one genuinely new constant-time-sensitive module, and one carefully-scoped X.509 extension — not a from-scratch protocol. That map was then walked *exactly*: each of [docs/v1.6-scope.md](docs/v1.6-scope.md), [docs/v1.7-scope.md](docs/v1.7-scope.md), [docs/v1.8-scope.md](docs/v1.8-scope.md), [docs/v1.9-scope.md](docs/v1.9-scope.md) is one cycle, and the arc closed where it was pre-registered to close.

What made scope binding rather than aspirational: each scope doc front-loads numbered **fork-questions** (`Q6.1`–`Q6.10`, `Q9.1`–`Q9.8`, …), and the cycle-shaping ones were *maintainer-locked at scope time* — the human picked the shape (one FFI cycle vs. several; opaque key-handles vs. by-value), the agent didn't get to drift mid-implementation. The "path validation is where small auditable cycles go to die" trap was named in the decomposition doc itself, and the defense — verify only what the TLCP handshake needs, name the permanent holes loudly — was a scope decision, not something the agent discovered under pressure. Pre-registering also makes scope *creep* visible: when v1.8 wanted to widen the known-critical-extension set, that was a flagged fork (`Q8.7b`), explicitly maintainer-signed as "the one widening of minimal" — a logged exception, not a silent expansion.

**Steal this:** size each cycle so its correctness is checkable against *one* oracle or spec section you can name up front — if a cycle needs two independent oracles, or a brand-new one you haven't built yet, split it. Write the cut-list and the deferral-list *before* the agent runs, as numbered fork-questions you answer; a fork the agent re-opens mid-cycle is scope you never actually closed.

### Step 2 — Encode the invariants the agent may not "improve"

The most dangerous agent behavior in security code is helpfulness. An agent will, unprompted, make an error message "more informative" — and turn a single failure mode into a padding oracle. The fix is to state the invariants as hard constraints the agent re-reads every session, and to make violating them a rejection-on-sight, not a discussion.

The headline invariant is the **single failure mode**, documented in [SECURITY.md](SECURITY.md): `verify_with_id` returns a `bool`, never a `Result`; DER decode returns `None`, never a typed error; and the workspace `Error` has exactly one variant, `Failed`, marked `#[non_exhaustive]` so no consumer can match on a richer set that doesn't exist. The doc closes the loop on the agent's instinct directly: *"PRs that distinguish failure modes — even 'helpfully' — will be rejected."* That sentence exists because the failure it forbids is the one an eager construction engine produces by default.

Two more invariants are stated the same way — non-negotiable, rationale attached so the agent can't argue around them:

- **`unsafe_code = "forbid"` on the crypto core.** Not "warn," not "discouraged." The `no_std` core literally cannot compile an `unsafe` block; the unavoidable raw-pointer FFI is quarantined to the `gmcrypto-c` shim, which runs `unsafe_code = "warn"` so every `unsafe` site surfaces in clippy. The core's forbid lint is the enforcement, not a style preference.
- **Constant-time discipline on secrets.** No `==` / `if` / Rust `bool` on a secret-derived value — `subtle`'s constant-time primitives instead, and the SM2 sign retry runs a fixed `K=2` iterations regardless of which candidate is valid ([SECURITY.md](SECURITY.md)). A "simplify the retry loop to bail on first success" refactor is exactly the plausible improvement this invariant is written to forbid.

These live in [CLAUDE.md](CLAUDE.md) — the project's living memory, re-read at the start of every session — alongside an explicit *Don't* list ("don't make `mode_cbc::decrypt` distinguish failure modes," "don't reduce the SM2 retry-loop iteration count"). The invariants aren't enforced by hoping the agent remembers; they're handed to the agent as input on every run, and the dudect and fuzz harnesses fail when one is broken.

**Steal this:** spell out the guardrails the agent must *not* optimize — with the attack each one prevents written next to it — in your agent's standing-memory file: the one the harness injects at every session start (here, [CLAUDE.md](CLAUDE.md)), not a doc it reads only when told. If your harness has no auto-reload, make re-reading the invariant file the first step of every task. Then "I made it more helpful" reads as a rejected diff, not a feature.

### Step 3 — Give bounded tasks, and the context to do them

The unglamorous half of "context engineering" is artifact discipline: the agent does bounded work well when the spec, the invariants, and the reference vectors are already written down and it is *forced* to read them. There is no prompt-wizardry layer here.

The receipt:

- [CLAUDE.md](CLAUDE.md) is the standing memory — hard constraints (`unsafe_code = "forbid"` on the core, the constant-time discipline, the single-`Error` rule), per-version history, the dudect gate table, an architecture map, and an explicit "Don't" list of the mistakes an agent makes by default. It is re-read at the top of each session, not summarized once and forgotten.
- Each cycle opens with its own scope doc — [docs/v1.6-scope.md](docs/v1.6-scope.md), [docs/v1.8-scope.md](docs/v1.8-scope.md), [docs/v1.9-scope.md](docs/v1.9-scope.md) — and a plan doc such as [docs/v1.9-tlcp-ffi-plan.md](docs/v1.9-tlcp-ffi-plan.md) that enumerates the tasks and the must-fixes for the cycle.
- The bigger arc was pre-registered: [docs/tlcp-decomposition.md](docs/tlcp-decomposition.md) §7 laid out a four-cycle map — key schedule (v1.6) → record protection (v1.7) → chain/pair verify (v1.8) → one FFI cycle (v1.9) — and the cycles were walked in that exact order.

The point of this discipline is that the task handed to the agent is *bounded*: not "implement TLCP," but "implement the key schedule specified in `docs/v1.6-scope.md`, against the KATs in `docs/v1.6-kat-sourcing.md`, without violating the invariants in `CLAUDE.md`." The agent re-reads the corpus; it does not re-derive the design.

**Steal this:** your leverage is the spec/invariant/KAT corpus you maintain, not the cleverness of any single prompt. Make the agent read it every time, and scope each task small enough that "done" is checkable against a document you wrote first.

### Step 4 — Require executable evidence, never assertions

An agent will happily tell you a thing passes. The standard that holds is an objective artifact — a known-answer test against a reference oracle, a timing-leak measurement, a fuzz run — that fails loudly when the agent is wrong. Crypto is a useful stress test for exactly this reason: the oracles are cheap and strong, so a wrong implementation has little room to hide on any behavior an oracle covers. (That makes failures visible where they're covered; it does not make the method generally safe in domains without such oracles.)

The receipt:

- A KAT caught a wrong assumption a read-through would have missed. Implementing SM2 key exchange, the agent reasonably assumed the GM/T 0003.5 worked example used the per-party IDs `ALICE123@YAHOO.COM` / `BILL456@YAHOO.COM`. Those belong to a *different* standard's annex. With them, every curve point still reproduces — but `Z_A`/`Z_B`, and therefore the shared key `K`, diverge, yielding the wrong `K = 3C136283 0B075A6F 891DBF65 1997BF50` instead of the correct `6C893473…`. The KAT forced the diagnosis; the fix was fixture-only. See [docs/v1.1-sm2kx-kat-sourcing.md](docs/v1.1-sm2kx-kat-sourcing.md).
- Behavior is pinned to external reference implementations, not self-certified: SM2/SM3/SM4 KATs cross-checked against gmssl 3.1.1 ([crates/gmcrypto-core/tests/interop_gmssl.rs](crates/gmcrypto-core/tests/interop_gmssl.rs)), SM4-GCM byte-identical to gmssl's `sm4 -gcm` ([crates/gmcrypto-core/tests/sm4_gcm_kat.rs](crates/gmcrypto-core/tests/sm4_gcm_kat.rs)), SM4-CCM byte-identical to OpenSSL 3.x EVP ([crates/gmcrypto-core/tests/sm4_ccm_kat.rs](crates/gmcrypto-core/tests/sm4_ccm_kat.rs)), and the TLCP key schedule against OpenSSL's `TLS1-PRF digest:SM3` ([crates/gmcrypto-core/tests/tlcp_key_schedule_kat.rs](crates/gmcrypto-core/tests/tlcp_key_schedule_kat.rs), recipe in [docs/v1.6-kat-sourcing.md](docs/v1.6-kat-sourcing.md)).
- Two standing harnesses make timing and parser behavior measurable rather than asserted: a dudect timing harness (23 functions — leak targets plus noise-floor probes and a negative control; four targets are currently demoted to telemetry, see the denominator) at [crates/gmcrypto-core/benches/timing_leaks.rs](crates/gmcrypto-core/benches/timing_leaks.rs), and 30 fuzz targets across the adversarial-input, decode/decrypt, differential (streaming vs single-shot), record, cert-chain, and C-ABI surfaces ([fuzz/](fuzz/)). The dudect harness is the same one that caught the original v0.1 timing leak at `|tau| ≈ 0.70` — see [SECURITY.md](SECURITY.md).

The human owns the evidence standard: which oracle counts, what the gate is, what "measured but not gated" means. The agent constructs the implementation and writes the tests; it does not get to declare its own work correct.

**Steal this:** pin every load-bearing behavior to a reference oracle the agent doesn't control — a KAT vector, an interop check, a measurable gate — and treat "the agent says it passes" as unverified until the artifact runs. When no external oracle exists (the TLCP layers were novel surface), either compose the new layer entirely from primitives each already pinned to an oracle, or hand-derive the expected value from the primary spec text and pin *that* — never let the agent self-certify a behavior no oracle judges.

### Step 5 — Adversarial review, executed

With the plan in hand, the next move is to *attack* it before any code lands. Two independent challenge mechanisms run every cycle, and the load-bearing word in both is **executed** — the reviewer doesn't read, it builds.

**(a) A second model as the scope adversary.** Each cycle's scope doc is consulted against a different model (Codex `gpt-5.5`, sometimes Grok), and the findings are folded back before TDD starts. This isn't a rubber stamp — the consults land real corrections: v1.6 folded 7 Codex findings (typed `[u8;48]` pre-master, loud completer names), v1.8 folded 5 including the maintainer-signed Q8.7b unknown-critical-extension reject, v1.9 folded 5, all recorded inline in [docs/v1.6-scope.md](docs/v1.6-scope.md), [docs/v1.8-scope.md](docs/v1.8-scope.md), and [docs/v1.9-scope.md](docs/v1.9-scope.md). The heaviest pass was the pre-1.0 re-audit: a multi-model adversarial review (Codex + Grok, **each finding source-verified by the human orchestrator**) that returned NO-GO with two API/ABI blockers plus should-fixes — see [docs/v1.0-reaudit.md](docs/v1.0-reaudit.md). The human picks the reviewer, runs it, and decides which findings are real; the models supply independent pressure.

**Steal this:** point a *different* model at your scope doc before you write code. A model that didn't draft the plan has no sunk cost in it.

**(b) Reviewers that compile the dangerous slice in a worktree.** A read-through review approves code that doesn't build. So the rule is: the reviewer checks out a worktree, **implements the riskiest slices**, and runs the suite. Bugs a prose review glides past surface as compiler errors.

- **v1.9** — an Opus executed review implemented the three riskiest FFI slices in an isolated worktree, ran 88/88 c_smoke plus the full workspace green, returned GO-WITH-FIXES with 6 must-fixes — and **two were caught live at compile time**: an `alloc::vec::Vec` `E0433` in the `std` shim, and a missing `at_time` reverse-`From` conversion. Both are in the review log at [docs/v1.9-tlcp-ffi-plan.md](docs/v1.9-tlcp-ffi-plan.md) (W2). Neither is the kind of thing a careful reader reliably spots; the compiler spots both instantly.
- **v1.8** — an Opus executed review built Tasks 1–5 against 26 tests using real minted SM2 certs, and caught 9 clippy lints including a hard `E0433` plus the S1 pair-binding gap (equal issuer *Name* alone doesn't pin the issuer *key*) — documented in the v1.8 paragraph of [CLAUDE.md](CLAUDE.md).
- **v1.7** — a Fable-5 executed review *ran the Lucky13 deprotect core* in a worktree and caught 2 compile bugs plus the missing-ceiling Lucky13 chain-break (the constant-time equalization silently degraded) and a false-assurance dudect axis — v1.7 paragraph of [CLAUDE.md](CLAUDE.md). A timing-side-channel bug the *tests still pass* with is exactly the class a read-through cannot catch.
- **v1.6** — the Fable-5 review went GO-WITH-FIXES (A1–A6) and **independently re-derived every KAT vector** in a scratch tree rather than trusting the plan's claimed values — v1.6 paragraph of [CLAUDE.md](CLAUDE.md).

The pattern across all four: the dangerous part is the part that gets built twice — once in the plan, once by the adversary — and the second build is where the bugs die. The division of labor stays intact throughout: the reviewer (agent) compiles and reports; the human reads the GO/NO-GO and decides what ships. The single-`Failed`-variant failure-mode invariant in [SECURITY.md](SECURITY.md) is one of the guardrails the reviewer is explicitly *not* allowed to "improve" away.

**Steal this:** a read-through review misses code that doesn't compile and timing bugs the tests still pass with — so make the reviewer *build* the dangerous slice in a throwaway worktree and run the suite. Pick the slice where a bug would be invisible to prose review: new `unsafe`/FFI boundaries, constant-time equalization, anything where the tests stay green while the code is wrong. Build those twice; trust the read-through for the mechanical rest.

### Step 6 — Keep failures *and* the denominator

A clean trail that only logs wins is a sales deck. What makes this method trustworthy is that the misses are in the repo, dated, and as legible as the catches — and that the catches are reported against the total, not in isolation.

Three failures are load-bearing in this repo:

- **The shipped-then-caught timing leak.** v0.1.0 went to crates.io on `crypto-bigint 0.6` with `|tau| ≈ 0.70` on `ConstMontyForm::invert` — a real, published constant-time leak. The fix wasn't a quiet amend; it was a harness expansion (`ct_fn_invert` / `ct_fp_invert` / `ct_sign_k_class`) that *diagnosed* the blind spot, plus the dependency upgrade. The four targets now run far under the `0.20` gate — `ct_fn_invert` 0.0071, `ct_fp_invert` 0.0063, `ct_sign_k_class` 0.0708, `ct_sign` 0.0044 — though `ct_sign_k_class` at 0.0708 sits an order of magnitude above the invert pair and was itself later demoted to the nightly sentinel (below). The whole arc is written up where a reader looks first, in [SECURITY.md](SECURITY.md) and [CLAUDE.md](CLAUDE.md)'s "v0.1 timing-leak narrative." The leak is not hidden behind the release; it *is* part of the documented posture.
- **A falsified idea, kept as evidence.** v0.19 built a self-calibrating relative gate — a fix-vs-fix noise-floor probe meant to re-promote the demoted invert targets — and it *failed its own test*: on the `sm4-bitsliced-simd` leg the probe measured median 0.0052 while `ct_fp_invert` measured 0.2606 (ratio 50, would-FAIL), because the runner's image-noise lives in the two-input class-split difference, not the operation duration a same-input probe can see. The gate was reverted to telemetry/sentinel — but the probes were **kept as telemetry**, because they're the evidence the next attempt needs. The full falsification, with the table, is in [docs/v0.5-dudect-recalibration.md](docs/v0.5-dudect-recalibration.md). A negative result documented well enough to stop the next person from re-running it is worth more than a feature.
- **A guardrail demotion that names what it costs.** On 2026-06-17 the `ct_hmac_sm3` dudect target was demoted to telemetry (PR) / sentinel@0.55 (nightly) — the fourth two-input class-split target hit by the same runner image-noise. The demotion didn't end at "demoted." It states the residual out loud: unlike the invert pair (backstopped by `ct_sign`), the HMAC primitive now has **no direct non-composite backstop**, so the TLCP key-schedule `p_sm3` path that rode it loses its direct guard until a class-split-aware "noise-twin" exists. That named hole is in [SECURITY.md](SECURITY.md) and [docs/v0.5-dudect-recalibration.md](docs/v0.5-dudect-recalibration.md). Weakening a check is sometimes correct; weakening it *silently* is how a suite rots.

The point underneath all three: assurance is reported as a **denominator**, not a highlight reel — 23 dudect harness functions, 30 fuzz targets, the per-target gate table in [CLAUDE.md](CLAUDE.md), each with its current posture (gated / telemetry / sentinel) and its known blind spot. (Crypto is what makes this cheap to enforce: on covered behavior the oracles are strong and unambiguous, so a regression can't hide. That's a stress test, not a proof the method is generally safe.) The catches only mean something against that total.

**Steal this:** commit your failures with the same care as your features — dated, diagnosed, and linked — and report assurance as a denominator (N targets, current posture, named residual), never as a list of wins. A falsified experiment kept as telemetry is a deliverable. And set a ship/block line for residuals: a named residual is shippable only if it is (a) written into the security-posture doc, (b) *bounded* — you can state what range of failure it leaves unguarded — and (c) not on a freshly-introduced surface. A residual you can't bound, or one covering new code, blocks.

### Step 7 — Reserve irreversible judgment for humans

The agent is the construction engine. It writes the code, writes the tests, runs the gates, drafts the scope. It does not get to make the moves that can't be taken back. The governance here is deliberately *light* — a few hard lines, not a process — but the lines are exactly the irreversible ones.

- **Publishing is the maintainer's authenticated call.** Every `cargo publish` and every SSH-signed tag is a deliberate human action; authority stayed with the maintainer throughout — the one v1.1.0 publish executed on delegation was recorded explicitly as a non-precedent. The agent's path stops at branch + PR — a hard rule in [CLAUDE.md](CLAUDE.md)'s workflow notes ("`cargo publish` is the irreversible step (the maintainer's authenticated call, not the agent's)").
- **No agent commit ever lands on `main` directly.** The agent branches and opens a PR, full stop — even where [CLAUDE.md](CLAUDE.md) notes the *maintainer* sometimes commits direct. The agent-produced path is always branch + PR; that's the workflow contract, and CI fires on the PR before anything reaches `main`.
- **The forks that shape a cycle are locked by a human, up front.** The cycle-shaping decisions — one FFI cycle vs. several, opaque handles vs. by-value, which trust the chain verifier does and does *not* assert — are surfaced as explicit questions and **maintainer-locked via AskUserQuestion** before planning starts. v1.9's four cycle-shaping forks and v1.8's were settled this way (see the v1.8/v1.9 paragraphs in [CLAUDE.md](CLAUDE.md) and [docs/v1.9-scope.md](docs/v1.9-scope.md)). The agent proposes a recommended shape; the human picks, and the pick is recorded in the scope doc before a line is written.
- **The judgment that can't be automated stays with the human.** Scope, invariant ("PRs that distinguish failure modes — even 'helpfully' — will be rejected," [SECURITY.md](SECURITY.md)), evidence standard, and the felt sense of whether a cycle is actually done — these are the maintainer's. A private reflective log holds the felt-fields; the agent drafts the historian's spine, never the judgment. The division is precise: the agent constructs and tests; the human decides what "good" means and when to ship.

**Steal this:** draw the human/agent line at *irreversibility*, not at *difficulty*. Let the agent build and test freely; reserve publish, merge-to-main, and the cycle-shaping forks for an explicit human action — recorded before the work, authenticated at the end. Light governance is enough if it sits on exactly the moves you can't undo.

## When NOT to use this

The method earned its receipts on a hard target, and that makes it easy to over-generalize. So be precise about the boundary. This worked **because SM2/SM3/SM4 and TLCP come with cheap, strong, externally-defined oracles** — published KAT vectors, `gmssl`/OpenSSL cross-checks, a dudect timing harness, a fuzz corpus. The agent could construct an implementation and the *oracle*, not the agent, decided whether it was wrong. Crypto here is a **stress test** — failures are loud once an oracle covers them — not a license. Where the failure is subtle *and* there's no cheap strong oracle, the same setup is dangerous.

Before adopting this on your own domain, run a three-question go/no-go test: **(1)** Can a wrong-but-plausible output be made to *fail* a binary, automatable check the agent doesn't control? **(2)** Is that check cheap enough to run every cycle? **(3)** Does the check come from outside the agent — a spec, a reference implementation, a measurement — and not a human eyeballing it? Three yeses make the method a candidate — not automatically safe (a KAT the agent miswired or that's too narrow still passes, so a human owns the oracle). Any no, and the agent is a drafting aid, not a construction engine you can trust under verification.

Concretely, do not trust the agent as the deciding party for:

- **Novel design vs. implementing a pinned standard.** The agent implemented standards that already existed; the human owned the map. The TLCP toolkit was novel surface — "No pure-Rust TLCP exists (only Tongsuo C bindings)" as of the project's v1.5 survey (2026-06; [CLAUDE.md](CLAUDE.md)) — and precisely because there was no pure-Rust reference to copy, a human pre-registered the decomposition into four cycles *before* any TLCP code (the standalone v1.5 doc-only cycle): [docs/tlcp-decomposition.md](docs/tlcp-decomposition.md) §7 mapped G1→G4 onto v1.6–v1.9, and the cycles were walked exactly. The agent built inside that map; it did not draw it.
- **Unaudited constant-time claims.** A timing harness tells you when something is *measurably* leaking, not that it's safe. The published v0.1.0 shipped a real leak — `|tau| ≈ 0.70` on `ConstMontyForm::invert` — that a read-through never caught; it took a human-directed harness expansion (`ct_fn_invert` / `ct_fp_invert` / `ct_sign_k_class`) to *diagnose* it ([SECURITY.md](SECURITY.md), v0.1 timing-leak narrative). And the harness itself is not self-validating: the v0.19 self-calibrating gate was **built, tested, and falsified** — the noise-floor probe sat at median 0.0052 while the real target spiked to 0.2606, ratio 50, so the gate was reverted to telemetry ([docs/v0.5-dudect-recalibration.md](docs/v0.5-dudect-recalibration.md), v0.19 resolution). A green timing gate is evidence a human calibrated, not a guarantee.
- **Protocol interpretation without KATs.** Reading a standard and getting it subtly wrong is the default failure mode, and it is invisible until a vector disagrees. The GM/T 0003.5 SM2-KX worked example uses the default ID `1234567812345678` for *both* parties, not the `ALICE`/`BILL` identities from a different annex; with the wrong IDs every curve point still reproduces but the derived key `K` silently diverges to `3C136283…` ([docs/v1.1-sm2kx-kat-sourcing.md](docs/v1.1-sm2kx-kat-sourcing.md), Task 1.5). The KAT — not the agent's reading — caught it. No KAT, no catch.
- **Release and security-API-semantics decisions.** Every `cargo publish` and SSH-signed tag stayed the maintainer's authenticated call. And the failure-mode invariant is a guardrail the agent is explicitly *not* allowed to "improve": `verify_with_id` returns `bool`, DER decode returns `None`, the workspace `Error` has exactly one `#[non_exhaustive]` variant, and "PRs that distinguish failure modes — even 'helpfully' — will be rejected" ([SECURITY.md](SECURITY.md), Failure-mode invariant). That rejection rule exists because the agent's instinct to be helpful is, in this domain, an oracle-leaking bug.

The honest summary: the agent was the construction engine. The oracle decided correctness, and a human chose the oracles, set the evidence bar, applied review pressure, and held release authority. Take away the cheap strong oracle and you take away the thing that made this safe.
