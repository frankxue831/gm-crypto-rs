# v1.0 pre-publish re-audit (GO/NO-GO)

**Date:** 2026-05-31 · **Workspace:** 0.16.0 · **Branch context:** `main` @ `475b856`
(post-v0.22) · **Status of this report:** the finding inventory + verdict that gates the
deliberate, irreversible `1.0.0` publish.

## 0. Why this exists

`docs/v1.0-readiness.md` (v0.21) produced a **CONDITIONAL GO** with one outstanding item
(§3.A `crypto-bigint` exposure), which **v0.22 resolved**. Before taking the irreversible
1.0 step, the maintainer asked for a **full pre-publish re-audit** — not a re-run of the
v0.21 single-pass tooling check, but an **independent, adversarial, multi-model review**,
on the research-backed principle that *self-assurance (KAT / fuzz / dudect) is necessary
but not sufficient — it cannot see logic-level invariant gaps or API-finality hazards an
adversarial reviewer can* (the v0.19 dudect falsification is the in-repo evidence).

### Methodology

- **Four dimensions:** A = API / SemVer finality; B = adversarial crypto-correctness;
  C = publish mechanics; D = honest-disclosure readiness.
- **Per security-critical dimension (A, B):** two **independent** model reviews
  (Codex `gpt-5.5` `--sandbox read-only`, Grok `--sandbox read-only`), each given the
  same neutral adversarial brief with **no** sight of the other's output, **then** every
  finding **adversarially verified against the source by the orchestrator** before being
  accepted or down-rated. This is the "diverse-lens panel + adversarial verify" pattern:
  it both corroborates true findings (two independent hits) and catches overstatement
  (the orchestrator down-rated 2 of Codex's 8 B-findings after reading the code).
- **Posture:** internal re-audit + **honest disclosure**, no external human-audit gate
  (RustCrypto-style); non-publishing until a clean re-review of the remediation.

## 1. Verdict

> **NO-GO for an immediate 1.0 publish as-is.** Remediate the BLOCKERs + the genuine
> should-fixes in a focused **v0.23 cycle**, re-run this multi-model review on the diff,
> **then** publish 1.0.

**The core cryptography is sound.** Both independent reviewers confirmed the load-bearing
constant-time / correctness properties (see §3, "mutually confirmed non-findings"): no
broken primitive, no exploitable runtime crypto BLOCKER. **The findings are about what
becomes *irreversible* at 1.0 (API/ABI shape) and defense-in-depth hardening
(zeroize-on-failure, a spec length ceiling, doc honesty)** — precisely the class a
pre-publish freeze review exists to catch, and all fixable in a bounded cycle.

The re-audit earned its keep: the v0.21 single pass declared GO with only §3.A
outstanding; this deeper multi-model pass surfaced **two genuine API BLOCKERs and a set
of correctness/hardening should-fixes it never raised.**

## 2. Dimension A — API / SemVer finality   (Codex + Grok: both CHANGES-NEEDED)

At 1.0 every public name, type, signature, generic bound, trait item, **and C ABI symbol
+ header** becomes a permanent contract. Findings, with the orchestrator's adjudicated
severity:

### BLOCKERs
- **A-1 — C ABI header ⟷ build-contract mismatch.** The committed `gmcrypto.h` is a
  "maximal union": cbindgen emits the `sm4-aead`/`sm4-xts`-gated GCM/CCM/XTS (single-shot
  + streaming) prototypes **unconditionally**, but a default `cargo build -p gmcrypto-c`
  (`default = []`) does **not** export those symbols. CI builds default + `--features
  sm4-aead` separately but never links a default artifact against the full header. A C
  consumer linking the default library against the shipped header gets link errors for
  declared symbols — a permanent ABI hazard at 1.0. *(Both reviewers ranked this #1.)*
  Files: `crates/gmcrypto-c/include/gmcrypto.h`; `src/lib.rs:978,1330,1449`;
  `Cargo.toml:54,60,68`; `.github/workflows/ci.yml:223,230,241`.
- **A-2 — raw `ProjectivePoint` + unchecked `from_point` frozen on the high-level key
  path.** `Sm2PrivateKey::public_key() -> ProjectivePoint`, `Sm2PublicKey::{from_point
  (no on-curve check), point(), From<ProjectivePoint>}`, plus the full `add/double/neg/
  generator/identity` arithmetic — all `pub`/re-exported (`sm2::ProjectivePoint`).
  Callers can construct a public key from an unchecked point, bypassing the on-curve
  guards `from_sec1_bytes`/`encrypt`/`decrypt` enforce, and the bare EC arithmetic is
  frozen as 1.0 contract. This was the deliberate v0.22 Group-C "decouple-only" choice
  (it names no `crypto-bigint` type once `to_affine` is hidden), but it **over-commits an
  internal RCB EC type** as permanent surface. *(Both reviewers flagged; Grok noted the
  missing on-curve check on `from_point` as the sharpest edge.)*
  Files: `sm2/point.rs:16`; `sm2/mod.rs:35`; `public_key.rs:19,25,64`;
  `private_key.rs:131,133`; `sec1.rs:143-151`.

### SHOULD-FIX-BEFORE-1.0
- **A-3 — `rand_core::{CryptoRng + Rng}` trait bounds in public signatures.**
  `sm2::sign_with_id<R>` / `encrypt<R>` name pre-1.0-churning `rand_core` traits in the
  stable surface — structurally the **same hazard class** as the `crypto-bigint`
  exposure v0.22 just resolved. *(Grok.)* Files: `sm2/sign.rs:158`, `encrypt.rs:103`.
  **Nuance:** unlike `crypto-bigint` (a numeric impl detail with no interop value),
  `rand_core` **is** the ecosystem interop point — see §5 Fork 4.
- **A-4 — 38 low-level `asn1::{reader,writer,oid}` items frozen forever.** TLV/tag
  consts, `read_*`/`write_*`, generic `collect_sequence_of`, const-generic OID encoder,
  panicking writers. v0.3 internal DER helpers made `pub` for cross-module reuse; freezing
  a DER toolkit as main-crate surface. *(Both.)* Files: `asn1/{reader,writer,oid}.rs`.
- **A-5 — public weak-shape `traits::{Hash, Mac, BlockCipher}`.** `&[u8]` args that panic
  on wrong length; the module doc *itself* says they are "NOT as bound-on-public-API
  constraints," yet they are `pub` and freeze at 1.0. *(Both.)* File: `traits.rs:35-94`.
- **A-6 — `sec1::EcPrivateKey.public: Option<ProjectivePoint>` public field.** Leaks the
  internal point type; resolved by the A-2 decision. *(Grok.)* File: `sec1.rs:143-151`.

### ACCEPTABLE-WITH-DOC
- `#[doc(hidden)] pub` Group-A items are removed from the SemVer *contract*
  (cargo-public-api / semver-checks exclude them) but remain **nameable/reachable** — the
  known, deliberate v0.22 posture (the `gmcrypto-simd` precedent). Disclose honestly:
  "contract-clean, **not** a privacy boundary."
- `asn1::writer` `panic!` on the documented 16 MiB ceiling (reachable from public helpers).
- C `gmcrypto_sm2_privkey_{to,from}_pkcs8` = *encrypted* PKCS#8 PEM; name non-specific
  (Codex; minor). `bytes_be` vs `sec1_*` naming, `EcPrivateKey` for an SM2-only decoder
  (Grok; minor).

## 3. Dimension B — adversarial crypto-correctness   (Codex + Grok: both CHANGES-NEEDED)

> **No exploitable runtime crypto BLOCKER on the primitives.** Every finding is a
> zeroize-on-failure gap, a spec length-ceiling gap, or a documentation item.

Each finding was verified against source; severities are the orchestrator's adjudication
after reading the code (Codex raw rating → adjudicated).

### Genuine — fix in v0.23
- **B-3 — CCM decrypt leaks tentative plaintext on tag-fail.** `mode_ccm::decrypt`
  (decrypt-before-verify, required by CCM) returns `None` on tag mismatch but drops
  `tentative_pt: Vec<u8>` **without zeroize** (a `Vec` drop frees, does not wipe). The
  code's own comment claims "no explicit zeroize needed," while the *GCM* decrypt comment
  says the opposite for exactly this case, and SM2 decrypt *does* zeroize on its fail arm
  — an internal inconsistency. *(Both; Grok rated BLOCKER, Codex SHOULD-FIX.)* Adjudicated:
  **SHOULD-FIX-HIGH** (one-line `tentative_pt.zeroize()`; memory-scraping is partly outside
  the trusted-host threat model, but it contradicts the project's own posture).
  File: `mode_ccm.rs:227,252`.
- **B-1 — single-shot SM4-GCM lacks the `2^36 − 32` byte ceiling.** `mode_gcm::encrypt`
  GCTRs any-length plaintext; past the limit the 32-bit counter wraps → keystream reuse.
  The **streaming** path enforces it (`GCM_MAX_PT_BYTES` + poison); single-shot does not.
  *(Codex; Grok did not flag — it is a spec-limit gap, not an output-correctness bug at
  sane sizes.)* **API-finality twist:** `encrypt` returns **infallible** `(Vec, [u8;16])`,
  so the spec-correct fix is a **breaking change to a fallible return** (panic is excluded
  by the no-panic posture) — see §5 Fork 3. Practical bar: a single ~68 GB in-memory
  buffer. File: `mode_gcm.rs:129-156`.
- **B-4 — SM2 sign nonce `k` + scalar intermediates not zeroized.** `try_sign_once` /
  `sample_nonzero_scalar` hold `k`, `(1+d)^-1`, `k−rd`, `s`, the RNG `buf`, `candidate`
  as plain `Fn`/`U256`/`[u8;32]` stack locals; `Fn` (`ConstMontyForm`) is not
  `ZeroizeOnDrop`. A nonce leak → private-key recovery from `(r,s)`. Inconsistent with the
  private key's own `ZeroizeOnDrop`. Fixable via `crypto-bigint`'s `zeroize` feature.
  *(Codex.)* SHOULD-FIX. File: `sm2/sign.rs:177-220`.
- **B-2 — `HmacSm3` zeroization doc overclaims.** The struct doc claims `finalize`/`verify`
  "zeroize it before returning" and "the `Drop` impl wipes the state" — but `finalize`
  does no zeroize and **there is no `impl Drop for HmacSm3`**. `new()`'s buffer-wipe claim
  *is* accurate (Grok confirmed the construction-time zeroize is correct). Residual: the
  folded keyed compression state in the inner/outer `Sm3`. *(Codex flagged "false claim";
  Grok rated HMAC zeroize correct — reconciled to: doc overclaims a Drop/finalize-zeroize
  that don't exist.)* SHOULD-FIX (add `Drop`+zeroize to honor the doc, or correct the doc).
  File: `hmac.rs:130-136,206-211`.
- **B-5 — SM2 KDF `u32` counter wrap.** Wraps above `32·(2^32−1) ≈ 137 GB` output (absurd
  for SM2; small payloads). `encrypt` is already fallible → a guard is cheap. *(Codex.)*
  SHOULD-FIX-LOW. File: `sm2/encrypt.rs:184-196`.
- **B-7 — SM2 nonce sampling `if bool::from(..)` on a `k`-derived `Choice`.** Reject
  probability ≈ `2^-32` (`n ≈ 2^256 − 2^224`; Codex corrected the reviewers' earlier
  `2^-224`) — ~1 in 4 billion calls does a second iteration, so not a meaningful timing
  leak, but it is a literal violation of the documented "no `if`/`bool` on secret-derived
  values" rule at the sampling site. *(Both; Grok refined; Codex corrected the magnitude.)*
  Adjudicated SHOULD-FIX (fix to a **fixed-budget masked sampler** — the K=2 sign-retry
  pattern — collapsing exhaustion to `Failed`). File: `sm2/sign.rs:215-217`.

### Document-only (ACCEPTABLE-WITH-DOC)
- **B-6 — streaming CBC emits plaintext before `finalize`.** `take_output()` is
  `#[doc(hidden)] pub` (C-FFI only); emitting non-final blocks before the final PKCS#7
  check is **inherent to streaming unauthenticated CBC** (confidentiality, not integrity —
  that is GCM/CCM). `finalize` returns a single `None`; the v0.20 differential fuzzer
  proves streaming ≡ single-shot. *(Codex rated BLOCKER → down-rated after verification;
  Grok did not flag.)* Clarify the C-FFI decryptor contract: incremental output is
  unverified until `finalize`.
- **B-8 — PKCS#8 decrypt failure timing** is already doc-acknowledged (`pkcs8.rs:47`) and
  dudect-gated on the password class; the standard PBES2-not-perfectly-CT limitation.
- **B-9 — short-plaintext SM2 encrypt KDF-zero retry timing.** For `L=1`, the spec-mandated
  KDF-zero retry leaks a ~`2^-8` predicate on the ephemeral `k` via retry timing
  (≤`2^-32` for `L≥4`). *(Grok; new.)* Negligible for realistic lengths; document.
- **B-10 — crate-wide `Choice → bool` final-validity branch.** After full CT work the
  unavoidable validity bit converts to `bool` to decide `Some`/`None` (zeroize on the fail
  arm). No differential work/timing precedes the decision. *(Grok.)* Document the posture:
  "constant-time until the one unavoidable public validity bit."

### Mutually confirmed NON-findings (core primitives sound)
C1 on-curve check **before** any `mul_var(d_B, C1)`; `ct_eq` tag/MAC compares (never
memcmp); XTS bit-reflected **masked** `mul_alpha` + CT `Key1==Key2` reject; RCB complete
addition formulas (no point-at-infinity special case); fixed-K=2 **masked** sign retry;
all inversions via `ConstMontyForm::invert → CtOption`; correct HMAC ipad/opad; SM2
decrypt **does** zeroize on its fail arm; single `Failed`/`None`/`false` everywhere
required; no early-exit padding scans, no secret-indexed table lookups, no `.len()`-derived
control flow on secrets.

## 5. v0.23 remediation — recommended fork resolutions (Codex to confirm; maintainer approves scope)

These are **design forks**, resolved per the v0.22 process (recommend → Codex-confirm →
single maintainer scope approval). Final wording lands in `docs/v0.23-scope.md`.

1. **A-2 ProjectivePoint depth — RECOMMEND: reshape the high-level path to speak keys/bytes.**
   `public_key() -> Sm2PublicKey`; `Sm2PublicKey::from_sec1_bytes` (already on-curve-checked)
   becomes the public constructor; `#[doc(hidden)]` `from_point`/`point`/`From<ProjectivePoint>`
   and the bare `add/double/neg` arithmetic (kept `pub` for internal/dev-crate use, the
   Group-A pattern). `ProjectivePoint` stays a public *opaque* type but is no longer an EC
   arithmetic API or an unchecked key constructor. Breaking, pre-1.0-only (ships with 1.0).
2. **A-1 C ABI contract — RECOMMEND: make the AEAD + XTS symbols always-on in `gmcrypto-c`.**
   Drop the forwarding `sm4-aead`/`sm4-xts` *cargo* features on the **C shim** so the
   committed header == the default build (one complete, permanent ABI). `gmcrypto-core`
   keeps its features (Rust callers opt in); `gmcrypto-c` (the integration layer, std-OK)
   always compiles every mode. Alternative (rejected unless Codex prefers): `#ifdef`-guard
   the header + a feature→symbol map (fragile for C consumers).
3. **B-1 GCM single-shot — RECOMMEND: make `encrypt`/`encrypt_with_tag_len` fallible**
   (return the workspace `Result`/`Option`, rejecting `> 2^36−32`), consistent with the
   streaming poison + `decrypt`'s `Option`. Breaking, pre-1.0-only. Lighter alternative
   (Codex to weigh): keep infallible + document the ceiling as a caller contract.
4. **A-3 `rand_core` bound — RECOMMEND: accept-and-document (do NOT decouple).** Unlike
   `crypto-bigint`, `rand_core` **is** the ecosystem interop point; naming `CryptoRng` is
   the idiomatic, interop-preserving choice (RustCrypto-wide convention). A custom RNG
   trait would hurt interop for net-negative benefit. Document the deliberate coupling
   (mirroring the `from_scalar` escape-hatch reasoning). *(This is the key fork to confirm.)*
5. **A-4/A-5 surface trim — RECOMMEND: `#[doc(hidden)]` `asn1::{reader,writer,oid}` +
   `traits::{Hash,Mac,BlockCipher}`** (keep `pub` for internal cross-module/dev use; the
   Group-A pattern). Keep `asn1::{encode_sig,decode_sig}` + `Sm2Ciphertext` public (the
   v0.22 wire types — intended public). Regenerate the cargo-public-api baseline.
6. **B-2/B-3/B-4/B-5 zeroization + ceiling fixes** (not forks): CCM `tentative_pt.zeroize()`
   on fail; enable `crypto-bigint` `zeroize` + wipe sign-nonce intermediates; add
   `HmacSm3` `Drop`+zeroize (honor the doc); KDF `u32` guard; GCM ceiling per Fork 3.
7. **B-7 nonce-sampling** — make the rejection CT or carve it out in the CT-posture docs.
8. **C-1 + D-1 + B-6/8/9/10 docs** — `description` fix; audit-status + CT-posture
   disclosure refresh.

**Verification gate for the cycle:** byte-output-identical runtime (KAT + gmssl interop
11/11) for everything except the *deliberately* changed signatures; full workspace tests +
per-feature clippy + `cargo doc` + 18 fuzz + MSRV-1.85 + wasm32 + `cargo deny`; cargo-public-api
baseline regenerated; the C header drift-check green under the new always-on contract; **a
clean re-run of this multi-model review on the diff**; non-publishing (workspace stays
`0.16.0`) until the deliberate 1.0 cut.
