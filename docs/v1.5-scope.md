# v1.5 scope — TLCP decomposition (non-publishing design cycle)

Status: direction maintainer-locked 2026-06-12 (Q5.1); deliverable is
`docs/tlcp-decomposition.md`. Codex-consulted (W2) and maintainer-signed
on the arc forks (Q5.3/Q5.4) before merge.

## 1. Sign-off decisions

- **Q5.1 Direction** — **TLCP decomposition** (maintainer-locked,
  2026-06-12). Both TLCP crypto prerequisites (v1.1 SM2-KX, v1.3 x509)
  plus their FFIs (v1.2, v1.4) are shipped; the v1.4 scope doc said TLCP
  "needs its own decomposition cycle" — this is that cycle. The
  alternatives considered: X.509 chain validation scoped cold (rejected:
  enters the Codex-flagged trap before the TLCP requirements are mapped;
  the decomposition derives the profile first — `tlcp-decomposition.md`
  §4) and a deferred-items consolidation cycle (rejected: everything on
  that list is recorded non-blocking).
- **Q5.2 Deliverable + posture** — **doc-only, non-publishing** (the
  v0.14/v0.21 precedent): `docs/tlcp-decomposition.md` (protocol
  anatomy, gap analysis, the derived chain-validation profile, end-state
  recommendation, cycle map, assurance strategy, D-item list) + this
  charter. No crate change; workspace stays **1.4.0**; whether crates.io
  ever carries a `1.5.0` is decided by whichever future cycle next
  publishes (the v0.14→v0.15 skip precedent — docs-cycle names are not
  reserved version numbers).
- **Q5.3 Arc end-state** — **O3: TLCP crypto toolkit in
  `gmcrypto-core`** (maintainer-signed); the sans-I/O protocol engine
  (O2, a fourth crate) stays an explicitly *uncommitted* later decision,
  revisited only after the toolkit ships. The full-stack option (O1) is
  rejected permanently. See `tlcp-decomposition.md` §5.
- **Q5.4 v1.6 pick** — **TLCP key schedule (G1 + G3)**: P_SM3 PRF +
  master-secret/key-block/Finished derivation + the no-confirmation
  SM2-KX path (maintainer-signed). Smallest gap, unblocks per-primitive
  KATs for the rest of the arc; the record-protection cycle (v1.7) and
  the chain cycle (v1.8) follow per the §7 cycle map.
- **Q5.5 Sourcing discipline** — wire-level facts in the decomposition
  are sourced from the gotlcp reference implementation cross-checked
  against RFC 8998 / openEuler docs / GmSSL 3.1.1, with every
  must-re-verify fact tagged **[D-n]** (`tlcp-decomposition.md` §8) and
  assigned to an owning cycle — the v0.8 CCM-sourcing posture. No D-item
  is resolved in this cycle.

## 2. Out of scope (this cycle)

- Any code, feature flag, or dep change (including the G1–G4 gaps the
  doc maps — they belong to v1.6+).
- Resolving D-items (each is owned by the cycle that implements the
  surface it affects).
- Committing to the O2 sans-I/O engine (deliberately left open).

## 3. Workflow record

W0 research (gotlcp source: cipher_suites/prf/key_agreement/common/conn/
key_schedule; gmssl 3.1.1 local oracle check; Rust ecosystem sweep —
no pure-Rust TLCP exists, only Tongsuo C bindings) → W1 docs → W2 Codex
consult (six findings, all folded: the §4 profile relabeled "chain +
role verification, NOT server authentication" with endpoint-identity
binding loudly the caller's + pair-binding check added; the record
`deprotect` API shape constrained pre-v1.7 incl. equalized HMAC work;
anti-oracle PMS handling + sequence/nonce lifecycle added to the gap
table; engine-shaped API discipline added to §5; per-cycle FFI-shape
constraint recording added to §7; D-7…D-12 added; cycle map ordering
and the O3 recommendation confirmed) → W3 maintainer sign-offs
(Q5.3/Q5.4) → branch + PR (doc-only).
