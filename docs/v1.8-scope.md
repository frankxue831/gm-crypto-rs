# v1.8 scope — TLCP certificate-pair / chain verification (G4)

Status: v1.8 W1 scope doc. The **third code cycle of the TLCP arc** (gap
G4 of `docs/tlcp-decomposition.md`; the §4 derived chain/role profile).
The last *core* cycle of the toolkit sequence — v1.9 is the FFI cadence
cycle. The four cycle-shaping forks were **maintainer-locked to the
recommended shape** (the v1.7 fork pattern); the owned D-items (D-1, D-4,
D-8, D-9, D-11) are resolved here against gotlcp + GM/T 0015 + GB/T 38636,
with three honest primary-text residuals tagged (the per-cycle
"facts-cross-checked-vs-oracles, residuals-tagged" posture).

This is the cycle the decomposition itself flagged as the trap: *"path
validation is where small auditable cycles go to die"* (Codex, v1.5 W2).
The defense is the **derived profile** — verify only what the TLCP
handshake actually requires, name the permanent holes loudly, and keep
the single failure mode.

## 0. The one thing a reader must not misread

**This is certificate-pair / chain *signature* verification, NOT
"validation" and NOT server authentication.** A `true` means: each cert
pair is issued (directly or via intermediates) by a CA the *caller*
trusts, each cert is usable for its TLCP role, and the sign/enc pair share
one identity. It does **NOT** mean "this is the server I dialed."
**Endpoint identity binding (the TLCP equivalent of hostname verification)
is the caller's job, permanently** — a consumer that treats "chain
verifies + keyUsage correct" as "server authenticated" accepts *any*
trusted-CA-issued pair, from anyone (the Codex W2 headline hole,
`tlcp-decomposition.md` §4). The module/function docs say this in the same
words the v1.3 `x509` module says "makes NO trust decisions".

## 1. Sign-off decisions

### The four cycle-shaping forks (maintainer-locked, recommended shape)

- **Q8.1 Placement (D-6 analog) — SPLIT.** The generic chain walk + the
  keyUsage / basicConstraints **readers** land in the existing **`x509`**
  feature (reusable by any X.509-with-SM2 consumer); the TLCP-specific
  **[sign, enc] pair profile** + the GB/T 38636 required-bit-sets live in a
  new **`tlcp::chain`** module. Mirrors how `tlcp::record` sits over the
  generic `sm4` primitives. `x509` gains its first *structural* trust check
  (chain linking + CA-ness) but the **role semantics** (which bits mean
  "signature cert") stay in `tlcp`, so `x509`'s "no trust decisions about
  *whose* cert this is" character is preserved — `verify_chain` answers
  "does this chain link to an anchor", never "is this the right peer".

- **Q8.2 Trust depth (the §4 trap; D-1/D-4/D-9) — MINIMAL + documented
  residual risk.** Exactly the decomposition's two interpreted extensions:
  **keyUsage** required-bit-sets (D-1) + **basicConstraints CA=TRUE** on
  issuers, plus a hard **depth cap**. `pathLenConstraint` is **NOT read**
  (D-4 — depth-cap only); **EKU is NOT interpreted** (D-9 — ignored); no
  policy / name-constraints processing. Each non-interpreted surface is a
  **named residual risk** the caller inherits, stated in the docs.

- **Q8.3 API shape (engine-shaped; sets the v1.9 FFI constraint) — FREE
  FNS.** A generic `x509::verify_chain(...)` + a thin
  `tlcp::chain::verify_pair(...)`; plus the `x509::{KeyUsage,
  BasicConstraints}` readers and `Certificate` accessors for them. No
  builder, no typestate, no hidden state, single `bool`. The lean
  `record.rs` / `key_schedule.rs` precedent. Caller supplies each chain
  **leaf-first, in issuing order**, the trust anchors, and an
  `Option<X509Time>` comparison time.

- **Q8.4 KAT oracle (the W0 sourcing doc) — GmSSL multi-level chain +
  hand-built DER negatives.** GmSSL `certgen`/`reqsign` builds a
  root → intermediate → [sign, enc] leaf pair with explicit
  keyUsage/basicConstraints (extends the v1.3 `x509_regen.md` recipe);
  hand-built DER via the existing `asn1::writer` pins the
  keyUsage/basicConstraints/depth/pair-binding **reject** edges with full
  control. See `docs/v1.8-kat-sourcing.md`.

### The owned D-item resolutions (W0 research — gotlcp + GM/T 0015)

**Headline W0 finding (shapes every resolution below):** the gotlcp
reference implementation does **almost no GM-specific certificate
checking** — it delegates to `emmansun/gmsm`'s `smx509.Verify` (a Go
`crypto/x509` fork), which **ignores all keyUsage/EKU bits** (the Gutmann
rationale), assigns the sign/enc roles **positionally** (`certs[0]` /
`certs[1]`), and does **no pair-binding**. It *does* enforce
basicConstraints `CA=TRUE` + `pathLenConstraint` via the Go machinery, but
not `keyCertSign`. So the v1.5-locked §4 profile is a **deliberate
strengthening over the reference** — the SDK's consistent "stricter than
the wild" posture (negative-serial reject in v1.3, `Key1==Key2` reject in
XTS). The bits we enforce come from the **issuance side** (GM/T 0015 §5.2.4
practice, pinned by Tongsuo `subca.cnf` + GmSSL), not from the
verification side gotlcp leaves empty.

- **Q8.5 / D-1 — keyUsage required bits (RESOLVED, stronger than gotlcp).**
  keyUsage **MUST be present** on every cert whose role we check (absent
  keyUsage → reject for that role — stricter than RFC 5280's "absent =
  unrestricted", and stricter than gotlcp). Required asserted bits:
  - **Signature leaf:** `digitalSignature` (bit 0) MUST be asserted.
    (`nonRepudiation`/`contentCommitment` bit 1 is conventionally also set
    on GM sign certs but is **not required** — present-or-absent both
    accepted; extra bits never reject.)
  - **Encryption leaf:** at least one of `keyEncipherment` (bit 2,
    ECC suites' SM2-encrypted PMS) **OR** `keyAgreement` (bit 4, ECDHE
    suites' static-KX half) MUST be asserted. The GM/T 0015 enc cert
    conventionally carries `keyEncipherment + dataEncipherment +
    keyAgreement` (a superset), so this OR is satisfied without per-suite
    certs. Suite-agnostic at verify time — we do not know the suite, so we
    accept the cert that can serve either family.
  - **Intermediate issuer (every non-leaf cert IN the chain — above the
    leaf, below the anchor):** `keyCertSign` (bit 5) MUST be asserted
    **AND** basicConstraints `CA=TRUE` (keyUsage MUST be present). Untrusted
    — authenticated only by its parent's signature, so it gets the full
    role check. Stricter than gotlcp (which skips `keyCertSign`); RFC 5280
    §4.2.1.3 compliant.
  - **Trust anchor:** caller-admitted by fiat — checked ONLY by Name-match +
    edge-signature-verify, **NOT** keyUsage/CA-checked (Codex consult #4).
    Imposing `keyCertSign` on the anchor would falsely reject legitimate bare
    roots that omit keyUsage; admitting only proper CAs as anchors is the
    caller's responsibility (the RFC 5280 trust-anchor-as-input model).
  - **Leaf MUST NOT be a CA:** a sign/enc leaf asserting basicConstraints
    `CA=TRUE` is rejected — a CA cert must not serve as a TLCP endpoint
    (Codex consult #5).
  - **Residual [D-1]:** the verbatim normative MUST/SHOULD status of each
    bit in GM/T 0015-2012 §5.2.4.2.2 was not retrievable from the paywalled
    primary PDF — confirmed by section location + universal Tongsuo/GmSSL
    practice. Our rule follows practice; tagged for live-oracle
    confirmation. The KAT fixtures are the residual guard.

- **Q8.6 / D-4 — pathLenConstraint (RESOLVED: depth-cap only).** We do
  **NOT** read `pathLenConstraint`; we enforce a hard total-chain depth cap
  `MAX_CHAIN_DEPTH` (leaf + intermediates). basicConstraints `CA=TRUE` on
  every issuer **is** enforced (the load-bearing one — an intermediate must
  be a CA). **Residual risk (documented):** a sub-CA that set
  `pathLenConstraint=0` (MUST-NOT issue further intermediate CAs) is treated
  the same as one without the constraint — mitigated by the small depth cap
  + the caller-controlled anchor set + the caller supplying the chain in
  order (no path discovery). gotlcp honors pathLen; this is a *weaker*
  choice there, deliberately traded for the smaller auditable surface.

- **Q8.7 / D-9 — ExtendedKeyUsage (RESOLVED: ignored).** EKU is **not
  interpreted** — neither required nor rejected. **Residual risk
  (documented):** under *delegated* CA trust, EKU + name-constraints are the
  next-biggest holes (a CA-issued cert intended for a different purpose is
  accepted for TLCP). The caller owns any EKU policy. (gotlcp's
  client-verifies-server path also tolerates absent EKU; it enforces
  serverAuth only *if present* — we don't even do that, keeping the
  interpreted surface at exactly two extensions. **Residual [D-9]:** no
  GB/T 38636 / GM/T 0015 clause *mandating* serverAuth EKU on TLCP leaves
  was confirmable from primary text.)

- **Q8.7b / RFC 5280 §4.2 — unknown CRITICAL extensions (RESOLVED:
  safe-reject; Codex consult #5; MAINTAINER-SIGNED — the deliberate one
  widening of "minimal", accepted as a safety hardening).** A cert carrying
  a **critical** extension
  whose `extnID` is **not** in our processed set — exactly
  `{keyUsage 2.5.29.15, basicConstraints 2.5.29.19}` — is **rejected**
  during chain verification (RFC 5280 §4.2's MUST). We do not *process* the
  unknown extension; we refuse the cert, because a critical constraint we
  cannot honor (e.g. `nameConstraints`, `policyConstraints`) must not be
  silently ignored. This keeps the *processed* surface at exactly two
  extensions while making the *ignored* surface **safe** rather than blind.
  (The v1.3 parser punted this obligation to the caller via
  `extensions_raw`; v1.8 IS the trust logic, so it owns it.) **Non**-critical
  unknown extensions are still ignored (RFC 5280 permits). Cost: the
  extension walk already needed (to find keyUsage/basicConstraints by OID)
  also reads each extension's `critical` flag — no new parse pass.

- **Q8.8 / D-8 — pair binding (RESOLVED: subject-DN byte-equality).**
  `verify_pair` requires the sign leaf and enc leaf to have **byte-identical
  raw `subject` Name TLVs** (the existing `is_self_issued` byte-match
  discipline — no Name parsing), so a caller's single endpoint-identity
  decision covers both certs. This is a **deliberate strengthening over
  gotlcp** (which does no pair-binding) and is the only structural identity
  check the profile makes; **whose** identity it is remains the caller's.
  Hardened per the Codex consult (#2): the sign/enc **subject MUST be
  non-empty** (a zero-length Name byte-equals another zero-length Name — the
  empty-DN bypass) **and** the two leaves MUST share a **byte-equal raw
  `issuer`** (a real TLCP pair is issued by one CA). **Hardened further per
  the W2 executed review (S1):** equal issuer *Name* alone does NOT pin the
  issuer *key* — two trusted CAs sharing a DN (key rollover, or an
  attacker-admitted colliding-DN CA) would each independently authenticate a
  leg — so `verify_pair` additionally requires the two legs to present the
  **same issuing chain**: equal length and byte-equal (`tbs_raw`) certs from
  index 1 upward. This pins both leaves to the **same actual issuing CA
  cert**, not just a same-named one. **Residual (documented):** when BOTH
  legs are length-1 (each leaf directly under a trusted anchor) the
  upper-chain check is vacuous and the bind falls back to issuer-Name +
  anchor verification — a caller who admits two same-DN anchors with
  different keys is the narrow remaining hole. AKI/SKI matching is **not**
  added (the parser does not extract them; the per-edge SM2 signature is the
  authoritative bind).

- **Q8.9 / D-11 — Certificate ordering (RESOLVED: caller supplies in
  order).** The leaf pair is fixed `[sign, enc]` (we take two separate
  caller-supplied chains, sign-chain and enc-chain, each **leaf-first in
  issuing order**); intermediates follow the leaf within each chain. We do a
  **single linear walk** linking by raw issuer↔subject Name byte-equality —
  **no unordered intermediate pool, no path discovery** (the
  decomposition's "caller supplies the chain in order, as the TLS
  Certificate message already does"). **Name equality is necessary, NOT
  sufficient** (Codex consult #3): each edge is authenticated by the **SM2
  signature verify** (the v1.3 primitive), with raw issuer↔subject Name
  equality only the candidate link. At the anchor step, **every** anchor
  whose `subject_raw` matches the topmost issuer Name is tried and **at
  least one must verify the edge signature** — so same-Name CA key-rollover
  and duplicate-Name anchors resolve by *which key actually signed*, never
  by Name first-match. Name-only linking never authenticates an edge.
  **Residual [D-11]:** whether GB/T
  38636 normatively fixes post-leaf ordering was not confirmable from
  primary text; gotlcp is order-agnostic on the tail (it pools
  intermediates). Our "caller supplies in order" contract is the stricter,
  simpler shape and is documented as the API's requirement.

### Remaining cycle decisions

- **Q8.10 Time.** `verify_chain`/`verify_pair` take `at_time:
  Option<X509Time>`. `Some(t)` enforces `not_before <= t <= not_after` on
  **every** cert in the chain (the existing `X509Time` `Ord`); `None` opts
  out of the validity-window check entirely. **The SDK never reads a
  clock** — the v1.3 "`X509Time` exposed, no clock" posture extended, not
  breached. Anchors used as issuers are time-checked too when `Some`.

- **Q8.11 Trust anchor model.** An anchor is a parsed `Certificate` the
  caller **declares** trusted; the walk uses only its `subject_raw()` (to
  match the topmost intermediate's issuer) + its `subject_public_key()` (to
  verify that edge). The anchor's **own** signature is **never re-verified**
  (it is trusted by assumption — the standard trust-store model). A
  self-signed root passed as an anchor works; so does a bare intermediate
  the caller chose to trust directly (cross-cert pruning is the caller's).

- **Q8.12 Failure mode.** Single `bool` throughout (`verify_chain` and
  `verify_pair` both return `bool`, never `Result`/`Option` with a reason)
  — the workspace failure-mode invariant + the v1.3 `verify_signature`
  precedent. A C caller / TLS stack must not learn *why* a chain was
  rejected (chain-rejection oracle defense). The `KeyUsage` /
  `BasicConstraints` **readers** return `Option` (parse success/failure of a
  *public* extension) — that's structural decode, not a trust-reason
  channel.

- **Q8.13 ID for edge signatures.** Each chain-edge `verify_signature` uses
  the GM/RFC 8998 default ID `1234567812345678` (the v1.3 `verify_signature`
  default); a `_with_id` variant is **out** for this cycle unless the plan
  surfaces a concrete need (CAs signing with non-default IDs is rare; can be
  added additively later).

- **Q8.14 Assurance — NO new dudect target.** G4 is **public-inputs-only**
  (certificates + anchors + a public comparison time are not secrets; the
  signature verify is over public bytes) — the v1.3 `x509` no-dudect
  rationale extends verbatim (first such since v1.3). `SECURITY.md`
  documents the rationale. **New fuzz target** `fuzz_x509_chain`
  (attacker-controlled chain + anchor bytes → no panic / no OOM / single
  `bool`; census 29 → 30). The `FUZZ_TARGETS` list in `fuzz-nightly.yml`
  must name it (the #102 census discipline).

- **Q8.15 FFI-shape constraints recorded for v1.9.** `verify_chain` /
  `verify_pair` take **slices of immutable `Certificate` handles** (the v1.4
  `gmcrypto_x509_certificate_t` const-handle model) + an anchor slice + an
  optional time struct (the v1.4 `gmcrypto_x509_time_t` repr(C) struct, here
  with a presence flag for the `Option`) → a single `int` status (0 =
  chain verifies, the v1.4 banner contract; the `bool` maps to OK/!OK, not
  an out-param predicate). No new opaque type; no consume-on-use (read-only
  verification). The `KeyUsage`/`BasicConstraints` readers may surface as
  copy-out accessors on the existing cert handle. **No C ABI is frozen now**
  — v1.9 owns it.

- **Q8.16 Version + posture — PUBLISHING cycle.** New public API
  (`x509::{KeyUsage, BasicConstraints, verify_chain}` +
  `tlcp::chain::verify_pair` + cert accessors) ⇒ a real crates.io release.
  Workspace **1.7.0 → 1.8.0**; sibling pins `=1.8.0`; lockstep publish
  simd → core → c is the maintainer's authenticated per-release call. The
  api-baseline (`docs/api-baseline/gmcrypto-core.txt`) is regenerated as the
  **last pre-PR step** (the v1.7 lesson — the drift-check isn't in the local
  gate). cargo-semver-checks: additive only.

## 2. Out of scope (this cycle)

- **Endpoint identity binding** (hostname/peer-identity) — the caller's,
  permanently; the module docs say so loudly. A convenience
  identity-predicate variant of `verify_pair` is **out** (Codex consult #1):
  the caller composes the identity decision against `subject_raw()`; baking
  in a predicate the SDK can't parse the Name behind would invite false
  assurance.
- **Revocation** (CRL/OCSP), **policy processing**, **name constraints**,
  **EKU interpretation**, **pathLenConstraint** reading (D-4 depth-cap
  only), **cross-signing / multiple-path discovery** (single linear,
  caller-ordered chain only).
- **Certificate / CSR generation**; **Name parsing** (raw-TLV byte-equality
  only, the v1.3 discipline).
- The **handshake / state machine** consuming this (G5 — not crypto, never
  `gmcrypto-core`).
- **C FFI** for the chain surface — deferred to the v1.9 toolkit FFI cycle
  (Q8.15 records the shape constraints).
- Any change to the v1.3 `Certificate::from_der` parse profile beyond
  **adding** the keyUsage/basicConstraints **readers** (parsing stays
  byte-identical; the readers operate on the already-captured
  `extensions_raw` span).
