# v1.4 scope ‚Äî C FFI for X.509-with-SM2 (leaf parse + signature verify)

Status: scope locked 2026-06-12 (maintainer sign-off on Q4.1‚ÄìQ4.2; the rest
follows settled precedent + the Codex design consult, which confirmed
copy-out / single-extensions-symbol / pubkey-handle-return / verify naming
and picked the out-param shape for `is_self_issued`). A Fable-5 adversarial
review of this scope + the companion plan (`docs/v1.4-x509-ffi-plan.md`)
returned **GO-WITH-FIXES** (2026-06-12); all findings are folded in below ‚Äî
headline: a 1/0 `is_self_issued` predicate would have falsified the header
banner's universal "every int return is 0 on success" contract (resolved by
adopting Codex's out-param shape, Q4.9), and the `% 8 ‚Üí % 9` fuzz op remap
silently moves the committed `sm3_abc` seed (Q4.13). Completes the
core-in-vN / FFI-in-vN+1 cadence for the v1.3 `x509` core (the v0.12‚Üív0.13,
v0.15‚Üív0.16 and v1.1‚Üív1.2 pattern).

## 1. Sign-off decisions

- **Q4.1 Direction** ‚Äî **C FFI for X.509** (maintainer-locked, 2026-06-12).
  The cadence candidate named in `docs/v1.3-scope.md` Q3.10. TLCP
  decomposition and chain validation were considered and deferred (TLCP is a
  protocol-sized arc needing its own decomposition cycle; chain validation is
  the Codex-flagged scope trap and belongs to that future arc).
- **Q4.2 Breadth** ‚Äî **full mirror** (maintainer-locked): every public
  accessor of the Rust `x509::Certificate` crosses the ABI. Verify-only
  without the raw accessors is crippled for real C callers (issuer/subject
  byte-matching and caller-owned clock checks are exactly what the raw
  surface is for).
- **Q4.3 Handle shape** ‚Äî ONE opaque handle `gmcrypto_x509_certificate_t`
  (immutable after construction ‚Äî unlike the v1.2 KX handles there is no
  state machine and no consume-on-use; every accessor takes `const *`).
  `_from_der` returns the handle pointer (NULL on ANY failure ‚Äî the
  `gmcrypto_sm2_pubkey_new` convention); `_free` (NULL no-op). Symbol names
  carry the full type prefix `gmcrypto_x509_certificate_*` (the
  `gmcrypto_sm4_gcm_encryptor_*` type‚Üîmethod binding convention).
- **Q4.4 Verify shape** ‚Äî two symbols mirroring Rust:
  `gmcrypto_x509_certificate_verify_signature(cert, issuer_pubkey)` and
  `_verify_signature_with_id(cert, issuer_pubkey, id, id_len)`; the issuer
  key rides the existing `gmcrypto_sm2_pubkey_t` handle. `id_len == 0` ‚Üí
  `DEFAULT_SIGNER_ID` (the v1.2 KX / sign-FFI precedent; an actually-empty
  ID is therefore unrepresentable through the FFI ‚Äî documented, same as KX).
  Return is `GMCRYPTO_OK` on a verifying signature, `GMCRYPTO_ERR` otherwise
  (the `gmcrypto_sm2_verify` precedent; valid-vs-error indistinguishable by
  design). The doc comment repeats the Rust module's blunt contract: **this
  is NOT certificate validation** ‚Äî no chain/time/extension/revocation
  logic.
- **Q4.5 Accessor memory model** ‚Äî **copy-out** `(out, out_capacity,
  out_actual_len)` for the five variable-length byte accessors
  (`tbs_raw`, `serial_raw`, `issuer_raw`, `subject_raw`, `extensions_raw`),
  riding the existing `write_output` helper (too-small buffer ‚Üí
  `GMCRYPTO_ERR` + required length in `*out_actual_len`, the v0.13 two-call
  discovery convention). Borrowed zero-copy views into the handle were
  considered and REJECTED: they would be the first borrowed-lifetime
  contract in the ABI, trading a new use-after-free footgun for a copy of at
  most a few KiB.
- **Q4.6 extensions absent** ‚Äî one symbol; a certificate parsed without the
  optional `[3]` extensions block returns `GMCRYPTO_OK` with
  `*out_actual_len = 0`. This is **unambiguous, not a conflation**: when
  present, `extensions_raw` is the Extensions SEQUENCE TLV and is never
  empty, so length 0 uniquely means "absent". Both committed fixtures carry
  extensions, so the absent path is pinned by a **strip-the-`[3]`-block
  surgery test** (remove the wrapper TLV, patch the two enclosing SEQUENCE
  lengths): the now-invalid signature is irrelevant because `from_der`
  never verifies (garbage-sig-parses-but-never-verifies is pinned core
  behavior) ‚Äî the review's D2 fix.
- **Q4.7 Validity times** ‚Äî new plain `#[repr(C)]` struct
  `gmcrypto_x509_time_t { uint16_t year; uint8_t month, day, hour, minute,
  second; }` mirroring the Rust `X509Time` field-for-field, written by
  `_not_before(cert, *out_time)` / `_not_after(cert, *out_time)`
  (status-return + out-param). NO comparison helper and NO clock ‚Äî the
  caller owns any validity decision (the core's no-trust-decisions posture
  crosses the ABI intact).
- **Q4.8 Subject key** ‚Äî `gmcrypto_x509_certificate_subject_public_key(cert)`
  returns a **newly allocated `gmcrypto_sm2_pubkey_t*`** (NULL only on NULL
  input ‚Äî the key was validated on-curve at parse), which the caller frees
  via the existing `gmcrypto_sm2_pubkey_free`. This composes the certificate
  surface with every existing pubkey consumer (`gmcrypto_sm2_verify`,
  `gmcrypto_sm2_encrypt`, the KX constructors) and is how a C caller chains
  leaf-vs-CA verification manually.
- **Q4.9 is_self_issued** ‚Äî **out-param + status return** (the Codex pick,
  resolving the Fable-5 C1 finding):
  `gmcrypto_x509_certificate_is_self_issued(cert, int *out_is_self_issued)`
  returns `GMCRYPTO_OK` for a successful evaluation and writes `1`
  (issuer/subject `Name`s byte-identical) or `0`; `GMCRYPTO_ERR` on
  null/bad args. Rationale: "not self-issued" is not a failure, so
  status-style `OK = self-issued` would read dangerously inverted in C
  `if (...)`; and a bare 1/0 predicate return would falsify the committed
  header banner's universal "every int return is 0 on success, non-zero on
  failure" contract (cbindgen.toml `header` + the `lib.rs` crate docs).
  The out-param shape keeps that contract universally true with zero text
  amendments, and matches the `not_before`/`not_after` out-param
  convention this same cycle introduces. The doc comment still says: NOT
  "self-signed" ‚Äî use `verify_signature` against the cert's own subject
  key for a self-signature check.
- **Q4.10 Feature posture** ‚Äî **always-on** (the v0.23 W3 posture): `x509`
  joins `sm4-aead`/`sm4-xts`/`sm2-key-exchange` unconditionally on the C
  shim's `gmcrypto-core` dep, so the committed `gmcrypto.h` == a default
  `cargo build -p gmcrypto-c` and the cbindgen cfg-gated-opaque-type trap
  (the v0.10 lesson) cannot fire. `gmcrypto-core`'s own `x509` feature stays
  opt-in/default-off.
- **Q4.11 Surface census** ‚Äî **13 new symbols + 1 opaque type + 1 plain
  struct, 0 new consts** (72 ‚Üí **85** FFI entry points):
  `gmcrypto_x509_certificate_{from_der, free, verify_signature,
  verify_signature_with_id, tbs_raw, serial_raw, issuer_raw, subject_raw,
  extensions_raw, not_before, not_after, is_self_issued,
  subject_public_key}`. No new consts: there are no fixed-size outputs (the
  copy-out two-call discovery covers sizing; serial is ‚â§ 20 bytes by parse
  contract, documented not const-ed).
- **Q4.12 Failure modes** ‚Äî single `GMCRYPTO_ERR` / NULL for everything
  (null pointers, malformed DER, wrong profile, short buffers, failed
  verify) ‚Äî the workspace failure-mode invariant. The FFI must not let C
  callers distinguish what `from_der`'s single `None` deliberately blinds.
- **Q4.13 Assurance** ‚Äî **no new dudect target** (thin shim over a
  public-inputs-only core ‚Äî v1.3 had no core target for the same reason;
  doubly covered). c_smoke additions (8 tests, 76 ‚Üí 84): fixture parse
  through the FFI + accessor byte-equivalence vs the Rust core on BOTH
  committed gmssl fixtures ‚Äî the **CA** fixture carries the real pad-strip
  pin (13 wire bytes ‚Üí 12 through the ABI, high bit set; the leaf serial
  was never padded ‚Äî review D3); CA self-verify + leaf-vs-CA OK;
  leaf-vs-own-key ERR; wrong-ID ERR; tampered-tbs ERR; truncated/garbage
  DER ‚Üí NULL handle; extensions present (len > 0) pin + the absent‚Üí0
  contract via the Q4.6 strip-`[3]` surgery test; `gmcrypto_x509_time_t`
  field equality vs Rust `X509Time`; subject-pubkey-handle composes (SEC1
  export equals the core's); copy-out too-small ‚Üí ERR + required len;
  NULL-argument sweep across constructors, accessors (incl. NULL cert and
  NULL `out_actual_len`), verifies and the predicate; `_free(NULL)` no-op.
  **fuzz**: extend `fuzz_c_abi` with an x509 op (op byte 8: attacker bytes ‚Üí
  `_from_der`; on parse, drive every accessor incl. undersized buffers +
  both verifies + subject-key free + `_free`), plus a committed seed
  embedding the leaf fixture ‚Äî **no new fuzz binary; the census stays 27**
  (the #102 must-match-`[[bin]]` invariant holds). The `% 8 ‚Üí % 9` dispatch
  change is audited against EVERY committed `fuzz_c_abi` seed:
  `kx_valid_transcript` (first byte `0x07`) is stable, but `sm3_abc`
  (first byte `0x41` = 65: 65 % 8 = 1 but 65 % 9 = 2) silently remaps to
  the wrong op ‚Äî its first byte is rewritten to `0x01` in the same commit
  (review D1). The `fuzz/Cargo.lock` 1.4.0 churn is committed per the
  v1.2/v1.3 precedent (review E1).
- **Q4.14 Versioning** ‚Äî additive minor ‚Üí **1.4.0** lockstep (`=1.4.0`
  sibling pins). Publish + SSH-signed tag + GitHub release remain the
  maintainer's authenticated per-release call. Agent path: branch + PR.
- **Q4.15 Docs** ‚Äî doc-only C example `crates/gmcrypto-c/examples/
  x509_verify.c` (argv: leaf DER + issuer DER ‚Üí parse both, pull the
  issuer's subject key, verify the leaf; compiled + run locally against the
  committed fixtures at dev time). README v1.4 section + history row;
  SECURITY.md FFI-census note; CHANGELOG `[1.4.0]`; CLAUDE.md header +
  architecture map; `gmcrypto-c/README.md` symbol table.

## 2. Out of scope (deliberate)

- Chain validation / trust anchors / time *decisions* ‚Äî the core's
  no-trust-decisions contract crosses the ABI unchanged.
- PEM certificate input (the Rust `x509` core is DER-only; C callers strip
  PEM armor themselves or convert with standard tooling).
- Name/RDN parsing, extension interpretation, revocation ‚Äî same as the core.
- Certificate / CSR generation (no emit path ‚Äî the `encode_c1c2c3_legacy`
  lesson).
- TLCP (the arc this cycle feeds; needs its own decomposition cycle).
- Any change to `gmcrypto-core` (the v1.3 surface is sufficient and the FFI
  is a pure consumer of it).
