# v1.7 scope — TLCP record protection (gap G2)

Status: direction set by the v1.5 decomposition cycle map
(`docs/tlcp-decomposition.md` §7) and the v1.6 umbrella-feature decision
(Q6.2). The four genuinely-open design forks — Q7.3 Lucky13 CT strategy,
Q7.4 record API shape, Q7.1 scope split, Q7.7 KAT oracle — were
maintainer-delegated to a Codex consult on 2026-06-14 ("consult with codex
and figure out the best solution for our case"). Codex **confirmed all four
leans** and added the refinements folded in below (risk order it assigned:
Q7.3 Lucky13 > Q7.7 oracle > Q7.4 API misuse > Q7.1 scope). The second
code cycle of the TLCP arc (gap G2 of the decomposition). Codex-consulted
(W0) before this scope; the verify-before-scoping oracle probe (D-5) is done.

## 0. What v1.7 is

The TLCP record-protection toolkit: byte-in / byte-out record protect +
deprotect primitives for the four SM2-family suites, behind the existing
opt-in **`tlcp`** umbrella feature (v1.6), as a new **`tlcp::record`**
module. Pure-core, **no new dep**, no_std, default build byte-identical.
TLCP is TLS-1.1-with-SM-algorithms; record version `0x0101`; max plaintext
`2^14`.

Two record cipher families (decomposition §1.5):

- **SM4-CBC** (`ECC/ECDHE_SM4_CBC_SM3`): TLS-1.1-style explicit per-record
  IV, **MAC-then-encrypt**, **TLS padding** (every pad byte == pad-length
  byte, NOT PKCS#7), HMAC-SM3 record MAC. The Lucky13-class constant-time
  deprotect is the arc's headline engineering item.
- **SM4-GCM** (`ECC/ECDHE_SM4_GCM_SM3`): **RFC 5288 TLS-1.2** AEAD shape —
  4-byte implicit salt ‖ 8-byte explicit nonce; TLS AAD; 16-byte tag.

**~80 % composes on the existing core (verified):**
`mode_gcm::{encrypt,decrypt}` already take a generic `nonce: &[u8]` +
first-class `aad: &[u8]` with commit-on-verify
([mode_gcm.rs:147/186](../crates/gmcrypto-core/src/sm4/mode_gcm.rs));
`derive_key_block` carves CBC 128 B / GCM 40 B server-first
([key_schedule.rs:105](../crates/gmcrypto-core/src/tlcp/key_schedule.rs));
`TlcpRole` exists ([key_schedule.rs:122](../crates/gmcrypto-core/src/tlcp/key_schedule.rs));
`Sm4Cipher::{encrypt,decrypt}_blocks` are directly reusable; `HmacSm3::verify`
(CT bool) is reusable for the **GCM** path only — its `finalize` compression
count is a function of the secret post-strip length, so it **MUST NOT** be
used on the CBC tag-check (Q7.3 uses the new constant-compression `pub(crate)`
`Sm3` capability + a hand-written CT scan instead). The genuinely-new work is
the CBC TLS-padding apply/strip and the **constant-compression-count Lucky13
deprotect MAC**.

## 1. Sign-off decisions

### Q7.1 Scope — both CBC + GCM, GCM-first staged (maintainer-delegated → Codex-confirmed)

G2 is decomposed as CBC + GCM together; **ship both in v1.7**. GCM is a
thin, zero-new-crypto wrapper, so it **stages first on the branch**
(validates the shared record API, key carve, seq/AAD framing, failure
semantics) before the CBC Lucky13 engine lands on top — a problem in the
shared scaffolding surfaces against the easy path first. Keeps the cycle
map intact (v1.8 = chain/pair verification). Size ≈ v0.8. Codex blind-spot
folded: "GCM is thin" must NOT weaken nonce discipline (Q7.5). Rejected:
CBC-only split (fragments a single gap, delays v1.8).

### Q7.2 Placement — `tlcp::record` module under the `tlcp` umbrella feature

Per v1.6 Q6.2: one pure-core opt-in feature carries the whole toolkit arc;
v1.7 adds `tlcp::record` under the same `tlcp` flag (no new dep, no_std,
wasm32 leg, default build byte-identical). `tlcp` does NOT imply
`sm2-key-exchange` (unchanged). The new SM3 `pub(crate)` capability (Q7.3)
is **`pub(crate)`-unconditional core** (invisible to the public API /
SemVer regardless of feature); the constant-compression-count deprotect MAC
logic lives in `tlcp::record` (cfg-gated on `tlcp`).

### Q7.3 Lucky13 constant-compression-count MAC — HEADLINE (maintainer-delegated → Codex pick A)

The CBC deprotect MAC must run an SM3-compression count **independent of the
secret post-strip plaintext length** — the Lucky13 signal. Verified blocker:
SM3 `compress` is module-private
([sm3.rs:180](../crates/gmcrypto-core/src/sm3.rs)); the extra-compression
boundary `if self.buffer_len > BLOCK_SIZE - 8`
([sm3.rs:108](../crates/gmcrypto-core/src/sm3.rs)) makes the count a
deterministic function of input length; `HmacSm3` exposes no
compression-count control (only `update`/`finalize`/`verify`).

**Decision (Codex A): widen SM3 `compress` to `pub(crate)`** and add a small
`pub(crate)` `Sm3` capability (process-one-block + a clone-and-discard
dummy-compress; `Sm3` is already `Clone` + Drop-wiping), **reusing the
single audited compression loop**. NOT a hand-written second
padding+compression loop (Codex: the duplicated path is the larger
cryptographic risk, and the highest-consequence place for a divergence bug).
The new surface stays **`pub(crate)`** (unreachable from the public API;
misuse-prone), guarded by a new dudect target (Q7.8) + a "constant-blocks
MAC over a public-length input == ordinary `HmacSm3`" equivalence test.

Design (Codex consult + adversarial-panel refinements, all MUSTs):

- **The real MAC keeps the true length; `length` is never a caller param on
  deprotect.** PROTECT computes the 2-byte `length` from `plaintext.len()`.
  DEPROTECT recomputes `length` **internally** from the stripped fragment
  after the constant-time pad-strip — it is the **secret post-strip
  plaintext length**, written into BOTH the SM3 bit-length AND the 2-byte
  MAC-header `length`, and is **never** a caller parameter and **never**
  `fragment_len`. (A "length = caller param" leaks the secret length back
  through the API; a "length = fragment_len" computes the wrong MAC and
  breaks interop. A KAT with `padding_length > 0` pins this.) Equalization
  is **separate dummy compressions on a clone/discard `Sm3` state**, outside
  the real MAC transcript — never by mis-padding the real hash.
- **Three constant-time surfaces must be equalized** — compression count
  alone is NOT the whole Lucky13 surface:
  1. **Inner-hash SM3 compression count.** The leaking quantity is the
     number of `compress` calls on the *inner* hash, whose input is
     `K'⊕ipad(64) ‖ header(13) ‖ plaintext(P)` plus the SM3 `0x80`/64-bit-
     length pad (9–72 B) → boundary `ceil((64 + 13 + P + 9) / 64)` (the
     outer hash is a constant 2 compressions — no equalization needed).
     Equalize to the count for the **public fragment length**:
     `max_plaintext_len = fragment_len − 32 − 1` (32-byte MAC + the
     mandatory 1-byte pad-length) is a **SAFE UPPER BOUND** that **MUST
     NEVER be reduced** toward a block-aligned or secret value — the ipad
     prefix is constant across classes, so the bound over-equalizes
     correctly. Top the real inner-hash count up to that target; the secret
     variation is only the ≤255-byte padding-length → a few dummy blocks.
     (Flat `2^14` is unnecessary perf/DoS cost — record length is public.)
     The 55/56/57-byte boundary neighborhood to test is relative to the
     **full** inner-hash input (`64 + 13 + P + 9` mod 64), NOT `13 + P`.
     Pin a test: `target ≥ actual` for every padding `0..=255` at a fixed
     fragment.
  2. **TLS pad-validity check.** TLS padding spans up to 255 pad bytes + 1
     length byte = up to **256** bytes (vs PKCS#7's ≤16), so the existing
     16-byte `strip_pkcs7_ct` window is **NOT** reused: a new check scans a
     **fixed window of `min(256, cbc_body_len)` trailing bytes**, masking
     each position's "in claimed-pad-region AND equals padlen" test into a
     single bad-accumulator (the `strip_pkcs7_ct` mask pattern widened
     16→256), examining **every** candidate regardless of claimed padlen —
     **no early return, no padlen-bounded loop.**
  3. **The 32-byte MAC-extraction scan** — the constant-time copy of the MAC
     from its secret offset within the fragment must be data-independent;
     the final tag compare is `subtle` CT.
- **Bad-padding path still runs the full MAC.** On padding-invalid (wrong
  pad byte, oversized/underflowing padlen, no valid plaintext when
  `padlen ≥ body − 32`), deprotect **still runs the equalized MAC over the
  fixed fallback length** (`max_plaintext_len`), THEN returns `None` — never
  short-circuit the MAC on bad pad (the textbook Lucky13 hole). The
  pad-validity `Choice` and the MAC-compare `Choice` combine into **ONE**
  final constant-time merge. The deprotect stays **ONE operation** (no
  public decrypt-then-check), **single `None`**, **no plaintext on failure**.
- **Public-length guards run BEFORE the secret arithmetic.** A CBC body
  (post-explicit-IV) whose length is not a positive multiple of 16, or is
  `< 48` bytes (the minimum: 0-plaintext + 32 MAC + one 16-byte padding
  block), is rejected by a **public-length branch** (single `None`) before
  any `fragment_len − 33` subtraction — no usize underflow.
- **`core::hint::black_box` against LTO elision** must wrap BOTH the dummy
  iteration bound AND the cloned `Sm3` state fed to `compress`, and the
  post-loop dummy state must be `black_box`-consumed (black_box is
  best-effort by its own docs). The **`ct_tlcp_cbc_deprotect` dudect target
  is built FIRST** as the empirical backstop (Q7.8).

### Q7.4 Record API shape — per-direction `ZeroizeOnDrop` keys struct + explicit seq (Codex-confirmed)

Per-direction key carriers, carved by the caller from `key_block`:
`RecordKeysCbc { mac_key: [u8; 32], enc_key: [u8; 16] }` /
`RecordKeysGcm { enc_key: [u8; 16], salt: [u8; 4] }`, **`ZeroizeOnDrop`, NO
internal counter**. `protect` / `deprotect` take an **explicit per-call
`seq: u64`** + (CBC) an injected IV via `rand_core::TryCryptoRng`.
Engine-shaped (decomposition §5: caller-held seq, injected RNG, no hidden
state). The party→direction mapping (a client writes with `client_*`, reads
with `server_*`) is on **public** role data — a role match is not a secret
branch; provide a role-aware `from_key_block(role, &key_block)` convenience
carver as sugar over the primitive triples. Codex blind-spot folded:
**doc/type that repeated `(direction_key, seq)` is forbidden** (catastrophic
for GCM); a convenience stateful wrapper may live OUTSIDE the core (not
v1.7). Rejected: a stateful `Protector`/`Opener` with an auto-incrementing
counter (hidden state violates §5; a sans-I/O engine cannot drive it).

**Signature-frozen details** (the MAC/AAD bytes ARE these fields, so they
freeze the protect/deprotect signatures): `type: u8` — the raw wire content
byte, NO enum (an enum is a policy leak the §5 engine should own and can't
express future record types); `version: [u8; 2]` passed by the caller
(optionally via a `TLCP_RECORD_VERSION: [u8; 2] = [0x01, 0x01]` const the
caller may pass) — NOT a hardcoded constant (engine-shaped explicit inputs);
`length` is NOT a parameter — it is computed internally on both sides
(protect from `plaintext.len()`, deprotect from the stripped fragment; Q7.3).
Deprotect returns `Option<Vec<u8>>`; on success the plaintext `Vec` is
**caller-owned and NOT zeroized by the SDK** (the `mode_gcm`/`mode_cbc`
precedent — caller wipes); on failure no plaintext `Vec` is ever
materialized.

**`from_key_block(role, &key_block)` carve offsets** (carve order =
client MAC ‖ server MAC ‖ client key ‖ server key ‖ client IV ‖ server IV):
CBC — client `mac 0..32` / `enc 64..80`, server `mac 32..64` / `enc 80..96`
(client/server IV `96..112`/`112..128` are produced but omitted from the
struct, Q7.6); GCM — client `enc 0..16` / `salt 32..36`, server
`enc 16..32` / `salt 36..40`. A wrong offset is silent (MAC fail, not a
compile error), so the carver is KAT-pinned.

### Q7.5 GCM record specifics (resolves part of D-12; RFC 5288 — anchor correction)

**RFC 5288 TLS-1.2 AEAD construction** (the 4-byte salt in
`derive_key_block`'s GCM carve confirms it), **NOT RFC 8998/TLS-1.3** — the
decomposition §1.5 was already right; this scope corrects the survey-brief
mis-anchor to **RFC 5288 §3 + RFC 5246 §6.2.3.3 + GB/T 38636 §6.3**.
Nonce = 4-byte salt ‖ 8-byte explicit per-record nonce; the explicit half is
**derived from the caller-held `u64` seq** (deterministic uniqueness — Codex:
`salt ‖ nonce` reuse under a write key is catastrophic). AAD =
`seq(8) ‖ type(1) ‖ version(2) ‖ length(2)`, length = fragment (plaintext)
length (== ciphertext length for GCM). Tag fixed 16. **Sequence/nonce
uniqueness is the caller's contract** (reconciled W2, maintainer-agreed): the
stateless `protect`/`deprotect` take `seq` explicitly and **cannot detect
reuse or wrap** — `(key, nonce)` reuse is catastrophic for GCM, so the module
docs LOUDLY warn the caller must never reuse `(key, seq)` (TLCP's
Finished-gated handshake guarantees this). The "hard reject on wrap" is a
future **stateful wrapper's** job, not the stateless primitive's. Thin wrapper
over `mode_gcm`: nonce/AAD shaping only, zero new GCM math. **Wire layout:**
`explicit_nonce(8) ‖ ciphertext ‖ tag(16)` — PROTECT prepends the
seq-derived 8-byte explicit nonce; DEPROTECT reads `nonce_explicit` **from
the wire** (RFC 5288) while `AAD.seq` uses the receiver's own caller-held
seq.

### Q7.6 CBC record specifics (resolves part of D-12; the Q7.6 IV-typing call)

Wire: `explicit_IV(16) ‖ CBC_encrypt(plaintext ‖ HMAC-SM3(…) ‖ tls_padding)`
— explicit IV at the FRONT (TLS-1.1), **MAC-then-pad-then-encrypt**. TLS
padding: pad to a 16-byte multiple, every padding byte equals the
padding-length byte, 0..255 pad bytes. The explicit IV is **caller-supplied
unpredictable, RNG-injected per record** (NIST SP 800-38A; the `mode_cbc`
precedent). The CBC keys struct **OMITS the vestigial key-block
`client_IV`/`server_IV`** — the 128 B carve still *produces* those bytes (so
caller carve offsets stay correct, KAT-pinned by §6.5.2), the record API just
never consumes them (documented on the keys struct). Raw CBC chaining is
driven over `Sm4Cipher::{encrypt,decrypt}_blocks` — the existing PKCS#7
`mode_cbc` is **NOT** reused (TLS padding ≠ PKCS#7). Deprotect: public-length guard (Q7.3) → strip explicit
IV → raw CBC decrypt → the Q7.3 constant-time TLS-pad check + internally-
recomputed `length` + the bad-pad-still-MACs equalized MAC over the recovered
fragment → single `None` on any failure, no plaintext on failure.

Boundary vectors (Codex): pad lengths, SM3 compression-boundary lengths

### Q7.8 Assurance

- **dudect**: NEW **`ct_tlcp_cbc_deprotect`** — class-split by
  TLS-padding-length interpretation / padding validity, fixed secret SM4 +
  MAC key, both classes reach the equalized MAC + CT tag compare via
  identical control flow (the `ct_sm4_gcm_decrypt` per-class-valid pattern).
  Gate `|tau| < 0.20`. **Build this target FIRST and use it to confirm the
  chosen dummy-compression window before locking the constant.** It is the
  residual guard — it does NOT certify equalization (the
  constant-compression-count math is a hard impl MUST, independent of the
  dudect result). Plus an **equivalence test** (constant-blocks MAC over a

### Q7.10 FFI-shape constraints recorded for v1.9 (nothing frozen)

Single-shot record protect/deprotect = plain copy-out (in-buffer +
out-buffer + `out_actual_len`), single error code, caller-held
`seq` (`uint64_t`) / IV / keys by value — **NO opaque handle**. The
equalization constant is derived INTERNALLY from the public ciphertext length
— the FFI must **NOT** take it as a parameter (a wrong value reintroduces the
leak). Single `GMCRYPTO_ERR` (never bad-pad vs bad-MAC vs wrong-length); no
separate strip-pad / verify-MAC entry points (the §6 MUST against
mis-assembly applies to the C surface too). CBC protect: caller-supplied
explicit IV OR a `CallbackRng` (the v0.5/v1.2 precedent) — a v1.9 fork, not
decided now. A streaming/connection handle is an O2 concern, deferred.

## 2. Out of scope (deliberate)

- 5-byte TLS record **header assembly/parsing** — the record API takes
  `type` (`u8`) + `version` (`[u8; 2]`) as explicit params for the MAC/AAD;
  `length` is computed internally on both sides (Q7.3/Q7.4), never a param.
  The API does NOT frame or emit the on-wire 5-byte header (caller/engine
  concern, §5).
- Record **fragmentation / reassembly** — single-shot over one record body.
  The `2^14` ceiling is a **length-range reject** (single `None`, mirroring
  `mode_gcm`'s `GCM_MAX_PT_BYTES` poison), NOT a chunking feature.
- **ClientKeyExchange / any handshake-message wire** (D-3) — the record
  layer consumes already-derived `key_block` + caller-held seq/role/IV; it
  never assembles or parses handshake messages. D-3 stays owned by a later
  handshake-message cycle.
- **Server authentication / chain / pair / keyUsage / basicConstraints**
  (G4) — deferred to **v1.8** (the §4 chain/pair profile).
- A stateful per-connection / streaming / sans-I/O **O2 engine** —
  explicitly uncommitted (decomposition O2); v1.7 ships engine-SHAPED
  stateless/policy-free primitives, not an engine.
- **Widening** any new ct-MAC / SM3 dummy-compression surface to the PUBLIC
  API — it stays `pub(crate)`, reachable only from `tlcp::record`.
- **C FFI** for the record layer — deferred to **v1.9** (the toolkit's
  single FFI cycle).
- **RFC 8998 / TLS-1.3** SM4-GCM construction (12-byte IV XOR seq, AAD =
  record header) — TLCP is TLS-1.1-based; the GCM record is RFC 5288
  TLS-1.2-shaped (the anchor correction above).
