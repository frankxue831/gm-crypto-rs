# v1.0 readiness report

**Status: AUDIT COMPLETE — GO** (the §3.A decision was **resolved in v0.22**; see
the §3.A note below). The public API is frozen and tooling-guarded, the docs are
complete, and the constant-time + fuzzing evidence is in place. The one load-bearing
API-design decision that the v0.21 audit left open — the `crypto-bigint` type
exposure (§3.A) — was resolved by the **v0.22 API-tightening cycle**
(`docs/v0.22-scope.md`): the always-on (default-features) 1.0 contract now names
**zero** `crypto-bigint` types. This report is the GO/NO-GO checklist + the publish
runbook; producing it was the deliverable of the v0.21 non-publishing readiness audit
(`docs/v0.21-scope.md`), updated by v0.22.

The actual `1.0.0` crates.io publish is a **separate, deliberate step** taken later
on explicit maintainer go-ahead (§4), gated by `docs/v0.1.0-release-review.md`.

## 2. What the audit locked (v0.21)

- **`cargo-public-api` baselines** (`docs/api-baseline/{gmcrypto-core,gmcrypto-simd}.txt`),
  drift-checked in CI with **pinned** `cargo-public-api 0.52.0` + `nightly-2026-05-23`
  (`--omit blanket-impls,auto-trait-impls,auto-derived-impls` for a stable surface).
  Bumping either pin is a **reviewed re-baseline** (regen with the pinned tool +
  nightly, commit, confirm the workflow stays green). `gmcrypto-core` = the full
  public surface (`#[doc(hidden)]` omitted by the tool; v0.22 regenerated it after
  the curve-surface hiding + the `[u8;32]` reshape — only the gated `from_scalar(U256)`
  remains `crypto-bigint`-typed); `gmcrypto-simd` = just `pub mod gmcrypto_simd`
  (its surface is entirely `#[doc(hidden)]`).
- **`cargo doc` gate** — `-D warnings -A rustdoc::private_intra_doc_links` (the
  one allowed lint = intentional internal cross-refs that render as code text on
  docs.rs, not broken links).
- **Feature matrix** — `--no-default-features` build + `--all-features` test.
- **API-surface finalization** (doc-attributes + tests only, no behavior change):
  the 3 `#[doc(hidden)]` core items + the `gmcrypto-simd` internal surface (§3.B/C).

The guard is a **drift gate**: an intentional baseline regeneration can pass any
change, so the committed `docs/api-baseline/*.txt` **diff is the review artifact**.
Because `--omit auto-trait-impls` is set, the public-api gate does **not** catch
`Send`/`Sync`/`Unpin` changes — `cargo-semver-checks` (enforced from 1.0) covers
those.

## 3. Known API considerations for 1.0

### 3.A — `crypto-bigint 0.7` types in the public API (RESOLVED in v0.22)

> **RESOLUTION (v0.22, `docs/v0.22-scope.md`).** The maintainer chose **Option 2**
> (tighten the surface) over Option 1 (accept-and-document). v0.22 made the
> always-on (default-features) 1.0 contract name **zero** `crypto-bigint` types:
> the low-level `sm2::curve` / `sm2::scalar_mul` / `ProjectivePoint::to_affine`
> surface is `#[doc(hidden)]` (kept `pub` for in-repo dev crates; out of the SemVer
> contract — Group A), and the always-on byte-adjacent types
> (`asn1::sig::{encode,decode}_sig`, `Sm2Ciphertext::{x,y}`) were reshaped from
> `U256` to `[u8; 32]` **byte-output-identically** (KAT + gmssl interop 11/11 —
> Group B). `ProjectivePoint` stays public + unchanged (it names no `crypto-bigint`
> type once `to_affine` is hidden — Group C). The **only** residual is the **opt-in**
> `crypto-bigint-scalar` feature's `Sm2PrivateKey::from_scalar(U256)`, a documented
> escape hatch: enabling that feature explicitly opts the caller into the
> `crypto-bigint 0.7` type contract. The committed (`--all-features`) public-api
> baseline (`docs/api-baseline/gmcrypto-core.txt`) records exactly that residual and
> nothing else `crypto-bigint`-typed; the default-features surface is clean. **1.0
> can publish a dependency-decoupled stable API.** The remainder of this section is
> retained for historical context (the audit finding as v0.21 saw it).

The audit found `crypto-bigint` types in the public surface in **16 places**, of
which several are **always-on** (not behind any feature):

- `asn1::sig::{encode_sig, decode_sig}` (+ the `asn1::{encode_sig, decode_sig}`
  re-exports) take / return `(crypto_bigint::U256, crypto_bigint::U256)`.
- `asn1::ciphertext::Sm2Ciphertext::{x, y}` are `pub … : crypto_bigint::U256`.
- `sm2::curve::{Fn, Fp}` are `pub type` aliases for
  `crypto_bigint::…::ConstMontyForm<…>`, used in the public signatures of
  `sm2::scalar_mul::{mul_g, mul_var}`, `sm2::point::ProjectivePoint::to_affine`,
  and `sm2::curve::{b, b3}`; `curve::{NMod, PMod}` impl `crypto_bigint`'s
  `ConstMontyParams`.
- `sm2::private_key::Sm2PrivateKey::from_scalar(U256)` — **opt-in** (behind
  `crypto-bigint-scalar`); the always-on `from_bytes_be(&[u8;32])` is the
  documented recommended path that avoids naming `U256`.

**Why it matters at 1.0:** `crypto-bigint` is itself **pre-1.0 (0.7.x)** and will
ship breaking releases. Any `crypto-bigint` type in gmcrypto-core's **stable** 1.0
public API means a `crypto-bigint` major bump forces a gmcrypto-core **major** bump
— i.e. gmcrypto-core 1.x is effectively pinned to `crypto-bigint 0.7` for its whole
1.x lifetime, or must cut 2.0 to upgrade. This is a genuine SemVer hazard a 1.0
commitment should not take on unexamined.

**Options (maintainer decision before the 1.0 publish):**
1. **Accept + document the version-pinning policy.** Declare that gmcrypto-core
   1.x pins `crypto-bigint 0.7.x` and a `crypto-bigint` major bump is a
   gmcrypto-core major bump. This is the **only** acceptable way to ship a pre-1.0
   dependency in a 1.0 API — it must be an *explicit, documented* acceptance, not
   an oversight. Lowest effort; locks the coupling for all of 1.x.
2. **Tighten the surface first, then wrap the residual.** Some of the exposure is
   the low-level `curve` / `scalar_mul` surface (`Fn`, `Fp`, `mul_g`, `mul_var`,
   `b`, `b3`, `NMod`, `PMod`) that is plausibly **not intended as stable public
   API** (analogous to the `gmcrypto-simd` backend) and could be `#[doc(hidden)]`/
   `pub(crate)`-ed. **But this is NOT purely mechanical:** `ProjectivePoint` is on
   the **high-level key path** — `Sm2PrivateKey::public_key() -> ProjectivePoint`,
   `Sm2PublicKey::{from_point, From<ProjectivePoint>}`, `sec1::EcPrivateKey.public:
   Option<ProjectivePoint>`, and `to_affine() -> (Fp, Fp)`. Hiding it requires
   **replacement byte-shaped public-key/point APIs** on those high-level types
   first. Likewise `encode_sig`/`decode_sig` (`(U256,U256)`) and
   `Sm2Ciphertext::{x,y}` would be wrapped (e.g. a `[u8;32]` / `Signature` shape).
   All of this is **breaking** and appropriate to land **at/just-before** 1.0 — a
   real design cycle, not a doc-attribute pass. The opt-in `from_scalar(U256)` can
   stay (callers opt into naming `U256`).
3. **Defer** — ship 1.0 with the coupling **without** an explicit policy and
   revisit in a 2.0. (Weakest: it's Option 1 minus the deliberate decision.)

**Recommendation:** **resolve this before the 1.0 publish** (it is a pre-1.0
decision, not a post-1.0 one). First **confirm maintainer intent on whether
`curve`/`scalar_mul`/`point` are meant to be public**. If the project is willing to
do the API work, Option 2 gives the cleanest 1.0 (a dependency-decoupled public
surface) — but it is a **non-trivial, breaking design cycle** (replacement
public-key/point + signature/ciphertext byte APIs), best run as its **own focused
pre-1.0 cycle (a "v0.22 API-tightening")**, NOT folded into this minimal audit
(v0.21 deliberately made **no signature changes** — Option A). If that work is out
of appetite, Option 1 (explicitly accept + document the `crypto-bigint 0.7` pin) is
the honest fallback. The decision is teed up here, not forced.

### 3.B — The 3 `#[doc(hidden)]` core items (finalized)
`sm2::sign::sign_raw_with_id` (dudect harness), `Sm4CbcEncryptor::take_output`,
`Sm4CbcDecryptor::take_output` (the `gmcrypto-c` FFI shim). All carry an explicit
"not part of the public API / not covered by SemVer; may change or be removed in
any release" note and are pinned by `crates/gmcrypto-core/tests/api_surface.rs`.
`cargo-public-api` and `cargo-semver-checks` both exclude `#[doc(hidden)]` items, so
they are correctly outside the 1.0 contract. **No cfg-gating** (rejected: too
invasive for nil gain — Q21.4).

### 3.C — `gmcrypto-simd` internal surface (finalized)
The crate's entire public surface is `#[doc(hidden)]` + carries a "no stable Rust
API; internal acceleration backend for `gmcrypto-core`" contract. It stays `pub`
only so `gmcrypto-core` can call it cross-crate; the public-api baseline is just
`pub mod gmcrypto_simd`. Pinned by `crates/gmcrypto-simd/tests/internal_surface.rs`.

### 3.D — Cross-crate coupling / sibling pinning
The hidden cross-crate-consumed items (the two `take_output` shims; the whole
`gmcrypto-simd` surface) stay sound under their "may change in any release" contract
via **lockstep publishing**: the three crates always release together at one
workspace version, and the 1.0 publish (§4) pins the intra-workspace deps **exactly**
(`=1.0.0`). Today's `version = "0.16"` (`^0.16` = `>=0.16.0,<0.17.0` in 0.x) is
already effectively exact; the runbook makes it explicit so the looser post-1.0
`^1.0` can't resolve a mismatched sibling pair.

### 3.E — C-ABI symbol-name backcompat & MSRV
- `gmcrypto_sm2_privkey_to_sec1_be` keeps its pre-rename name for C-ABI backcompat
  (the Rust method is `to_bytes_be`); do not rename the FFI symbol. The C ABI's 1.0
  guard is the cbindgen header drift-check.
- **MSRV = 1.85** (edition 2024). Policy: an MSRV bump is a minor (not patch); CI
  builds the MSRV on every run.

## 4. The `1.0.0` publish runbook (separate step — NOT executed by v0.21)

Pre-flight: **§3.A is resolved** (v0.22 — the surface-tightening cycle decoupled
`crypto-bigint` from the always-on contract); confirm `api-stability` + `ci` + the
dudect/fuzz nightlies are green on `main`.

1. Bump `[workspace.package].version` `0.16.0 → 1.0.0` (root `Cargo.toml`).
2. Pin the intra-workspace path-deps **exactly**: `gmcrypto-core` →
   `gmcrypto-simd = { …, version = "=1.0.0" }`; `gmcrypto-c` → `gmcrypto-core =
   { …, version = "=1.0.0" }` (§3.D).
3. Regenerate the cbindgen header + the `cargo-public-api` baselines (versions in
   item paths don't change, but regenerate to be safe) and confirm the drift-checks
   stay green.
4. `cargo publish --dry-run` + package-content review for each crate, in order
   **`gmcrypto-simd` → `gmcrypto-core` → `gmcrypto-c`** (a crate must be on
   crates.io before its dependents publish).
5. Walk the release-review checklist as the final gate — **adapt/clone**
   `docs/v0.1.0-release-review.md` to a `docs/v1.0.0-release-review.md` first (the
   existing one is v0.1.0-specific).
6. `cargo publish` in the same order; SSH-signed `v1.0.0` tag; GitHub release.
7. **Flip `cargo-semver-checks` to enforced** (drop `continue-on-error`,
   current-vs-`1.0.0`) in `api-stability.yml` — the forward breaking-change gate.
