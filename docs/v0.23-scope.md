# v0.23 scope — pre-1.0 re-audit remediation

**Theme:** fix the BLOCKERs + genuine should-fixes surfaced by the v1.0 pre-publish
re-audit (`docs/v1.0-reaudit.md`), so the deliberate `1.0.0` publish freezes a *correct,
minimal, honestly-documented* contract. **Non-publishing** (workspace stays `0.16.0`; the
breaking API/ABI changes ship **with** 1.0, never reaching a published 0.x crate — the
v0.14/v0.17–v0.22 precedent). Input: `docs/v1.0-reaudit.md` §2–§5.

## 1. Context

The re-audit (two independent model reviews — Codex `gpt-5.5` + Grok — each finding
adversarially source-verified) returned **NO-GO as-is**: the **core primitives are sound**
(mutually-confirmed CT/correctness non-findings — `docs/v1.0-reaudit.md` §3), but there are
**2 API BLOCKERs** + a set of API-finality and zeroize-on-failure / spec-ceiling / doc
should-fixes that become irreversible or harder to fix after 1.0. This cycle remediates
them, then a **clean re-run of the multi-model review on the diff** is the gate to publish.

## 2. Hard contract (non-negotiable)

- **Non-publishing.** No version bump (stays `0.16.0`), no tag, no `cargo publish`. The
  breaking API/ABI *shape* changes ship with the deliberate 1.0 cut.
- **Runtime output byte-identical** for everything except the *deliberately* changed
  signatures (B-1 GCM fallibility): full KAT + gmssl interop **11/11** must stay green;
  the SM2 DER signature + ciphertext wire bytes, SM4 mode outputs, PKCS#8/SPKI fixtures
  are all unchanged. Zeroization changes are not observable in output.
- **All standing invariants preserved:** `unsafe_code = "forbid"` on core; constant-time
  discipline (no new `==`/`if`/`bool` on secrets — and B-7 *removes* one); single
  `Failed`/`None` failure-mode invariant (B-1's new `None`/`Err` is a *length-range* reject,
  not a failure-mode distinction); MSRV 1.85 / edition 2024; `Cargo.lock` gitignored.
- **Each workstream gets a `codex exec --sandbox read-only` milestone review** (the
  established cadence). Branch + PR; never direct to `main`.

## 3. Decisions (Q23.x) — recommended resolutions for Codex confirmation + maintainer scope approval

Forks per `docs/v1.0-reaudit.md` §5. **Bold = recommended.**

- **Q23.1 — A-2 `ProjectivePoint` depth.** **Reshape the high-level path so
  `ProjectivePoint` appears in ZERO visible signatures, then doc-hide the type + its
  re-export (Codex-confirmed: hide the type itself, not just methods).**
  - `Sm2PrivateKey::public_key() -> Sm2PublicKey` (was `-> ProjectivePoint`);
    `Sm2PublicKey::from_sec1_bytes` (already on-curve-checked) is the public constructor.
  - `#[doc(hidden)]` `Sm2PublicKey::{from_point, point}`, `From<ProjectivePoint> for
    Sm2PublicKey`, the bare `ProjectivePoint` arithmetic (`add/double/neg`,
    `generator`/`identity`/`is_identity` — kept `pub` for internal/dev use), **and the
    `sm2::point` module / `sm2::ProjectivePoint` re-export themselves** (module-level
    hiding, the v0.22 Group-A pattern).
  - **Audit + reshape EVERY remaining public signature that names `ProjectivePoint`**
    (Codex flagged the scope under-specified these): `sec1::EcPrivateKey.public:
    Option<ProjectivePoint>` → `Option<Sm2PublicKey>` (or `Option<[u8; SEC1_UNCOMPRESSED_LEN]>`);
    any `spki::{encode,decode}` / `sec1` API that returns/takes a point → speak
    `Sm2PublicKey`/SEC1 bytes. W1 enumerates the full list via the cargo-public-api baseline.
  - `ProjectivePoint` stays a public *opaque* type (decouple-only posture preserved) but is
    out of the documented contract and out of every visible signature.
- **Q23.2 — A-1 C ABI contract.** **Make the AEAD + XTS FFI symbols always-on in
  `gmcrypto-c`:** drop the forwarding `sm4-aead`/`sm4-xts` cargo features on the C shim so
  the committed `gmcrypto.h` == the default `cargo build -p gmcrypto-c` (one complete,
  permanent ABI). `gmcrypto-core` keeps its feature gates (Rust callers opt in); the C
  shim (std-OK integration layer) always compiles every mode. `regen-header` no longer
  needs to "imply" anything. *Alternative (rejected unless Codex prefers): `#ifdef`-guard
  the header + ship a feature→symbol map (fragile for C consumers).*
- **Q23.3 — B-1 single-shot GCM fallibility.** **Make `mode_gcm::{encrypt,
  encrypt_with_tag_len}` fallible** (return the workspace `Result`/`Option`, rejecting
  `> 2^36 − 32` bytes), matching the streaming poison + `decrypt`'s `Option`. The C FFI
  GCM encrypt already returns an error code, so the ABI is unaffected. *Lighter alternative
  (Codex to weigh): keep infallible + document the ceiling as a caller contract.* The
  matching `decrypt` length-guard is additive (already `Option`).
- **Q23.4 — A-3 `rand_core` bound.** **Accept the `rand_core` coupling (do NOT decouple),
  but switch the public signatures to the *fallible* `TryCryptoRng` path
  (Codex-confirmed refinement).** `rand_core` **is** the ecosystem RNG-interop point (unlike
  `crypto-bigint`, a numeric impl detail); a custom RNG trait would break interop for
  net-negative benefit. **However**, the repo is already on `rand_core 0.10.1` — so rather
  than freezing the *infallible* `CryptoRng + Rng` bound (which forces callers into
  `UnwrapErr`/panic adapters for fallible RNGs, and leaves RNG-failure behavior undefined —
  Grok-B's RNG-failure-surface concern), freeze `R: TryCryptoRng` (the fallible trait) and
  **collapse RNG errors to the single `Failed`** (failure-mode invariant). This accepts the
  ecosystem coupling AND gives a defined, no-panic RNG-failure path. Breaking signature
  change, pre-1.0-only. (W2 pins the exact 0.10 trait name — `TryCryptoRng`/`TryRngCore`.)
  Document the deliberate coupling (mirroring the `crypto-bigint-scalar` escape-hatch).
- **Q23.5 — A-4/A-5 surface trim.** **`#[doc(hidden)]` at the MODULE/re-export level
  (Codex: hide modules + re-exports, not only leaf items):** the `pub mod
  asn1::{reader,writer,oid}` declarations and the whole `pub trait
  traits::{Hash,Mac,BlockCipher}` surface (keep `pub` for internal cross-module/dev-crate
  use — the Group-A pattern). Keep `asn1::{encode_sig,decode_sig}` +
  `asn1::ciphertext::Sm2Ciphertext` public (the v0.22 wire types). Regenerate the
  cargo-public-api baseline.
- **Q23.6 — zeroization + ceiling fixes (not forks, confirm approach).** CCM
  `tentative_pt.zeroize()` on the tag-fail arm (B-3); enable `crypto-bigint`'s `zeroize`
  feature and wipe the SM2 sign nonce `k` + scalar intermediates + RNG buffer (B-4); SM2
  KDF `u32` length guard (B-5, `encrypt` already fallible). For B-2 (HMAC), **do NOT add
  `impl Drop for HmacSm3`** — `finalize(self)` moves the inner/outer `Sm3` fields out, which
  conflicts with `Drop` (Codex). Instead make **`Sm3` itself `ZeroizeOnDrop`** (or a manual
  `Drop`+zeroize on `Sm3`), so the keyed compression state wipes on drop at the source — and
  then correct the `HmacSm3` doc to describe the actual (field-layer) behavior. All
  defense-in-depth; no output change.
- **Q23.7 — B-7 nonce sampling.** **Make `sample_nonzero_scalar`'s rejection CT via a
  fixed-budget masked sampler** (Codex): a fixed number of draws, masked-select the first
  valid candidate, collapse exhaustion to `Failed` — NOT an unbounded `loop` with
  `if bool::from(..)`, so the documented "no `bool` on secrets" invariant holds literally
  and there is no data-dependent loop count. (Reject prob ≈ `2^-32` — `n ≈ 2^256 − 2^224`;
  Codex corrected the earlier `2^-224` — so a fixed 2–4-draw budget has negligible failure
  ≈ `2^-64`..`2^-128`, mirroring the K=2 sign-retry pattern.)
- **Q23.8 — docs/disclosure (C-1 + D-1 + B-6/8/9/10).** Fix `gmcrypto-core` `description`
  (add SM4); sharpen README audit-status to "no *third-party* audit; assurance is internal
  (multi-model re-audit + KAT + gmssl interop + 18 fuzz + dudect)"; add SECURITY notes for
  the streaming-CBC contract, short-PT encrypt timing, and the "CT-until-the-validity-bit"
  posture; keep GM/T-certification-out-of-scope.
- **Q23.9 — publish posture.** **Non-publishing** (stays `0.16.0`; ships with 1.0).
