# gmcrypto v0.6 scope

**Crates:** `gmcrypto-core`, `gmcrypto-c`, `gmcrypto-simd` (the
workspace as of v0.5.1).
**Status:** v0.5.0 published to crates.io 2026-05-13; v0.5.1 awaiting
publish at the time of writing (signed tag at `f76a46a`).
**Document scope:** the maintainer signs off on this scope before any
implementation work begins. Implementation plans for each workstream
are drafted *after* sign-off.
**Hard constraints:** all carry-overs from `CLAUDE.md` apply
unchanged — `unsafe_code = "forbid"` on `gmcrypto-core`, `#![no_std]`
+ `alloc`, MSRV 1.85, edition 2024, constant-time discipline on
secrets, failure-mode invariant, fixed-K masked-select sign retry,
`Cargo.lock` gitignored, `sign_raw_with_id` doc-hidden / not-SemVer,
`unsafe_code = "warn"` on `gmcrypto-c` and `gmcrypto-simd` with
`// SAFETY:` comments on every block. Don't restate them; defer to
`CLAUDE.md`.

**v0.6 is a minor release on the 0.x track. The Rust public API
does NOT break** — the workstream below is purely additive on the
internal SIMD plumbing and `Sm4CbcDecryptor`'s internal state
machine. Q5.10's "no new public Rust surface" carries through
unchanged into v0.6.

## 1. Context

v0.5.0 shipped 2026-05-13: W1 streaming SM4-CBC FFI, W2 raw byte-
concat SM2 ciphertext on the C ABI, W3 caller-supplied RNG callback
on the C ABI, W4 phase 1 (`sm4-bitsliced-simd` scaffolding, no SIMD
intrinsics), W5 BREAKING (`gmcrypto_core::Error` unification, U256
escape hatch, `std` feature removal).

v0.5.1 followed quickly (2026-05-14 tag): W4 **phase 2** added the
real AVX2 8-way bitsliced SM4 S-box backend in a new sibling crate
`gmcrypto-simd` (rlib-only, `unsafe_code = "warn"`, mirroring the
`gmcrypto-c` precedent). The Q5.11 addendum recorded the
architectural reset: `safe_arch` + `is_x86_feature_detected!` did
not compose with `forbid(unsafe_code)` + `no_std` on MSRV 1.85, so
the SIMD intrinsics live in the sibling crate behind `// SAFETY:`
comments. Runtime AVX2 detection uses `cpufeatures` (no_std-OK).
v0.5.1 also bundled a dudect calibration update (`ct_fn_invert` +
`ct_fp_invert` moved to telemetry on PR-smoke + gross-regression
sentinel at `|tau| ≥ 0.55` on nightly) after the 2026-05-12 GitHub
Actions `ubuntu-24.04` runner-image change shifted the noise floor.

What v0.5.1 did NOT deliver:

- The AVX2 backend exists and is exercised end-to-end (dudect target
  `ct_sm4_encrypt_block_bitsliced_simd` gated at `|tau| < 0.20`),
  but **`Sm4Cipher`'s `tau` dispatch still feeds the AVX2 path with
  the single input replicated across all 8 lanes (7 wasted per
  call).** Production throughput on x86_64 matches v0.4 single-
  block bitslice.
- NEON 4-way on aarch64 — phase 2's scope stayed x86_64-only by
  design (the sibling-crate architectural reset was the only
  cross-platform claim required for phase 2). aarch64 hosts fall
  back to the scalar single-block bitslice path under
  `sm4-bitsliced-simd`.
- `Sm4CbcDecryptor::process_chunk` SIMD fanout per Q5.10 — the
  surface that would batch parallel ciphertext blocks through the
  packed S-box. Deferred to v0.6.

**v0.6 closes both gaps.** W6 (the only v0.6 workstream) wires the
SIMD path into the `Sm4CbcDecryptor` fanout where the lanes carry
real parallel data, and adds the NEON 4-way backend for `aarch64`.
This is **the throughput-win release**.

Scope boundary: v0.6 does NOT change any cryptographic primitive's
on-wire output. SM2 / SM3 / SM4 byte paths are preserved bit-for-
bit. The SIMD-packed fanout path is byte-identical to per-block
processing (exhaustive equivalence tests cross-checked against the
GB/T 32907-2016 §6.2 S-box table). The dudect-gated constant-time
properties carry through unchanged; one new target may be added
under `cfg(feature = "sm4-bitsliced-simd")` to gate the fanout
path (Q6.7).

## 2. In-scope workstreams

### W6 — Multi-block SIMD fanout for CBC decrypt + NEON 4-way

The headline v0.6 workstream. Three components:

**Component (a) — AVX2 32-byte packed S-box** in `gmcrypto-simd`:

- New public-to-workspace entry `sbox_x32(input: &[u8; 32]) ->
  [u8; 32]` that byte-parallels the v0.4 W3 Boyar-Peralta gate
  sequence across the full 256-bit `__m256i` register width.
  Eight SM4 blocks × 4 `tau` bytes per round = 32 bytes per
  call; zero wasted lanes (vs phase 2's `sbox_x8` 7-of-8 waste).
- Runtime AVX2 detection via the existing
  `cpufeatures::new!(cpuid_avx2, "avx2")` in `detect.rs` (cached
  after first call). On non-AVX2 x86_64 hosts: silent fallback to
  a 32-byte scalar loop calling the existing `sbox_byte`. The
  fallback must NOT be slower than the v0.4 single-block bitslice
  path — codex flag #1 from the phase 3 scope consultation.
- Existing `sbox_x8` stays for `Sm4Cipher::tau`'s per-byte
  dispatch (no churn on phase 2's plumbing).

**Component (b) — NEON 16-byte packed S-box** in `gmcrypto-simd`:

- New public-to-workspace entry `sbox_x16(input: &[u8; 16]) ->
  [u8; 16]` that byte-parallels the same gate sequence across the
  128-bit `uint8x16_t` NEON register width. Four SM4 blocks × 4
  `tau` bytes per round = 16 bytes per call.
- NEON is baseline on `aarch64` (per Q5.12 decision A), so this
  path is compile-time gated under `cfg(target_arch = "aarch64")`
  with no runtime detect — codex's design hint from the phase 2
  consultation.
- For consistency with `sbox_x32`'s scalar-fallback shape, a
  scalar `sbox_x16_scalar` is also exposed (used by non-aarch64,
  non-x86_64 targets and by tests that want to bypass NEON).

**Component (c) — `Sm4CbcDecryptor::process_chunk` SIMD fanout**
in `gmcrypto-core`:

When the buffered ciphertext holds `N` full blocks (where `N` = 8
on `x86_64`, 4 on `aarch64`, 1 on every other target), decrypt
them in a single batch through the packed S-box. The held-back-
final-block invariant is preserved: the LAST decrypted block of
the chunk stays in `held_back: Option<[u8; 16]>` so `finalize`
can strip PKCS#7 padding. Preceding decrypted blocks (including
the prior chunk's `held_back`) are emitted to `output`.

Per Q5.10 the surface is internal — no new public Rust API. The
fanout path is `pub(crate) fn decrypt_batch` in
`cbc_streaming.rs`, gated under
`cfg(feature = "sm4-bitsliced-simd")`. Default builds and
`sm4-bitsliced`-only builds keep the existing single-block loop.

**State-machine invariant the batch helper preserves** (codex's
phase 3 flag #2):

```
Before fanout call:
  prev = ciphertext of last block before this batch (or IV)
  held_back = Some(plaintext of last block before this batch),
              or None on first batch

For input batch ct[0..N]:
  1. saved[i] = ct[i]                   (chaining input snapshot)
  2. pt[0..N] = decrypt_in_parallel(ct[0..N])
  3. pt[0] ^= prev                      (chain to pre-batch state)
     pt[i] ^= saved[i-1]   for i in 1..N
  4. Emit prior held_back to output, if Some
  5. Emit pt[0..N-1] to output
  6. held_back = Some(pt[N-1])          (defer last for padding)
  7. prev = saved[N-1]                  (= ct[N-1])
```

This is the per-block state machine extended to N blocks; the
**only** difference from N=1 is that the SIMD decrypt is amortized
across N blocks of S-box work. Behavior is byte-identical to N
sequential `decrypt_one` calls.

### What v0.6 explicitly does NOT change

- The `Sm4CbcEncryptor` single-block path. CBC encryption chains
  on the previous ciphertext output, so SIMD packing cannot win.
  Q5.10 decision unchanged.
- The `Sm4Cipher` public API. `encrypt_block` / `decrypt_block`
  signatures are unchanged. No `encrypt_blocks` / `decrypt_blocks`
  public batch surface is added (per Q5.10's "no new public Rust
  surface" — and per the absence of an overrideable batch hook in
  the RustCrypto `cipher` 0.4 trait).
- The phase 2 `tau` dispatch path through `sbox_x8`. `Sm4Cipher`'s
  per-byte SIMD route stays as v0.5.1 shipped.
- Workflow files (CI YAML) — except for one new `[[bench]]` entry
  for criterion CBC-decrypt-throughput benchmarks (Q6.6) if the
  decision lands on yes.

## 7. Decisions (autonomous-mode sign-off, 2026-05-14)

### Q6.1 Batch size — fixed 8 / 4 / 1 by arch, or runtime-selectable?

- **A: Compile-time, by target arch.** `BATCH_SIZE` constant in
  `gmcrypto-core` set to 8 on `x86_64`, 4 on `aarch64`, 1 on
  every other arch (`cfg(any(target_arch = "x86_64",
  target_arch = "aarch64"))` selects the SIMD path; everything
  else stays on the existing single-block loop).
- **B: Runtime-selectable.** `gmcrypto-simd` exposes
  `max_batch_size()` that returns 8 / 4 / 1 based on
  `cpufeatures` / arch. `Sm4CbcDecryptor` consults it before
  each batch.
- **C: Variable-N batch.** SIMD entry takes `&mut [[u8; 16]]`
  with any length; processes 8-at-a-time internally with a
  tail loop.

**Recommended:** **A.**

**Decision:** **A.** Compile-time selection is the simplest
shape that respects codex's "non-AVX2 fallback must not be
slower than scalar" (codex flag #1 from phase 3 consultation):
on non-AVX2 x86_64 hosts the SIMD entry's internal scalar
fallback runs N=8 times per batch with no per-iteration
overhead, but the OUTER batch loop in `Sm4CbcDecryptor` still
operates on 8-block chunks, so amortization of the per-batch
setup cost stays favorable. On `aarch64` NEON is baseline so
no runtime branch is needed. Other arches keep N=1, identical
to current behavior. Runtime-selectable (B) adds a branch per
batch with no real benefit on the dominant deploy targets.
Variable-N (C) is the right shape for a generic batch trait
but premature for one specific consumer.

### Q6.2 SIMD entry shape — low-level S-box or high-level decrypt?

- **A: Low-level packed S-box.** `gmcrypto-simd` exposes
  `sbox_x32` / `sbox_x16`. `gmcrypto-core` drives the SM4
  round loop and the per-round byte transposition.
- **B: High-level batched decrypt.** `gmcrypto-simd` exposes
  `decrypt_blocks_x8(round_keys: &[u32; 32], blocks: &mut
  [[u8; 16]; 8])`. `gmcrypto-core` calls once per batch; the
  SIMD crate owns the SM4 round loop.

**Recommended:** **A.**

**Decision:** **A (low-level packed S-box).** Reasons:

1. SM4's round loop is short (~30 LOC); duplicating it across
   `gmcrypto-core` and `gmcrypto-simd` adds a maintenance
   surface for a savings of one function-call boundary.
2. The SM4 round structure (`F` function, `T` / `T'` linear
   transforms, round-key XOR) is byte-level data manipulation
   that `gmcrypto-core` already knows. Pushing it down to
   `gmcrypto-simd` would mean exposing those internals to a
   crate whose only legitimate concern is the SIMD primitive.
3. Codex's hint (phase 2 consultation, pitfall #3) was
   "dispatch once per outer batch, not per S-box or per
   round" — which option A achieves: `gmcrypto-simd::sbox_x32`
   is called once per round (32 rounds total per 8-block
   batch). The cached `cpufeatures` check is one atomic load
   plus one branch per call; at 32 calls per 8-block batch =
   4 calls per block effective, which is acceptable for a
   batch path. The per-batch overhead does NOT scale with N
   (a single `Sm4Cipher` clone + round-key broadcast cost).
4. Keeps `gmcrypto-simd` focused on the SIMD primitive (the
   S-box). Phase 4 (if ever) could add `sbox_x64` for AVX-512
   without rewriting the SM4 round loop.

### Q6.3 NEON detection — runtime detect or compile-time baseline?

- **A: Compile-time baseline** under `cfg(target_arch =
  "aarch64")`. NEON is mandatory on aarch64 (the ARMv8
  baseline includes Advanced SIMD).
- **B: Runtime detect** via `cpufeatures::new!(neon, "neon")`,
  same shape as the AVX2 path.

**Recommended:** **A.**

**Decision:** **A (compile-time baseline).** Per Q5.12 and
codex's phase 2 consultation: NEON is baseline on aarch64; the
ARMv8 architectural reference manual mandates Advanced SIMD
support for every conforming implementation. Runtime detect
on aarch64 is a no-op that just adds a branch. The
`gmcrypto-simd::detect::has_avx2` shape stays x86_64-only;
aarch64's `sbox_x16` is unconditionally available under
`cfg(target_arch = "aarch64")`.

### Q6.4 `Sm4CbcDecryptor` state-machine extension — in-place or copy-out?

The current `decrypt_one` takes `&[u8; 16]` and produces an
internal `[u8; 16]` plaintext that gets moved into
`held_back`. For 8-block batching, two options:

- **A: Stack array of 8 blocks.** `decrypt_batch(&mut self,
  ct: &[[u8; 16]; 8])` operates on a `[[u8; 16]; 8]` buffer
  on the stack (256 bytes), decrypts in parallel, then emits
  7 to `output` and stores 1 in `held_back`. Existing
  `decrypt_one` stays for the N=1 / tail-end case.
- **B: Mutate the input buffer.** Inline the batch decrypt
  inside `process_chunk` against the input slice directly,
  no intermediate stack array. Skips one 256-byte copy.

**Recommended:** **A.**

**Decision:** **A (stack array).** 256 bytes on the stack is
negligible; the readability win of an explicit
`[[u8; 16]; 8]` buffer + a contained `decrypt_batch` helper
is worth more than one memcpy. The hot path is dominated by
the SIMD round loop, not the buffer move. Inlining (B) makes
the chunk-boundary state machine harder to test and review.

### Q6.5 Batch helper visibility

- **A: `pub(crate) fn decrypt_batch`** in `cbc_streaming.rs`.
  Internal to `gmcrypto-core`; not in the public API.
- **B: `pub fn decrypt_batch`** on `Sm4Cipher`. Could be
  useful for non-CBC callers (CTR, ECB) in v0.7+.

**Recommended:** **A.**

**Decision:** **A (pub(crate)).** Per Q5.10's "no new public
Rust surface" carry-over. v0.6 is purely additive on
internal plumbing. If v0.7 ships SM4-CTR mode that wants the
batch primitive, the API can be promoted at that point — but
not speculatively in v0.6.

- **B: Defer to v0.7.** Phase 3 ships without throughput

### Q6.7 New dudect target for the CBC-fanout path?

The existing `ct_sm4_encrypt_block_bitsliced_simd` target
exercises the single-block SIMD path (the v0.5.1 `tau`
dispatch with 7 wasted lanes). v0.6's batch path is a
different code shape (32-byte packed S-box vs 8-byte
replicated S-box) and warrants its own CT proof.

- **A: Add `ct_sm4_cbc_decrypt_fanout`** cfg-gated on
  `sm4-bitsliced-simd`. Class-split by recipient master key
  bytes. Gate at `|tau| < 0.20`. Both PR-smoke (10K) and
  nightly (100K) matrices add the entry.
- **B: Don't add.** Rely on the existing
  `ct_sm4_encrypt_block_bitsliced_simd` + the byte-
  equivalence test against single-block. The new gate
  sequence under SIMD is structurally the same as phase 2.

**Recommended:** **A.**

**Decision:** **A.** Same posture as Q4.10 and Q5.14: new
code path gets its own gate. The byte-equivalence test
proves correctness but not constant-time; the new dudect
target is the CT regression guard. One new entry in the
table at `CLAUDE.md` "Dudect harness gate" section,
modeled on the existing `ct_sm4_encrypt_block_bitsliced_simd`
row.

### Q6.8 SIMD lane-test strategy

Codex flag #4 from the phase 3 consultation: "for SIMD tests,
exhaustively verify every lane position, not just low lanes;
lane-order bugs are the easy miss in `x32` / `x16`."

- **A: Lane-position-shifted sweeps.** For each input byte
  `b ∈ 0..=255` and each lane position `l ∈ 0..32` (AVX2) /
  `0..16` (NEON): place `b` at lane `l`, fill remaining
  lanes with sentinel `0x00`, run `sbox_x32` / `sbox_x16`,
  assert lane `l` of output equals `SM4_SBOX[b]` and every
  other lane equals `SM4_SBOX[0x00] = 0xD6`. Verifies no
  lane-permutation bugs.
- **B: Replicated only.** Existing
  `tests/lane_equivalence.rs` shape: replicate `b` across
  all lanes, verify every lane equals `SM4_SBOX[b]`. Misses
  lane-permutation bugs that affect only some lanes.

**Recommended:** **A.**

**Decision:** **A.** Per codex's phase 3 flag. The
replicated-only test from phase 2 ships forward for the
`sbox_x8` entry; phase 3 adds lane-position-shifted tests
for both `sbox_x32` and `sbox_x16`. Coverage size: 256 ×
32 = 8192 cases for AVX2; 256 × 16 = 4096 for NEON. Each
test runs in <100ms locally.

### Q6.9 `sbox_x8` deprecation — remove in v0.6?

Phase 2's `sbox_x8` is the per-byte dispatch entry; phase 3's
`sbox_x32` is the per-batch entry. Once `Sm4CbcDecryptor` is
wired to `sbox_x32` and `tau` continues to use `sbox_x8`, do
we keep both, or unify on `sbox_x32`?

- **A: Keep both.** `sbox_x8` continues to serve `tau`'s per-
  byte dispatch (the path used by single-block encrypt /
  non-fanout decrypt under `sm4-bitsliced-simd`).
- **B: Remove `sbox_x8`.** Reroute `tau` to call `sbox_x32`
  with the single input padded across 32 lanes (30 wasted
  lanes instead of 7). Simpler API surface; worse single-
  block throughput.

**Recommended:** **A.**

**Decision:** **A.** Same throughput trade-off as phase 2
already accepted with `sbox_x8`'s 7-wasted-lanes shape. The
`tau` per-byte path is not the throughput-critical path in
v0.6 (CBC decrypt is); leaving `sbox_x8` alone preserves
phase 2's measured performance characteristics. `sbox_x32`
is purely additive.

  release notes.
- **B: v0.5.2.** Patch bump. Frames phase 3 as the
  continuation of the v0.5 W4 milestone series.

**Recommended:** **A.**

**Decision:** **A (v0.6.0).** Per the maintainer's directive
to target v0.6.0 as the next release. The W4 milestone
formally closes with v0.6.0 (all three phases shipped). v0.7
opens a new scope-doc cycle for the candidates listed in §8
below. Patch bump (B) would also be defensible (the changes
are additive only, no breaking) but the dedicated minor-
version moniker matches the project's "release-as-narrative"

## 8. v0.7 candidate Q-list (provisional — for v0.7-scope.md)

Not for sign-off in this document; recording so the v0.7 scope-
doc cycle can pick up the thread.

- **Q7.1** Pinned / noise-isolated dudect runner (the
  authoritative fix the v0.5.1 recalibration deferred). Dedicated
  self-hosted machine with CPU isolation (`isolcpus`), fixed
  governor, pinned bench process affinity. Goal: lift the noise
  floor below 0.006 at 100K samples so the original `|tau| <
  0.20` gate can come back on `ct_fn_invert` / `ct_fp_invert`.
- **Q7.2** SM4-CTR mode. Q5.10 originally floated this for v0.6;
  pushed to v0.7 to keep v0.6 focused on the W6 fanout
  milestone. CTR is fully parallelizable so it would benefit
  from the SIMD fanout machinery W6 lands.
- **Q7.3** AEAD modes — SM4-GCM and SM4-CCM. Net-new
  cryptographic surface. Requires constant-time GHASH
  implementation (GCM) or fixed-iteration CMAC (CCM). Owns its
  own scope-doc cycle.
- **Q7.4** RustCrypto `digest = 0.11` / `cipher = 0.5`
  migration when those upstream lines stabilize. Currently
  pinned at major+minor (`0.10` / `0.4`) per Q4.4; both are
  candidate moves once their stable releases are out.
- **Q7.5** `wasm-bindgen-test`-driven KAT runner. Currently
  the `wasm32-unknown-unknown` build is compile-tested only;
  no functional tests run on wasm32 in CI. A `wasm-bindgen-test`
  runner would close that gap with KAT-level functional coverage.
- **Q7.6** AVX-512 16-way variant. Q5.12 decision A deferred
  AVX-512 to "v0.5.x or v0.6 once AVX2 + NEON paths are battle-
  tested." v0.6 closes the AVX2 + NEON path; v0.7 is the
  earliest natural slot for AVX-512.
- **Q7.7** Public batch API on `Sm4Cipher`. Q6.5's `pub(crate)`
  helper could promote to a public `Sm4Cipher::decrypt_blocks`
  if v0.7 adds CTR mode or other multi-block consumers. Worth
  designing alongside Q7.2 if both land in v0.7.
