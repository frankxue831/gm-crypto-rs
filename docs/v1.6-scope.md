# v1.6 scope — TLCP key schedule (G1) + no-confirmation SM2-KX (G3)

Status: direction maintainer-signed at `docs/v1.5-scope.md` Q5.4; the two
API-placement forks (Q6.2/Q6.3) maintainer-signed 2026-06-13. The first
code cycle of the TLCP arc mapped in `docs/tlcp-decomposition.md` (§7
cycle map; §2 gaps G1+G3). Codex-consulted (W1) before planning.

## 1. Sign-off decisions

- **Q6.1 Direction** — **G1 + G3** (maintainer-signed, Q5.4): the TLCP
  key schedule (TLS-1.2-style `P_hash` over SM3: master secret, key
  block, Finished verify_data — GB/T 38636 §6.5) + the no-confirmation
  SM2-KX path (GB/T 32918.3 makes confirmation optional; TLCP ECDHE
  omits the tags). Smallest gap first; unblocks per-primitive KATs for
  the rest of the arc.
- **Q6.2 Feature naming (resolves D-6)** — **`tlcp = []` umbrella**
  (maintainer-signed): one pure-core opt-in feature carries the whole
  toolkit arc; v1.6 ships `tlcp::key_schedule`, v1.7/v1.8 add modules
  under the same flag. Per-piece flags rejected (matrix explosion).
  No new dep; default build byte-identical; no_std-compatible
  (wasm32 leg added).
- **Q6.3 KX placement** — the no-confirmation path lands on the
  existing **`sm2-key-exchange`** feature (maintainer-signed): it is
  standard-permitted generality (32918.3 optional confirmation), not
  TLCP-specific. `tlcp` does NOT imply `sm2-key-exchange`; a TLCP
  consumer enables both (documented).
- **Q6.4 Key-schedule API shape** — module `tlcp::key_schedule`, three
  public functions + two consts + one role enum, **engine-shaped**
  (decomposition §5 discipline: explicit inputs, caller-held buffers,
  no hidden state, injected nothing):
  - `derive_master_secret(pre_master: &[u8; 48], client_random:
    &[u8; 32], server_random: &[u8; 32], out: &mut [u8; 48])` — seed
    order `client ‖ server`, label `"master secret"`. `pre_master` is
    **typed `[u8; 48]`** (Codex W1 High-1): TLCP pins the PMS to 48
    bytes in BOTH key-exchange variants (ECC `version(2)‖random(46)`;
    ECDHE KX klen = 48, double-sourced from gotlcp), and an
    arbitrary-slice input would silently accept truncated secrets
    while the doc claims "typed inputs, zero failure modes".
  - `derive_key_block(master_secret: &[u8; 48], client_random:
    &[u8; 32], server_random: &[u8; 32], out: &mut [u8])` — seed order
    **`server ‖ client`** (the §6.5.2 flip — a KAT-pinned trap),
    label `"key expansion"`, arbitrary `out` length (caller carves per
    suite: CBC 2×(32+16+16)=128 B, GCM 2×(0+16+4)=40 B — documented,
    not hard-coded; suite-agnostic = engine-shaped + resumption-
    agnostic).
  - `finished_verify_data(master_secret: &[u8; 48], role: TlcpRole,
    transcript_hash: &[u8; 32], out: &mut [u8; 12])` — labels
    `"client finished"` / `"server finished"` via `TlcpRole`.
  - `MASTER_SECRET_LEN = 48`, `FINISHED_VERIFY_DATA_LEN = 12`.
  - `P_hash` (`p_sm3`) stays **private** (every TLCP use is one of the
    three derivations; exposing the raw PRF invites misuse). Secret
    outputs go to **caller-supplied buffers** (the `pbkdf2_hmac_sm3`
    discipline — caller owns placement + wiping, documented); internal
    `A(i)` chain values + intermediate HMAC outputs are wiped.
  - **No `Result` anywhere**: with typed inputs there is no failure
    mode at all (`out.len() == 0` on `derive_key_block` is vacuously
    fine). Stronger than single-failure: zero-failure.
- **Q6.5 No-confirmation KX API** — two additive completers on the
  existing v1.1 typestates, both reusing `shared_secret` verbatim
  (same on-curve/identity/all-zero-K rejects, same zeroize set, same
  CT discipline; the only delta is skipping tag computation). Naming
  is the misuse defense (Codex W1 Medium-3: releasing K without tag
  verification is a real misuse-resistance regression vs v1.1, and
  doc warnings alone are weak) — the methods carry the warning at
  every call site:
  - `Sm2KxInitiatorWaiting::derive_without_key_confirmation(self,
    r_b: &Sm2KxEphemeralPoint) -> Result<Sm2SharedKey>` —
    consume-on-use preserved; no `S_B` input, no `S_A` output;
    `x_U`/`y_U` wiped immediately (no tag hashing needs them).
  - `Sm2KxResponder::respond_without_key_confirmation<R: TryCryptoRng>(
    self, r_a: &Sm2KxEphemeralPoint, rng: &mut R) ->
    Result<(Sm2KxEphemeralPoint, Sm2SharedKey)>` — the key releases
    immediately (no Waiting state: with no `S_A` to verify, holding it
    would be commit-on-nothing).
  - A distinct `Sm2UnconfirmedSharedKey` type was considered and
    REJECTED: the key is consumed as bytes either way (PRF input, FFI
    `key_out`), so the type adds surface without a checkpoint; the
    loud method names put the warning where it is read. The v1.1
    confirmed flow is unchanged byte-for-byte. Doc comments state the
    trade plainly: without tags, key confirmation is the caller's
    protocol (TLCP: the Finished exchange).
- **Q6.6 KAT sourcing** — two independent standard-rooted sources, no
  new toolchain:
  - **Key schedule: OpenSSL 3.x `openssl kdf … TLS1-PRF -kdfopt
    digest:SM3`** — verified working locally (3.6.2) before this scope
    was written; the v0.8/v0.12 EVP-oracle precedent (gmssl 3.1.1 has
    no PRF CLI). Recipe pins (Codex W1, per `EVP_KDF-TLS1_PRF` docs):
    TLS1-PRF has **no label parameter — the label bytes are part of
    the seed** (`seed = label ‖ seed-parts`); binary material goes via
    `hexsecret:`/`hexseed:` (plain `secret:`/`seed:` are ASCII);
    `digest:SM3` selects the TLS-1.2-style single-`P_hash` mode
    (OpenSSL special-cases only `MD5-SHA1` for TLS 1.0/1.1); pin the
    **default (non-FIPS) provider**. Vectors: master-secret (48 B),
    key-block 128 B (CBC carve) + 40 B (GCM carve), Finished
    client+server (12 B), plus a long-output multi-iteration case.
    Regen recipe in `docs/v1.6-kat-sourcing.md` + `tests/data/`.
  - **No-confirm KX: the GM/T 0003.5 recommended-curve worked example
    already in-repo** — `K = 6C893473…` is derived *before* the
    confirmation tags, so the existing v1.1 KAT values pin both new
    completers byte-for-byte (fixed standard ephemerals via the
    in-repo `FixedRng`). The worked example uses klen = 16 while TLCP
    uses 48 (Codex W1 Medium-5), so the suite adds a **48-byte
    confirmed-vs-unconfirmed differential** alongside the 16-byte
    standard vector: unconfirmed K == confirmed K on random exchanges,
    both directions, at both klens.
  - D-item resolutions recorded (W0, gotlcp source): **D-2** = ECDHE
    passes nil IDs → default ID `1234567812345678` both sides,
    premaster klen = 48. The 48 is frozen into the v1.6 API signature
    NOW (Codex W1: an API-frozen fact cannot ride a deferred
    cross-check); only the ID convention — a caller-supplied
    *parameter*, not API shape — keeps its v1.7 full-transcript
    cross-check; **D-7** = handshake signatures are plain
    SM2 message-mode signs with the default ID, DER-encoded
    (CertificateVerify signs the SM3 transcript hash *as a message* —
    Z-preprocessing applies); **D-10** = gotlcp aborts on PMS decrypt
    failure — acceptable for SM2 (GM/T 0009 ciphertext is
    integrity-protected; no RSA-style padding oracle), documented in
    the module docs rather than code.
- **Q6.7 Assurance** —
  - **No new dudect target.** Key schedule: HMAC-SM3 chains over
    secrets with public lengths/labels/structure — `ct_hmac_sm3`
    covers the primitive (the v0.3 Q7.6 streaming-HMAC precedent).
    Initiator-side no-confirm KX: a strict work-subset of
    `ct_sm2_key_exchange`'s measured path (same `t`, same `mul_var`,
    same KDF; minus public-input tag ops); the responder side is
    **structurally covered** by the same target (identical
    `shared_secret` core), not literally measured (Codex W1 wording
    fix — `ct_sm2_key_exchange` is initiator-side). SECURITY.md
    records both rationales.
  - **Fuzz: `fuzz_sm2_kx` extended in place, no dispatch byte** — the
    harness additionally drives BOTH new completers on every input
    (Codex W1 High-2): `derive_without_key_confirmation` with the
    `[R_B:65]` bytes, and `respond_without_key_confirmation` feeding
    the same 65 bytes as an adversarial `R_A` into a fixed-RNG
    responder. No seed format change, no remap audit needed (the v1.4
    lesson respected by construction), census stays **27**. Key
    schedule gets NO fuzz target: typed fixed-length inputs, not an
    untrusted-input decoder (the fuzz surface contract from v0.14).
  - Standard legs: per-feature clippy (`tlcp`), deny (features list +
    `tlcp`), MSRV build, wasm32 `--features tlcp --no-default-features`,
    full KAT suite.
- **Q6.8 Versioning** — additive minor → workspace **1.4.0 → 1.6.0**,
  sibling pins `=1.6.0`; **crates.io skips `1.5.0`** (the v0.13→v0.15
  precedent — v1.5 was the unpublished decomposition cycle). Publish +
  SSH-signed tag remain the maintainer's authenticated per-release
  call; agent path = branch + PR.
- **Q6.9 FFI-shape constraints recorded for v1.9** (decomposition §7
  corollary; nothing frozen): key-schedule = stateless
  byte-in/byte-out free functions (no handles, no opaque types;
  `write_output` not even needed — fixed/caller-sized outs);
  no-confirm KX = two symbols on the existing v1.2 handle model
  (`…_derive_unconfirmed` consumes the initiator handle;
  `…_respond_unconfirmed` consumes the responder handle and writes
  `key_out` directly — the caller-wipes-`key_out` contract carries
  over).
- **Q6.10 Docs** — module docs carry the D-2/D-7/D-10 pinned facts +
  the "confirmation is the caller's protocol" trade;
  `docs/v1.6-kat-sourcing.md`; SECURITY.md (dudect rationales, fuzz
  note); CHANGELOG `[1.6.0]`; CLAUDE.md header + commands (clippy/
  deny/wasm32 legs) + architecture map; README feature-table row
  (`tlcp`) + intro feature list.

## 2. Out of scope (deliberate)

- Record protection (v1.7), chain/pair verification (v1.8), toolkit
  FFI (v1.9) — per the decomposition cycle map.
- Handshake message parsing/assembly, transcript management, state
  machines — not crypto (decomposition §3).
- A public raw-PRF entry point (see Q6.4; revisit only with a concrete
  consumer).
- Session-resumption claims: `derive_key_block` is resumption-agnostic
  by construction (it never sees handshake messages), so no
  abbreviated-handshake vectors are owed (decomposition §6 condition
  not triggered).
- Any change to the four published-API surfaces outside
  `sm2/key_exchange.rs` additions + the new `tlcp` module.
