# v0.8 W0 — SM4-CCM KAT sourcing

## Why this doc exists

Per [`docs/v0.7-aead-scope.md`](v0.7-aead-scope.md) §Q8.4: SM4-CCM has no
established interop oracle in the gmcrypto-rs validation toolchain.

- **GM/T 0002-2012 §5.4** publishes one SM4-ECB KAT (Appendix A.1) and no
  CCM-specific vectors.
- **GM/T 0009 / RFC 8998** define the SM4-CCM OID (`1.2.156.10197.1.104.9`)
  but do not ship test vectors.
- **gmssl 3.1.1** ships `sm4 -cbc / -ctr / -gcm / -cbc_sm3_hmac / -ctr_sm3_hmac`
  but **not** `-ccm`. Confirmed locally on 2026-05-15:
  ```
  $ gmssl sm4 -help 2>&1 | head -1
  usage: sm4 (-cbc|-ctr|-gcm|-cbc_sm3_hmac|-ctr_sm3_hmac) ...
  ```

Without an external oracle, v0.8 W3's [`sm4::mode_ccm`] cannot be
cross-validated against a second independent implementation — the same
"two-implementations-agree" property that gmssl + gmcrypto-rs gives us
for SM4-CBC / SM4-CTR / SM4-GCM through the v0.3 W3 and v0.7 W2 interop
test suites.

This doc resolves the gap.

## Decision: OpenSSL 3.x libcrypto EVP `SM4-CCM`

OpenSSL 3.0+ ships `SM4-CCM` in the libcrypto EVP cipher table under
the GB/T 0009 OID. Verified on the development host (2026-05-15):

```
$ openssl version
OpenSSL 3.6.2 7 Apr 2026 (Library: OpenSSL 3.6.2 7 Apr 2026)

$ openssl list -cipher-algorithms 2>&1 | grep -i "sm4-ccm"
  { 1.2.156.10197.1.104.9, SM4-CCM } @ default
```

OpenSSL is chosen over the alternatives surveyed in the v0.8 plan
file because:

1. **Vendor-neutral and broadly audited.** OpenSSL has the largest
   crypto-deployment surface of any open implementation; SM4-CCM
   landed via the broader GM-family addition tracked in `openssl/openssl`
   issues + PRs and is reviewed under the same audit cadence as the
   rest of libcrypto.
2. **OID matches the GB/T 0009 / RFC 8998 spec.** No ambiguity about
   which SM4-CCM variant is being tested.
3. **Already on the development host.** No new toolchain to install or
   pin; reproducibility doesn't depend on Python / Go / language-bound
   crypto libraries.
4. **Auditable harness.** The reference oracle (below) is ~80 lines of
   C against the public EVP API. No magic, no opaque library bindings.

OpenSSL is **not** used at runtime by gm-crypto-rs (the SDK is pure
Rust and `#![no_std]` in core); OpenSSL is only used here at
*test-time* to generate fixture vectors that get committed to the
repo as binary blobs. v0.8 W3's test harness reads those blobs
without depending on OpenSSL.

### Why not the alternatives

- **`tjfoc/gmsm` Go library** — vendor-specific (single Chinese
  open-source project), narrower audit surface than OpenSSL.
- **`pycryptodome` Python wrapper** — pulls in a Python dep into the
  fixture-regen workflow; weaker reproducibility than a stand-alone
  C harness.
- **Hand-derived spec-only** — circular: we'd be validating our
  primitives with our own primitives. Documented in
  [`docs/v0.7-aead-scope.md`](v0.7-aead-scope.md) §Q8.4 as the
  "no external validation" fallback. **Not chosen** — we have
  OpenSSL.

The cross-validation we **do** still want: hand-derive at least one
SM4-CCM vector by composing our own `Sm4Cipher::encrypt_block` calls
(CBC-MAC + CTR), confirm the W3 `mode_ccm::encrypt` output matches it,
*and* confirm both match the OpenSSL oracle's output. Two-of-three
agreement gives us strong confidence that the v0.8 implementation is
correct.

## Reference oracle

The C harness below uses OpenSSL 3.0+'s EVP API to do SM4-CCM
encrypt and decrypt. Output is hex-encoded so it composes cleanly
with shell pipelines and Python fixture-regen scripts.

`crates/gmcrypto-core/tests/data/sm4_ccm_oracle.c` (committed in v0.8 W3
alongside the fixtures):

```c
/* SM4-CCM reference oracle via OpenSSL 3.x libcrypto EVP API.
 *
 * Used to derive KAT vectors for gm-crypto-rs v0.8 SM4-CCM mode_ccm.
 * Reproducibility: see docs/v0.8-ccm-kat-sourcing.md.
 *
 * Build:
 *   cc -O2 -Wall -I$(brew --prefix openssl@3)/include sm4_ccm_oracle.c \
 *      $(brew --prefix openssl@3)/lib/libcrypto.dylib -o sm4_ccm_oracle
 *
 * Encrypt: ./sm4_ccm_oracle enc <key-hex> <nonce-hex> <aad-hex> <pt-hex> <tag-len-bytes>
 *          -> stdout: "<ciphertext-hex> <tag-hex>"
 *
 * Decrypt: ./sm4_ccm_oracle dec <key-hex> <nonce-hex> <aad-hex> <ct-hex> <tag-hex>
 *          -> stdout: "<plaintext-hex>"  or  "TAG_FAIL"
 */
#include <openssl/evp.h>
#include <stdio.h>
#include <stdlib.h>
#include <string.h>

static int hex2bin(const char *h, unsigned char **out, size_t *out_len) {
    size_t l = strlen(h);
    if (l % 2) return -1;
    *out_len = l / 2;
    *out = malloc(*out_len ? *out_len : 1);
    if (!*out) return -1;
    for (size_t i = 0; i < *out_len; i++) {
        unsigned int b;
        if (sscanf(h + 2*i, "%2x", &b) != 1) return -1;
        (*out)[i] = (unsigned char)b;
    }
    return 0;
}

static void print_hex(const unsigned char *b, size_t n) {
    for (size_t i = 0; i < n; i++) printf("%02x", b[i]);
}

int main(int argc, char **argv) {
    if (argc < 7) {
        fprintf(stderr, "usage: %s enc|dec key nonce aad data tag_len_or_tag\n", argv[0]);
        return 2;
    }
    int encrypt = (strcmp(argv[1], "enc") == 0);

    unsigned char *key, *nonce, *aad, *data, *tag = NULL;
    size_t key_len, nonce_len, aad_len, data_len, tag_len;
    if (hex2bin(argv[2], &key, &key_len) ||
        hex2bin(argv[3], &nonce, &nonce_len) ||
        hex2bin(argv[4], &aad, &aad_len) ||
        hex2bin(argv[5], &data, &data_len)) {
        fprintf(stderr, "hex decode failed\n");
        return 2;
    }
    if (encrypt) {
        tag_len = (size_t)atoi(argv[6]);
    } else {
        if (hex2bin(argv[6], &tag, &tag_len)) { fprintf(stderr, "tag hex bad\n"); return 2; }
    }

    EVP_CIPHER_CTX *ctx = EVP_CIPHER_CTX_new();
    if (!ctx) return 1;
    const EVP_CIPHER *cipher = EVP_CIPHER_fetch(NULL, "SM4-CCM", NULL);
    if (!cipher) { fprintf(stderr, "SM4-CCM not in OpenSSL\n"); return 1; }

    int outl = 0;
    unsigned char *outbuf = malloc(data_len + tag_len + 16);

    if (encrypt) {
        EVP_EncryptInit_ex2(ctx, cipher, NULL, NULL, NULL);
        EVP_CIPHER_CTX_ctrl(ctx, EVP_CTRL_AEAD_SET_IVLEN, (int)nonce_len, NULL);
        EVP_CIPHER_CTX_ctrl(ctx, EVP_CTRL_AEAD_SET_TAG, (int)tag_len, NULL);
        EVP_EncryptInit_ex2(ctx, NULL, key, nonce, NULL);
        /* CCM needs total plaintext length up front */
        EVP_EncryptUpdate(ctx, NULL, &outl, NULL, (int)data_len);
        if (aad_len) EVP_EncryptUpdate(ctx, NULL, &outl, aad, (int)aad_len);
        int ct_len = 0;
        EVP_EncryptUpdate(ctx, outbuf, &ct_len, data, (int)data_len);
        int final_len = 0;
        EVP_EncryptFinal_ex(ctx, outbuf + ct_len, &final_len);
        ct_len += final_len;
        unsigned char tag_buf[16];
        EVP_CIPHER_CTX_ctrl(ctx, EVP_CTRL_AEAD_GET_TAG, (int)tag_len, tag_buf);
        print_hex(outbuf, ct_len);
        printf(" ");
        print_hex(tag_buf, tag_len);
        printf("\n");
    } else {
        EVP_DecryptInit_ex2(ctx, cipher, NULL, NULL, NULL);
        EVP_CIPHER_CTX_ctrl(ctx, EVP_CTRL_AEAD_SET_IVLEN, (int)nonce_len, NULL);
        EVP_CIPHER_CTX_ctrl(ctx, EVP_CTRL_AEAD_SET_TAG, (int)tag_len, tag);
        EVP_DecryptInit_ex2(ctx, NULL, key, nonce, NULL);
        EVP_DecryptUpdate(ctx, NULL, &outl, NULL, (int)data_len);
        if (aad_len) EVP_DecryptUpdate(ctx, NULL, &outl, aad, (int)aad_len);
        int pt_len = 0;
        int rv = EVP_DecryptUpdate(ctx, outbuf, &pt_len, data, (int)data_len);
        if (rv <= 0) { printf("TAG_FAIL\n"); EVP_CIPHER_CTX_free(ctx); return 0; }
        print_hex(outbuf, pt_len);
        printf("\n");
    }
    EVP_CIPHER_CTX_free(ctx);
    EVP_CIPHER_free((EVP_CIPHER *)cipher);
    return 0;
}
```

## Parametric coverage matrix for v0.8 W3 fixtures

Per RFC 3610 / NIST SP 800-38C, CCM has three parametric axes:

- **`nonce_len`**: `{7, 8, 9, 10, 11, 12, 13}` (7 lengths).
- **`tag_len`**: `{4, 6, 8, 10, 12, 14, 16}` (7 lengths).
- **`aad_len`** + **`pt_len`**: tested at edges that exercise the AAD
  length-encoding path (2-byte prefix for `aad_len < 2^16 - 2^8`,
  6-byte prefix for `aad_len ≥ 2^16 - 2^8`; W3 likely won't hit the
  6-byte path in unit tests due to memory, but the boundary is
  documented in the CCM spec).

W3 will generate a fixture set covering at minimum:

| Scenario | nonce_len | tag_len | aad_len | pt_len | Why |
|---|---|---|---|---|---|
| baseline | 12 | 16 | 16 | 48 | most-common shape; matches the validation example above |
| minimum-nonce | 7 | 16 | 0 | 16 | edge of allowed nonce range; empty AAD |
| maximum-nonce | 13 | 16 | 0 | 16 | edge of allowed nonce range |
| minimum-tag | 12 | 4 | 16 | 16 | shortest allowed tag |
| middle-tag | 12 | 10 | 16 | 16 | non-default tag length |
| empty-pt-with-aad | 12 | 16 | 16 | 0 | tag-only output (auth path with no encryption) |
| empty-aad-and-pt | 12 | 16 | 0 | 0 | M=0, A=0 degenerate case |
| long-aad-crosses-block | 12 | 16 | 33 | 50 | AAD spans 3 blocks; PT spans 4 blocks |

Each scenario contributes one binary fixture row to
`crates/gmcrypto-core/tests/data/sm4_ccm_kat.bin` (or the equivalent
text-encoded `.txt`) along with the regen-recipe shell snippet that
invokes the oracle.

## Fixture file layout (committed in W3)

W3 will commit the fixtures alongside the v0.3 W2 pattern (binary +
regen-recipe README), but written as a text format since CCM
fixtures are small (~few hundred bytes each) and reviewability
matters more than byte-density:

```
crates/gmcrypto-core/tests/data/
├── README.md                     # existing — v0.3 W2 pkcs8/spki
├── sm4_ccm_oracle.c              # NEW (W0/W3) — the C harness above
└── sm4_ccm_kat.txt               # NEW (W3) — text fixtures + recipe header
```

`sm4_ccm_kat.txt` will be a `# `-commented self-documenting file:

```
# v0.8 W3 SM4-CCM KAT fixtures, generated by sm4_ccm_oracle.c
# Sourcing rationale: docs/v0.8-ccm-kat-sourcing.md
# Re-generate: see header comment of sm4_ccm_oracle.c
#
# Format: one row per blank-line-separated record:
#   scenario = <label>
#   key      = <hex>
#   nonce    = <hex>
#   aad      = <hex>
#   pt       = <hex>
#   tag_len  = <decimal>
#   ct       = <hex>
#   tag      = <hex>

scenario = baseline
key      = 0123456789abcdeffedcba9876543210
nonce    = 000102030405060708090a0b
aad      = feedfacedeadbeeffeedfacedeadbeef
pt       = aaaaaaaaaaaaaaaabbbbbbbbbbbbbbbbcccccccccccccccc
tag_len  = 16
ct       = bdc07260cb3c00cd1144bb9bd492353fbc9391492c2fc81a
tag      = 575b95ef901b592c5f6dc0235c897a3c

scenario = minimum-nonce
...
```

The test harness in `crates/gmcrypto-core/tests/sm4_ccm_kat.rs`
parses this file at test-time, runs `mode_ccm::encrypt` /
`mode_ccm::decrypt` against each row, and asserts byte-identity for
`(ct, tag)` and round-trip identity for `decrypt`.

## Cross-validation plan

For at least the `baseline` scenario, W3 will:

1. Generate `(ct, tag)` via the OpenSSL oracle (committed to
   `sm4_ccm_kat.txt`).
2. Re-derive `(ct, tag)` by composing `Sm4Cipher::encrypt_block` calls
   directly per the NIST SP 800-38C algorithm: B0 + AAD-formatting +
   CBC-MAC chain + CTR-stream + tag-XOR. Done inside a test as a
   secondary reference implementation; ~30 lines of straightforward
   spec-following Rust.
3. Assert OpenSSL output == hand-derived output == `mode_ccm::encrypt`
   output, byte-for-byte.

Three-way agreement gives us strong confidence that the v0.8 CCM
implementation is correct *and* that the OpenSSL oracle is producing
spec-conformant vectors. Any disagreement is a bug somewhere — the
test will tell us which path is wrong.
