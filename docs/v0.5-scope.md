# gmcrypto v0.5 scope

**Crates:** `gmcrypto-core` and `gmcrypto-c` (the workspace as of v0.4.0).
**Status:** `gmcrypto-core 0.4.0` and `gmcrypto-c 0.4.0` published to
crates.io 2026-05-12.
**Document scope:** the maintainer signs off on this scope before any
implementation work begins. Implementation plans for each workstream are
drafted *after* sign-off, the same chunked-PR cadence v0.2 / v0.3 / v0.4
used.
**Hard constraints:** all carry-overs from `CLAUDE.md` apply unchanged
(`unsafe_code = "forbid"` on `gmcrypto-core`, `#![no_std]` + `alloc`,
MSRV 1.85, edition 2024, constant-time discipline on secrets, failure-
mode invariant, fixed-K masked-select sign retry, `Cargo.lock`
gitignored, `sign_raw_with_id` doc-hidden / not-SemVer, `unsafe_code =
"warn"` on `gmcrypto-c` with `// SAFETY:` comments on every block).
Don't restate them; defer to `CLAUDE.md`. **v0.5 is a minor release on
the 0.x track and is permitted to break the public Rust API in service
of W5 ergonomic cleanup.** The break boundaries are enumerated in
Q5.16–Q5.18; nothing else may break.

---

## 1. Context

v0.4.0 shipped 2026-05-12: `wasm32-unknown-unknown` build target, opt-in
RustCrypto-trait fit (`digest-traits` / `cipher-traits`), opt-in
bitsliced (table-less, gate-only) SM4 S-box (`sm4-bitsliced`), and a
brand-new `gmcrypto-c` C ABI crate exposing 31 FFI entry points for
SM2/SM3/SM4/HMAC/PBKDF2 to C / C++ / Go / Zig callers. Plus a CI
optimisation pass (~33% runner-time saved) and a dudect gate
recalibration for the ubuntu-24.04 runner noise floor (`ct_sign_k_class`
PR-smoke gate dropped; nightly stays at 0.25).

What v0.5 inherits as constraint surface:

- `gmcrypto-core` and `gmcrypto-c` API surface as of `v0.4.0`. The two
  v0.5 break boundaries (Q5.16, Q5.17) are scoped to the error-type
  shape and the `crypto_bigint::U256` public-surface coupling. The C
  ABI in `gmcrypto-c` stays additive — every v0.4 entry point keeps
  its signature unchanged in v0.5. New v0.5 entry points are added,
  none removed.
- Failure-mode invariant carries forward. The C ABI's
  `GMCRYPTO_FAILED` int-return convention carries through every new
  W1/W2/W3 entry point unchanged.
- Dudect targets: 12 today (no new targets in v0.4; the v0.3 W2
  `ct_pkcs8_decrypt` was the last addition). v0.5 W4 adds at most 1
  new target (`ct_sm4_encrypt_block_bitsliced_simd`, gated under
  `cfg(feature = "sm4-bitsliced-simd")`). All new targets gate at the
  same per-target thresholds documented in `.github/workflows/dudect-
  *.yml`.
- Dep landscape today: `crypto-bigint = 0.7.3`, `subtle = 2.6.1`,
  `zeroize = 1.8.2`, `rand_core = 0.10.1`, `getrandom = 0.4.2`,
  `spin = 0.10` (`once` feature only); opt-in `digest = "0.10"` and
  `cipher = "0.4"` (W2 feature-gated); `gmcrypto-c` adds `cbindgen
  0.29` (build-time only, `regen-header`-gated). v0.5 adds **zero new
  runtime deps**: W1–W5 are all internal or compose over existing
  primitives. W4's SIMD intrinsics are `core::arch` (built-in).
- v0.5 W5 cleans up the `crypto_bigint::U256` coupling (Q5.17). After
  W5, `Sm2PrivateKey::new(d: U256)` is replaced by `from_bytes_be(&[u8;
  32])` as the recommended constructor; the `U256` constructor remains
  available as `from_scalar` behind a `crypto-bigint-scalar` feature
  flag for callers that already have the dep.

The bridge: v0.2 was *primitive-complete*; v0.3 was *interop-complete*;
v0.4 was *ecosystem-complete* (other languages, other targets,
RustCrypto-trait fit, faster SM4). **v0.5 is the *parity-and-polish*
release.** The C ABI catches up with the Rust public surface (the three
v0.4 W4 deferrals), the SM4 fast path catches up with what hardware can
actually deliver (multi-block SIMD packing), and the public Rust API
gets the one breaking cleanup before the v1.0 stabilization track.

Scope boundary: v0.5 does NOT change any cryptographic primitive's on-
wire output. SM2 / SM3 / SM4 byte paths are preserved bit-for-bit; the
SIMD-packed bitsliced path is constant-time-equivalent to the v0.4
single-block bitsliced path (Q5.10), which is byte-identical to the
linear-scan default. The dudect-gated constant-time properties and the
failure-mode invariant carry through unchanged.

## 2. In-scope workstreams

### W1 — Streaming SM4-CBC FFI (`Sm4CbcEncryptor` / `Sm4CbcDecryptor`
over the C ABI)

**Rationale.** v0.4 W4 shipped single-shot `gmcrypto_sm4_cbc_encrypt`
and `gmcrypto_sm4_cbc_decrypt`. Rust callers have had streaming
`Sm4CbcEncryptor` / `Sm4CbcDecryptor` since v0.3 W5; C callers are
forced to accumulate the entire plaintext in memory before encrypting.
Per `docs/v0.4-scope.md:660,892`, this was explicitly deferred to v0.5
to keep W4 small. Now is the right moment.

**Scope.**

- Add opaque-handle types `gmcrypto_sm4_cbc_encryptor_t` and
  `gmcrypto_sm4_cbc_decryptor_t` to `crates/gmcrypto-c/src/lib.rs`,
  matching the existing `gmcrypto_sm3_t` / `gmcrypto_hmac_sm3_t`
  shape.
- Eight new FFI entry points (four per direction):
  - `gmcrypto_sm4_cbc_encryptor_new(key, iv) -> *mut t`
  - `gmcrypto_sm4_cbc_encryptor_update(t, in_ptr, in_len, out_ptr,
    out_capacity, out_actual_len) -> int`
  - `gmcrypto_sm4_cbc_encryptor_finalize(t, out_ptr, out_capacity,
    out_actual_len) -> int`
  - `gmcrypto_sm4_cbc_encryptor_free(t)`
  - Same four for `_decryptor_*`.
- Update `gmcrypto-c/tests/c_smoke.rs` with at least four new tests:
  streaming encrypt matches single-shot, streaming decrypt matches
  single-shot, chunked-update with arbitrary boundaries produces the
  same output, decryptor's buffer-back-by-one behaviour returns
  `GMCRYPTO_FAILED` cleanly on a truncated final block.
- Regenerate `crates/gmcrypto-c/include/gmcrypto.h` via cbindgen;
  CI's existing header-drift gate enforces correctness.

**Hard contract.** Single `GMCRYPTO_FAILED` return code on any error
path. Internal buffer-back-by-one logic (the v0.3 W5 padding-oracle
defense) is preserved across the FFI boundary — the decryptor's
`_finalize` never returns plaintext if the final block's padding is
invalid.

**Scope boundary.** This workstream does not change the underlying
Rust types or their behaviour. It's a marshalling layer.

### W2 — Raw byte-concat SM2 ciphertext on the C ABI

**Rationale.** v0.3 W4 shipped `encode_c1c3c2` / `decode_c1c3c2` /
`decode_c1c2c3_legacy` in Rust. Some Chinese-standard interop
deployments (older gmssl, certain HSM drivers) expect raw byte-concat
SM2 ciphertext, not GM/T 0009 DER. v0.4 W4 shipped the C ABI with DER
only per Q4.17. C callers needing raw byte-concat have had no path
short of compiling against the Rust crate directly.

**Scope.**

- Three new FFI entry points:
  - `gmcrypto_sm2_encrypt_c1c3c2(pubkey, msg, msg_len, out_ptr,
    out_capacity, out_actual_len) -> int` — emits `C1 || C3 || C2`.
  - `gmcrypto_sm2_decrypt_c1c3c2(privkey, ct, ct_len, out_ptr,
    out_capacity, out_actual_len) -> int` — modern format only.
  - `gmcrypto_sm2_decrypt_c1c2c3_legacy(privkey, ct, ct_len, out_ptr,
    out_capacity, out_actual_len) -> int` — **decrypt-only** legacy
    `C1 || C2 || C3` ordering; the contract that this is non-auto-
    detecting is documented in the cbindgen-generated header
    (Q5.5).
- No `gmcrypto_sm2_encrypt_c1c2c3_legacy` entry point — same posture
  as the Rust crate (no `encode_c1c2c3_legacy`). Don't propagate the
  legacy ordering forward.
- Three new `c_smoke` tests: round-trip C1||C3||C2, round-trip via
  legacy decrypt of an emitted-by-Rust C1||C2||C3 (constructed in the
  test setup), corruption tests confirming `GMCRYPTO_FAILED` on every
  failure path.

**Hard contract.** Identical failure-mode invariant to W4's DER path
(`GMCRYPTO_FAILED` on every distinguishable failure mode collapses to
the same return code). Same `point_on_curve` invalid-curve defense as
the DER path — preserved because both routes go through `sm2::decrypt`
internally.

### W3 — Caller-supplied RNG callback on the C ABI

**Rationale.** v0.4 used `getrandom::SysRng` internally for
`gmcrypto_sm2_sign` and `gmcrypto_sm2_encrypt` (Q4.18). HSM, SDF, and
SKF integrations need their own RNG plumbed through. Per Q4.18 this
was deferred to "post-v0.4 when a use case surfaces"; the use case is
ecosystem fit for the Chinese commercial crypto stack (HSM-issued
random for sign / encrypt operations).

**Scope.**

- Two new FFI entry points (variant `_with_rng` of the existing two):
  - `gmcrypto_sm2_sign_with_rng(privkey, msg, msg_len, id, id_len,
    rng_callback, rng_context, out_ptr, out_capacity, out_actual_len)
    -> int`
  - `gmcrypto_sm2_encrypt_with_rng(pubkey, msg, msg_len, rng_callback,
    rng_context, out_ptr, out_capacity, out_actual_len) -> int`
- RNG callback ABI (Q5.6):
  ```c
  typedef int (*gmcrypto_rng_callback)(void *context, uint8_t *buf,
                                       size_t buf_len);
  ```
  Returns `0` on success, non-zero on failure. On failure, the
  enclosing `gmcrypto_sm2_*_with_rng` call returns
  `GMCRYPTO_FAILED`.
- Existing `gmcrypto_sm2_sign` / `gmcrypto_sm2_encrypt` keep using
  `getrandom::SysRng` internally — additive surface (Q5.7). v0.4
  callers see no change.
- Callback re-entrancy contract (Q5.9): callbacks MUST NOT call back
  into `gmcrypto-c` (no nested locks; simplifies the implementation;
  fails fast in tests if violated via a runtime debug-only check).
- Three new `c_smoke` tests: deterministic callback producing fixed
  bytes for KAT verification, callback-returns-error propagates to
  `GMCRYPTO_FAILED`, callback context pointer correctly passed
  through.

**Hard contract.** The callback ABI is **stable in v0.5** —
`extern "C"`, no Rust types crossing the boundary. The internal
implementation wraps the callback in an `impl rand_core::Rng` adapter
that satisfies the existing `sm2::sign_raw_with_id` / `sm2::encrypt`
signatures. The fixed-K masked-select retry contract on sign is
preserved: a callback that returns the same bytes twice still gets
masked-select retry on both, exactly the same as a real RNG.

### W4 — Multi-block SIMD-packed SM4 bitslicing (`sm4-bitsliced-simd`
feature)

**Rationale.** v0.4 W3 shipped a single-block Boyar-Peralta bitsliced
S-box. Same throughput as the linear-scan default (~1-2M blocks/s)
because the throughput floor is the linear-scan-equivalent count of
gate operations applied to a single block, not the S-box choice. Per
Q4.11 the single-block scope was deliberate to keep W3 small. The
multi-block path packs 8 (AVX2) or 16 (AVX-512) or 4 (NEON) blocks
into the same gate sequence, getting linear-in-N throughput. Target:
~50-100M blocks/s on a server-class CPU.

**Scope.**

- New module `crates/gmcrypto-core/src/sm4/sbox_bitsliced_simd.rs`.
  Provides:
  - Architecture-specific packed S-box: AVX2 (`__m256i`), AVX-512
    (`__m512i`), NEON (`uint8x16_t`).
  - Block-packing / unpacking primitives that lay out N parallel
    blocks across SIMD lanes per the bitsliced layout.
  - SM4-CBC parallel-decrypt and SM4-CTR (when available — see scope
    boundary below) fast paths.
- Runtime CPU detection (Q5.11): `is_x86_feature_detected!("avx2")` /
  `is_x86_feature_detected!("avx512f")` on x86_64; `target_arch
  = "aarch64"` (NEON is baseline on aarch64) on ARM. Compile-time
  fallback to the v0.4 single-block bitslice when neither is
  available. Q5.13 pins the fallback policy.
- New `sm4-bitsliced-simd` feature flag in
  `crates/gmcrypto-core/Cargo.toml`. Independent from `sm4-bitsliced`
  (the single-block opt-in stays available). Default-off (Q5.15).
- One new dudect target `ct_sm4_encrypt_block_bitsliced_simd` gated
  under `cfg(feature = "sm4-bitsliced-simd")`, gating at `|tau| <
  0.20` matching the existing `ct_sm4_encrypt_block_bitsliced` (Q5.14
  applies the same Q4.10 reasoning: new impl gets its own gate).
- Throughput benches via `criterion` (separate from the dudect
  harness) measuring single-block vs SIMD-packed vs OpenSSL/gmssl SM4
  on the same hardware. Bench is `examples/`-only, not gated in CI
  (avoids CPU-class-dependent flakes).
- The `Sm4CbcEncryptor` / `Sm4CbcDecryptor` streaming types do NOT
  switch to the SIMD path in v0.5 — SM4-CBC's chain-of-blocks
  serialization defeats per-block SIMD packing for *encryption*.
  Decryption CAN parallelize because each ciphertext block decrypts
  independently before XOR with the preceding ciphertext. Q5.10 pins
  the surface that opts in: `Sm4CbcDecryptor::process_chunk` uses
  the SIMD path when `sm4-bitsliced-simd` is enabled; encryption
  stays on the single-block path even under the feature flag.

**Hard contract.** Byte-identical output to the v0.4 single-block
bitsliced path on every input. Exhaustive equivalence test added at
`sm4::sbox_bitsliced_simd::tests::simd_matches_single_block` —
brute-forces all 256 inputs through the packed S-box and confirms
each lane matches the single-block bitslice. Constant-time by
construction (the same gate sequence as v0.4 W3, just SIMD-widened);
the new dudect target gates it under load.

**Scope boundary.** No new public Rust API surface — `Sm4Cipher`'s
public methods are unchanged; the feature flag swaps the S-box
implementation under the hood. No SM4-CTR mode in v0.5 (would
require a new public surface for counter-mode block construction —
deferred to v0.6 alongside AEAD modes).

### W5 — Error-type unification + `crypto_bigint::U256` escape hatch +
`std` feature decision (BREAKING)

**Rationale.** Three v0.4 audit findings (G1, G2, D3) bundled into one
breaking workstream. v0.5 is the right release boundary: callers
migrate once, get all three cleanups, before v1.0's API stabilization.

**Scope (G1 — error-type unification, Q5.16).**

- Replace `sm2::SignError`, `sm2::EncryptError`, `sm2::DecryptError`,
  `pem::Error`, `pkcs8::Error` with a single workspace-wide
  `gmcrypto_core::Error::Failed` enum (single variant).
- Re-export at module level for callers: `sm2::Error`, `pem::Error`,
  `pkcs8::Error` all aliasing the workspace-wide type. Callers
  matching on the type name keep compiling without a path change;
  callers matching on variant names see only `Failed`.
- C ABI is unaffected — `GMCRYPTO_FAILED` is the only error code
  already.

**Scope (G2 — `crypto_bigint::U256` escape hatch, Q5.17).**

- `Sm2PrivateKey::new(d: U256) -> CtOption<Self>` becomes
  `Sm2PrivateKey::from_scalar(d: U256) -> CtOption<Self>` (rename,
  unchanged behaviour). Gated behind a `crypto-bigint-scalar`
  feature.
- New `Sm2PrivateKey::from_bytes_be(d: &[u8; 32]) -> CtOption<Self>`
  / `to_bytes_be(&self) -> [u8; 32]` as the recommended public
  constructor / serializer. Always-available (no feature flag).
- `asn1::sig::{encode_sig, decode_sig}` and `asn1::ciphertext::*`
  keep `U256` because they're internal-ASN.1 helpers tied to the
  bigint representation. Mark `#[doc(hidden)]` so future callers
  don't grow new dependencies on them.

**Scope (D3 — `std` feature decision, Q5.18).**

- The `std` feature has been a no-op reservation since v0.3. Remove
  the feature flag entirely; the `Cargo.toml` line is gone in v0.5.
  If a v0.5+ file-I/O helper materializes, it gets a new specific
  feature flag (`std-file-io`) — naming a reservation for a future
  is worse than removing it.

**Hard contract.** v0.5 is permitted to break these three surfaces.
Every break is enumerated in `CHANGELOG.md`'s `[0.5.0]` "Breaking
changes" section. Migration guidance is documented in the same
section. Sub-workstreams ship in one PR (W5) so callers do one
migration, not three.

**Scope boundary.** Does not change the C ABI (already collapses to
`GMCRYPTO_FAILED`). Does not change the `sm2::raw_ciphertext`,
`sm3`, `sm4`, `hmac`, `kdf` surfaces (their existing error returns
were already `bool`, `Option`, or `CtOption` — no `Error` enums
there). Does not change the `compute_z` / `sign_with_id` /
`verify_with_id` signatures.

## 3. Sequencing

The five workstreams are mostly independent. Recommended PR order:

1. **`docs/v0.5-scope.md`** — this doc, sign-off PR. No code.
2. **W1 — Streaming SM4-CBC FFI.** Smallest, most established
   pattern. Builds confidence on the FFI scaffold.
3. **W2 — Raw byte-concat SM2 ciphertext on C ABI.** Trivial
   continuation of W1's pattern. Could land before or after W1.
4. **W3 — RNG callback on C ABI.** New FFI surface (callback ABI);
   land after W1/W2 so the c_smoke test patterns are established.
5. **W4 — Multi-block SIMD-packed SM4 bitslicing.** Largest. Land
   in parallel with W1-W3 if possible — independent code path.
6. **W5 — Error-type unification + escape hatch + std cleanup.**
   Last because it's the breaking change. Callers reading the
   CHANGELOG land on v0.5.0 and migrate once.
7. **CI optimisation pass.** Mirror v0.4's pre-release CI tightening.
8. **Release-prep PR.** `docs/v0.1.0-release-review.md` checklist;
   STOP before `cargo publish`.

W1/W2/W3 share the FFI scaffolding; if they land sequentially the
cabi job's cbindgen header-drift gate catches any duplicate-symbol
mistakes early. W4 is independent of W1-W3 entirely (different
crate, different code path). W5 is the breaking change; landing it
last gates the v0.5 release tag.

## 4. Out of scope for v0.5

- **AEAD SM4 modes (SM4-GCM, SM4-CCM).** Big functional gap; warrants
  its own v0.6 release with proper primitive sourcing, KAT vectors
  from authoritative sources, dudect targets, and gmssl interop.
- **`wasm-bindgen-test`-driven KAT runner on wasm32.** Builds
  correctly today (v0.4 W1). Cross-validation is a v0.5.x or v0.6
  chore; not blocking on us.
- **`digest 0.11` / `cipher 0.5` major-bump trait fit.** Blocking on
  upstream timing. When the lines stabilize, a v0.5.x patch can add a
  second pair of feature flags (`digest-traits-0.11` /
  `cipher-traits-0.5`) or a clean swap if v0.4.x callers don't object.
- **C-toolchain CI integration for `gmcrypto-c/examples/`.** Small
  chore; can ride along any v0.5 workstream without its own scope row.
- **HKDF-SM3, Argon2-with-SM3, other KDFs.** v0.5+ candidates per
  `docs/v0.4-scope.md:844`. PBKDF2-HMAC-SM3 remains the only KDF.
- **C++ class wrappers / Python ctypes wrapper / Go cgo wrapper.**
  Out-of-tree binding work; the cdylib + staticlib + header already
  enables external binding crates.
- **File-I/O wire-format helpers behind a new `std-file-io` feature.**
  Q5.18 removes the no-op `std` feature; a future file-I/O helper
  would land in v0.5.x or v0.6 with the new specific feature name.
- **SM2 sign nonce-loop CT proof (G5 in the system-design review).**
  Probabilistically bounded; `ct_sign_k_class` empirically covers it.
  A formal proof would require a new `sample_nonzero_scalar` shape
  that ablates the iteration count entirely; deferred until a real
  CT-proof effort runs.
- **SM4-CTR mode** (mentioned in W4 scope boundary). Deferred to v0.6
  alongside AEAD.

## 5. Definition of Done — v0.5.0

### CHANGELOG `[0.5.0]` checklist (skeleton)

```markdown
## [0.5.0] — YYYY-MM-DD

### Breaking changes (v0.5 is permitted to break — v0.5 only)

- **Error-type unification.** `sm2::SignError`, `sm2::EncryptError`,
  `sm2::DecryptError`, `pem::Error`, `pkcs8::Error` collapse to a
  single workspace-wide `gmcrypto_core::Error::Failed`. Migration:
  match on `Error::Failed` instead of variant-specific names; the
  module-level `sm2::Error` / `pem::Error` / `pkcs8::Error` type
  aliases keep import paths compiling.
- **`Sm2PrivateKey` constructor surface.** `Sm2PrivateKey::new(d:
  U256)` is renamed to `Sm2PrivateKey::from_scalar(d: U256)` and
  gated behind the `crypto-bigint-scalar` feature. New `from_bytes_be`
  / `to_bytes_be` constructors are always-available and are the
  recommended migration target. Callers using `U256` directly: add
  the feature flag or migrate to `from_bytes_be`.
- **`std` feature flag removed.** It was a no-op reservation since
  v0.3. If you had `gmcrypto-core = { features = ["std"] }` in your
  manifest, drop the flag — no behavioural change. A future file-I/O
  helper will land under a new specific feature flag, not `std`.

### Added

- **v0.5 W1** — Streaming SM4-CBC over the C ABI. Adds
  `gmcrypto_sm4_cbc_encryptor_*` and `gmcrypto_sm4_cbc_decryptor_*`
  entry points to `gmcrypto-c`. Same buffer-back-by-one padding-
  oracle defense as the v0.3 W5 Rust streaming surface.
- **v0.5 W2** — Raw byte-concat SM2 ciphertext on the C ABI:
  `gmcrypto_sm2_encrypt_c1c3c2`, `gmcrypto_sm2_decrypt_c1c3c2`,
  `gmcrypto_sm2_decrypt_c1c2c3_legacy` (decrypt-only).
- **v0.5 W3** — Caller-supplied RNG callback on the C ABI:
  `gmcrypto_sm2_sign_with_rng` and `gmcrypto_sm2_encrypt_with_rng`
  variants taking a `gmcrypto_rng_callback` function pointer.
  Existing `gmcrypto_sm2_sign` / `gmcrypto_sm2_encrypt` keep using
  `getrandom::SysRng` internally — additive.
- **v0.5 W4** — Multi-block SIMD-packed SM4 bitslicing behind the
  new `sm4-bitsliced-simd` feature. AVX2 / AVX-512 / NEON paths with
  runtime CPU detection; compile-time fallback to the v0.4 W3
  single-block bitslice. New dudect target
  `ct_sm4_encrypt_block_bitsliced_simd` (gated under the feature).
  `Sm4CbcDecryptor` uses the SIMD path; `Sm4CbcEncryptor` stays on
  single-block (CBC's chain serializes encryption).
- **v0.5 W5** — Always-available `Sm2PrivateKey::from_bytes_be` /
  `to_bytes_be` constructors (G2 escape hatch).

### Changed

- New `c_smoke` integration tests cover every new FFI entry point.
- cbindgen-regenerated `crates/gmcrypto-c/include/gmcrypto.h`.
- Dudect harness: 13 targets (+1 SIMD-packed). PR-smoke and nightly
  matrices add a third entry under `features=sm4-bitsliced-simd`.
- README and CLAUDE.md updated for v0.5 scope, dudect-table delta,
  breaking-change migration notes.

### Decided (v0.5 scope)

See `docs/v0.5-scope.md` Section 7 (Q5.1–Q5.20).

### Posture (unchanged)

- `unsafe_code = "forbid"` on `gmcrypto-core`; `"warn"` on
  `gmcrypto-c` (W4's SIMD intrinsics on `gmcrypto-core` use only
  safe `core::arch` wrappers — no new `unsafe` blocks).
- `#![no_std]` + `alloc`-only inside `crates/gmcrypto-core/src/`.
- Constant-time discipline on all secret-touching paths. The new
  SIMD-packed bitsliced path is constant-time by construction (same
  gate sequence as v0.4 W3, SIMD-widened).
- Failure-mode invariant: every public surface that can fail returns
```

### Interop test additions

- Streaming SM4-CBC C-ABI tests cross-validated against single-shot
  CBC on the same plaintext/key/iv.
- Raw byte-concat C-ABI tests cross-validated against the Rust crate
  via `extern "C"` interop in `c_smoke`.
- RNG-callback C-ABI tests use a deterministic callback that supplies
  a fixed nonce `k` for KAT verification of `gmcrypto_sm2_sign_with_rng`
  against a known-vector signature.

### Dudect target additions

## 7. Decisions (autonomous-mode sign-off, 2026-05-12)

Each question below is answered by the autonomous-execution agent
per the prompt's instruction to "make the calls and document them"
with each Q defaulted to its "Recommended:" option (matching the v0.3
/ v0.4 sign-off pattern). Pro/con bullets remain so future readers
can see the trade-offs that were considered.

### Q5.1 Streaming CBC FFI handle model — `Box::into_raw` again?

The v0.4 W4 baseline (Q4.6) used `Box::into_raw` for SM3, HMAC, SM4,
SM2 handles. Reuse for streaming CBC?

- **Pro:** consistency; no new pattern to audit; same `_free`
  contract.
- **Con:** the streaming types are short-lived (`_new` → multiple
  `_update` → `_finalize` → `_free`); they're heap-allocated each
  time. Alternative: stack-allocated opaque buffer with the encryptor
  state inline.

**Recommended:** `Box::into_raw`.

**Decision:** **`Box::into_raw`.** Same pattern as every other v0.4
W4 handle. Streaming CBC's per-handle state is small (~80 bytes:
round keys + IV chain + back-by-one buffer) but consistency wins.
The single-`free` contract is the right invariant.

### Q5.2 Two separate streaming types or one unified `_cbc_t`?

The Rust surface is `Sm4CbcEncryptor` and `Sm4CbcDecryptor` — two
types with separate state machines (encrypt buffers a partial block;
decrypt buffers back-by-one). C ABI could expose one
`gmcrypto_sm4_cbc_t` taking a mode enum.

- **Pro of unified:** smaller type-name surface.
- **Con of unified:** mode-enum dispatch creates a code path in the
  C boundary that the Rust side doesn't have. Easy to mis-use
  (encrypt-with-decryptor handle).

**Recommended:** two separate types.

**Decision:** **two separate types.** `gmcrypto_sm4_cbc_encryptor_t`
and `gmcrypto_sm4_cbc_decryptor_t`. Matches the Rust surface
1:1. Different `_update` signatures naturally — encryptor's update
appends ciphertext when a block fills; decryptor's update appends
plaintext only when the back-by-one buffer fills. The compiler
catches the mis-use.

### Q5.3 Streaming output buffer convention — single (out, capacity,
actual_len) on each update?

Each `_update` call may emit zero or more full blocks of output
depending on how much input fills the current partial block.

- **Recommended:** every `_update` / `_finalize` call uses the same
  `(out, out_capacity, out_actual_len)` convention as v0.4 W4. On
  insufficient capacity, the function returns `GMCRYPTO_FAILED` and
  writes the required capacity to `out_actual_len` for the caller to
  retry.

**Decision:** **standard (out, capacity, actual_len) on every
streaming entry point.** Matches v0.4 W4; documented contract.

### Q5.4 Raw byte-concat C ABI — mirror all three Rust entry points?

`encode_c1c3c2`, `decode_c1c3c2`, `decode_c1c2c3_legacy`.

- **Recommended:** mirror all three.

**Decision:** **mirror all three.** No legacy *encode* path — same
posture as Rust (per `CLAUDE.md` "Don't" entry). The cbindgen-
generated header carries a doc-comment on the legacy decode entry
warning callers that auto-detection between the two byte orderings
is not provided; the caller must know which format the wire-data
follows.

### Q5.5 Legacy decrypt-only contract communication in the C header

How to make it impossible for a casual reader to interpret
`gmcrypto_sm2_decrypt_c1c2c3_legacy` as a default-recommended path?

- **Option A:** doc-comment in cbindgen-generated header (visible
  but easy to miss).
- **Option B:** name prefix (`_legacy_` already in the entry point
  name).
- **Option C:** both A and B.

**Recommended:** both.

**Decision:** **both A and B.** Entry point name includes
`_legacy_`; cbindgen doc-comment warns callers in three lines about
the byte-ordering ambiguity and the absence of auto-detection.

### Q5.6 RNG callback ABI shape

Options:

- **A: function pointer + void* context.** Standard C-ABI pattern;
  matches OpenSSL's `RAND_method` shape.
- **B: vtable struct.** More extensible; supports multiple methods
  (init, get_bytes, deinit) — overkill for v0.5's "fill a buffer
  with random bytes" surface.

**Recommended:** A.

**Decision:** **A (function pointer + opaque `void*` context).**
Single entry point per callback: `int callback(void *ctx, uint8_t
*buf, size_t buf_len) -> int`. Returns 0 on success, non-zero on
failure. The context pointer is opaque to gmcrypto-c — callers stash
HSM session handles, SDF/SKF context, whatever's needed. ABI shape
documented in `gmcrypto-c/include/gmcrypto.h` via cbindgen.

### Q5.7 Coexistence: keep existing `_sign` / `_encrypt`, add `_with_rng`?

- **Pro:** v0.4 callers see no breaking change.
- **Con:** larger surface (4 entry points instead of 2 for
  sign+encrypt); two RNG sources to maintain.

**Recommended:** keep both.

**Decision:** **keep both.** Additive. `gmcrypto_sm2_sign` keeps
`getrandom::SysRng`; `gmcrypto_sm2_sign_with_rng` takes the
callback. Same for encrypt. v0.4 callers see no change. Internally
both routes flow through the same `sm2::sign_raw_with_id` /
`sm2::encrypt` core; the difference is the `Rng` adapter passed in.

### Q5.8 RNG callback failure mode

Callback returns non-zero. What does the enclosing `_with_rng` entry
point return?

- **Recommended:** `GMCRYPTO_FAILED` (single failure code). No
  distinction between callback-error and crypto-failure exposed to
  the caller.

**Decision:** **`GMCRYPTO_FAILED`.** Failure-mode invariant
preserved. The caller's callback can log its own errors; the C ABI
collapses to the same single failure code.

### Q5.9 Re-entrancy: can callback call back into `gmcrypto-c`?

E.g., callback calls `gmcrypto_sm3_new` to derive bytes from a
session token.

- **Recommended:** **forbidden**. The callback contract is "fill
  the supplied buffer with `buf_len` random bytes and return". No
  re-entrancy into `gmcrypto-c`. The Rust `Rng` adapter does not
  hold any locks across the callback; re-entrancy is technically
  safe but the policy is "don't" for clarity.

**Decision:** **forbidden.** Documented in the cbindgen-generated
header doc-comment. No runtime check in v0.5 (would add a thread-
local flag); v0.6 may add a debug-build assertion if anyone violates
the contract.

### Q5.10 `sm4-bitsliced-simd` scope — SM4-CBC encryption included?

SM4-CBC encryption chains: each block's input depends on the previous
ciphertext. SIMD packing only wins on independent blocks (CBC
*decryption*, CTR, ECB).

- **Recommended:** **CBC decryption opts in to SIMD; CBC encryption
  stays on single-block.** No new public Rust surface — the feature
  flag swaps the underlying S-box implementation; `Sm4CbcDecryptor`'s
  internal loop fans out across SIMD lanes when `sm4-bitsliced-simd`
  is enabled.

**Decision:** **CBC decryption only.** Encryption stays
single-block (no win available). Future SM4-CTR mode (v0.6) will get
the SIMD fast path on both directions.

### Q5.11 SIMD CPU detection — runtime or compile-time?

- **A: runtime detection** via `is_x86_feature_detected!("avx2")` /
  `is_x86_feature_detected!("avx512f")`. One binary works on every
  CPU class; fallback at runtime.
- **B: compile-time only** via `#[cfg(target_feature = "avx2")]`.
  Requires callers to compile with `RUSTFLAGS="-C
  target-feature=+avx2"` for the fast path. Smaller binary; no
  runtime dispatch.

**Recommended:** A.

**Decision:** **runtime detection.** Most binaries ship to mixed CPU
fleets; runtime dispatch is the right default. Compile-time
target-feature is still respected via `cfg!` macros: a build with
`+avx2` skips the detection check (the compiler can prove the
intrinsics are available). On `aarch64`, NEON is baseline — no
runtime detection needed.

### Q5.12 SIMD width — start with AVX2 only, or AVX2 + AVX-512 + NEON?

- **A: AVX2 + NEON.** Covers most server CPUs and ARM. ~8-way / 4-way
  packing.
- **B: AVX2 + AVX-512 + NEON.** Adds 16-way packing on AVX-512 CPUs.
  More implementation surface.
- **C: AVX2 only.** Simplest. Loses ARM and AVX-512.

**Recommended:** A.

**Decision:** **A (AVX2 + NEON).** AVX-512 is a v0.5.x or v0.6
extension once AVX2 + NEON paths are battle-tested. Wider lanes
require careful gate-sequence tuning; AVX2 has 5 years of
public-domain reference implementations to crib from. NEON is
baseline on aarch64 so no runtime check is needed there.

### Q5.13 Fallback policy when no SIMD available

A binary built with `sm4-bitsliced-simd` runs on a CPU without
AVX2 / NEON.

- **A: panic at runtime.** Simplest; matches "you asked for it, the
  CPU can't deliver."
- **B: silent fallback to v0.4 single-block bitslice.** Hides the
  performance trade-off but keeps the binary working.
- **C: compile-time guard via `#[cfg(any(target_arch = "x86_64",
  target_arch = "aarch64"))]`.** Compile fails on unsupported
  architectures.

**Recommended:** B + C combined.

**Decision:** **B + C.** Compile-time guard: feature is only
buildable on x86_64 and aarch64 (cfg-gates the `sbox_bitsliced_simd`
module). Runtime: on x86_64 without AVX2, fall back to v0.4
single-block bitslice silently — the throughput regresses but
correctness and CT properties are preserved. Document the silent
fallback in the README so capacity-planners can read it.

### Q5.14 New dudect target for SIMD-packed path?

Same question as Q4.10 (was "new dudect target for bitsliced?").
Bitsliced reused the same CT-by-construction property; SIMD-packed
applies the same gate sequence across SIMD lanes. New impl OR same
gate sequence?

- **Recommended:** ship a separate target.

**Decision:** **ship `ct_sm4_encrypt_block_bitsliced_simd`.** New
implementation (SIMD intrinsics, lane packing/unpacking, possibly
new branches around runtime detection), so it gets its own gate.
Same `|tau| < 0.20` threshold; one new bench function gated under
`cfg(feature = "sm4-bitsliced-simd")`.

### Q5.15 `sm4-bitsliced-simd` default-on?

Same trade-off as Q4.9 was for the v0.4 single-block.

- **Recommended:** default-off.

**Decision:** **default-off.** Same posture as `sm4-bitsliced`.
Callers opt in. README updates to document the recommended
production-deployment recipe (enable both `sm4-bitsliced-simd` and
runtime-detect-fallback).

### Q5.16 Error-type unification target

- **A: workspace-wide `gmcrypto_core::Error::Failed`.** Module-level
  type aliases keep `sm2::Error` / `pem::Error` / `pkcs8::Error`
  importable.
- **B: per-module `Error::Failed` with no central type.** Less
  centralized; more migration.
- **C: leave them alone.** No breaking change in v0.5.

**Recommended:** A.

**Decision:** **A (workspace-wide `gmcrypto_core::Error::Failed`).**
Module-level re-exports keep the existing import paths working. The
new central type makes the failure-mode invariant visually obvious
in the public API.

### Q5.17 `crypto_bigint::U256` public-surface escape hatch shape

The system-design audit (G2) flagged that `Sm2PrivateKey::new(d:
U256)` forces transitive dep on `crypto-bigint` major. The
`asn1::sig::{encode_sig, decode_sig}` and `asn1::ciphertext::*`
surfaces have the same coupling but are internal-helper enough that
making them `#[doc(hidden)]` is the right v0.5 move.

- **A: rename `Sm2PrivateKey::new(d: U256)` to `from_scalar` behind a
  feature flag + add `from_bytes_be` / `to_bytes_be` always-on.**
  Breaking, but the breaking is bounded.
- **B: add `from_bytes_be` only, keep `new(d: U256)` as the primary
  constructor.** Non-breaking but the coupling stays as the canonical
  recipe.
- **C: hide `new(d: U256)` behind `crypto-bigint-scalar` feature
  without renaming.** Less migration, but the unfeature-gated path
  becomes `from_bytes_be` while the historical `new` becomes
  feature-gated — odd naming.

**Recommended:** A.

**Decision:** **A.** Rename → `from_scalar` (under
`crypto-bigint-scalar` feature flag), add `from_bytes_be` always-on
as the recommended path. The rename eliminates the "which is the
canonical constructor?" ambiguity. The CHANGELOG migration sed
recipe is:
```
s/Sm2PrivateKey::new(d)/Sm2PrivateKey::from_bytes_be(&d.to_be_bytes())/
```
(or, if the caller already has `U256` in scope, enable
`crypto-bigint-scalar` and `s/new/from_scalar/`).

### Q5.18 `std` feature flag — ship file-I/O helper or remove?

- **A: ship a first file-I/O helper.** E.g.,
  `pem::read_to_string_from_path(path: &Path) -> Result<String,
  std::io::Error>`. Then the feature has concrete content.
- **B: remove the feature flag entirely.** No-op since v0.3.
- **C: keep as no-op reservation.** Status quo.

**Recommended:** B.

**Decision:** **B (remove).** A reserved feature flag without
contents is worse than no feature flag — callers wonder what it does
and the documentation has to explain "nothing yet". If a future v0.5.x
file-I/O helper materializes, it gets a new specific feature flag
name like `std-file-io`. The `std` namespace is too generic.

### Q5.19 v0.5 PR series order — C ABI first or SIMD first?

- **A: C ABI first (W1, W2, W3), then SIMD (W4), then W5.** Closer
  to the v0.4 W4 momentum; easier to land while FFI patterns are
  fresh.
- **B: SIMD first (W4), then C ABI (W1, W2, W3), then W5.** Lets
  the SIMD investigation happen first; if it stalls, C ABI work
  isn't blocked.
- **C: parallel.** W4 is independent of W1-W3; both proceed at
  once.

**Recommended:** C.

**Decision:** **C (parallel).** W4 and W1-W3 touch disjoint code
paths. The cabi job's header-drift gate catches FFI mistakes
incrementally; the SIMD work is on a separate branch entirely. W5
lands last regardless. If W4 stalls on SIMD-feature detection or
gate-sequence calibration, W1-W3 still land and v0.5 ships W4 as
v0.6 carry-over.

### Q5.20 Breaking-change boundary — what's allowed to break in v0.5?

`gmcrypto-core` is on the 0.x track; SemVer 0.x conventions permit
breaking changes on every minor bump. But the actual policy is
governed by caller pain.

- **A: Only the three W5 items.** Tight, predictable.
- **B: Anything that surfaced in the system-design audit.** Includes
  the `sign_with_id` / `sign_raw_with_id` naming asymmetry (G3),
  potentially others.
- **C: Anything goes — v0.5 is the cleanup release.** Risk: callers
  delay upgrading.

**Recommended:** A.

**Decision:** **A (only the three W5 items).** Every other breaking
opportunity defers to v1.0 prep. The audit's G3 (`sign_with_id`
naming) is doc-only confusion — fixable with a doc patch in v0.5.x,
not a rename. v0.5's breaking boundary is exactly:
- workspace-wide error type
- `Sm2PrivateKey::new(d: U256)` → `from_scalar` + new `from_bytes_be`
- `std` feature flag removal

No other public-API breaks in v0.5. This is enforceable via the
`cargo public-api diff` step in Section 6.
