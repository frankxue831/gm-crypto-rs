# v0.10 — Scope doc

**Status:** Lands as v0.10 W0. Mirrors the structure of `docs/v0.9-scope.md`
— Q-list with provisional answers, hard contract, posture, deferred items.
**Anchor: streaming AEAD FFI for SM4-GCM** in `gmcrypto-c` (the item v0.9
deferred per Q9.6). Five workstreams: W0 doc → W1 encryptor FFI → W2
decryptor FFI → W3 `c_smoke` + header + example → W4 release prep.

Top-level decisions baked into the Q-list below:

- **Anchor only.** v0.10 ships *only* the streaming AEAD FFI for SM4-GCM.
  Infra/perf ride-alongs (pinned dudect runner, AVX-512 `sbox_x64`) and the
  remaining consumer-driven items (CCM incremental input, RustCrypto `aead`
  trait fit) are all deferred to their own later cycles. Tightly-scoped
  release, matching the v0.7 / v0.8 / v0.9 cadence.
- **The deferral precondition is now met.** v0.9 deferred streaming FFI
  because "C ABI is permanent and the Rust-side streaming semantics are
  unproven" (Q9.6). The Rust incremental API (`Sm4GcmEncryptor` /
  `Sm4GcmDecryptor`) shipped in v0.9.0 with 11 differential KATs, the
  `ct_sm4_gcm_decrypt_buffered` dudect gate, and a post-release consolidation
  pass (PR #60, internal-only). The semantics are now proven; v0.10 cashes
  in the deferral.
- **Independent encryptor / decryptor opaque types** — *not* a unified
  `gcm_ctx_t` with a direction enum. This re-applies the pin from the v0.5
  CBC-streaming FFI (`docs/v0.5-scope.md` Q5.2), which chose independent
  `Sm4CbcEncryptor` / `Sm4CbcDecryptor` opaque types over a unified
  `_cbc_t`. Consistency with the established C-ABI shape.
- **`aead` trait fit stays blocked.** As of 2026-05-21 the `aead` crate is
  still `0.6.0-rc.10` (latest stable `0.5.2`), unchanged on crates.io since
  2026-02-04 — ten release candidates in, no stable. v0.10 does **not**
  target the older-but-stable `0.5.2`: building against a soon-to-be-
  superseded trait API would just be redone when `0.6` lands. Recheck each
  cycle.
- **No streaming CCM FFI.** CCM's CBC-MAC needs the total message length
  up-front in `B0`, which collapses the streaming benefit on unknown-length
  inputs (`docs/v0.9-scope.md` §5). No consumer has surfaced. Deferred.

## 1. Context

v0.9.0 closed AEAD ergonomics on the **Rust** side:

1. **GCM tag-length parameterization** — `GcmTagLen` newtype (valid set
   `{4, 8, 12, 13, 14, 15, 16}`) + `mode_gcm::encrypt_with_tag_len` /
   `decrypt_with_tag_len` (NIST SP 800-38D §5.2.1.2 truncated tags).
2. **Incremental-input buffered SM4-GCM** —
   `sm4::gcm_streaming::{Sm4GcmEncryptor, Sm4GcmDecryptor}`. Encryptor
   output-streaming; decryptor input-incremental / output-buffered with
   commit-on-verify. Differential-KAT-equal to single-shot across arbitrary
   chunking.
3. **Single-shot AEAD C FFI** — 6 symbols (`gmcrypto_sm4_gcm_*` /
   `gmcrypto_sm4_ccm_*`) behind a forwarding `sm4-aead` feature on
   `gmcrypto-c`.

v0.9 explicitly deferred **streaming AEAD FFI** to v0.10 (Q9.6) and dropped
**streaming CCM** (Q9.1 / §5). The deferral was risk-driven, not
value-driven: the C ABI is permanent, and at v0.9 W4 time the Rust streaming
semantics were brand-new.

The gap v0.10 closes: today a C / C++ / Go / Zig / Python caller can only
reach *single-shot* AEAD. The incremental encryptor/decryptor — the headline
of v0.9 — is **Rust-only**. Exposing the crypto to non-Rust callers is the
entire reason `gmcrypto-c` exists, so the streaming path being unreachable is
the obvious missing piece. v0.10 ships it.

## 2. Hard contract

The same byte-identical invariant v0.6–v0.9 hold to, carried across the C
boundary:

- The streaming AEAD FFI, driven with arbitrary chunking, produces the
  **same ciphertext/tag (encrypt) and plaintext (decrypt)** as the Rust
  `Sm4GcmEncryptor` / `Sm4GcmDecryptor` would on the same chunking — which is
  itself differential-KAT-equal to the v0.8 single-shot `mode_gcm` path,
  which is byte-identical to gmssl 3.1.1 `sm4 -gcm`. Transitively, the C
  streaming FFI is byte-identical to gmssl.
- **Commit-on-verify preserved across the C boundary.** The decryptor's
  `_update` emits **no plaintext**; `_finalize_verify` releases the full
  plaintext only on a successful constant-time tag check. On any tag
  mismatch / invalid tag length / length-ceiling overflow it returns
  `GMCRYPTO_ERR` and writes `0` to `*out_actual_len` — no plaintext ever
  materializes on the failure path. No tag-oracle surface.
- **Single failure code.** Every error path returns `GMCRYPTO_ERR`. No
  distinguishing-error codes (padding/tag-oracle defense, per the
  workspace failure-mode invariant).
- **`_finalize` consumes and frees the handle** (`Box::from_raw`), mirroring
  the v0.5 CBC-streaming FFI lifecycle. `_free` is the abort path; calling
  `_free` after `_finalize` is use-after-free (documented, caller's
  responsibility — same contract as every existing opaque handle).
- **Constant-time on key bytes and tag comparison preserved.** The FFI is a
  thin wrapper over the *same* Rust primitives already gated by
  `ct_sm4_gcm_decrypt_buffered` (v0.9 W3). No new timing surface is
  introduced (see Q10.8).
- **cbindgen header is deterministic.** cbindgen does not evaluate
  `#[cfg(feature)]`, so the streaming prototypes appear in `gmcrypto.h`
  whether or not `sm4-aead` is enabled at regen time — identical to v0.9 W4.
  The drift gate (`git diff --exit-code`) stays green.

## 3. Q-list (Q10.1–Q10.11) with provisional answers

### Q10.1 — Scope: anchor only

The streaming AEAD FFI for SM4-GCM is the sole feature of v0.10. The
infra/perf items (pinned dudect runner [`v0.9-Q10.4`], AVX-512 `sbox_x64`
[`v0.9-Q10.5`]) and consumer-driven items (CCM incremental [`v0.9-Q10.2`],
`aead` trait fit [`v0.9-Q10.3`]) are deferred — each warrants its own focused
cycle (the dudect runner is CI-provisioning work orthogonal to crypto;
AVX-512 needs a perf/benchmark cycle and a capable runner; the trait fit is
upstream-blocked; CCM incremental has no consumer).

**Provisional decision: anchor only — streaming SM4-GCM AEAD FFI, full
stop.**

### Q10.2 — Opaque-type shape

Two encryptor/decryptor C types vs. one unified `gmcrypto_sm4_gcm_ctx_t` with
a direction parameter. The v0.5 CBC-streaming FFI explicitly chose
*independent* opaque types (`docs/v0.5-scope.md` Q5.2) over a unified
`_cbc_t` + mode enum, because the unified shape mushes two different
contracts (encrypt emits as it goes; decrypt withholds) into one symbol.
GCM has the *same* asymmetry (stronger, even — the decryptor emits nothing
per `_update`). Re-introducing a unified type for GCM would contradict the
pinned precedent.

**Provisional decision: two independent opaque types,
`gmcrypto_sm4_gcm_encryptor_t` and `gmcrypto_sm4_gcm_decryptor_t`.**

### Q10.3 — Encryptor finalize variants

The Rust `Sm4GcmEncryptor` exposes both `finalize() -> [u8; 16]` and
`finalize_with_tag_len(GcmTagLen) -> Vec<u8>`. The v0.9 single-shot FFI also
ships both a fixed-16 form (`gmcrypto_sm4_gcm_encrypt`) and a truncated form
(`gmcrypto_sm4_gcm_encrypt_with_tag_len`). For consistency the streaming
encryptor mirrors this: a `_finalize` that writes a bare 16-byte tag (the
common case, ergonomic — no output triple needed) and a
`_finalize_with_tag_len` that uses the `(out, out_capacity, out_actual_len)`
triple for truncated tags.

The *decryptor* needs only **one** finalize: `_finalize_verify` already takes
`(tag, tag_len)` as input, so a single symbol covers every valid tag length
(`tag_len` is validated against `GcmTagLen`'s set; invalid → `GMCRYPTO_ERR`).

**Provisional decision: encryptor gets `_finalize` (bare 16-byte tag) +
`_finalize_with_tag_len` (truncated, output triple). Decryptor gets one
`_finalize_verify` taking `(tag, tag_len)`.**

### Q10.4 — Decryptor `_update` output shape

The Rust decryptor buffers ciphertext internally and emits nothing until
`finalize_verify` (commit-on-verify; output-buffered, `O(message)` memory).
The C `_update` mirrors this exactly: it takes `(dec, ct, ct_len)` and has
**no output parameters at all**. Plaintext appears only at
`_finalize_verify`, whose output length equals the total ciphertext fed.
This is the C-level expression of the encryptor/decryptor asymmetry, and it
is *simpler* than the CBC decryptor (which emits held-back blocks per
`_update`) — GCM's decryptor emits strictly nothing mid-stream.

**Provisional decision: `gmcrypto_sm4_gcm_decryptor_update(dec, ct, ct_len)
-> int` with no output params. All plaintext released at
`_finalize_verify`.**

### Q10.5 — Lifecycle: `_finalize` consumes + frees

Mirror the v0.5 CBC-streaming FFI lifecycle (`docs/v0.5-scope.md` Q5.3):

- `_new(...)` → `Box::into_raw`; returns `NULL` on invalid pointer/length
  input (wrapped in `catch_unwind`).
- `_update(...)` → `&mut *handle`; returns `GMCRYPTO_ERR` on null handle or
  bad input pointer.
- `_finalize` / `_finalize_with_tag_len` / `_finalize_verify` →
  `Box::from_raw` (takes ownership, runs the Rust `finalize*(self)`, **frees
  the handle**). Caller must not touch the handle afterward.
- `_free(...)` → abort path; `Box::from_raw` + drop. `_free(NULL)` is a
  no-op. Do **not** call `_free` after a `_finalize*` (use-after-free, UB,
  documented).

**Provisional decision: lifecycle identical to the v0.5 CBC-streaming
handles. `_finalize*` consume+free; `_free` is the abort path.**

### Q10.6 — Nonce + AAD at construction

The Rust constructors take `(key, nonce, aad)`; nonce is variable-length
(12-byte canonical + arbitrary), AAD is supplied whole at construction (it
is the message header, known up-front — there is no incremental-AAD path in
the Rust API). The C `_new` mirrors this:
`_new(key, nonce, nonce_len, aad, aad_len)`. No separate incremental-AAD
symbol — that would be a Rust-API change out of scope for an FFI cycle.

**Provisional decision: `_new(key, nonce, nonce_len, aad, aad_len)`. Nonce
as `(ptr, len)`; AAD whole at construction. No incremental-AAD symbol.**

### Q10.7 — Failure-mode invariant

Unchanged from v0.8 / v0.9. Every error path returns `GMCRYPTO_ERR`;
`*out_actual_len` is set to `0` on failure. Specifically:

- Encryptor `_update` returns `GMCRYPTO_ERR` once the cumulative plaintext
  would exceed the GCM ceiling (`2^36 − 32` bytes); the underlying Rust
  encryptor is poisoned, so every subsequent `_update` also returns
  `GMCRYPTO_ERR`. The caller should `_free` the handle.
- Decryptor `_finalize_verify` returns `GMCRYPTO_ERR` on tag mismatch,
  invalid `tag_len` (not in `GcmTagLen`'s set), or length-ceiling overflow.
  No plaintext written.
- On insufficient output capacity, `_update` / `_finalize_with_tag_len` /
  `_finalize_verify` return `GMCRYPTO_ERR` (same `(out, out_capacity,
  out_actual_len)` convention as the existing CBC-streaming symbols). For
  the encryptor `_update` the exact required capacity is `pt_len` (GCM
  ciphertext is the same length as plaintext — no padding/buffering), so a
  correctly-sized caller never hits this.

**Provisional decision: single `GMCRYPTO_ERR`; `*out_actual_len = 0` on
failure. No distinguishing codes.**

### Q10.8 — No new dudect target

The streaming AEAD FFI is a thin wrapper that calls straight into the v0.9
Rust `Sm4GcmEncryptor` / `Sm4GcmDecryptor`, whose decrypt path is already
gated by `ct_sm4_gcm_decrypt_buffered` (v0.9 W3, `|tau| < 0.20`). The FFI
adds pointer marshalling and a `Box` indirection — not secret-dependent
control flow. The dudect harness lives in `gmcrypto-core` and measures the
Rust primitives directly; wrapping them in `extern "C"` introduces no new
timing surface to gate.

**Provisional decision: no new dudect target in v0.10. The existing
`ct_sm4_gcm_decrypt_buffered` gate covers the constant-time property the FFI
relies on.**

### Q10.9 — Testing + cbindgen

- **`c_smoke` Rust-equivalence tests** (extend `crates/gmcrypto-c/tests/c_smoke.rs`,
  cfg-gated on `sm4-aead`), mirroring the existing CBC-streaming + single-shot
  AEAD smoke patterns:
  - encrypt-via-C / decrypt-via-Rust and encrypt-via-Rust / decrypt-via-C
    (cross-direction equivalence);
  - chunked `_update` (multiple chunk sizes straddling block boundaries)
    equals single-shot output;
  - truncated tag (`_finalize_with_tag_len` + `_finalize_verify` with
    `tag_len ∈ {12, 16}`);
  - tamper → `GMCRYPTO_ERR` (flip a tag byte; flip a ciphertext byte);
  - invalid `tag_len` → `GMCRYPTO_ERR`;
  - `_finalize_verify` releases nothing before verify (decryptor `_update`
    has no output param — structurally enforced).
- **cbindgen header regen** under the `regen-header` feature; CI gates drift
  via `git diff --exit-code crates/gmcrypto-c/include/gmcrypto.h`. Header is
  deterministic w/ or w/o `sm4-aead` (cbindgen ignores `cfg`).
- **Example:** extend `crates/gmcrypto-c/examples/` (or its README) with a
  chunked GCM streaming round-trip in C.
- **MSRV / clippy / deny** under `--features sm4-aead`, per the existing
  `gmcrypto-c` command set in CLAUDE.md.

**Provisional decision: `c_smoke` cross-direction + chunked + truncated +
tamper + invalid-len coverage; deterministic header regen; one C example;
MSRV/clippy/deny under `sm4-aead`.**

### Q10.10 — Workstream ordering

The encryptor and decryptor FFI are independent (separate opaque types,
separate symbols), so they split cleanly into two implementation
workstreams. Tests + header + example consolidate in W3; release prep in W4.

**Provisional decision: W0 (doc) → W1 (encryptor FFI: `_new` / `_update` /
`_finalize` / `_finalize_with_tag_len` / `_free`) → W2 (decryptor FFI:
`_new` / `_update` / `_finalize_verify` / `_free`) → W3 (`c_smoke` + cbindgen
header + C example) → W4 (release prep 0.9.0 → 0.10.0). Five workstreams,
five PRs.**

### Q10.11 — Deferred-item posture (aead trait fit / CCM incremental)

Restated for the record so the deferral is auditable:

- **`aead` trait fit** (`v0.9-Q10.3`): `aead 0.6.0-rc.10`, no upstream
  movement since 2026-02-04, latest stable `0.5.2`. v0.10 does not target
  `0.5.2` (would be redone on `0.6`). Recheck next cycle.
- **CCM incremental input** (`v0.9-Q10.2`): CBC-MAC needs total length in
  `B0`; no consumer. Deferred.

**Provisional decision: both remain deferred; no v0.10 work.**

## 4. Hard-contract checklist (for v0.10 PR review)

- [ ] `gmcrypto-c` ships **9 new FFI symbols** behind the `sm4-aead` feature:
      encryptor `_new` / `_update` / `_finalize` / `_finalize_with_tag_len` /
      `_free` (5); decryptor `_new` / `_update` / `_finalize_verify` / `_free`
      (4). Two new opaque types.
- [ ] Encryptor `_update` emits ciphertext of length exactly `pt_len`;
      `_finalize` writes a bare 16-byte tag; `_finalize_with_tag_len` writes
      `tag_len` bytes via the output triple.
- [ ] Decryptor `_update` takes no output params; `_finalize_verify` releases
      the full plaintext **only** on tag-verify success, else `GMCRYPTO_ERR`
      + `*out_actual_len = 0`.
- [ ] `_finalize*` consume+free the handle; `_free` is the abort path;
      `_free(NULL)` is a no-op. Lifecycle documented per the CBC-streaming
      precedent.
- [ ] Every error path returns `GMCRYPTO_ERR` (single failure code). Encryptor
      poisons on overflow; decryptor `GMCRYPTO_ERR` on tamper / invalid
      `tag_len` / overflow.
- [ ] Constant-time on key + tag preserved (wrapper over the
      `ct_sm4_gcm_decrypt_buffered`-gated Rust path; no new dudect target).
- [ ] `c_smoke` covers cross-direction equivalence, chunked `_update`,
      truncated tag `{12, 16}`, tamper → `ERR`, invalid `tag_len` → `ERR`.
- [ ] cbindgen header regenerated; drift gate (`git diff --exit-code`) green;
      header identical with/without `sm4-aead`.
- [ ] C example: chunked GCM streaming round-trip.
- [ ] `cargo test -p gmcrypto-c --features sm4-aead` passes; `cargo clippy -p
      gmcrypto-c --features sm4-aead --all-targets -- -D warnings` clean.
- [ ] MSRV build of `gmcrypto-c --features sm4-aead`; `cargo deny check
      --exclude-dev` clean under `--features gmcrypto-core/sm4-aead`.
- [ ] Workspace version `0.9.0 → 0.10.0`; `gmcrypto-c` core dep `"0.9" →
      "0.10"`; CHANGELOG / README / CLAUDE.md updated.
