# v0.22 — Scope doc

**API-tightening cycle: decouple `crypto-bigint 0.7` from the stable 1.0 public API.** The v0.21
readiness audit produced a **CONDITIONAL GO** for 1.0 with exactly one load-bearing decision
outstanding — **§3.A: `crypto-bigint 0.7` types in `gmcrypto-core`'s always-on public API**
(`docs/v1.0-readiness.md` §3.A). v0.22 resolves it via §3.A **Option 2 ("tighten the surface, wrap
the residual")**: the **always-on (default-features) 1.0 contract names zero `crypto-bigint`
types**, achieved with the smallest, **byte-output-preserving** change set. v0.22 is a **repository
/ infra-assurance milestone, non-publishing**: the workspace stays `0.16.0`, crates.io skips
`0.22.0` (the v0.14/v0.17/v0.18/v0.19/v0.20/v0.21 precedent). The breaking API-*shape* change ships
with the deliberate `1.0.0` publish (the runbook in `docs/v1.0-readiness.md` §4), now unblocked.

> **Decision provenance.** The v0.21 audit teed up §3.A as a pre-1.0 maintainer decision; the
> maintainer chose **Option 2** (decouple) over Option 1 (accept-and-document the pin). For v0.22,
> a focused **Codex consult (2026-05-31, `codex exec --sandbox read-only`, gpt-5.5)** reviewed the
> plan and **confirmed all four load-bearing decisions** with no blockers (depth = decouple-only;
> `from_scalar` = documented escape hatch; posture = non-publishing; exposure map = complete). Its
> doc-clarity asks are folded into §1/§3 below.

## 1. Context

The published library has been API-stable since **v0.5**. The v0.21 audit froze + CI-guarded the
surface (committed `cargo-public-api` baselines + drift-check, informational `cargo-semver-checks`,
a `cargo doc` gate, a feature matrix) and finalized the `#[doc(hidden)]` posture (the 3 core hidden
items + the whole `gmcrypto-simd` surface). It found **one** thing a 1.0 commitment should not take
on unexamined: the always-on public API **names `crypto-bigint 0.7` types**. Since `crypto-bigint`
is itself pre-1.0 (0.7.x) and will ship breaking releases, a `crypto-bigint` major bump would force
a gmcrypto-core **major** bump — pinning gmcrypto-core 1.x to `crypto-bigint 0.7` for its whole 1.x
lifetime, or forcing a 2.0 to upgrade.

### The exposure surface — three groups

`cargo-public-api` (`docs/api-baseline/gmcrypto-core.txt`) + a source sweep classify every
`crypto-bigint`-naming public item into three groups:

- **Group A — low-level EC arithmetic surface, no intended/stable or C-FFI consumers.**
  - `sm2::curve::{Fn, Fp}` (`pub type` aliases for `crypto_bigint::…::ConstMontyForm<…>`),
    `sm2::curve::{NMod, PMod}` (impl `crypto_bigint`'s `ConstMontyParams`), `sm2::curve::{b, b3}`
    (return `Fp`).
  - `sm2::scalar_mul::{mul_g, mul_var}` (take `&Fn`).
  - `sm2::point::ProjectivePoint::to_affine() -> Option<(Fp, Fp)>` (the **only** crypto-bigint-
    naming method on `ProjectivePoint`).
  - The `sm2::{Fn, Fp, mul_g, mul_var}` re-exports (`sm2/mod.rs`).

  The **C FFI (`gmcrypto-c`) touches none of these** — it uses high-level functions + opaque
  handles only. The only consumers are the **dudect bench, integration tests, and fuzz targets** —
  all *separate crates* that see `gmcrypto-core` as an external consumer (only `pub` items are
  visible to them). → **`#[doc(hidden)]` (kept `pub`)** + a "not public API / not SemVer" contract,
  exactly the v0.21 `gmcrypto-simd` precedent. No reshape; internal callers unchanged.

- **Group B — always-on byte-adjacent public APIs that genuinely need reshaping.**
  - `asn1::sig::{encode_sig, decode_sig}` (+ the `asn1::{encode_sig, decode_sig}` re-exports) —
    `(&U256, &U256) -> Vec<u8>` / `&[u8] -> Option<(U256, U256)>`.
  - `asn1::ciphertext::Sm2Ciphertext::{x, y}` — `pub … : U256` fields.

  The sweep confirmed the `U256` here is **always an immediate `bytes ↔ U256` intermediary**: sign
  produces `U256` → `encode_sig` → `to_be_bytes`; verify `decode_sig` → reconstructs field/scalar
  from bytes; encrypt stores `Fp::retrieve()`; decrypt reads via `Fp::new(&U256::from_be_slice)`.
  The encoders/decoders already operate on byte slices. The **FFI never reads `.x`/`.y`**. →
  **reshape to `[u8;32]`** — byte-output-identical wire format, no `U256` in the public shape.

- **Group C — the high-level key path, which uses `ProjectivePoint` (our own struct, NOT a
  crypto-bigint type).**
  - `Sm2PrivateKey::public_key() -> ProjectivePoint`; `Sm2PublicKey::{from_point, point,
    From<ProjectivePoint>}`; `spki::{decode, encode}(ProjectivePoint)`;
    `sec1::EcPrivateKey.public: Option<ProjectivePoint>`.

  **Key realization:** `ProjectivePoint` only names `crypto-bigint` via `to_affine() -> (Fp, Fp)`.
  Once that one method (Group A) is `#[doc(hidden)]` and `Fp` is hidden, `ProjectivePoint` is
  `crypto-bigint`-free in its **documented** surface. So the **entire high-level key path stays
  public and unchanged.** We do **not** remove `ProjectivePoint` — the §3.A "Option 2 is non-
  trivial" caveat only applies if you delete the point type and replace it with byte-shaped key
  APIs; we keep `ProjectivePoint` as an opaque curve point. **Codex confirmed decouple-only is
  sufficient to call §3.A resolved.**

  > **Audit note (Codex SHOULD-FIX).** A reviewer reading the regenerated baseline will still see
  > `ProjectivePoint` on `spki`/`sec1`/`public_key`/`from_point` (Group C) and the
  > `asn1::{encode_sig, decode_sig}` re-exports. These are **intentional, not missed `crypto-bigint`
  > exposure**: Group C's `ProjectivePoint` is `crypto-bigint`-free once `to_affine` is hidden, and
  > the `asn1` re-exports are reshaped to `[u8;32]` by Group B. The decoupling target is
  > *`crypto-bigint` TYPES in the contract*, not the `ProjectivePoint` type.

### The one opt-in escape hatch stays

`Sm2PrivateKey::from_scalar(U256)` is behind the **`crypto-bigint-scalar`** feature (off by
default). §3.A explicitly says it "can stay (callers opt into naming `U256`)." We keep it as a
**documented opt-in escape hatch**: the **always-on** surface is decoupled (Option 2); the single
opt-in feature **explicitly accepts** the `crypto-bigint 0.7` type coupling (a narrow, documented
Option 1). Enabling `crypto-bigint-scalar` is the caller opting into that contract. This is a
coherent split, not a contradiction (Codex confirmed; the README must state the opt-in semantics).

### Intended outcome

After W1+W2, the **default-features** public-API surface names **zero** `crypto-bigint` types. The
**committed** baseline `docs/api-baseline/gmcrypto-core.txt` is regenerated by `api-stability.yml`
with **`--all-features`** (so it can never go stale on a feature-gated change), so it **retains
exactly the two `from_scalar(crypto_bigint::…::U256)` lines** (the inherent method + the `sm2::`
re-export) — the documented `crypto-bigint-scalar` opt-in residual, and **nothing else**
`crypto-bigint`-typed. The "zero `crypto-bigint`" guarantee is therefore about the **default-
features contract** and is proven in W3 by a **separate ad-hoc default-features `public-api` grep**
(NOT the committed all-features baseline). §3.A flips to **GO**; the eventual `1.0.0` publishes a
dependency-decoupled stable surface.

## 2. Hard contract

- **Non-publishing.** No version bump, no tag, no `cargo publish`. Workspace stays `0.16.0`;
  crates.io skips `0.22.0`. The breaking API-shape change ships with the deliberate `1.0.0` publish
  — it never reaches a published 0.x crate, so no 0.x consumer ever sees the break (the only
  migration is 0.16 → 1.0, a major bump anyway).
- **Byte-output-identical runtime.** Unlike v0.21 (which was byte-identical at the API level too),
  v0.22 **changes the public API *shape* (breaking)**. But the **runtime output** — SM2 signature
  and ciphertext **wire bytes** — stays byte-identical. The KAT + gmssl interop (11/11) gate this;
  W2's commit is blocked until interop passes.
- **Zero `crypto-bigint` in the default-features contract.** The W3 baseline regen is the machine-
  checkable proof; the residual is the opt-in `crypto-bigint-scalar` feature only.
- **Constant-time + single-failure invariants preserved.** W2 introduces **no** `==`/`if`/`bool` on
  secret-derived values — the `[u8;32]` arrays are public wire material; the secret field/scalar
  elements stay in `subtle`/`Fp`/`Fn` form. `decrypt`/`decode_sig` keep their single `None` (no new
  distinguishing failure modes — the failure-mode invariant).
- **`unsafe_code = "forbid"` on `gmcrypto-core` stays.** MSRV 1.85, edition 2024.
- **The C ABI is untouched** (FFI uses none of this surface — verified). `gmcrypto.h` drift-check
  stays green; `gmcrypto-simd.txt` baseline unchanged (already fully `#[doc(hidden)]`).

## 3. Q-list (Q22.1–Q22.8) with answers

### Q22.1 — Anchor (theme)
**§3.A Option 2 API-tightening** — decouple `crypto-bigint` from the always-on 1.0 API, before the
publish. Confirmed primary from the v0.21 readiness report (`docs/v1.0-readiness.md` §3.A).

### Q22.2 — Depth
**Decouple-only (keep `ProjectivePoint` public, opaque), NOT full point-type removal.** Hiding
`to_affine` + the curve/scalar surface already makes `ProjectivePoint` `crypto-bigint`-free;
deleting it would force replacement byte-shaped public-key/point APIs across the whole key path — a
larger redesign §3.A flagged as non-trivial and unnecessary to meet the goal. **Codex: SHOULD-FIX
→ decouple-only is sufficient; do not remove `ProjectivePoint`.**

### Q22.3 — Group A disposition
`#[doc(hidden)]` (kept `pub`) + the contract note **"Not part of the public API / not covered by
SemVer; may change or be removed in any release. Internal low-level SM2 curve arithmetic — Rust
users use the high-level `sm2` API; C users use `gmcrypto-c`."** on `curve::{Fn, Fp, NMod, PMod, b,
b3}`, `scalar_mul::{mul_g, mul_var}`, `ProjectivePoint::to_affine`, and the `sm2/mod.rs` re-exports.
Items stay `pub` because the bench / integration tests / fuzz are separate crates that reach them.
`cargo-public-api` + `cargo-semver-checks` both exclude `#[doc(hidden)]`, so this removes them from
the 1.0 *contract* without a structural change — the v0.21 simd / `take_output` pattern. (Internal
soundness comes from lockstep exact-version sibling pinning, like v0.21 §2/§6 — but Group A is
consumed only by *dev-only* same-repo crates, not by `gmcrypto-c`, so even that is moot here.)

### Q22.4 — Group B disposition
**Reshape to `[u8;32]`.** `encode_sig(r: &[u8;32], s: &[u8;32]) -> Vec<u8>`; `decode_sig(&[u8]) ->
Option<([u8;32], [u8;32])>`; `Sm2Ciphertext::{x, y}: [u8;32]`. The strict-canonical /
zero-rejection logic and the wire format are **unchanged** (byte-output-identical); single `None`
preserved; no new failure modes.

**W2 hazards to hold exactly (Codex SHOULD-FIX flags):**
- **The decoded values are NOT pure byte intermediaries in verify.** `verify_with_id` consumes
  `decode_sig`'s `(r, s)` *numerically*: it checks `r/s != 0`, `r/s < n`, builds `Fn::new(...)`,
  and checks `t != 0`. So `decode_sig`'s `[u8;32]` outputs must be **reconstructed into `U256`
  internally inside `verify`** (`U256::from_be_slice`), and **all of `decode_sig`'s existing rejects
  stay in `decode_sig`** (the strict-canonical INTEGER rules + the single-byte `0x00` / `len > 32`
  zero-and-range rejects). W2 moves the *type at the boundary*, never the *failure behavior*.
- **`Sm2Ciphertext { x, y }` has public fields and stays directly constructible**, so `[u8;32]`
  fields are **not inherently canonical**. The `< p` reduction (`Fp::new`) and the **on-curve check
  on C1** (`point_on_curve` in `decrypt`) and the DER/raw decoders' field-element validation must be
  **preserved at exactly their current boundaries** — the reshape does not relocate or weaken any of
  them. (Today `x:U256` is already an unreduced 256-bit int reduced by `Fp::new`; `[u8;32]` →
  `U256::from_be_slice` → `Fp::new` is the same reduction, same guard.)

### Q22.5 — `from_scalar(U256)` / `crypto-bigint-scalar`
**Keep as documented opt-in escape hatch** (narrow Option 1). It is **not** part of the always-on
1.0 contract (off by default). The README + this doc **state that enabling `crypto-bigint-scalar`
opts the caller into the `crypto-bigint 0.7` type contract** (a `crypto-bigint` major bump would
break *that feature's* API). **Codex: NIT → keep as escape hatch.** (Fully-clean-contract fallback,
not taken: also `#[doc(hidden)]` `from_scalar` — the bench still reaches it — a one-line change if
the maintainer later prefers it.)

### Q22.6 — Publish posture
**Non-publishing** (stays `0.16.0`; ships with 1.0). **Codex: NIT → acceptable**; a breaking
`0.17.0` only buys downstream rehearsal time, not needed for 1.0 cleanliness given the migration
notes + KAT + gmssl interop cover the break. Matches the v0.14–v0.21 defer-all-publishing-to-1.0
pattern.

### Q22.7 — Verification
Byte-output-identical (full KAT + gmssl interop **11/11** + the v0.3 PKCS#8/SPKI KAT fixtures +
`gmcrypto-c` round-trips through the new shapes); the **ad-hoc default-features** `cargo-public-api`
run names **zero** `crypto_bigint`/`U256` (grep-verified); the committed **all-features** baseline
retains only the two gated `from_scalar(U256)` residual lines; per-feature clippy `-D warnings`;
`cargo +nightly fuzz build`; MSRV 1.85 (`--all-features`); `--no-default-features` + wasm32 builds;
`cargo doc -D warnings` clean.

### Q22.8 — Deferred → post-1.0
(Unchanged from v0.21 §5.) AVX-512 16-way `sbox_x64`; SM4-CCM incremental/buffered input; `aead
0.6` trait fit (still `0.6.0-rc.10` — blocked); the class-split-aware "noise-twin" dudect
reference; round-trip/differential **parser** fuzzing; the `dudect-nightly` leg-cancellation
CI-health fix. The actual **`1.0.0` publish** is the deliberate next step *after* this tightening
merges.
