# v0.11 — Scope doc

**Status:** Lands as v0.11 W0. Mirrors the structure of `docs/v0.10-scope.md`
— Q-list with provisional answers, hard contract, posture, deferred items.
**Anchor: RustCrypto trait-fit modernization** — migrate the opt-in
`digest-traits` / `cipher-traits` impls from `digest 0.10` / `cipher 0.4` to
`digest 0.11` / `cipher 0.5`, in-place, both deps together. Cashes in the
candidate formerly listed as `v0.10-Q11.7` in `docs/v0.10-scope.md` §6 (which
read "blocked on upstream" — now unblocked: both crates published stable at
MSRV 1.85). Four workstreams: W1 digest → W2 cipher → W3 supply-chain +
cross-feature verification → W4 release prep.

Top-level decisions baked into the Q-list below:

- **Anchor only.** v0.11 ships *only* the `digest 0.11` / `cipher 0.5`
  migration. The RustCrypto `aead` trait fit stays deferred (still
  `0.6.0-rc.10`, no upstream movement since 2026-02-04). Infra/perf items
  (pinned dudect runner, AVX-512 `sbox_x64`) and other consumer-driven items
  (CCM incremental input, SM4-XTS) ride their own later cycles. Tightly-scoped
  release, matching the v0.7 → v0.10 cadence.
- **The upstream block cleared.** `v0.10-Q11.7` was deferred "blocked on
  upstream." As of 2026-05-23: `digest 0.11.3` and `cipher 0.5.2` are both
  published stable, and the whole new dependency line declares
  `rust-version = 1.85` (`digest 0.11.3`, `cipher 0.5.2`, `crypto-common
  0.2.2`, `inout 0.2.2`, `block-buffer 0.12.0`) — matching the repo's MSRV
  exactly. The deferral precondition is met; v0.11 cashes it in.
- **In-place, both deps together — not dual-version, not split.** Drop
  `digest 0.10` / `cipher 0.4` entirely. The two new majors share
  `crypto-common 0.2` (the `generic-array → hybrid-array` transition), so they
  are naturally coupled; migrating them in one release is cleaner than
  straddling two. RustCrypto majors do not co-exist (trait incompatibility),
  so a dual-version compat shim is a non-starter (YAGNI).
- **Zero default-build impact.** The trait fit is opt-in behind the
  `digest-traits` / `cipher-traits` features (`docs/v0.4-scope.md` Q4.3). The
  `#![no_std]` default-features build, the in-crate `traits.rs`, and every
  cryptographic primitive use plain `[u8; N]` / slices and do not touch
  `generic-array` or `hybrid-array`. The migration is confined to the opt-in
  modules.
- **Byte-for-byte output invariant.** The migration must not change a single
  byte of cryptographic output — SM3 digests, HMAC-SM3 tags, SM4 block
  output all identical before/after. The existing KAT + gmssl-interop suites
  are the regression oracle. This is the spine of the release.

> **Q-numbering note:** this doc's `Q11.x` are the *design decisions* for
> v0.11. They **supersede** the provisional `Q11.x` *candidate* list in
> `docs/v0.10-scope.md` §6 (a different, forward-looking list). Where this
> doc references that older candidate list, it disambiguates with the
> `v0.10-Q11.x` prefix — the same disambiguation convention v0.10 used for
> `v0.9-Q10.x`. v0.11 anchors on the former candidate `v0.10-Q11.7`.

## 1. Context

The opt-in RustCrypto trait fit landed in **v0.4 W2** (`docs/v0.4-scope.md`
Q4.3, Q4.4), pinned at the then-current stable lines:

- `digest = "0.10"` — `Sm3` impls `Digest` (`OutputSizeUser` / `Update` /
  `FixedOutput` / `FixedOutputReset` / `HashMarker`); `HmacSm3` impls `Mac`
  (`MacMarker` / `KeyInit` / `OutputSizeUser` / `Update` / `FixedOutput`),
  in `sm3.rs` / `hmac.rs`.
- `cipher = "0.4"` — `Sm4Cipher` impls `BlockEncrypt` / `BlockDecrypt` via the
  `BlockBackend` / `BlockClosure` closure machinery (`Sm4Backend` +
  `proc_block`), in `sm4/cipher.rs`.

Q4.4 flagged the major-version bump as a future cycle: *"digest 0.10.x and
cipher 0.4.x are the current stable lines; future major-bumps are a v0.5
candidate."* It was carried as a deferred item every cycle since (most
recently `v0.10-Q11.7`), each time "blocked on upstream" — the `digest 0.11`
/ `cipher 0.5` / `crypto-common 0.2` generation was in pre-release.

That block is now cleared. The new line is published stable and pins MSRV
1.85. The cost of staying on the old line is concrete ecosystem friction:
downstream users on the current RustCrypto generation cannot compose
`gmcrypto`'s trait impls with their own `digest 0.11` / `cipher 0.5`
dependencies (trait-version mismatch). v0.11 closes that gap and, as a
side benefit, puts the crate on the `crypto-common 0.2` line that the
eventual `aead 0.6` trait fit (a v0.12+ candidate) will require — turning
that future work from a double migration into a small delta.

## 2. Hard contract

- **Byte-identical cryptographic output.** Every SM3 / HMAC-SM3 / SM4 KAT in
  the suite produces identical bytes before and after the migration. The
  trait migration changes *how* the impls are spelled (associated array
  types, backend trait names), never *what* they compute. A KAT shift is a
  migration bug, full stop.
- **Default-features build unchanged.** `cargo build -p gmcrypto-core
  --no-default-features` and the wasm32 target build are byte-for-byte
  unaffected — the migration touches only code behind `digest-traits` /
  `cipher-traits`. No new always-on dependency; `generic-array` /
  `hybrid-array` never enter the default dep graph.
- **Feature-flag names stable.** The public feature names `digest-traits` and
  `cipher-traits` do not change. A downstream caller's `features = [...]`
  list does not churn; only the *version* of the trait they get changes.
- **MSRV stays 1.85.** Verified against the published crates' declared
  `rust-version` (see Q11.6). No MSRV bump.
- **Failure-mode invariant untouched.** No fallible API gains or loses a
  failure mode. `verify_with_id` still returns `bool`; DER decode still
  returns `Option`; the single `Error::Failed` variant is unchanged. The
  RustCrypto traits' own error types (`InvalidLength` on `KeyInit`) are
  unchanged in shape across the bump.
- **Constant-time discipline untouched.** No secret-dependent control flow is
  added or moved. The migration is a trait-spelling change over the same
  primitives; the dudect harness needs no new target (see Q11.8).
- **Breaking change for trait-fit consumers, documented.** A downstream crate
  that enabled `digest-traits` / `cipher-traits` and used `gmcrypto`'s impls
  against `digest 0.10` / `cipher 0.4` MUST bump its own `digest` / `cipher`
  deps to the 0.11 / 0.5 line to keep compiling. This is an expected,
  SemVer-permitted breaking change for a `0.x` minor bump; the CHANGELOG
  flags it prominently. Default-features users are unaffected.

## 3. Q-list (Q11.1–Q11.11) with provisional answers

### Q11.1 — Scope: anchor only

The `digest 0.11` / `cipher 0.5` migration is the sole feature of v0.11. The
`aead` trait fit (`v0.10-Q11.1`) stays deferred — `aead` is still
`0.6.0-rc.10` (latest stable `0.5.2`), unchanged since 2026-02-04; a `0.5.2`
fit would be redone on `0.6`. Infra/perf items (pinned dudect runner
[`v0.10-Q11.3`], AVX-512 `sbox_x64` [`v0.10-Q11.4`]) and other
consumer-driven items (CCM incremental [`v0.10-Q11.2`], SM4-XTS
[`v0.10-Q11.5`]) each warrant their own cycle.

**Provisional decision: anchor only — `digest 0.11` + `cipher 0.5`
migration, full stop.**

### Q11.2 — In-place vs. dual-version

Replace `digest 0.10` / `cipher 0.4` outright, or keep both old and new
behind separate feature flags? RustCrypto trait majors are mutually
incompatible (a type cannot impl both `cipher 0.4::BlockEncrypt` and
`cipher 0.5::BlockCipherEncrypt` in a way callers can disambiguate without
pain), and dual-support would double the test/clippy/deny surface for no
real consumer benefit. The opt-in feature is precisely the seam that makes a
clean replacement safe.

**Provisional decision: in-place replacement. Drop `digest 0.10` /
`cipher 0.4` entirely; no dual-version compat shim.**

### Q11.3 — Both deps together vs. split across releases

`digest 0.11` and `cipher 0.5` both depend on `crypto-common 0.2` (the
`hybrid-array` transition). Splitting them across v0.11 / v0.12 would leave
`main` transiently carrying *both* `crypto-common 0.1` (from the un-migrated
crate) and `crypto-common 0.2` (from the migrated one) — a `cargo deny`
multiple-versions smell and an awkward half-migrated state. Doing both in one
release keeps the dependency tree single-versioned and matches the
one-theme-per-release cadence.

**Provisional decision: both deps in one v0.11. No *tagged* release ships a
split-`crypto-common` tree; any single-dep intermediate state lives only on a
feature branch (see Q11.10).**

### Q11.4 — `digest 0.11` API reshape (hybrid-array + `Mac`/`KeyInit` split)

Two distinct digest-side API changes (both confirmed by the 2026-05-23 codex
review of this doc):

**(a) `generic-array → hybrid-array`.** `digest 0.11` / `cipher 0.5` /
`crypto-common 0.2` replace `generic-array` (`GenericArray<u8, N>`) with
`hybrid-array` (`Array<u8, N>`, still `typenum`-dimensioned —
`crypto-common 0.2` re-exports `hybrid_array as array`). The trait associated
types (`Output<Self>`, `Key<Self>`, `Block<Self>`) become `Array` aliases. The
impls that construct or return these — `FixedOutput::finalize_into`,
`Update::update`, the cipher backend's block processing — adapt to the new
array type's constructors (`Array::from` / indexing / `as_slice`).

**(b) `Mac` no longer requires `KeyInit`.** In `digest 0.10` the `Mac` trait
carried `KeyInit`-derived convenience constructors (`new` / `new_from_slice`).
In `digest 0.11`, `Mac` wraps only `Update + FixedOutput + MacMarker`;
construction moves to a **separate `KeyInit` impl** —
`HmacSm3` keeps `impl KeyInit`, and callers construct via
`digest::KeyInit::new_from_slice(key)` (variable-length key, per the existing
`hmac.rs` over-length-key contract) then drive update/finalize/verify through
`Mac`. The existing `rustcrypto_traits.rs` HMAC tests that construct through
`Mac::new_from_slice` must move to `KeyInit::new_from_slice`, or they will not
compile. This is a compile-surface change, not just a rename.

Both are confined to the opt-in modules. `traits.rs` (the in-crate
`Hash` / `Mac` / `BlockCipher` traits) and every primitive operate on plain
`[u8; N]` / `&[u8]` and are untouched. The conversion boundary is purely at
the RustCrypto-trait impl surface.

**Provisional decision: adopt `hybrid-array` `Array` at the trait-impl
boundary; keep `HmacSm3: KeyInit` and migrate construction call sites to
`KeyInit::new_from_slice`; in-crate primitives stay on plain arrays. Exact
constructor calls confirmed against `digest 0.11.3` / `crypto-common 0.2.2`
during W1.**

### Q11.5 — `cipher 0.5` backend rewrite

`cipher 0.5` reorganized the block-cipher trait family. The renames /
reshape (exact names confirmed by the 2026-05-23 codex review):

- `BlockEncrypt` → `BlockCipherEncrypt`, `BlockDecrypt` →
  `BlockCipherDecrypt`.
- The closure/backend machinery (`BlockBackend` / `BlockClosure` /
  `encrypt_with_backend` / `decrypt_with_backend`) is restructured into the
  `cipher 0.5` backend trait family: `BlockCipherEncBackend` /
  `BlockCipherDecBackend` and `BlockCipherEncClosure` /
  `BlockCipherDecClosure`. (`cipher 0.5` also has a separate
  `BlockMode{Enc,Dec}*` family for mode/mutable use — **not** what we want;
  `Sm4Cipher` is a raw block cipher and uses the `BlockCipher*` family.)
- The standalone `BlockCipher` marker is folded into `BlockSizeUser` +
  the enc/dec traits.
- `ParBlocksSizeUser` is retained for the multi-block backend (see Q11.8 —
  this is the path single-block KATs do *not* exercise).

Crucially, the rewrite is *mechanical at the crypto level*: `Sm4Backend`'s
`proc_block` still delegates to the existing `Sm4Cipher::encrypt_block` /
`encrypt_blocks` batch API (v0.7 W1), which rides the v0.6 SIMD fanout under
`sm4-bitsliced-simd`. No cryptographic code moves; only the trait wrapper
around it is re-spelled. The SM4 KATs catch any behavioral drift immediately.

**Provisional decision: rewrite `sm4/cipher.rs`'s trait-fit module to the
`cipher 0.5` backend traits; `Sm4Backend` continues to re-wrap the unchanged
`encrypt_blocks` batch API. Exact trait/method names pinned during W2.**

### Q11.6 — MSRV stays 1.85

Verified 2026-05-23 against the published crates' declared `rust-version`:

| crate | version | `rust-version` |
|---|---|---|
| `digest` | 0.11.3 | 1.85 |
| `cipher` | 0.5.2 | 1.85 |
| `crypto-common` | 0.2.2 | 1.85 |
| `inout` | 0.2.2 | 1.85 |
| `block-buffer` | 0.12.0 | 1.85 |

The whole new line pins MSRV 1.85 — the repo's exact MSRV. No bump, no
conflict with the hard constraint.

**Provisional decision: MSRV remains 1.85. The `cargo +1.85 build` gate
(with the trait features) is part of the W3 verification matrix.**

### Q11.7 — Feature-flag names unchanged

The public features stay `digest-traits` and `cipher-traits`. Renaming them
(e.g. to encode the version) would churn every downstream caller's
`Cargo.toml` for no benefit — the feature *names* a capability ("RustCrypto
digest trait fit"), not a version. The version is an implementation detail
recorded in `Cargo.toml` + CHANGELOG.

**Provisional decision: `digest-traits` / `cipher-traits` names unchanged.
Only the underlying pinned versions move (`0.10 → 0.11`, `0.4 → 0.5`).**

### Q11.8 — Testing + the output-unchanged oracle, no new dudect target

- **Existing KATs are the regression oracle.** SM3, HMAC-SM3, SM4-ECB/CBC/CTR
  KATs and the gmssl-interop suite must produce identical bytes post-migration.
  No KAT values change.
- **`tests/rustcrypto_traits.rs`** (required-features-gated on
  `digest-traits` + `cipher-traits`) is updated for the new trait names /
  associated types and the UFCS call sites (the
  `<Sm4Cipher as CipherBlockEncrypt>::encrypt_block` helper renames to
  `BlockCipherEncrypt`). Coverage parity with the v0.4 set (9 trait
  integration tests) is preserved, plus two **new** test surfaces the codex
  review flagged as gaps single-block KATs do not cover:
  - **cipher 0.5 multi-block backend** (Q11.5 `ParBlocksSize` path): drive
    `BlockCipherEncrypt` / `BlockCipherDecrypt` over multi-block buffers
    (back-to-back blocks + tail lengths straddling `ParBlocksSize`) and assert
    byte-equality with the inherent `Sm4Cipher::encrypt_blocks` /
    `decrypt_blocks` batch API. A `ParBlocks`/backend wiring mistake is
    invisible to single-block tests but caught here.
  - **HMAC `KeyInit` construction** (Q11.4(b)): construct `HmacSm3` via
    `digest::KeyInit::new_from_slice` across empty / short / block-length /
    over-block-length keys (exercising the `hmac.rs` over-length-key
    contract), then `Mac` update/finalize/verify — confirming the
    `Mac`/`KeyInit` split compiles and round-trips.
- **No new dudect target.** The migration adds no secret-dependent control
  flow; it re-spells trait wrappers over primitives the harness already gates
  (`ct_sm4_*`, `ct_hmac_sm3`). The thirteen/fifteen/eighteen-target matrix is
  unchanged.

**Provisional decision: KATs + interop as the output-unchanged oracle;
`rustcrypto_traits.rs` updated for new names/UFCS + a cipher-0.5 multi-block
backend test + an HMAC `KeyInit` construction test; no new dudect target.**

### Q11.9 — Supply chain (`deny.toml`)

`digest` / `cipher` / `crypto-common` / `inout` are already on the `deny.toml`
allowlist (added in v0.4 W2 for the trait fit). The migration bumps their
allowed versions and may surface `hybrid-array` + `typenum` (and
`block-buffer 0.12`) as newly-reachable transitive crates under the trait
features — these get added to the allowlist with a comment pointing back to
this Q. The exact gate (after `cargo generate-lockfile`):

```bash
cargo deny --features gmcrypto-core/digest-traits,gmcrypto-core/cipher-traits,\
gmcrypto-core/sm4-bitsliced,gmcrypto-core/sm4-bitsliced-simd,\
gmcrypto-core/sm4-aead,gmcrypto-core/crypto-bigint-scalar check --exclude-dev
```

must return "advisories ok, bans ok, licenses ok, sources ok". Plus two
explicit dedup assertions the codex review called for (the real risk is a
*stale old-major* surviving in the all-features graph, not crypto behavior):

- `cargo tree -d --all-features` shows **no duplicate** `crypto-common`,
  `inout`, or `block-buffer` majors (no `crypto-common 0.1` lingering from a
  half-migrated path — Q11.3).
- `cargo tree -i crypto-common --all-features` shows only the `0.2` line
  reaching `digest` / `cipher`; `cargo tree -i generic-array --all-features`
  shows **no** `generic-array` pulled in via the migrated trait path.

**Provisional decision: extend the `deny.toml` allowlist for the new
transitive crates; the spelled-out `cargo deny` clean under the trait
features; `cargo tree -d` / `-i` confirm a single `crypto-common 0.2` line and
no lingering `generic-array`; no transient dual-`crypto-common` in any tagged
release.**

### Q11.10 — Workstream ordering

The digest and cipher migrations touch disjoint files (`sm3.rs` / `hmac.rs`
vs. `sm4/cipher.rs`) and gate on disjoint features, so they split cleanly.
Because both pull `crypto-common 0.2`, the cleanest integration is to land
them as a stacked / combined change so `main` never carries a half-migrated
split-`crypto-common` tree (Q11.3) — the W1-only state is a transient on the
feature branch, never tagged.

**Provisional decision: W0 (this doc) → W1 (`digest 0.11`: `sm3.rs` +
`hmac.rs` impls + digest-side `rustcrypto_traits.rs`) → W2 (`cipher 0.5`:
`sm4/cipher.rs` backend rewrite + cipher-side tests + UFCS renames) → W3
(supply-chain + verification: `deny.toml`, the `cargo tree -d` / `-i` dedup
checks (Q11.9), the full cross-feature test/clippy/MSRV/wasm matrix, and a
`cargo doc -p gmcrypto-core --all-features --no-deps` docs.rs rehearsal — the
manifest sets `all-features = true`, so the migration must prove the rendered
all-feature docs still build) → W4 (release prep `0.10.0 → 0.11.0`). Four
implementation workstreams; W1 + W2 stacked or combined to avoid a tagged
split-`crypto-common` state.**

### Q11.11 — Deferred-item posture

Restated for the record so the deferral is auditable:

- **`aead` trait fit** (`v0.10-Q11.1`): `aead 0.6.0-rc.10`, no upstream
  movement since 2026-02-04, latest stable `0.5.2`. v0.11 does not target
  `0.5.2` (would be redone on `0.6`). v0.11 *does* put the crate on the
  `crypto-common 0.2` line `aead 0.6` will want, shrinking the future fit.
  Recheck next cycle.
- **Pinned / noise-isolated dudect runner** (`v0.10-Q11.3`): CI-provisioning;
  own infra cycle. Strong v0.12 candidate.
- **AVX-512 `sbox_x64`** (`v0.10-Q11.4`), **SM4-XTS** (`v0.10-Q11.5`),
  **CCM incremental** (`v0.10-Q11.2`), **Argon2-with-SM3** (`v0.10-Q11.6`),
  **`wasm-bindgen-test` KAT runner** (`v0.10-Q11.8`): unchanged deferral.

**Provisional decision: all remain deferred; no v0.11 work.**
