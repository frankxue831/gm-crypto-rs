# v1.7 KAT sourcing — TLCP record protection vectors

Status: v1.7 W0 sourcing-decision doc (the v0.8 CCM / v0.12 XTS
"verify-the-oracle-before-scoping" precedent). Records WHERE the
framed-record KAT vectors come from and the recipe to regenerate them; the
harnesses are stood up + validated during W3 (implementation). Pairs with
`docs/v1.7-scope.md` Q7.7.

## 0. Posture

TLCP (GB/T 38636-2020) has **no freely-redistributable English text**, so
the record-layer wire facts (MAC field widths `seq(8)‖type(1)‖version(2,
=0x0101)‖length(2)`, MAC-then-encrypt order, TLS padding, explicit-IV
placement, GCM salt‖nonce/AAD) are sourced from **TLS-1.1/1.2 inheritance**
(RFC 4346 §6.2.3, RFC 5246 §6.2.3.3, RFC 5288 §3) cross-checked against a
**real TLCP implementation** and **must be transcript-confirmed** before the
KATs freeze. This is the same posture as v0.8 (SM4-CCM sourced from OpenSSL
EVP because gmssl lacked `-ccm`).

**Why OpenSSL alone is insufficient:** OpenSSL EVP `SM4-CBC` / `SM4-GCM`
validate only the **cipher primitive** — they know nothing about the
`seq/type/version/length`-bound MAC + AAD framing, TLS padding, explicit-IV
placement, or the GCM `salt‖nonce` derivation. A framed-record KAT needs a
source that implements the **record layer**.

## 1. The three sources (three-way agreement, gotlcp primary)

| Source | Role | Covers |
|---|---|---|
| **gotlcp** (Trisia/gotlcp, Go) | **primary** — deterministic FRAMED vectors | the full record layer for all four SM2 suites (CBC + GCM): MAC framing, TLS padding, explicit IV, GCM salt‖nonce + AAD |
| **OpenSSL 3.x EVP** SM4-CBC / SM4-GCM | cipher-**primitive** cross-check | the raw CBC chaining + the GCM `(ct, tag)` under a fixed `(key, nonce, aad)` |
| **GmSSL 3.1.1** `tlcp_client`/`tlcp_server` | live CBC **interop** only | a real `ECC_SM4_CBC_SM3` handshake end-to-end (sanity, not a vector source) |

### D-5 probe result (2026-06-14)

- `gmssl tlcp_client` / `tlcp_server` usage exposes only
  `-host/-port/-cert/-key/-cacert` — **no cipher-suite selector and no
  key-log flag**. So the GmSSL CLI **cannot select the GCM suites**
  (`0xE051/0xE053`) and **cannot export record keys**. It is a live CBC
  interop check, not a framed-vector source. D-5 resolved.
- Go is **brew-installable** (`go` 1.26.4 bottled; not currently present) →
  gotlcp is feasible as the primary oracle. Go stays **dev-tooling only** —
  never a build/runtime/published dependency (the `fuzz/` separate-workspace
  precedent).
- OpenSSL 3.6.2 exposes both `SM4-CBC` and `SM4-GCM` EVP algorithms locally.

## 2. gotlcp — deterministic framed vectors (primary)

One-time dev setup (W3): `brew install go` (pin the formula version + record
it here at stand-up). Fetch a pinned gotlcp + `emmansun/gmsm` (the SM crypto
backend; irrelevant to vector *values*, which are spec-deterministic).

Recipe (validated in W3): a small Go harness drives the gotlcp **record
layer** directly with fixed inputs — fixed `key_block`-derived per-direction
keys, a fixed `seq`, a fixed explicit CBC IV / GCM explicit nonce — and dumps
the framed record bytes for both `ECDHE/ECC_SM4_CBC_SM3` and
`ECDHE/ECC_SM4_GCM_SM3`. Deterministic by construction (no live RNG). Each
emitted vector pins: inputs (keys, seq, type, version, length, plaintext, IV/
nonce) → output (full record body). Provenance (gotlcp commit + Go version)
checked in alongside the vectors.

## 3. OpenSSL EVP — cipher-primitive cross-check

- **SM4-CBC raw blocks** (the chaining under `Sm4Cipher::{encrypt,decrypt}_
  blocks`, no padding):
  `openssl enc -sm4-cbc -nopad -K <key1_hex> -iv <iv_hex> -in pt -out ct`.
- **SM4-GCM** `(ct, tag)` under a fixed `(key, nonce, aad)`: via the EVP API
  in a small C oracle (reuse the v0.8 `tests/data/sm4_ccm_oracle.c`
  template, swapping the cipher to `EVP_sm4_gcm()` and setting the 12-byte
  nonce + AAD). Confirms the v1.7 GCM record's inner `(ct, tag)` independent
  of the gotlcp framing.

These two cross-check the **cipher half** of each framed gotlcp vector — if
gotlcp and OpenSSL agree on the primitive and gotlcp owns the framing, the
framed vector is trustworthy (the v0.8 three-way-agreement logic).

## 4. Boundary-vector matrix (Codex)

Beyond the nominal CBC + GCM round-trip vectors, the KAT set pins:

- **TLS padding shapes** — minimal (1 pad byte) through a full 256-byte pad
  block; the all-equal-padlen valid form **and** each invalid form (wrong
  pad byte, pad length > fragment) → single `None`.
- **SM3 compression-boundary lengths** — plaintext lengths whose
  `13 + len` transcript lands at 55/56/57 mod 64 (the `buffer_len > 56`
  extra-block neighborhood), to exercise the equalization top-up math.
- **Tamper sweep** — per-byte tamper of explicit IV / ciphertext / MAC /
  tag → single `None`, no plaintext on failure.
- **Sequence boundary** — `seq = u64::MAX` still produces a well-formed
  record and round-trips (the stateless layer cannot reject reuse/wrap; the
  hard-reject is owed by a future stateful wrapper, reconciled W2).
- **Short** (< one block, padding-only) and **max-size** (`2^14`) records.

## 5. Provenance + format (v1.6 A5 posture)

Short vectors are **inlined in the test files** (`tests/tlcp_record_kat.rs`),
not stored under `tests/data/`; only the oracle harness sources
(`sm4_gcm_oracle.c`, the gotlcp Go harness) live under `tests/data/`. Each
committed vector carries a comment naming its source (gotlcp commit + Go
version, or the OpenSSL command). The gotlcp + Go versions are pinned and
recorded here at W3 stand-up so a regen is reproducible.
