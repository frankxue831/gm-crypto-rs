## 1. Context & motivation

`gm-crypto-rs` is a published **v1.0.1** pure-Rust SM2/SM3/SM4 SDK (no_std core +
C FFI + SIMD backend). The "what's next" decision was made by triangulating a Codex
consult against a market scan:

- **The SM2/SM3/SM4 primitive space is saturated** (RustCrypto/sm2, libsm, gmsm,
  smcrypto, gm-rs) and **SM9/ZUC are already served** by existing Rust crates вЂ” so the
  opening is *not* another primitive.
- **The real gap is the trust dimension** (no existing GM crate is audited / has
  CT-verified assembly) and **the protocol/cert layer** (TLCP has *zero* pure-Rust
  implementations; everyone shells out to OpenSSL/Tongsuo).
- **SM2 key exchange (GM/T 0003.3)** is the highest *fit-adjusted* next brick: Codex
  rated it "excellent fit, medium effort." It **completes the SM2 family** (we ship
  GM/T 0003.2 sign + 0003.4/.5 encrypt; 0003.3 key exchange is the missing third),
  **reuses nearly all existing assets**, stays pure-core/no_std, and is the
  **prerequisite for an eventual TLCP capability** (TLCP `ECDHE_SM4_*` suites need it).
  X.509-with-SM2 is the natural *follow-on* (v1.2 candidate), deferred here because its
  path-validation scope creep doesn't fit a "small, auditable" cycle.

### Locked decisions (from brainstorming)

| # | Decision | Choice |
|---|---|---|
| 1 | Direction | **SM2 key exchange (GM/T 0003.3)** for v1.1 |
| 2 | Key confirmation | **Included** вЂ” full authenticated AKE (`S_A`/`S_B` SM3 tags) |
| 3 | API shape | **Role state-machines** (`Sm2KxInitiator` / `Sm2KxResponder`) |
| 4 | Packaging | **Opt-in feature `sm2-key-exchange`** (default-off, pure-core) |

## 2. Standard & protocol

Implement **GM/T 0003.3-2012 в‰Ў GB/T 32918.3-2016** SM2 key agreement, **with key
confirmation**. SM2 cofactor `h = 1`. Let `n` be the curve order, `G` the generator.

**Associated value (avf):** with `w = вЊ€вЊ€logв‚‚ nвЊ‰ / 2вЊ‰ в€’ 1 = 127` for SM2,
`xМ„ = avf(x) = 2В№ВІвЃ· + (x mod 2В№ВІвЃ·)` (i.e. force bit 127 set, keep low 127 bits).

**Parties:** A = initiator, B = responder. Each holds a static key pair
(`d_A,P_A` / `d_B,P_B`), the peer's static public key, both identity strings, and the
party hashes `Z_A = compute_z(P_A, id_A)`, `Z_B = compute_z(P_B, id_B)`. `klen` = desired
key length in bytes.

**Flow (3 messages):**

```
A:  r_A в€€ [1,nв€’1];  R_A = [r_A]G                         в”Ђв”ЂR_Aв”Ђв”Ђв–¶
B:  r_B в€€ [1,nв€’1];  R_B = [r_B]G
    xМ„в‚‚ = avf(x_{R_B});  t_B = (d_B + xМ„в‚‚В·r_B) mod n
    xМ„в‚Ѓ = avf(x_{R_A});  assert R_A on curve
    V = [hВ·t_B](P_A + [xМ„в‚Ѓ]R_A);  assert V в‰  O
    K_B = KDF(x_V вЂ– y_V вЂ– Z_A вЂ– Z_B, klen)
    S_B = Hash(0x02 вЂ– y_V вЂ– Hash(x_V вЂ– Z_A вЂ– Z_B вЂ– x_{R_A} вЂ– y_{R_A} вЂ– x_{R_B} вЂ– y_{R_B}))
    (also precompute S_2 with prefix 0x03 for verifying A's S_A)
                                                          в—Ђв”ЂR_B,S_Bв”Ђв”Ђ
A:  xМ„в‚Ѓ = avf(x_{R_A});  t_A = (d_A + xМ„в‚ЃВ·r_A) mod n
    xМ„в‚‚ = avf(x_{R_B});  assert R_B on curve
    U = [hВ·t_A](P_B + [xМ„в‚‚]R_B);  assert U в‰  O
    K_A = KDF(x_U вЂ– y_U вЂ– Z_A вЂ– Z_B, klen)            # K_A == K_B since U == V
    verify  S_B == Hash(0x02 вЂ– y_U вЂ– inner)            # inner = Hash(x_U вЂ– Z_A вЂ– Z_B вЂ– x1 вЂ– y1 вЂ– x2 вЂ– y2)
```

## 4. API design (role state-machines)

Mirrors the repo's streaming-cipher idiom (`Sm4GcmEncryptor`: methods advance state,
the final step consumes `self`). The ephemeral secret `r` lives in `ZeroizeOnDrop`
state; advancing consumes the prior state so an ephemeral **cannot be reused** and the
**key cannot be fetched before confirmation**.

```rust
// ---- Initiator (party A) ----
pub struct Sm2KxInitiator { /* d_A, P_B, Z_A, Z_B, klen */ }
pub struct Sm2KxInitiatorWaiting { /* + r_A (ZeroizeOnDrop), R_A */ }

impl Sm2KxInitiator {
    /// `klen` and id lengths validated here; over-long id в†’ Err(Failed).
    pub fn new(d_a: &Sm2PrivateKey, p_b: &Sm2PublicKey,
               id_a: &[u8], id_b: &[u8], klen: usize) -> Result<Self>;
    /// Sample r_A, compute R_A. RNG failure в†’ Err(Failed).
    pub fn produce_ephemeral<R: TryCryptoRng>(self, rng: &mut R)
        -> Result<(Sm2KxEphemeralPoint, Sm2KxInitiatorWaiting)>;
}
impl Sm2KxInitiatorWaiting {
    /// Verify S_B (constant-time), derive K, emit S_A. Any failure в†’ Err(Failed).
    pub fn confirm(self, r_b: &Sm2KxEphemeralPoint, s_b: &Sm2KxConfirm)
        -> Result<(Sm2SharedKey, Sm2KxConfirm)>;   // (K, S_A)
}

// ---- Responder (party B) ----
pub struct Sm2KxResponder { /* d_B, P_A, Z_A, Z_B, klen */ }
pub struct Sm2KxResponderWaiting { /* + S_2 expected tag, K (held) */ }

impl Sm2KxResponder {
    pub fn new(d_b: &Sm2PrivateKey, p_a: &Sm2PublicKey,
               id_a: &[u8], id_b: &[u8], klen: usize) -> Result<Self>;
    /// Sample r_B, validate R_A, derive K, emit (R_B, S_B). Holds K until finish().
    pub fn respond<R: TryCryptoRng>(self, r_a: &Sm2KxEphemeralPoint, rng: &mut R)
        -> Result<(Sm2KxEphemeralPoint, Sm2KxConfirm, Sm2KxResponderWaiting)>;
}
impl Sm2KxResponderWaiting {
    /// Verify S_A (constant-time); only then release K.
    pub fn finish(self, s_a: &Sm2KxConfirm) -> Result<Sm2SharedKey>;
}
```

Supporting types:
- `Sm2KxEphemeralPoint` вЂ” the on-wire `R` (uncompressed SEC1 `04вЂ–XвЂ–Y`, reusing the
  existing public-key encoding helpers). Caller-constructible from bytes в†’ validated.
- `Sm2KxConfirm` вЂ” a 32-byte SM3 tag (`S_A`/`S_B`); compared with `ConstantTimeEq`.
- `Sm2SharedKey` вЂ” `klen`-byte secret, `ZeroizeOnDrop`, exposes `as_bytes()`.

All errors collapse to the single workspace `Error::Failed` (failure-mode invariant вЂ”
**no distinguishing** off-curve vs. bad-tag vs. RNG-fail; that would be an oracle).

## 5. Constant-time & security invariants

- Ephemeral `r` via the fixed-budget masked sampler вЂ” **no secret-dependent branch/loop**.
- `t = (d + xМ„В·r) mod n` and the scalar mults via `mul_var` вЂ” **no `if`/`==`/`bool` on
  secrets**; selection via `subtle`.
- `avf` reduction is pure bit-masking (set bit 127, mask low 127) вЂ” already constant-time;
  **must not branch** on `x`.
- Peer `R` validated with `point_on_curve` **before** use (invalid-curve / small-subgroup
  defense вЂ” the same guard `sm2::decrypt` uses); reject identity and `U/V = O`.
- Confirmation tags compared via `ConstantTimeEq` only.
- **Zeroize:** `r`, `t`, `(1)`-style intermediates, `x_UвЂ–y_U`, and the KDF input buffer;
  `Sm2SharedKey` is `ZeroizeOnDrop`.
- `id.len()` bounded by `MAX_ID_LEN` (inherited via `compute_z`).

## 6. Testing & KAT

**gmssl 3.1.1 has no SM2 key-exchange CLI** (verified: only `sm2keygen/sign/verify/
encrypt/decrypt`), so вЂ” exactly as v0.8 CCM and v0.12 XTS вЂ” KAT is **not** sourced from
gmssl. A `docs/v1.1-sm2kx-kat-sourcing.md` records this decision.

- **Primary KAT:** the GM/T 0003.5 / GB/T 32918.3 **Annex worked example**
  (`ALICE123@YAHOO.COM` / `BILL456@YAHOO.COM`, fixed `r_A`/`r_B`) driven through
  `FixedScalarRng` вЂ” asserts `R_A`, `R_B`, `K`, `S_B`, `S_A` **byte-for-byte**.
- **Cross-check:** against `libsm`'s published key-exchange vectors.
- **Round-trip property:** random `(d_A, d_B)` в†’ both sides derive equal `K`, both
  confirmations pass.
- **Negative tests:** off-curve `R` в†’ `Failed`; flipped `S_B`/`S_A` byte в†’ `Failed`;
  over-long `id` в†’ `Failed`; identity/`O` results в†’ `Failed`.

## 7. CI / dudect / fuzz

- New matrix leg `sm2-key-exchange` across **clippy**, **cargo-deny** (no new dep вЂ” pure
  allowlist no-op), **MSRV-1.85**, **wasm32** (`--no-default-features --features
  sm2-key-exchange`), and the **dudect** PR/nightly matrices.
- New cfg-gated dudect target **`ct_sm2_key_exchange`** вЂ” class-split by static private
  key `d`; both classes valid so both reach the confirmation compare via **identical
  control flow**; gate `|tau| < 0.20`. (Add the row to the CLAUDE.md dudect table; the
  base/feature counts shift accordingly.)
- New nightly fuzz target **`fuzz_sm2_kx`** вЂ” adversarial peer `R`/`S` bytes в†’ no panic /
  no OOM / no hang (failure-mode invariant). Added to the `fuzz-nightly.yml` sweep + the
  `fuzz-build.yml` PR build.
- `cargo-public-api` baseline + `cargo-semver-checks`: new surface is **additive** behind
  a feature в†’ passes as non-breaking; regenerate the `--all-features` baseline.

## 11. Open questions / risks

- **avf width:** confirm `w = 127` against the chosen Annex example before locking
  (off-by-one on `w` silently breaks interop but passes round-trip).
- **Masked-sampler visibility:** may need a narrow widening to reach from
  `key_exchange.rs` вЂ” keep it `#[doc(hidden)] pub`/`pub(crate)`, not public API.
- **Confirmation-tag misuse:** document that omitting `confirm`/`finish` (not calling
  them) forfeits authentication вЂ” but the typestate makes the *key* unreachable without
  them, which is the mitigation.
- **libsm vector availability:** if libsm's vectors aren't byte-extractable, the Annex
  worked example is the sole KAT (acceptable вЂ” it's the normative source).
