# v1.8 KAT sourcing — TLCP chain/pair verification fixtures

Status: v1.8 W0 sourcing-decision doc (the v0.8 CCM / v1.3 x509 / v1.7
record "verify-the-oracle-before-scoping" precedent). Records WHERE the
chain/pair test fixtures come from + the recipe; the harnesses are stood
up + validated during W3. Pairs with `docs/v1.8-scope.md` Q8.4 and extends
`crates/gmcrypto-core/tests/data/x509_regen.md` (v1.3).

## 0. Posture

G4 verification is **public-inputs-only** — every input (certificates,
trust anchors, the comparison time) is public, and the per-edge signature
verify is the v1.3 `verify_signature` primitive already KAT-pinned against
gmssl-generated fixtures. So v1.8's KAT job is narrower than a new crypto
primitive's: prove the **chain-walk + role-check policy** accepts a
well-formed TLCP pair-chain and **rejects** every structural deviation
(broken edge, wrong issuer Name, non-CA intermediate, missing/wrong
keyUsage, mismatched pair subject, over-depth, expired-at-time). Two
fixture sources cover that:

| Source | Role | Covers |
|---|---|---|
| **GmSSL 3.1.1** `sm2keygen`/`certgen`/`reqgen`/`reqsign` | realistic 3-level chain + the [sign, enc] pair | root → intermediate CA → [sign, enc] leaf with explicit keyUsage/basicConstraints; gmssl-self-verified before commit |
| **hand-built DER** via `asn1::writer` (in-test) | the negative edges | missing/wrong keyUsage bits, non-CA intermediate, broken issuer↔subject linking, depth overflow, pair-subject mismatch — full byte control |

gotlcp/Tongsuo were considered (Q8.4) as a third "real server pair" oracle
but **not adopted**: gotlcp ignores keyUsage at verification time (W0
finding), so it cannot cross-check the *policy* we strengthen over it, and
its value is interop (already covered by the v1.3 `verify_signature` KAT +
the GmSSL chain here). A future gotlcp full-handshake replay stays the same
documented follow-up the v1.7 record cycle left open.

## 1. GmSSL multi-level chain recipe (validated W0; re-run + commit at W3)

GmSSL 3.1.1 (`/opt/homebrew/bin/gmssl`). `reqsign` was confirmed (W0) to
support `-ca -path_len_constraint num`, repeated `-key_usage str`, and
`-ext_key_usage str` — so an **intermediate CA signed by the root** (the
piece the v1.3 self-signed-CA-only recipe lacked) is buildable. All parties
use the default SM2 ID `1234567812345678` (the v1.3 + RFC 8998 default).

```bash
cd "$(mktemp -d)"
P=-pass; PW=P@ss
DN="-C CN -ST Beijing -L Beijing -O gmtest"          # shared org

# 1. Root CA (self-signed, CA=TRUE, pathlen 1, keyCertSign+cRLSign)
gmssl sm2keygen $P $PW -out root.pem -pubout rootpub.pem
gmssl certgen $DN -OU ca -CN "GMTEST Root CA" -serial_len 12 -days 3650 \
  -key root.pem $P $PW -ca -path_len_constraint 1 \
  -key_usage keyCertSign -key_usage cRLSign -out rootcert.pem

# 2. Intermediate CA (signed by root, CA=TRUE, pathlen 0)
gmssl sm2keygen $P $PW -out int.pem -pubout intpub.pem
gmssl reqgen $DN -OU ca -CN "GMTEST Intermediate CA" -key int.pem $P $PW -out int.req
gmssl reqsign -in int.req -serial_len 12 -days 1825 -ca -path_len_constraint 0 \
  -key_usage keyCertSign -key_usage cRLSign \
  -cacert rootcert.pem -key root.pem $P $PW -out intcert.pem

# 3. Signature leaf + 4. Encryption leaf — SAME subject DN (pair binding!)
LEAF_DN="$DN -OU server -CN gmtest.example"
gmssl sm2keygen $P $PW -out sign.pem -pubout signpub.pem
gmssl reqgen $LEAF_DN -key sign.pem $P $PW -out sign.req
gmssl reqsign -in sign.req -serial_len 12 -days 365 \
  -key_usage digitalSignature -key_usage nonRepudiation \
  -cacert intcert.pem -key int.pem $P $PW -out signcert.pem

gmssl sm2keygen $P $PW -out enc.pem -pubout encpub.pem
gmssl reqgen $LEAF_DN -key enc.pem $P $PW -out enc.req     # identical DN to sign leaf
gmssl reqsign -in enc.req -serial_len 12 -days 365 \
  -key_usage keyEncipherment -key_usage dataEncipherment -key_usage keyAgreement \
  -cacert intcert.pem -key int.pem $P $PW -out enccert.pem

# 5. gmssl self-check each edge (MUST print "Verification success")
gmssl certverify -in intcert.pem  -cacert rootcert.pem
gmssl certverify -in signcert.pem -cacert intcert.pem
gmssl certverify -in enccert.pem  -cacert intcert.pem
gmssl certparse  -in signcert.pem    # confirm keyUsage = digitalSignature,nonRepudiation
gmssl certparse  -in enccert.pem     # confirm keyUsage = keyEncipherment,dataEncipherment,keyAgreement
# PEM -> DER for the 5 certs (rootcert, intcert, signcert, enccert) + the
# root SPKI, via the v1.3 x509_regen.md python one-liner (label CERTIFICATE
# / PUBLIC KEY). Committed under tests/data/ as x509_chain_*.der.
```

Committed fixtures (names pinned at W3): `x509_chain_root.der`,
`x509_chain_int.der`, `x509_chain_sign.der`, `x509_chain_enc.der` (+ the
root SPKI if a test needs the anchor key directly — usually the anchor is
the parsed `rootcert.der`). Throwaway keys are NOT committed; regen produces
a fresh chain, so the KAT asserts **structural + verification** properties
(chain verifies, roles match, pair binds, edges link), never key bytes —
the v1.3 posture.

## 2. Hand-built DER negatives (in-test, `asn1::writer`)

The reject edges need byte-precise control the gmssl flow can't give
cheaply. Built in-test from a minimal valid cert template (the v1.3
`tlv`/`extension` test helpers + `asn1::writer`), each mutating exactly one
property:

- **keyUsage rejects:** sign leaf missing `digitalSignature`; enc leaf
  asserting neither `keyEncipherment` nor `keyAgreement` (e.g.
  `digitalSignature` only); **keyUsage extension absent** on a leaf whose
  role we check (→ reject, the stricter-than-RFC rule, Q8.5/D-1);
  intermediate/anchor missing `keyCertSign`.
- **basicConstraints rejects:** intermediate with `CA=FALSE` (or absent
  basicConstraints) used as an issuer (→ reject, Q8.6/D-4); a leaf
  presented as an intermediate.
- **Linking rejects:** chain whose `issuer_raw()` ≠ the next cert's
  `subject_raw()` (broken Name link); chain that never reaches an anchor;
  edge whose signature is valid-shaped but does not verify under the
  issuer's key (rides the v1.3 garbage-sig-parses-never-verifies pin).
- **Depth reject:** chain exceeding `MAX_CHAIN_DEPTH` → reject (DoS /
  over-restriction guard, Q8.6).
- **Pair-binding reject:** sign leaf + enc leaf with **different** raw
  `subject` Name TLVs → `verify_pair` returns `false` (Q8.8/D-8). Built by
  giving the two hand-built leaves different CN bytes.
- **Time rejects:** `at_time = Some(t)` with `t` before `not_before` or
  after `not_after` of any chain cert → reject; `t` inside every window →
  accept; `None` → window ignored (Q8.10). Uses the existing `X509Time`.

Positive control from the same hand-built template (single self-issued
anchor + one leaf) cross-checks the gmssl chain's structural assertions
without depending on fresh gmssl key bytes.

## 3. Provenance + format (v1.6 A5 / v1.7 posture)

DER chain fixtures live under `tests/data/x509_chain_*.der` (binary, like
the v1.3 `x509_*.der`). Hand-built negatives are constructed **in the test
file** (`tests/x509_chain_kat.rs` + `tests/tlcp_chain_kat.rs`), not stored.
This recipe file is the regen provenance; gmssl version + the
`certparse`-confirmed keyUsage strings are recorded here at W3 stand-up so a
regen is reproducible. No new C oracle harness is needed (no new crypto
primitive — the signature verify is the v1.3 primitive).
