freeze.** Lock the public API surface, wire the SemVer/doc guards a 1.0 contract
needs, and freeze the user-facing docs — **without** the irreversible crates.io

## 1. Context

The published library has been API-stable since **v0.5** (the last breaking
renames: `Sm2PrivateKey::new`→`from_scalar`, `from_sec1_be`→`from_bytes_be`,
`to_sec1_be`→`to_bytes_be`). Eleven minor releases later (v0.16.0, 2026-05-29) the
surface has only grown additively under opt-in features. Five consecutive cycles
(v0.14/0.17/0.18/0.19/0.20) were non-publishing assurance work; v0.20 just closed
with streaming-decryptor differential fuzzing, a `cargo fuzz coverage` report, and
the **codified v1.0 constant-time baseline** in `SECURITY.md`.

What a 1.0 contract still needs, and what an audit found missing:

- **No public-API change guard.** There is no `cargo-public-api` baseline and no
  `cargo-semver-checks` in CI. Nothing mechanically prevents an accidental
  breaking change to the Rust surface. (The **C ABI** *is* already guarded — the
  committed `crates/gmcrypto-c/include/gmcrypto.h` is drift-checked via
  `git diff --exit-code` on every CI run.)
- **No `cargo doc` gate.** `#![deny(missing_docs)]` is enforced at compile time, and
  `[package.metadata.docs.rs] all-features = true` is set, but CI never builds the
  docs — broken intra-doc links / rustdoc warnings can land silently.
- **Two un-finalized surface ambiguities** that a 1.0 freeze must resolve:
  - **3 `#[doc(hidden)] pub` items** in `gmcrypto-core`:
    `sm2::sign::sign_raw_with_id` (the dudect harness hook), and
    `Sm4CbcEncryptor::take_output` / `Sm4CbcDecryptor::take_output` (drain shims
    used cross-crate by `gmcrypto-c`). CLAUDE.md already states `sign_raw_with_id`
    is "not covered by SemVer," but the contract is not on the items themselves.
  - **`gmcrypto-simd`'s public surface** — `pub mod ghash` / `pub mod sm4`,
    `pub use detect::{has_avx2, has_pclmulqdq, has_pmull}`, `pub use
    ghash::ghash_mul` (and the `sbox_x{8,16,32}` submodules). The crate doc says
    "small Rust-internal API surface only," yet these are fully `pub` on a
    crates.io-published crate. They **must stay crate-`pub`** so `gmcrypto-core`
    can call them across the crate boundary — but their external SemVer status is
    ambiguous.

The CHANGELOG is current and the `deny.toml` dependency policy is hard-lined (the
5-core-dep rule) — both are 1.0-ready and only need a freeze note.

## 2. Hard contract

- **Non-publishing.** No version bump, no tag, no `cargo publish`. Workspace stays
  `0.16.0`; crates.io skips `0.21.0`. The milestone lives in `docs/` + `CLAUDE.md`
  (+ Notion). The `1.0.0` publish is a **separate later step** (the
  `docs/v1.0-readiness.md` runbook + `docs/v0.1.0-release-review.md` gate).
- **Option-A finalization discipline.** v0.21 *does* touch published-crate source,
  but only for **doc attributes + tests** — **no behavioral or codegen-intent
  change**. The default-features build's *output* is unaffected (KAT + interop
  unchanged). The accepted, bounded departure from the v0.17–v0.20 "zero
  crate-source change" pattern: the invariant is *"nothing publishes,"* not *"main
  stays byte-identical to 0.16.0 forever"* (the changes de-risk the 1.0 surface and
  are reviewable now rather than concentrated into the irreversible publish PR).
- **Sibling-crate coupling stays sound.** The hidden cross-crate-consumed items
  (the two `take_output` shims used by `gmcrypto-c`; the whole `gmcrypto-simd`
  surface used by `gmcrypto-core`) are kept matched not by being "stable" but by
  **lockstep, exact-version sibling pinning**: the three crates are always published
  together at one workspace version, and the intra-workspace deps pin **exactly**
  (`=`) to it (§6). So "may change in any release" is a contract for *external*
  consumers; the internal coupling never resolves a mismatched sibling pair. Today's
  `version = "0.16"` (`^0.16` = `>=0.16.0,<0.17.0` in 0.x) is already effectively
  exact; the 1.0 runbook makes it explicit so the looser post-1.0 `^1.0` can't drift.
- **The C ABI's stability guard stays the cbindgen header drift-check.**
  `cargo-public-api` covers the Rust surface (`gmcrypto-core`, `gmcrypto-simd`);
  `gmcrypto-c`'s contract is the committed `gmcrypto.h` diff, already in CI.
- **dudect untouched.** No change to the dudect workflows, the gate logic, or the
  invert-target posture. `sign_raw_with_id` keeps its `#[doc(hidden)] pub` shape so
  the bench harness compiles unchanged (no cfg-gating — see Q21.4).
- **`unsafe_code = "forbid"` on `gmcrypto-core` stays.** MSRV 1.85, edition 2024.

## 3. Q-list (Q21.1–Q21.9) with answers

### Q21.1 — Anchor (theme)
**v1.0 readiness audit, non-publishing.** Lock the public Rust API surface, add the
SemVer/doc CI guards, finalize the two surface ambiguities, and freeze the docs —
producing a GO/NO-GO readiness report and a `1.0.0` publish runbook. The confirmed
primary from the v0.20 Codex+Grok discussion (`docs/v0.20-scope.md` §6 Q21.1).

### Q21.2 — Publish posture
**Non-publishing** (maintainer-decided). v0.21 stays `0.16.0`; the `1.0.0` crates.io
publish is a deliberate later step. Rationale: the project reserves the irreversible
`cargo publish` for its own reviewed step (`docs/v0.1.0-release-review.md`) and never
bundles it with audit cleanup — the v0.14–v0.20 caution pattern.

### Q21.3 — API-finalization depth
**Option A: minimal finalization on `main`, disciplined** (Codex-decided, focused
2nd consult). Land the smallest non-behavioral surface changes (doc attributes +
compile-time existence tests) so the committed public-api baseline records the
*intended* 1.0 surface. **Rejected: Option B** ("docs/CI-only," zero crate-source
change) — it would commit a baseline capturing the internal SIMD items + harness
hooks as "public," and then `#[doc(hidden)]`-ing them at the 1.0 PR reads to tooling
as a public-API *removal* → churn exactly where stability matters; it also
concentrates too much semantic judgment into the irreversible publish step.
**Rejected: Option C** (B/A + cfg-gating `sign_raw_with_id` behind an internal
feature) — too invasive (churns the `[[bench]]` required-features + the dudect CI
matrix) for nil gain, since `cargo-semver-checks` already excludes `#[doc(hidden)]`
items from the modeled public API.

### Q21.4 — The 3 `#[doc(hidden)] pub` items
Keep all three `#[doc(hidden)] pub` (no cfg-gating, no promotion). Add an explicit
item-level doc note: **"Not part of the public API / not covered by SemVer; may
change or be removed in any release."** Add a compile-time existence/regression test
that references each (so the harness/FFI consumers can't silently lose them).
`sign_raw_with_id` stays dudect-only; the two `take_output` shims stay
`gmcrypto-c`-internal (the FFI crate's existing `c_smoke` tests already exercise
them). `cargo-semver-checks` treats `#[doc(hidden)]` as outside the public API, so
this gives the "not SemVer-covered" guarantee in tooling terms without any structural
change. (The two `take_output` shims are consumed cross-crate by `gmcrypto-c`; like
the simd surface in Q21.5, their internal soundness comes from lockstep exact-version
sibling pinning — §2 / §6 — not from the items being stable.)

### Q21.5 — `gmcrypto-simd` public surface
Keep every entry point `pub` (so `gmcrypto-core` calls it cross-crate) but add
`#[doc(hidden)]` + an explicit **"internal acceleration backend for `gmcrypto-core`;
no stable Rust API is promised here — Rust users use `gmcrypto-core`, C users use
`gmcrypto-c`"** contract on the crate-root exports (`pub mod ghash`, `pub mod sm4`,
the two `pub use` lines). Rust has no workspace-private visibility, so
`pub + #[doc(hidden)] + explicit non-SemVer contract` is the correct compromise
(Codex). Doing this **before** committing the public-api baseline (Q21.6) keeps the
baseline clean and avoids the later-removal churn.

### Q21.6 — Tooling (the v1.0 guard set)
- **`cargo-public-api` baselines** for `gmcrypto-core` + `gmcrypto-simd`, committed
  under `docs/api-baseline/*.txt` and **drift-checked** via `git diff --exit-code`
  in CI (the established cbindgen-header pattern). This is the **enforced** gate —
  any unintended public-surface change fails CI. **Pin both the `cargo-public-api`
  version and the nightly toolchain date** (it depends on nightly rustdoc JSON; a
  floating nightly would re-flake a committed drift gate — record the pinned
  toolchain in the workflow + the baseline regen recipe). Run in its own job, not in
  `cargo test --workspace` — no dev-dep added to a published crate.
- **`cargo-semver-checks`** core+simd vs crates.io `0.16.0`, run **informational**
  (logged, non-gating): 0.x SemVer permits breakage and `0.16→1.0` is a major bump,
  so a backward diff is expected (e.g. the Q21.5 simd doc-hiding). It becomes the
  **enforced forward gate from 1.0 onward** (current-vs-last-release).
- **`cargo doc`** with `RUSTDOCFLAGS="-D warnings -D rustdoc::broken_intra_doc_links"`,
  `--no-deps --all-features` and an explicit **`--workspace`** (not the default
  members — an unqualified `cargo doc` would skip `gmcrypto-c`/`gmcrypto-simd`) —
  gate broken links / rustdoc warnings across the whole published surface.
- **Feature-matrix legs** — `--no-default-features` + `--all-features` build/test
  (complements the existing per-feature clippy passes + the wasm32 `--no-default-features`
  build).

All of this lands in a new `.github/workflows/api-stability.yml`. The **C ABI**
guard remains the existing `gmcrypto.h` drift-check in `ci.yml` (unchanged).

- **CHANGELOG** — a `## [Unreleased] — v1.0 prep` entry (guards added, surface

## 6. The 1.0.0 publish runbook (deliverable, not executed here)

`docs/v1.0-readiness.md` (W3) specifies the mechanical cut: bump
`[workspace.package].version` `0.16.0 → 1.0.0`; update the intra-workspace path-dep
pins to **exact** (`gmcrypto-core` → `gmcrypto-simd = "=1.0.0"`; `gmcrypto-c` →
`gmcrypto-core = "=1.0.0"`) so the hidden cross-crate-consumed APIs (Q21.4/Q21.5)
can never resolve a mismatched sibling pair under the looser post-1.0 `^1.0`
(today's 0.x `^0.16` is already effectively exact — the runbook just makes the
lockstep explicit); `cargo publish` order **simd → core → c**; `cargo publish
--dry-run` + package-content review per crate; SSH-signed `v1.0.0` tag; the
`docs/v0.1.0-release-review.md` checklist as the final gate. Flip
`cargo-semver-checks` to enforced (current-vs-`1.0.0`) in the follow-up.
