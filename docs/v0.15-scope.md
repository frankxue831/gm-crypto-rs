# v0.15 — Scope doc

**SM4-XTS multi-sector (disk) helper** — add `sm4::mode_xts::{encrypt_sectors,
decrypt_sectors}`: an **in-place** convenience API that encrypts/decrypts a run
of equal-size data units (disk sectors) under auto-incrementing tweaks derived
from a starting sector number. It rides the v0.12 single-shot `mode_xts` core
100% — **no new dependency, no new feature flag** (lives under the existing
`sm4-xts` feature), no new SIMD. Additive; default builds unchanged. A
publishable single-feature minor release on the v0.7→v0.13 feature cadence.

## 1. Context

SM4-XTS (`crates/gmcrypto-core/src/sm4/mode_xts.rs`, v0.12; C FFI v0.13) is a
length-preserving, tweakable, **confidentiality-only** block mode designed for
random-access **disk/sector encryption** per GB/T 17964-2021. Its single-shot
API encrypts **one** data unit under **one** caller-supplied 16-byte tweak:

```rust
pub fn encrypt(key: &[u8; XTS_KEY_SIZE], tweak: &[u8; 16], data_unit: &[u8]) -> Option<Vec<u8>>;
pub fn decrypt(/* same */) -> Option<Vec<u8>>;
```

But the *actual* disk use case is a **run of sectors**: a 4 KiB page is eight
512-byte sectors, a partition is millions of them, each encrypted independently
under tweak = its sector number. With only the single-shot API, the caller must
(a) loop over sectors and (b) **hand-encode each sector number into a 16-byte
tweak**. The module docstring explicitly punts (b) to the caller ("The encoding
of a sector number into the 16-byte tweak (endianness/width) is the caller's
responsibility"). That encoding is the easy-to-get-wrong, security-relevant part
(a wrong/duplicated tweak silently leaks equality structure — XTS is
deterministic). A multi-sector helper that owns the loop *and* the standard
tweak encoding removes the footgun and is the natural ergonomic completion of
the mode.

### Building blocks already in place (grounded 2026-05-26)

- **Single-shot core** (`mode_xts.rs`): `split_keys(key, len)` validates length
  ∈ `[16, 16 MiB]` + the constant-time `Key1 == Key2` weak-key reject and
  returns the two `Sm4Cipher`s; `xts_encrypt` / `xts_decrypt` do the per-data-
  unit work (whole-block bulk via `Sm4Cipher::encrypt_blocks` / `decrypt_blocks`
  — the SIMD fanout under `sm4-bitsliced-simd` — plus the CTS tail). `mul_alpha`
  is the constant-time GB/T 17964 α-doubling. `XTS_KEY_SIZE = 32` is re-exported.
- **`Sm4Cipher` batch API** (`cipher.rs`, v0.7 W1): `encrypt_block` /
  `decrypt_block` (single) + `encrypt_blocks` / `decrypt_blocks`
  (`&mut [[u8; 16]]`, SIMD-fanout). The sector helper reuses these directly.
- **LE sector-tweak precedent already shipped**: the v0.13 C example
  `crates/gmcrypto-c/examples/sm4_xts_sector.c` encodes the sector number as a
  **little-endian** 16-byte tweak (`tweak[i] = sector >> (8*i)`). The helper
  adopts the same convention so the Rust API and the documented C usage agree.
  LE-128 is also the IEEE 1619 / NIST SP 800-38E data-unit-sequence-number
  convention.
- **dudect coverage**: `ct_sm4_xts_decrypt` (cfg `sm4-xts`) already times
  `mode_xts::decrypt` over a CTS-length data unit; the sector helper rides the
  identical block/α-doubling machinery (Q15.9).
- **Failure-mode invariant**: `mode_xts` returns a single `None` for *all*
  validation failures; the helper extends that posture verbatim (Q15.7).

## 2. Hard contract

Two new public functions in `mode_xts.rs`, under the existing `sm4-xts` feature:

```rust
/// Encrypt a contiguous run of equal-size sectors in place. Sector `i`
/// (0-based within `buf`) is encrypted under tweak = LE-128(`start_sector` + i).
pub fn encrypt_sectors(
    key: &[u8; XTS_KEY_SIZE],
    sector_size: usize,   // multiple of 16, in [16, 16 MiB]
    start_sector: u128,   // first sector's number; tweak_i = LE128(start_sector + i)
    buf: &mut [u8],       // 0 or more whole sectors: buf.len() % sector_size == 0
) -> Option<()>;          // Some(()) on success; single None on any validation failure
pub fn decrypt_sectors(/* identical signature */) -> Option<()>;
```

- **In-place** (`&mut [u8]`, `-> Option<()>`), not `-> Vec<u8>` (Q15.3).
- **Validation → single `None`** (Q15.7): `sector_size` not a multiple of 16, or
  `< 16`, or `> 16 MiB`; `buf.len()` not a whole multiple of `sector_size`;
  `Key1 == Key2` (constant-time compare, reused from `split_keys`); sector-number
  overflow `start_sector + i` exceeding `u128::MAX` (checked; §Q15.4). No
  distinguishing variants. XTS has no tag → no MAC-failure path.
- **`buf.len() == 0` ⇒ `Some(())`** (zero sectors, vacuous success), provided
  `sector_size` **and the key** are valid — the key is validated even with zero
  sectors, so empty `buf` under a weak key (`Key1 == Key2`) is still `None`
  (Q15.5).
- **Confidentiality only** — `decrypt_sectors` returning `Some(())` does **not**
  authenticate. Same warning as the single-shot module docstring.
- **Differential identity** (Q15.8): `encrypt_sectors` over a `buf` of N sectors
  is **byte-identical** to, for each sector `i`, copying
  `mode_xts::encrypt(key, &le128(start_sector + i), &buf[i*ss..(i+1)*ss])` back
  into that sector slice. This is the spec and the primary test oracle.
- **Default-features build byte-unchanged**; `sm4-xts` build gains only the two
  fns + their tests/doctest. `no_std` / `wasm32` posture preserved (pure-core).

## 3. Q-list (Q15.1–Q15.12) with answers

### Q15.1 — Anchor
**SM4-XTS multi-sector helper, single feature.** A user-visible, *publishable*
crypto-feature release after the unpublished v0.14 assurance cycle. Rejected for
this cycle (all remain deferred — §6): the `v0.14-Q15.x` provisional menu —
round-trip / streaming-decryptor fuzzing (assurance; v0.14 just shipped the
no-panic baseline), `aead` trait fit (blocked: `aead 0.6.0-rc.10`), AVX-512
`sbox_x64` (needs an x86_64 AVX-512 env; the runner is aarch64), pinned dudect
runner (entangled with the GitHub-billing block), v1.0 readiness (premature —
finish the XTS ergonomics first); and **CCM incremental input** (Q15.11).
Single-anchor cadence, matching v0.7→v0.13.

### Q15.2 — API location & gating
New `pub fn encrypt_sectors` / `decrypt_sectors` **in `mode_xts.rs`**, under the
existing `#[cfg(feature = "sm4-xts")]`. **No new feature flag, no new dependency,
no new SIMD** — the helper is pure orchestration over `split_keys` + the
`Sm4Cipher` batch API + `mul_alpha`. (`sm4-xts = []` stays empty.)

### Q15.3 — API shape: in-place `&mut [u8]` → `Option<()>`
**In-place**, not a returned `Vec`. Rationale: disk encryption reads a region
into a buffer and writes the transformed bytes back to the *same* region;
returning a `Vec` doubles peak memory for a whole-partition buffer. Matches the
no-alloc discipline already in the SDK (`pbkdf2_hmac_sm3`'s caller-supplied
`&mut [u8]`). The single-shot `encrypt`/`decrypt` keep their `-> Vec` shape
(they're length-*preserving* per call but return-by-value is fine for one unit);
the sector helper is the bulk/in-place path. `Option<()>` carries the single
validation-failure signal with no payload.

### Q15.4 — Tweak derivation: `start_sector: u128` → little-endian 128-bit
The caller passes a **`u128` starting sector number**; the helper derives
sector `i`'s tweak as the **little-endian 16-byte encoding of
`start_sector + i`**. LE-128 is the de-facto disk-XTS data-unit convention
(IEEE 1619 / NIST SP 800-38E) and matches the already-shipped
`sm4_xts_sector.c` example, so the Rust and C surfaces agree. Owning the
encoding is the whole point — it removes the "encode the sector number
yourself" footgun the single-shot leaves open. **Overflow:** `start_sector + i`
is computed with `checked_add`; if it would exceed `u128::MAX` (only reachable
with `start_sector` astronomically close to the max), the call returns `None`
(validated up front against `buf.len() / sector_size`, before any crypto).
**Tweak-namespace contract (caller-owned, stated bluntly per W0 codex):**
sector numbers must be **unique within the XTS-key namespace**, not merely
within a single call. Reusing a `(key, sector_number)` pair across *any* two
plaintexts leaks block-equality structure (XTS is deterministic). In
particular: **do not encrypt multiple devices / partitions / snapshots that all
start at sector 0 under the same key** unless that cross-device equality leakage
is explicitly acceptable — use **absolute LBAs** (a global sector address) or a
**separate key per device**. The helper owns the *encoding* of `start_sector`
into the tweak; it cannot and does not enforce global uniqueness of the sector
numbers the caller supplies.

### Q15.5 — `sector_size` & buffer contract
`sector_size` must be a **multiple of `BLOCK_SIZE` (16)** and in
`[16, 16 MiB]` (the existing per-data-unit `MAX_LEN`); `buf.len()` must be a
**whole multiple of `sector_size`** (0 allowed). Because each sector is a whole
number of 16-byte blocks, **ciphertext stealing never triggers inside the
helper** — the CTS tail of `xts_encrypt`/`xts_decrypt` is simply never reached
on a sector run (real sectors are 512 / 4096 bytes). `buf.len() == 0` with a
valid `sector_size` **and key** is a **vacuous `Some(())`** (zero sectors,
nothing to do) — strictly defined, no surprising panic on an empty region. The
key is validated even at zero sectors (`split_keys` runs unconditionally), so
empty `buf` under a weak key (`Key1 == Key2`) is still `None` (W0 codex finding,
pinned by a test). Any other shape violation → `None`.

### Q15.6 — Ciphers constructed once for the whole run
`split_keys(key, sector_size)` is called **once** (validating the key + a
representative sector length), and the two `Sm4Cipher`s are reused across every
sector — not reconstructed per sector. This is both the efficiency win
(key-schedule once, not N times) and keeps the constant-time / zeroize posture
identical to the single-shot path (`Sm4Cipher` is `ZeroizeOnDrop`). The weak-key
(`Key1 == Key2`) reject is therefore evaluated exactly once, constant-time, via
the existing `subtle` compare in `split_keys`.

### Q15.7 — Failure-mode invariant
Single `Option<()>` `None` for **every** validation failure (bad `sector_size`,
non-multiple `buf.len()`, `Key1 == Key2`, sector-number overflow). No
distinguishing variants — same oracle-defense posture as the rest of the SDK.
XTS is unauthenticated, so there is no MAC-mismatch path; `decrypt_sectors`
never fails on *content*, only on the same up-front validation as
`encrypt_sectors`.

### Q15.8 — Correctness / KAT strategy (no new oracle needed)
The helper's spec is **differential identity with the single-shot core**: a
sector run equals the per-sector loop of `mode_xts::encrypt`/`decrypt` with
LE-128 tweaks. The v0.12 single-shot is already byte-pinned to **OpenSSL 3.x EVP
`SM4-XTS` `xts_standard=GB`**, so a differential test against it transitively
inherits the OpenSSL KAT — **no new external oracle, no new fixture**. Tests
(W1, TDD):
1. **Differential** vs the single-shot loop over several N (1, 2, 8, 1000) and
   sector sizes (16, 512, 4096), with a non-zero `start_sector` (e.g. crossing
   an α-doubling carry / a 256-boundary in the LE counter).
2. **Round-trip**: `decrypt_sectors ∘ encrypt_sectors == identity` in place.
3. **Validation**: each `None` path (non-multiple `sector_size`, `< 16`,
   `> 16 MiB`, non-multiple `buf.len()`, `Key1 == Key2`, overflow) returns
   `None` and (where applicable) leaves `buf` untouched.
4. **Vacuous**: empty `buf` → `Some(())`, `buf` unchanged.
5. A small **literal KAT**: one multi-sector vector pinned to the OpenSSL oracle
   directly (re-running the v0.12 `sm4_xts_oracle.c` per sector) so the file
   carries an absolute, not only relative, anchor.

### Q15.9 — dudect: reuse `ct_sm4_xts_decrypt`, no new target
**No new dudect target.** The sector helper's secret-dependent work is exactly
the single-shot path (`split_keys` → per-block `encrypt_blocks`/`decrypt_blocks`
→ `mul_alpha`), already gated by `ct_sm4_xts_decrypt` (`|tau| < 0.20`). The only
*new* per-sector logic is the **sector-number arithmetic + LE-128 encode**,
which operates on **public** data — sector addresses are not secret (an attacker
who can time disk access already knows which sector). Adding a target over public
arithmetic would dilute, not sharpen, the leak signal. Same posture as v0.13
(thin FFI → no new target). The dudect gate table / matrix is unchanged.

### Q15.10 — Performance / SIMD
Within each sector, whole-block bulk rides `Sm4Cipher::encrypt_blocks` /
`decrypt_blocks` (the v0.6 SIMD fanout under `sm4-bitsliced-simd`), exactly as
the single-shot path. The helper adds no new SIMD and no per-block dispatch in
`gmcrypto-core` (the `cpufeatures` cache lives in `gmcrypto-simd`). Cross-sector
parallelism (rayon / threads) is **out of scope** — `no_std` core, caller's
concern.

### Q15.11 — Docs, examples, and CCM
- **Docs**: a new "Multi-sector (whole-disk) helper" section in the `mode_xts`
  module docstring + a runnable Rust doctest. The tweak-uniqueness and
  no-authentication caveats are restated for the sector API.
- **C example / FFI**: **deferred to v0.16** (Q15.12). The existing
  single-shot `sm4_xts_sector.c` already shows the LE sector-tweak idiom.
- **CCM incremental input**: **dropped from v0.15**, deferred with rationale.
  `mode_ccm` itself documents that CCM "requires the caller to commit to
  plaintext length up front (the length is encoded in the first CBC-MAC block) —
  incompatible with streaming." An "incremental-input" CCM API could therefore
  only *buffer* the whole message and process it at finalize — **zero** memory
  benefit over the single-shot, a misleading "streaming" name, and a permanent
  SemVer surface for a non-feature. If a buffered convenience is ever wanted it
  should be named honestly (e.g. `Sm4CcmBuffered`) and justified on ergonomics
  alone; not this cycle. Carried to §6 as a v0.16 candidate.

### Q15.12 — FFI deferral & publish
The **C FFI** for `encrypt_sectors` / `decrypt_sectors` is **deferred to v0.16**,
matching the established **core-in-vN / FFI-in-vN+1** cadence (XTS core v0.12 →
FFI v0.13; AEAD core v0.8 → streaming FFI v0.10). v0.15 ships core-only but **is
a real crates.io release**: workspace `version` `0.13.0 → 0.15.0` (+ inter-crate
dep refs `gmcrypto-core → gmcrypto-simd` and `gmcrypto-c → gmcrypto-core`),
`CHANGELOG.md` `[0.15.0]`, README roadmap, then — **after explicit user
confirmation at the W4 gate** (irreversible per `CLAUDE.md`) — SSH-signed tag
`v0.15.0` + `cargo publish` triplet `simd → core → c`. crates.io skips `0.14.0`
(see the Version note above).

  `0.13.0 → 0.15.0`. Direction unchanged: private feature cadence toward v1.0
  (the self-hosted→hosted CI swap + public flip stay parked until just before
  publication). Auto-mode execution with codex milestone reviews; publish step
  (Q15.12) pending user confirmation at the W3 gate.
- **2026-05-26 (W0 codex review)**: design assessed "basically sound." Five
  findings; one substantive. (1) **Tweak-namespace wording strengthened** —
  Q15.4 now states bluntly that sector numbers must be unique within the
  *XTS-key namespace* (don't reuse one key across devices/partitions starting at
  sector 0; use absolute LBAs or per-device keys). (2) Confirmed:
  no-partial-mutation requires **pre-flighting all validation** (sector_size,
  buf-multiple, overflow) *before* `split_keys`/the loop — no `checked_add(...)?`
  inside the loop after sectors are mutated. (3) Implementation guidance: under
  `forbid(unsafe_code)` / MSRV 1.85, **do not** reinterpret `&mut [u8]` →
  `&mut [[u8; 16]]` via unsafe or unstable `as_chunks_mut`; use safe
  gather/scatter (`copy_from_slice`) into a reused `[[u8; 16]]` scratch, and
  **do not** call the single-shot public API per sector (rebuilds ciphers).
  (4) Failure mode + empty-`buf` → `Some(())` confirmed correct. (5) No new
  dudect target confirmed defensible; CCM-drop / FFI-to-v0.16 / `0.13.0 →
  0.15.0` skip all endorsed.
- **2026-05-26 (W1 codex review)**: implementation assessed sound — **no
  higher-severity findings**. `xts_sector_in_place` faithfully mirrors the
  `rem == 0` single-shot path (no block-ordering / scatter-gather bug); the
  no-partial-mutation invariant holds by inspection (all shape + overflow checks
  precede `split_keys`, which precedes any mutation; the in-loop `wrapping_add(i)`
  is provably non-wrapping given the `checked_add(last)` precheck); constant-time
  posture + zeroization match the single-shot path; MSRV/no_std clean (no
  `is_multiple_of` / `as_chunks_mut` / new panic path). One **LOW** finding:
  empty-`buf` semantics were under-specified — the impl validates the key even at
  zero sectors (so empty `buf` + weak key → `None`). Resolved by pinning
  "valid sector_size **and key**" in Q15.5 + a dedicated test
  (`empty_buffer_is_vacuous_success` now asserts the weak-key `None`).
