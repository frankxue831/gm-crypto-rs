# v0.13 — Scope doc

**C FFI for SM4-XTS** — expose the v0.12 `sm4::mode_xts::{encrypt, decrypt}`
single-shot tweakable-mode core through the `gmcrypto-c` C ABI, behind a
forwarding `sm4-xts` feature. The deferred half of v0.12 (Q12.11), landed on
the established **core-in-vN, FFI-in-vN+1** cadence (SM4-GCM/CCM core in v0.8 →
streaming FFI in v0.10). Additive only; default builds unaffected.

> **Q-numbering note:** this doc's `Q13.x` are the *design decisions* for v0.13.
> They **supersede** the provisional `Q13.x` *candidate* list in
> `docs/v0.12-scope.md` §6 (a forward-looking menu). Where this doc references

> on the former candidate `v0.12-Q13.1` (C FFI for SM4-XTS).

## 1. Context

The C ABI (`gmcrypto-c`: cdylib + staticlib + cbindgen header) is the
workspace's single language-binding surface for C / C++ / Go / Zig / Python
callers. Every cipher mode reaches it **except SM4-XTS**, which shipped
core-only in v0.12 (Q12.11 deferred the FFI to let the Rust API soak one
cycle, mirroring the v0.8→v0.10 AEAD precedent). v0.13 closes that gap so the
disk/sector-encryption mode is reachable from non-Rust callers — leaving every
mode FFI-complete, a clean state ahead of an eventual v1.0.

SM4-XTS is **single-shot, length-preserving, confidentiality-only** (no tag,
no expansion). That makes its FFI the *simplest* of the cipher modes: it is the
single-shot SM4-GCM shape (`crates/gmcrypto-c/src/lib.rs:975`) minus the
nonce / AAD / tag — just `key + tweak + data → equal-length data`. No opaque
handle types, no streaming state machine.

### Building blocks already in place (grounded 2026-05-24)

- **Core API** (`gmcrypto_core::sm4::mode_xts`):
  `encrypt(key: &[u8; 32], tweak: &[u8; 16], data_unit: &[u8]) -> Option<Vec<u8>>`
  and the matching `decrypt`, plus `XTS_KEY_SIZE = 32`. Single `None` failure
  mode (len ∉ `[16, 16 MiB]`, or `Key1 == Key2`). Already dudect-gated
  (`ct_sm4_xts_decrypt`).
- **FFI conventions** (`gmcrypto-c/src/lib.rs`): `ffi_guard` (catch_unwind →
  `GMCRYPTO_ERR`), `try_slice` (`(NULL,0)`→empty, null+len→`None`),
  `write_output` (the `(out, out_capacity, out_actual_len)` convention — writes
  required length even when the buffer is too small, returns `GMCRYPTO_ERR`),
  `GMCRYPTO_OK = 0` / `GMCRYPTO_ERR = -1`, size constants
  `GMCRYPTO_SM4_{KEY,BLOCK}_SIZE`.
- **Feature-forwarding precedent**: `sm4-aead = ["gmcrypto-core/sm4-aead"]`
  gates the single-shot AEAD FFI. XTS mirrors this exactly.
- **Test precedent**: `tests/c_smoke.rs` AEAD block (`:1183+`) —
  import the symbol via `gmcrypto_c::{…}`, call through Rust's `extern "C"`
  interop, assert byte-equivalence with the core API + round-trip + error code.
- **Header-regen precedent**: `regen-header` runs cbindgen in `build.rs`;
  CI gates drift via `git diff --exit-code include/gmcrypto.h`.

## 2. Hard contract

- Two new symbols, behind `#[cfg(feature = "sm4-xts")]`:
  - `gmcrypto_sm4_xts_encrypt(key, tweak, data, data_len, out, out_capacity, out_actual_len) -> c_int`
  - `gmcrypto_sm4_xts_decrypt(key, tweak, data, data_len, out, out_capacity, out_actual_len) -> c_int`
- `key` = 32 bytes (`Key1 ‖ Key2`); `tweak` = 16 raw bytes; output length ==
  `data_len` (length-preserving). Returns `GMCRYPTO_OK` / `GMCRYPTO_ERR`.
- **Single failure mode**: any of null pointer, `data_len < 16`,
  `data_len > 16 MiB`, `Key1 == Key2`, or buffer-too-small →
  `GMCRYPTO_ERR` (failure-mode invariant; no distinguishing — same posture as
  the core `None`). On buffer-too-small, `*out_actual_len` receives the
  required length (== `data_len`, public).
- Byte-identical to `gmcrypto_core::sm4::mode_xts` (which is byte-identical to
  OpenSSL EVP `SM4-XTS` `xts_standard=GB`). **Confidentiality only — no auth.**
- **No new dependency**, no new `gmcrypto-core` API, **no public API breakage**
  (additive, opt-in). Default-features build of both crates byte-unchanged.
- `unsafe_code = "warn"` stays; every new `unsafe` block carries `// SAFETY:`.
  `gmcrypto-core` remains `forbid`.

## 3. Q-list (Q13.1–Q13.12) with answers

### Q13.1 — Scope: anchor only
**C FFI for SM4-XTS, nothing else.** No streaming, no per-sector batch helper,
no other v0.12-§6 candidate. Tightly-scoped, matching the v0.7→v0.12 cadence.

### Q13.2 — API shape
Two free functions mirroring single-shot `gmcrypto_sm4_gcm_encrypt`/`_decrypt`
minus nonce/AAD/tag. Signature (both directions identical shape):
```c
int gmcrypto_sm4_xts_encrypt(const uint8_t *key,    /* 32 bytes */
                             const uint8_t *tweak,  /* 16 bytes */
                             const uint8_t *data, size_t data_len,
                             uint8_t *out, size_t out_capacity,
                             size_t *out_actual_len);
```
No opaque handle (single-shot, stateless). Rejected: a `gmcrypto_sm4_xts_t`
key-scheduled handle (premature — the core API is stateless per call; a handle
is a future consumer-driven optimization, like the SM4-CBC streaming handles).

### Q13.3 — Output buffer convention
The existing `(out, out_capacity, out_actual_len)` convention via
`write_output`. Length-preserving, so callers size `out_capacity = data_len`.
On too-small buffer: `*out_actual_len = data_len`, return `GMCRYPTO_ERR`
(in-place output into `data` is **not** offered — the C ABI is uniformly
out-of-place; matches every other single-shot symbol).

### Q13.4 — Key-size constant
Add `pub const GMCRYPTO_SM4_XTS_KEY_SIZE: usize = 2 * GMCRYPTO_SM4_KEY_SIZE;`
(= 32). **Always-on** (not `#[cfg]`-gated), like the other `GMCRYPTO_SM4_*`
size constants — a bare integer is harmless to always export and cbindgen
emits `#define`s unconditionally regardless of cfg.

### Q13.5 — Feature flag
`sm4-xts = ["gmcrypto-core/sm4-xts"]` on `gmcrypto-c` (forwarding; no new
transitive dep — core's `sm4-xts` is pure-core). The two functions and the
`use gmcrypto_core::sm4::mode_xts;` import are `#[cfg(feature = "sm4-xts")]`.

### Q13.6 — Header regen (the key contrast with v0.10)
`regen-header` does **NOT** need to imply `sm4-xts`. cbindgen parses *source*
(it does not evaluate `#[cfg(feature)]` for **free functions** or `const`s), so
the two XTS prototypes + the always-on size `#define` appear in the regenerated
header **unconditionally** — exactly the v0.9 single-shot-AEAD situation.
v0.10 had to make `regen-header` imply `sm4-aead` only because it added cfg-gated
**opaque struct types** (which cbindgen *does* drop when inactive). XTS adds
**no structs**, so the existing `regen-header` (which already implies
`sm4-aead`) regenerates a complete, deterministic header with the documented
`--features regen-header` command, and the drift gate stays green.

### Q13.7 — Failure modes
Single `GMCRYPTO_ERR` for every error: null pointer, `data_len < 16`,
`data_len > 16 MiB`, `Key1 == Key2`, buffer too small. No distinguishing across
the C boundary (failure-mode invariant; padding-/weak-key-oracle defense). The
`Key1 == Key2` constant-time reject lives in core; the FFI just forwards `None`.

`key` (32 B) and `tweak` (16 B) are **fixed-size by ABI contract**: there are
no `key_len`/`tweak_len` params (matching the GCM single-shot convention), so
`try_slice(key, 32)` reads exactly 32 bytes and the subsequent `&[u8; 32]`
`try_into` is infallible — passing an undersized buffer is caller UB, not a
detectable error. The defensive `try_into` match is kept in code only to mirror
the existing GCM shape.

### Q13.8 — Testing + dudect
- `tests/c_smoke.rs`, cfg-gated on `sm4-xts`: (a) encrypt byte-equivalence vs
  `mode_xts::encrypt` + FFI round-trip on a **whole-block** data unit; (b) the
  same on a **CTS (non-block-multiple)** data unit (exercises the steal across
  the FFI boundary); (c) `data_len < 16` → `GMCRYPTO_ERR`; (d) `Key1 == Key2`
  → `GMCRYPTO_ERR`; (e) buffer-too-small → `GMCRYPTO_ERR` with
  `*out_actual_len == data_len`.
- **No new dudect target.** The FFI is a thin shim with no new secret-touching
  logic; the core path is already gated by `ct_sm4_xts_decrypt` (v0.12 W3).
- C example `examples/sm4_xts_sector.c` (doc-only; CI does not build C
  examples) — a sector-style encrypt/decrypt round-trip.

### Q13.9 — Supply chain + MSRV
No new dependency (`sm4-xts` forwards to pure-core `gmcrypto-core/sm4-xts`).
MSRV stays **1.85**. `cargo deny` opt-in pass gains
`gmcrypto-core/sm4-xts` (no new crates in the graph).

### Q13.10 — No streaming / per-sector batch FFI
Single-shot per data unit covers the disk-sector use, same as the Rust core.
A multi-sector batch or key-scheduled handle is a future consumer-driven
candidate, not v0.13.

### Q13.11 — Workstream ordering
- **W1** — `sm4-xts` feature + `GMCRYPTO_SM4_XTS_KEY_SIZE` const + the two FFI
  functions + `c_smoke` tests (TDD: write the equivalence/round-trip/error
  tests, then the symbols). Milestone codex review.
- **W2** — regenerate `include/gmcrypto.h` (drift-check) + C example +
  `gmcrypto-c/README.md` XTS section. Milestone codex review.
- **W3** — release prep (version `0.12.0`→`0.13.0`; CHANGELOG / README /
  CLAUDE.md) + full verification matrix + PR. Final codex review.

### Q13.12 — Deferred-item posture
v0.14+ candidates unchanged from v0.12 §6 minus the now-done FFI. Top of list:
**parser fuzzing** (`cargo-fuzz` on DER/PEM/PKCS#8 decode + AEAD/XTS decrypt) —
the recommended pre-v1.0 assurance gate. Also: `aead` trait fit (still
`0.6.0-rc.10`), pinned dudect runner, AVX-512 `sbox_x64`.

## 4. Hard-contract checklist (for the v0.13 PR review)

- [ ] `gmcrypto_sm4_xts_encrypt` / `_decrypt` byte-identical to core `mode_xts`.
- [ ] Single `GMCRYPTO_ERR` on every failure (null / short / long / weak-key /
      small-buffer); `*out_actual_len` set to required length on small-buffer.
- [ ] `GMCRYPTO_SM4_XTS_KEY_SIZE == 32` exported (always-on).
- [ ] `sm4-xts` forwarding feature; functions + import cfg-gated; **default
      build of `gmcrypto-c` byte-unchanged**.
- [ ] Header regenerated; `git diff --exit-code include/gmcrypto.h` clean after
      `cargo build -p gmcrypto-c --features regen-header`.
- [ ] Every new `unsafe` block has a `// SAFETY:` comment; clippy clean under
      `--features sm4-xts --all-targets -- -D warnings`.
- [ ] `cargo test -p gmcrypto-c --features sm4-xts` passes (new c_smoke tests).
- [ ] No new dependency (cargo deny opt-in pass with `gmcrypto-core/sm4-xts`).
- [ ] MSRV 1.85 build of `gmcrypto-c --features sm4-xts`.

## 5. Out of scope / deferred to v0.14+

- **Parser fuzzing** — `cargo-fuzz` on the untrusted-input decode surface; the
  recommended next assurance cycle.
- **RustCrypto `aead` trait fit** — `aead 0.6.0-rc.10`, still stalled.
- **Pinned / noise-isolated dudect runner** — restore authoritative
  `ct_fn_invert` / `ct_fp_invert` gating (`docs/v0.5-dudect-recalibration.md`).
- **AVX-512 16-way `sbox_x64`** — needs an x86_64 AVX-512 CI env.
- **SM4-XTS streaming / per-sector batch / key-scheduled FFI handle** —
  consumer-driven.
- **CCM incremental input**, **Argon2-with-SM3**, **`wasm-bindgen-test` KAT
  runner** — unchanged from v0.12 §6.
