# v0.9 — Scope doc

**Status:** Lands as v0.9 W0. Mirrors the structure of `docs/v0.7-aead-scope.md` — Q-list with provisional answers, hard contract, posture, deferred items. Six workstreams (W0 doc → W1 tag-len param → W2 incremental-input buffered AEAD for GCM → W3 dudect target → W4 single-shot AEAD FFI → W5 release prep).

This scope was reviewed by codex before landing; key concessions vs the initial proposal:

- **Streaming AEAD FFI deferred to v0.10** — ABI permanence risk before the Rust-side incremental API is battle-tested. v0.9's `gmcrypto-c` work ships only the four single-shot AEAD entry-point pairs.
- **CCM "streaming" dropped from v0.9** — CCM's CBC-MAC requires the total message length up-front in `B0`, which collapses the streaming benefit on unknown-length inputs. Re-enter in v0.10 if a real consumer surfaces.
- **Terminology pin: "incremental-input buffered AEAD"** — NOT "streaming AEAD". The decryptor still buffers the full plaintext (memory scales with message size); only the **input** is chunked. RustCrypto's segmented STREAM construction (`aead::stream`) is a different primitive and is **not** what v0.9 ships.
- **`GcmTagLen` newtype** — valid-length policy centralized in `GcmTagLen::new(usize) -> Option<Self>` instead of accepting raw `usize` throughout. Preserves the single-failure-mode invariant.

---

## 1. Context

v0.8.0 closed the AEAD core: single-shot SM4-GCM (NIST SP 800-38D / GM/T 0009 / RFC 8998) and SM4-CCM (NIST SP 800-38C / RFC 3610 / GM/T 0009), both behind the opt-in `sm4-aead` feature flag, both byte-identical to their reference oracles (gmssl 3.1.1 `sm4 -gcm` for GCM; OpenSSL 3.x EVP `SM4-CCM` for CCM — gmssl 3.1.1 lacks `-ccm`). GHASH primitive landed in `gmcrypto-simd` (CLMUL on `x86_64` / PMULL on `aarch64` / software bit-serial fallback).

v0.8 explicitly deferred three items per Q8.6 / Q8.8 / `docs/v0.7-aead-scope.md` §6:

1. **GCM tag-length parameterization** — v0.8 hardcodes 16-byte tags. NIST SP 800-38D §5.2.1.2 admits `{4, 8, 12, 13, 14, 15, 16}`-byte tags; common short-tag deployments (e.g. constrained-IoT, some VPN profiles) need this.
2. **Streaming AEAD** — v0.8 ships one-shot encrypt/decrypt only. Callsites with chunk-at-a-time framing (file-IO, stream protocols) currently must buffer the full message before invoking the AEAD primitive.
3. **C FFI for AEAD** — v0.8's `gmcrypto-c` shim covers the non-AEAD surface but not GCM/CCM. C/C++/Go/Zig callers cannot reach the AEAD primitives.

v0.9 ships items 1 and 2 (GCM only for item 2) and the single-shot part of item 3. Streaming FFI and CCM streaming are deferred to v0.10.

## 2. Hard contract

The same byte-identical invariant that v0.6, v0.7, v0.8 hold to:

- `Sm4GcmEncryptor` driven with arbitrary input chunking emits the **same `(ciphertext, tag)` pair** as the v0.8 single-shot `sm4::mode_gcm::encrypt` would on the concatenated input. Verified via differential KAT in W2.
- `Sm4GcmDecryptor` accepting arbitrary chunking on a (valid `(ciphertext, tag)`) pair returns `Some(plaintext)` byte-identical to v0.8 `sm4::mode_gcm::decrypt`. On any tag mismatch / length-ceiling violation / overflow, returns single `None` per the failure-mode invariant.
- GCM tag-length parameterization (`tag_len ∈ {4, 8, 12, 13, 14, 15, 16}`): the new `encrypt_with_tag_len` API produces output where the first `tag_len` bytes of the v0.8-spec-full-tag are the output tag, matching the truncated-tag rule in NIST SP 800-38D §5.2.1.2. KAT against gmssl `sm4 -gcm` with explicit `-tag <len>` flag for `tag_len ∈ {12, 14}` (mid-range; 16 is already covered by v0.8 KAT).
- Constant-time on key bytes and tag-comparison preserved end-to-end.
- C FFI: every error path returns `GMCRYPTO_FAILED` (single failure code per the invariant). No padding-oracle / tag-oracle surface across the C boundary.

The hard-contract bar is the same shape as v0.8's W2/W3: differential KAT against the v0.8 single-shot path (which is itself oracle-validated), plus a new dudect gate on the incremental-input decryptor.

## 3. Q-list (Q9.1–Q9.10) with provisional answers

### Q9.1 — Streaming terminology

Per codex review: "streaming AEAD" overloads three distinct primitives —

- (a) Raw `update(chunk) → output_chunk` + `finalize() → tag` (output-streaming encrypt). For decrypt this is a chosen-plaintext-distinguisher vector unless the caller is contract-bound not to release pre-tag bytes.
- (b) **Incremental-input buffered AEAD** — `update(chunk)` accepts arbitrary chunking on input; output bytes are buffered internally; `finalize_verify(tag) → Option<Vec<u8>>` releases the full plaintext **only on successful tag verify**. Memory scales with plaintext.
- (c) Segmented STREAM AEAD (RustCrypto `aead::stream`) — framed/segmented construction with per-segment verification. Different primitive; not interchangeable with raw GCM.

**Provisional decision: v0.9 ships only (b) — "incremental-input buffered AEAD" for SM4-GCM.** The decryptor never returns pre-verify plaintext bytes; the failure-mode invariant is preserved. Documentation must use this term consistently; the word "streaming" appears only as a contrast in the scope doc and rust-doc.

### Q9.2 — Decryptor commit-on-verify shape

Two API shapes considered:

(a) `update(chunk) → ()`; `finalize_verify(tag) → Option<Vec<u8>>` returns the entire plaintext or `None`.
(b) `update(chunk) → TentativeBytes`; the caller is contract-bound not to release the returned bytes until a separate `verify_tag(tag) → bool` succeeds.

(a) is fail-safe — the API itself prevents a caller from accidentally releasing pre-tag bytes. (b) is faster on large messages (output-streaming) but trades the project's failure-mode invariant for performance.

**Provisional decision: shape (a).** Memory cost is acceptable for v0.9's use cases (file-blob decrypt + stream-protocol framing where messages fit in memory). Document the memory implication in the rust-doc. v0.10+ may revisit shape (b) if a concrete consumer surfaces with a hard memory ceiling that shape (a) violates.

### Q9.3 — GCM tag-length parameterization

NIST SP 800-38D §5.2.1.2 admits `tag_len ∈ {4, 8, 12, 13, 14, 15, 16}` (32, 64, 96, 104, 112, 120, 128 bits). Three API shapes considered:

(i) New top-level fn `encrypt_with_tag_len(key, nonce, aad, pt, tag_len) → Option<(Vec<u8>, Vec<u8>)>` (variable-length tag).
(ii) `GcmTagLen` newtype with `GcmTagLen::new(usize) → Option<Self>` constructor; the type is the only way to express a valid tag length.
(iii) Const-generic `<const TAG_LEN: usize>` on the function.

Codex pushed back on raw-`usize` (i): the valid-length policy gets scattered across the API surface. Codex's recommended tweak: combine (i) and (ii) — use `GcmTagLen` as the parameter type. **Provisional decision: shape (i) with `GcmTagLen` as the parameter (codex's combined recommendation).**

```rust
pub struct GcmTagLen(usize);
impl GcmTagLen {
    pub const fn new(bytes: usize) -> Option<Self> {
        match bytes {
            4 | 8 | 12 | 13 | 14 | 15 | 16 => Some(Self(bytes)),
            _ => None,
        }
    }
    pub const fn as_usize(self) -> usize { self.0 }
}

pub fn encrypt_with_tag_len(
    key: &[u8; KEY_SIZE],
    nonce: &[u8],
    aad: &[u8],
    pt: &[u8],
    tag_len: GcmTagLen,
) -> (Vec<u8>, Vec<u8>);  // tag is variable-length; first tag_len bytes of full tag
```

The existing v0.8 `encrypt(...) -> (Vec<u8>, [u8; 16])` and `decrypt(..., tag: &[u8; 16])` APIs stay; v0.9 adds the parameterized variants alongside them (non-breaking).

### Q9.4 — Length ceilings + overflow paths

NIST SP 800-38D §5.2.1.1 caps GCM plaintext at `2^39 − 256` bits (≈ 64 GiB minus 32 bytes; 2^36 − 32 bytes). AAD capped at `2^64 − 1` bits. Counter rollover protection in `GCTR` (32-bit counter wraps at 2^32 blocks ≈ 64 GiB per the same bound — the bounds are consistent).

For v0.9's incremental-input variant the running total must be tracked and the encryptor / decryptor must return single `None` on overflow. Per codex's hidden-risk call-out: **add explicit length-ceiling state to the encryptor / decryptor structs, check on every `update`, and unit-test the overflow path returns `None`.**

CCM lengths are bounded by the `q` parameter in `B0` (per RFC 3610 §2.2); v0.9's CCM work stays single-shot so the existing v0.8 length checks suffice.

**Provisional decision: v0.9 W2 ships explicit `running_pt_len: u64` + `running_aad_len: u64` fields in `Sm4GcmEncryptor` and `Sm4GcmDecryptor`. Per-`update` checked-add returns single `None` on overflow. Unit tests cover both the AAD-overflow and PT-overflow paths.**

### Q9.5 — Dudect targets

Mirror the v0.8 W4 pattern. Per Q9.1's terminology pin, the target is named `ct_sm4_gcm_decrypt_buffered` (not `_streaming`).

- `ct_sm4_gcm_decrypt_buffered` — class-split by master key. Both classes' `(chunked_ct, tag)` tuples are valid encrypts under their own keys; both reach `finalize_verify` via identical control flow. Gate `|tau| < 0.20`. cfg-gated on `sm4-aead`.
- Single target (not two). CCM has no v0.9-shipped incremental variant so no `ct_sm4_ccm_decrypt_buffered` target.

CI matrix entries already cover `sm4-aead`; no new matrix slot needed.

**Provisional decision: 1 new dudect target (`ct_sm4_gcm_decrypt_buffered`), cfg-gated on `sm4-aead`. PR-smoke + nightly workflows already pick up the feature in the existing matrix.**

### Q9.6 — `gmcrypto-c` AEAD FFI scope

Per codex review: streaming AEAD FFI is the *highest* risk in this cycle because C ABI is permanent and the Rust-side streaming semantics are unproven. Defer streaming FFI to v0.10.

v0.9 ships single-shot AEAD FFI only:

- `gmcrypto_sm4_gcm_encrypt(key, nonce, nonce_len, aad, aad_len, pt, pt_len, ct_out, tag_out) -> int` (full 16-byte tag).
- `gmcrypto_sm4_gcm_decrypt(key, nonce, nonce_len, aad, aad_len, ct, ct_len, tag, pt_out) -> int`.
- `gmcrypto_sm4_gcm_encrypt_with_tag_len(..., tag_len, tag_out) -> int`.
- `gmcrypto_sm4_gcm_decrypt_with_tag_len(..., tag, tag_len, pt_out) -> int`.
- CCM single-shot equivalents (4 more entries; CCM already takes `tag_len` so no `_with_tag_len` split): `gmcrypto_sm4_ccm_encrypt` / `_decrypt`.
- Total: 6 new FFI symbols.

Every error path returns `GMCRYPTO_FAILED`. cbindgen header regen + drift gate. C smoke tests live alongside v0.8's `c_smoke.rs` patterns.

**Provisional decision: 6 single-shot AEAD FFI symbols. Streaming FFI deferred to v0.10.**

### Q9.7 — Failure-mode invariant

Same as v0.8 (Q8.5): `Sm4GcmDecryptor::finalize_verify` returns `Option<Vec<u8>>`. `None` covers: tag mismatch, length-ceiling overflow, nonce-length-out-of-range (set at constructor). No distinguishing variants.

`GcmTagLen::new(usize) -> Option<Self>` returns `None` on invalid length. Single failure mode preserved.

C FFI: every error path returns `GMCRYPTO_FAILED`. No distinguishing-error codes.

**Provisional decision: unchanged from v0.8.** Single `None` / `GMCRYPTO_FAILED`. PRs that distinguish failure modes get rejected on sight per the project posture.

### Q9.8 — Feature flag shape

v0.8 introduced `sm4-aead` (additive, default-off). v0.9's additions:

- Tag-length parameterization (W1) — same `sm4-aead` flag.
- Incremental-input GCM (W2) — same `sm4-aead` flag.
- Single-shot AEAD FFI (W4) — gated on `gmcrypto-c`'s existing dependency chain. The `gmcrypto-core` `sm4-aead` feature transitively enables the AEAD FFI surface in `gmcrypto-c` (matching the existing pattern where `gmcrypto-c` re-exposes whatever's enabled in core).

No new feature flag introduced in v0.9.

**Provisional decision: `sm4-aead` remains the single feature gate.**

### Q9.9 — Streaming-encryptor `update` return type

The encryptor (unlike the decryptor) can safely output ciphertext chunks as they're computed — GCM's `GCTR` is just a counter mode over the block cipher, and the running GHASH state lets the encryptor commit each ciphertext byte to the tag deterministically. So the encryptor can be output-streaming:

```rust
impl Sm4GcmEncryptor {
    pub fn update(&mut self, chunk: &[u8]) -> Vec<u8>;     // emits CT chunk
    pub fn finalize(self) -> [u8; 16];                      // emits tag
    // or, with tag-len param:
    pub fn finalize_with_tag_len(self, tag_len: GcmTagLen) -> Vec<u8>;
}
```

Asymmetry note: the *encryptor* is output-streaming, the *decryptor* is input-incremental + output-buffered. This matches the inherent asymmetry of AEAD decrypt's commit-on-verify property. **Document this asymmetry prominently in the module docstring** — it's a common source of API confusion.

**Provisional decision: encryptor is output-streaming (returns `Vec<u8>` per `update`); decryptor is input-incremental, output-buffered.** Asymmetry documented in the module docstring + rust-doc.

### Q9.10 — Workstream ordering

Codex confirmed W1 (tag-len param) before W2 (incremental-input GCM): the tag policy affects `Sm4GcmEncryptor::finalize` and `Sm4GcmDecryptor::finalize_verify` signatures, plus the FFI shape in W4 and the dudect coverage in W3. Retrofitting tag-len into W2's types post-hoc would be expensive.

**Provisional decision: W0 (doc) → W1 (tag-len) → W2 (incremental GCM) → W3 (dudect) → W4 (single-shot AEAD FFI) → W5 (release prep).** Six workstreams, six PRs.

## 4. Hard-contract checklist (for v0.9 PR review)

- [ ] `GcmTagLen::new` accepts `{4, 8, 12, 13, 14, 15, 16}` only; returns `None` otherwise.
- [ ] `sm4::mode_gcm::encrypt_with_tag_len` produces output where the first `tag_len.as_usize()` bytes of the full 16-byte tag are the output tag (NIST SP 800-38D §5.2.1.2 truncation rule).
- [ ] `sm4::mode_gcm::decrypt_with_tag_len` constant-time-compares the truncated tag via `subtle::ConstantTimeEq`.
- [ ] KAT for `tag_len ∈ {12, 14}` against gmssl 3.1.1 `sm4 -gcm` (bidirectional interop).
- [ ] `Sm4GcmEncryptor` driven with arbitrary input chunking emits the same `(ct, tag)` as the v0.8 single-shot `encrypt` would. Differential KAT covers chunking patterns `{1, 7, 15, 16, 17, 31, 32, 33, 100, full}` bytes.
- [ ] `Sm4GcmDecryptor::finalize_verify` returns `Some(plaintext)` only on tag-verify success; `None` otherwise.
- [ ] `Sm4GcmEncryptor` / `Sm4GcmDecryptor` enforce GCM length ceilings (`pt_len ≤ 2^36 − 32`, `aad_len ≤ 2^61 − 1` bytes); overflow returns single `None`. Unit-tested.
- [ ] `Sm4GcmEncryptor` is output-streaming (returns CT chunks from `update`); `Sm4GcmDecryptor` is input-incremental + output-buffered. Asymmetry documented in module docstring.
- [ ] `ct_sm4_gcm_decrypt_buffered` dudect target lands at `|tau| < 0.20` on PR-smoke + nightly under `--features sm4-aead`.
- [ ] `gmcrypto-c` ships 6 new single-shot AEAD FFI entry points. cbindgen header regen runs only under `regen-header` feature; CI gates header drift via `git diff --exit-code`.
- [ ] C smoke test exercises each new FFI entry point against the Rust-side single-shot AEAD.
- [ ] `cargo build -p gmcrypto-core --no-default-features` unchanged (no_std posture).
- [ ] `cargo build -p gmcrypto-core --target wasm32-unknown-unknown --no-default-features --features sm4-aead` compiles.
- [ ] `cargo deny check --exclude-dev` clean under `--features gmcrypto-core/sm4-aead`.

## 5. Out of scope for v0.9 / explicitly deferred to v0.10+

- **CCM incremental input** — CBC-MAC's total-length-up-front in `B0` collapses the streaming benefit for unknown-length inputs (Q9.1 / codex review). Re-enter in v0.10 if a real consumer surfaces.
- **Streaming AEAD FFI** — ABI permanence risk before the Rust streaming API is battle-tested (Q9.6 / codex review). v0.9's `gmcrypto-c` work covers single-shot only.
- **RustCrypto `aead = 0.6` trait fit** — `aead` is at `0.6.0-rc.10` (10 release candidates in, no stable). Defer until upstream stabilizes.
- **Pinned / noise-isolated dudect runner** (Q9.1 from `docs/v0.7-aead-scope.md` §6 — collides with the new Q9.x numbering; refer to it as `v0.7-Q9.1` to disambiguate).
- **AVX-512 16-way `sbox_x64`** — performance-only; no user-visible API surface change. Defer.
- **`digest 0.11` / `cipher 0.5` migration** — blocked on upstream.
- **`wasm-bindgen-test`-driven KAT runner** — tooling, not crypto.
- **SM4-XTS** — new mode; substantial new surface; defer to v0.10+ with its own scope doc cycle.
- **Argon2-with-SM3 alternative KDF** — research-only; check downstream demand first.
- **Output-streaming `Sm4GcmDecryptor`** (raw `update` returning `TentativeBytes` — Q9.2 shape (b)). Re-enter if a hard-memory-ceiling consumer surfaces.
