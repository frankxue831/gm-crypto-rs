# v0.12 — Scope doc

**Status:** Lands as v0.12 W0. Mirrors the structure of `docs/v0.11-scope.md`
— Q-list with provisional answers, hard contract, posture, deferred items.
**Anchor: SM4-XTS** — a new tweakable block-cipher mode for disk/sector
encryption (IEEE Std 1619-2007, generalized from XTS-AES to SM4; GM/T OID
`1.2.156.10197.1.104.10`), single-shot, with **full ciphertext stealing**,
byte-identical to OpenSSL 3.x EVP `SM4-XTS`. Cashes in the candidate listed as
`Q12.4` in `docs/v0.11-scope.md` §6 (`v0.10-Q11.5`). Five workstreams:
W0 (this doc) → W1 KAT sourcing → W2 `mode_xts` core → W3 dudect + CI +
verification → W4 release prep.

Top-level decisions baked into the Q-list below:

- **Anchor only.** v0.12 ships *only* SM4-XTS in `gmcrypto-core`. The
  RustCrypto `aead` trait fit stays deferred (re-verified `0.6.0-rc.10` on
  2026-05-23, no upstream movement). Infra/perf items (pinned dudect runner,
  AVX-512 `sbox_x64`) and other consumer-driven items (CCM incremental input)
  ride their own later cycles. **C FFI for XTS is deferred to v0.13** — the
  v0.8 AEAD-core precedent (land the Rust core, let the API soak, add the C ABI
  next cycle). Tightly-scoped release, matching the v0.7 → v0.11 cadence.
- **Full ciphertext stealing — not whole-block-only.** Data units of any
  length ≥ 16 bytes are supported, including non-block-multiples via the
  standard XTS last-two-block ciphertext-stealing swap. This is what makes the
  mode standard-complete and what lets every OpenSSL EVP vector validate
  directly. Whole-block-only was rejected (non-standard, surprises callers,
  can't validate non-aligned vectors).
- **Pure `gmcrypto-core`, no new dependency.** XTS's per-block tweak update is
  a GF(2¹²⁸) multiply by the primitive element α — a trivial ×2-with-carry,
  **not** GHASH's full multiply (which is bit-reversed/big-endian and lives in
  `gmcrypto-simd`). So XTS needs **no `gmcrypto-simd` dependency**; the α-doubling
  lives in-core. The whole-block bulk rides the existing
  `Sm4Cipher::encrypt_blocks` batch API (v0.7 W1), which transparently picks up
  the v0.6 SIMD fanout when `sm4-bitsliced-simd` is also enabled.
- **Zero default-build impact.** XTS is opt-in behind a new `sm4-xts` feature.
  The `#![no_std]` default-features build is byte-for-byte unaffected; no new
  always-on dependency; no change to any existing module.
- **Constant-time discipline on the tweak.** The XTS tweak `T = SM4_E(Key2,
  tweak)` is **secret-derived** (Key2 is secret). The α-doubling's carry
  reduction must be a **masked** XOR, never an `if` on `T`. Branching on the
  *length* of the data unit is fine — length is public input, like every other
  mode. A new dudect target `ct_sm4_xts_decrypt` gates this.

## 1. Context

The cipher-modes arc has shipped, one mode-family per cycle, on the v0.7 W1
batch API (`Sm4Cipher::encrypt_blocks` / `decrypt_blocks`):

- **v0.7** — SM4-CTR (single-shot + streaming).
- **v0.8** — AEAD core: SM4-GCM + SM4-CCM single-shot (+ the GHASH primitive
  in `gmcrypto-simd`).
- **v0.9 / v0.10** — AEAD ergonomics + streaming AEAD FFI.
- **v0.11** — RustCrypto `digest 0.11` / `cipher 0.5` trait-fit modernization
  (no new mode).

SM4-XTS is the natural next mode: tweakable, length-preserving (no expansion,
no tag), built for random-access sector/disk encryption. It is the GM-T
counterpart to the ubiquitous AES-XTS used by dm-crypt / BitLocker / FileVault.
Unlike the AEAD modes, XTS provides **confidentiality only** — it is *not*
authenticated. That is by design for the disk-sector threat model (no room for
a tag in a fixed-size sector), and the module docstring will state the
non-authentication contract loudly.

The building blocks already exist: `Sm4Cipher` block ops + the batch API, the
`subtle` constant-time toolkit, the `Option`-return failure-mode discipline,
and the `mode_ctr` / `mode_ccm` module templates. The one genuinely new
primitive is the GF(2¹²⁸) α-doubling of the tweak — small, in-core, and
constant-time by construction.

### KAT sourcing reality (resolved at W0)

The project's KAT north star is "byte-identical to gmssl 3.1.1." For XTS that
star is unavailable, exactly as it was for SM4-CCM in v0.8:

- **gmssl 3.1.1 has no XTS** (`sm4` supports only `-cbc|-ctr|-gcm|
  -cbc_sm3_hmac|-ctr_sm3_hmac`). So there is **no gmssl interop test** for XTS.
- **OpenSSL 3.x registers SM4-XTS** — verified on OpenSSL 3.6.2:
  `{ 1.2.156.10197.1.104.10, SM4-XTS } @ default`. The `openssl enc` CLI
  blocks XTS ("enc XTS ciphers not supported"), but the cipher is reachable via
  the **EVP API**. So KAT vectors are sourced via a small C oracle harness
  (`tests/data/sm4_xts_oracle.c`, mirroring v0.8's `sm4_ccm_oracle.c`), and the
  validation surface is "byte-identical to OpenSSL 3.x EVP `SM4-XTS`."

This mirrors the v0.8 W0 CCM sourcing decision precisely; `docs/v0.8-ccm-kat-
sourcing.md` is the template for `docs/v0.12-xts-kat-sourcing.md` (W1).

## 2. Hard contract

- **Byte-identical to OpenSSL 3.x EVP `SM4-XTS`.** Every KAT vector — whole-block
  and ciphertext-stealing — produced by the W1 oracle harness must match our
  output byte-for-byte. A mismatch is a bug, full stop.
- **Confidentiality only — no authentication.** XTS provides no integrity tag.
  `decrypt` returning `Some` does **not** mean the ciphertext is authentic. The
  module docstring states this prominently; callers needing integrity use an
  AEAD mode (GCM/CCM) instead. This is not a failure-mode the API hides — it is
  the defining property of the mode.
- **Tweak-uniqueness contract (caller-owned).** The caller MUST supply a
  **unique** 16-byte tweak per data unit under a given key — in disk use, the
  tweak is the data-unit (sector) number. Reusing a `(key, tweak)` pair across
  different plaintexts leaks equality structure (XTS is deterministic). The
  encoding of a sector number into the 16-byte tweak (endianness, width) is the
  caller's responsibility; this module takes the raw 16 bytes as-is. The module
  docstring states the uniqueness contract loudly (same posture as the CTR/GCM
  nonce contracts).
- **Full ciphertext stealing.** Data units of any length ≥ 16 bytes encrypt and
  decrypt correctly, round-trip to identity, and match OpenSSL.
- **Data-unit length bounds.** `16 ≤ data_unit.len() ≤ 2²⁰ · 16` (16 MiB) — the
  IEEE 1619 / NIST SP 800-38E ceiling of 2²⁰ blocks per data unit, which
  OpenSSL also enforces. Lengths outside this range → `None` (input
  validation).
- **Single `None` failure mode.** `encrypt` / `decrypt` return
  `Option<Vec<u8>>`; `None` is returned only for input-validation failures
  (`data_unit.len() < 16` or `> 16 MiB`, or weak key `Key1 == Key2`). No
  distinguished error variants — per the project's failure-mode invariant. XTS
  has no tag, so there is no MAC-failure path at all.
- **Constant-time discipline.** The α-doubling carry is a masked XOR, never a
  branch on the secret-derived tweak `T`. The `Key1 == Key2` weak-key check
  uses `subtle::ConstantTimeEq`. Branching on the public data-unit *length* is
  permitted. The dudect target `ct_sm4_xts_decrypt` gates the discipline.
- **Default-features build unchanged.** XTS lives behind the new opt-in
  `sm4-xts` feature; `cargo build -p gmcrypto-core --no-default-features` and
  the wasm32 target build are byte-for-byte unaffected. No new always-on
  dependency; no `gmcrypto-simd` dependency added by this feature.
- **MSRV stays 1.85.** No new dependency; nothing moves the MSRV.
- **Additive only.** No existing public API changes; no existing module is
  touched except the `pub mod` line in `sm4/mod.rs` and the feature/version
  bookkeeping. v0.11's other invariants (`unsafe_code = "forbid"` on
  `gmcrypto-core`, `Cargo.lock` gitignored) are untouched.

## 3. Q-list (Q12.1–Q12.13) with provisional answers

### Q12.1 — Scope: anchor only

SM4-XTS in `gmcrypto-core` is the sole feature of v0.12. The `aead` trait fit
(`v0.11-Q12.1`) stays deferred — `aead` re-verified `0.6.0-rc.10` (latest
stable `0.5.2`) on 2026-05-23, unchanged since 2026-02-04. Infra/perf items
(pinned dudect runner [`v0.11-Q12.2`], AVX-512 `sbox_x64` [`v0.11-Q12.3`]) and
CCM incremental input [`v0.11-Q12.5`] each warrant their own cycle. **C FFI for
XTS is explicitly deferred to v0.13** (Q12.11).

**Provisional decision: anchor only — SM4-XTS Rust core, full stop.**

### Q12.2 — Ciphertext stealing vs. whole-block-only

Support arbitrary data-unit lengths ≥ 16 bytes (full CTS), or restrict to
multiples of 16? Real-world sectors (512 / 4096 bytes) are block-multiples, so
whole-block-only covers the common case — but it is non-standard-complete,
surprises callers who expect standard XTS, and cannot validate the non-aligned
OpenSSL vectors. A correctness-first library ships the full standard.

**Provisional decision: full ciphertext stealing. Data units of any length
in `[16, 2²⁰·16]` bytes (16 B .. 16 MiB); the last partial block is handled by
the standard XTS last-two-block steal. Lengths < 16 (XTS undefined for sub-block
data units) or > 16 MiB (IEEE 1619 / SP 800-38E 2²⁰-block ceiling, also
enforced by OpenSSL) → `None`.**

### Q12.3 — Public API shape

`mode_ctr` takes `key: &[u8; KEY_SIZE]` + a 16-byte counter; `mode_ccm` returns
`Option<Vec<u8>>`. XTS needs 32 bytes of key (`Key1 ‖ Key2`) and a 16-byte
tweak (the data-unit number). The 32-byte combined key + raw 16-byte tweak maps
1:1 to OpenSSL EVP's `(key=32, IV=16)` inputs, so KAT vectors validate without
a translation layer. Single-shot per data unit — XTS is inherently
random-access; streaming is not a natural fit and is not in scope.

```rust
pub const XTS_KEY_SIZE: usize = 32;

pub fn encrypt(
    key: &[u8; XTS_KEY_SIZE],
    tweak: &[u8; BLOCK_SIZE],   // BLOCK_SIZE = 16; the data-unit number
    data_unit: &[u8],           // len >= 16
) -> Option<Vec<u8>>;

pub fn decrypt(
    key: &[u8; XTS_KEY_SIZE],
    tweak: &[u8; BLOCK_SIZE],
    data_unit: &[u8],
) -> Option<Vec<u8>>;
```

The tweak is consumed as **raw 16 bytes** — the caller owns any sector-number→
tweak encoding (endianness/width) and the per-data-unit uniqueness contract
(§2). `data_unit.len()` is validated to `[16, 16 MiB]` (§2 / Q12.5).

**Provisional decision: 32-byte combined key + raw 16-byte tweak + single-shot
`Option<Vec<u8>>`. `XTS_KEY_SIZE = 32`; reuse `BLOCK_SIZE = 16` for the tweak.
Tweak encoding + uniqueness are caller-owned; length bounded `[16, 16 MiB]`.**

### Q12.4 — KAT sourcing

gmssl 3.1.1 lacks XTS; OpenSSL 3.x EVP has `SM4-XTS` (OID
`1.2.156.10197.1.104.10`) reachable only via the EVP API (the `enc` CLI blocks
XTS). Source vectors via a C oracle harness (`tests/data/sm4_xts_oracle.c`,
`EVP_CIPHER_fetch("SM4-XTS")`), mirroring v0.8's `sm4_ccm_oracle.c`. Fixtures
land as `tests/data/sm4_xts_kat.bin` (the v0.3 W2 binary-fixture pattern). The
W1 deliverable `docs/v0.12-xts-kat-sourcing.md` records provenance (OpenSSL
version, OID, C source, command lines, expected outputs) and a reproducibility
recipe. **No gmssl interop test** (gmssl lacks the mode).

Oracle pinning (codex review): the harness fetches `SM4-XTS` from OpenSSL's
**default (non-FIPS) provider** — a `fips=yes` context may refuse to fetch XTS
(and would reject `Key1 == Key2`). Each vector is a **single-shot**
`EVP_EncryptUpdate` (no `EVP_EncryptFinal` padding; XTS is length-preserving,
no block padding). The OpenSSL IV (16 bytes) is the **raw tweak**, matching our
API 1:1. The weak-key (`Key1 == Key2`) and length-bound (`<16`, `>16 MiB`)
rejection cases are **local** assertions (our `None` contract), **not** oracle
KATs — OpenSSL rejects those inputs, so there is no ciphertext to compare.

**Provisional decision: OpenSSL 3.x EVP `SM4-XTS` (default/non-FIPS provider,
single-shot, padding off, IV = raw tweak) via a C oracle harness; binary KAT
fixtures + sourcing doc in W1; weak-key/length rejections tested locally; no
gmssl interop.**

### Q12.5 — Algorithm + ciphertext-stealing semantics

Per IEEE 1619 with SM4 as the block cipher:

1. Split `key` into `Key1 = key[..16]` (data), `Key2 = key[16..]` (tweak).
2. `T = SM4_E(Key2, tweak)` — the initial tweak for block 0.
3. For each 16-byte block `j`: `C_j = SM4_E(Key1, P_j ⊕ T_j) ⊕ T_j`, then
   `T_{j+1} = T_j · α`.
4. **α-doubling** (`T · α`): a GF(2¹²⁸) multiply by the primitive element,
   little-endian byte order, reducing polynomial `x¹²⁸ + x⁷ + x² + x + 1`.
   Shift the 128-bit value left by one bit; if the bit shifted out of bit 127
   is 1, XOR `0x87` into byte 0. (This is **not** GHASH's multiply.)
5. **Ciphertext stealing (encrypt)** for `len = 16·q + r`, `r > 0`: encrypt the
   first `q-1` whole blocks normally; for the final full+partial pair —
   `CC = SM4_E(Key1, P_{q-1} ⊕ T_{q-1}) ⊕ T_{q-1}`; the last (partial)
   ciphertext is `CC[..r]`; `PP = P_partial ‖ CC[r..]`;
   `C_{q-1} = SM4_E(Key1, PP ⊕ T_q) ⊕ T_q`. Output order is
   `C_0 … C_{q-2} ‖ C_{q-1} ‖ CC[..r]`.
   **Ciphertext stealing (decrypt)** — explicit steps (not just "swap tweaks"):
   decrypt the first `q-1` whole blocks normally; then
   `PP = SM4_D(Key1, C_{q-1} ⊕ T_q) ⊕ T_q` (the second-to-last *ciphertext*
   block, under `T_q`); the last (partial) plaintext is `PP[..r]`;
   `CC = C_partial ‖ PP[r..]`; `P_{q-1} = SM4_D(Key1, CC ⊕ T_{q-1}) ⊕ T_{q-1}`
   (under `T_{q-1}`).
6. **Weak-key rejection:** IEEE 1619 mandates `Key1 ≠ Key2`; reject equal
   halves with `None` (constant-time compare — Q12.6). Length bounds
   `16 ≤ len ≤ 2²⁰·16` (16 MiB) else `None` (§2).

The exact CTS block indexing is pinned against the OpenSSL oracle during W2;
the non-block-multiple KAT vectors (e.g. 17, 31, 100, 4101 bytes) are the
acceptance gate.

**Provisional decision: IEEE 1619 XTS-SM4 with little-endian α-doubling
(poly `…+x⁷+x²+x+1`, reduce constant `0x87`) and standard last-two-block CTS
(tweaks swapped on decrypt); `Key1 ≠ Key2` enforced; min length 16.**

### Q12.6 — Constant-time posture

- **α-doubling carry.** `T = SM4_E(Key2, tweak)` is secret-derived (Key2 is
  secret), so the carry reduction is a **masked** XOR, never an `if`:
  `t[0] ^= 0x87 & carry.wrapping_neg()` (carry ∈ {0,1} → mask 0x00/0xFF). No
  branch on any byte of `T`.
- **Weak-key check.** `Key1 == Key2` via `subtle::ConstantTimeEq` over the two
  16-byte halves. The boolean *outcome* gates the reject (a usage error,
  acceptable to reveal); the comparison itself leaks no byte positions.
- **Length branching is public.** `q`, `r`, and the CTS branch derive from
  `data_unit.len()`, which is public input — branching on it is fine, exactly
  as in every existing mode.
- **Zeroize.** Intermediate tweak/working buffers on the secret path are
  zeroized where the existing modes do (consistent with `mode_ccm` /
  `decrypt.rs` discipline).
- **Dudect.** A new target `ct_sm4_xts_decrypt` (Q12.9), class-split by key,
  gates the discipline at `|tau| < 0.20`.

**Provisional decision: masked α-doubling carry; `subtle` weak-key compare;
length branching public; `ct_sm4_xts_decrypt` added.**

### Q12.7 — Block processing: batched, no `gmcrypto-simd` dependency

The whole-block bulk goes through `Sm4Cipher::encrypt_blocks` /
`decrypt_blocks` (v0.7 W1): precompute the tweak sequence (cheap sequential
doublings), XOR into all plaintext blocks, one batch cipher call (which rides
the v0.6 SIMD fanout under `sm4-bitsliced-simd`), then XOR the tweaks back. The
CTS tail (≤ 2 blocks) is handled with single-block calls. The α-doubling is a
trivial ×2, so XTS does **not** depend on `gmcrypto-simd`'s GHASH — unlike
`sm4-aead`, the `sm4-xts` feature pulls in **no** sibling crate.

**Provisional decision: batched whole-block path via the existing batch API;
CTS tail single-block; α-doubling in-core; no `gmcrypto-simd` dependency.**

### Q12.8 — Feature flag

A new opt-in `sm4-xts = []` feature (pure-core, no `dep:` entries), default-off,
gating `pub mod mode_xts;` in `sm4/mod.rs`. Mirrors the `sm4-bitsliced` /
`sm4-aead` opt-in posture; default-features build unchanged.

**Provisional decision: `sm4-xts = []`, default-off; `#[cfg(feature =
"sm4-xts")] pub mod mode_xts;` in `sm4/mod.rs`.**

### Q12.9 — Testing + dudect target

- **`tests/sm4_xts_kat.rs`** (cfg `sm4-xts`): byte-identical to the W1 OpenSSL
  oracle across whole-block (32, 512 B), CTS lengths (17, 31, 100, 4101 B),
  min length 16, encrypt→decrypt round-trip, and `len < 16 → None` +
  `Key1 == Key2 → None`.
- **Round-trip property test** exhaustively over `len ∈ 16..=80` plus a few
  large lengths, asserting `decrypt(encrypt(x)) == x`.
- **Dudect `ct_sm4_xts_decrypt`** cfg-gated on `sm4-xts`, class-split by master
  key (both classes' data units are valid encrypts under their own key, so both
  decrypt via identical control flow — the `ct_sm4_ctr` / `ct_sm4_gcm` shape).
  Fixed tweak + a fixed **CTS (non-block-multiple) data-unit length**
  (e.g. 100 bytes = 6 whole blocks + 4) — **mandatory, not budget-optional**
  (codex review): a whole-block-only length never executes the final-pair
  `T_{q-1}/T_q` CTS path, so the partial-block branch (the riskiest tweak
  arithmetic) would go ungated. Gate `|tau| < 0.20` at both 10K (PR-smoke) and
  100K (nightly).
- **CI dudect matrix** gains an `sm4-xts`-bearing slot so the target gates; the
  `required_low` dict adds `ct_sm4_xts_decrypt = 0.20` under an `"sm4-xts"`
  `MATRIX_FEATURES` substring match.

**Provisional decision: `sm4_xts_kat.rs` (OpenSSL-equal scenarios, weak-key +
length-bound rejections local) + round-trip property test + `ct_sm4_xts_decrypt`
dudect target at a **CTS (non-block-multiple) length** + a CI matrix slot
carrying `sm4-xts`.**

### Q12.10 — Supply chain + MSRV

XTS adds **no new dependency** (pure-core; no `gmcrypto-simd`, no new
transitive crate). `deny.toml`'s allowlist is unchanged; only the `cargo deny
--features …` invocation gains `gmcrypto-core/sm4-xts` so the feature is
exercised. MSRV stays 1.85 (nothing new to raise it). The `cargo +1.85 build`
gate adds `sm4-xts` to its feature list.

**Provisional decision: no allowlist change; `sm4-xts` added to the `cargo
deny` and `cargo +1.85 build` feature lists; MSRV 1.85 unchanged.**

### Q12.11 — C FFI deferred to v0.13

Following the v0.8 precedent (AEAD core shipped with no FFI; the C ABI landed
in v0.9/v0.10 once the Rust API soaked), the ~2 XTS C ABI symbols
(`gmcrypto_sm4_xts_encrypt` / `_decrypt`) are **deferred to v0.13**. Bundling
them would freeze a C ABI before the Rust API has any real-world use.
`gmcrypto-c` gets only its path-dep version bump in W4 (no new symbols, no
header regen).

**Provisional decision: no XTS C FFI in v0.12; deferred to v0.13.**

### Q12.12 — Workstream ordering

XTS needs its KAT fixtures **before** the impl (TDD), exactly as CCM did in
v0.8. So sourcing is its own workstream ahead of the implementation.

**Provisional decision: W0 (this doc) → W1 (KAT sourcing: oracle harness +
`sm4_xts_kat.bin` fixtures + `docs/v0.12-xts-kat-sourcing.md`) → W2 (`mode_xts`
encrypt/decrypt + full CTS + constant-time α-doubling + `sm4-xts` feature +
`sm4_xts_kat.rs`, TDD against W1 fixtures) → W3 (`ct_sm4_xts_decrypt` dudect +
CI matrix slot + full cross-feature clippy/deny/MSRV/wasm verification) → W4
(release prep `0.11.0 → 0.12.0`). Five workstreams; codex milestone reviews at
W1 (sourcing), W2 (core), W3 (full diff).**

### Q12.13 — Deferred-item posture

Restated for the record so the deferral is auditable:

- **C FFI for SM4-XTS** — deferred to v0.13 (Q12.11).
- **`aead` trait fit** (`v0.11-Q12.1`): `aead 0.6.0-rc.10`, no upstream movement
  since 2026-02-04, latest stable `0.5.2`. v0.11 already put the crate on the
  `crypto-common 0.2` line `aead 0.6` will want. Recheck next cycle.
- **Pinned / noise-isolated dudect runner** (`v0.11-Q12.2`): CI-provisioning;
  own infra cycle (needs a dedicated noise-isolated runner box). Strong v0.13
  candidate.
- **AVX-512 `sbox_x64`** (`v0.11-Q12.3`): perf-only; needs x86_64 AVX-512
  hardware the current macOS aarch64 self-hosted runner lacks — first milestone
  must be a validation environment, not the code.
- **CCM incremental input** (`v0.11-Q12.5`), **Argon2-with-SM3**
  (`v0.11-Q12.6`), **`wasm-bindgen-test` KAT runner** (`v0.11-Q12.7`):
  unchanged deferral.

**Provisional decision: all remain deferred; no v0.12 work.**

  `0.6.0-rc.10` (stalled) and re-deferred. Re-open via a "Q12.x amendment"
  section appended below if a workstream surfaces a blocker (e.g. the OpenSSL

  implementation — the v0.8 W0 fallback posture).
- **2026-05-23 (codex review)**: `codex exec` reviewed this doc. **No BLOCKER** —
  the cryptographic core (little-endian α-doubling with reduce constant `0x87`,
  the last-two-block CTS construction, the secret-derived-tweak constant-time
  posture, the min-length-16 / key-as-secret treatment) confirmed correct. Four
  SHOULDs + one NIT folded in: (1) explicit per-data-unit **tweak-uniqueness
  contract**, caller-owned encoding (Q12.3 / §2); (2) the IEEE 1619 / SP 800-38E

  dudect case is mandatory**, not budget-optional — whole-block-only never
  exercises the final-pair CTS path (Q12.9); (5) NIT — spelled out the explicit
  CTS *decrypt* steps in place of "swap the final two tweaks" (Q12.5).
