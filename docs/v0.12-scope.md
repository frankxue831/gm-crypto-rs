# v0.12 ‚Äî Scope doc

**Status:** Lands as v0.12 W0. Mirrors the structure of `docs/v0.11-scope.md`
‚Äî Q-list with provisional answers, hard contract, posture, deferred items.
**Anchor: SM4-XTS** ‚Äî a new tweakable block-cipher mode for disk/sector
encryption (**GB/T 17964-2021**, the SM4 national standard for the XTS mode;
GM/T OID `1.2.156.10197.1.104.10`), single-shot, with **full ciphertext
stealing**, byte-identical to OpenSSL 3.x EVP `SM4-XTS` (`xts_standard=GB`).
**Note (W1 finding):** GB/T 17964 is *not* byte-compatible with IEEE 1619 (the
AES-XTS standard) ‚Äî they differ in the GF(2¬π¬≤‚Å∏) tweak-doubling convention (GB
uses the bit-reflected / GHASH-style representation) and consequently in CTS;
v0.12 targets **GB**. Cashes in the candidate listed as
`Q12.4` in `docs/v0.11-scope.md` ¬ß6 (`v0.10-Q11.5`). Five workstreams:
W0 (this doc) ‚Üí W1 KAT sourcing ‚Üí W2 `mode_xts` core ‚Üí W3 dudect + CI +
verification ‚Üí W4 release prep.

Top-level decisions baked into the Q-list below:

- **Anchor only.** v0.12 ships *only* SM4-XTS in `gmcrypto-core`. The
  RustCrypto `aead` trait fit stays deferred (re-verified `0.6.0-rc.10` on
  2026-05-23, no upstream movement). Infra/perf items (pinned dudect runner,
  AVX-512 `sbox_x64`) and other consumer-driven items (CCM incremental input)
  ride their own later cycles. **C FFI for XTS is deferred to v0.13** ‚Äî the
  v0.8 AEAD-core precedent (land the Rust core, let the API soak, add the C ABI
  next cycle). Tightly-scoped release, matching the v0.7 ‚Üí v0.11 cadence.
- **Full ciphertext stealing ‚Äî not whole-block-only.** Data units of any
  length ‚â• 16 bytes are supported, including non-block-multiples via the
  standard XTS last-two-block ciphertext-stealing swap. This is what makes the
  mode standard-complete and what lets every OpenSSL EVP vector validate
  directly. Whole-block-only was rejected (non-standard, surprises callers,
  can't validate non-aligned vectors).
- **Pure `gmcrypto-core`, no new dependency.** XTS's per-block tweak update is
  a GF(2¬π¬≤‚Å∏) multiply by the primitive element Œ± ‚Äî a trivial √ó2 (one
  shift-and-reduce). GB/T 17964 uses the **bit-reflected** representation (the
  same bit convention GHASH/PMULL use: a right-shift with reduce constant
  `0xE1`), but it is *not* GHASH's full carryless multiply ‚Äî just multiply-by-x.
  So XTS needs **no `gmcrypto-simd` dependency**; the Œ±-doubling lives in-core.
  The whole-block bulk rides the existing
  `Sm4Cipher::encrypt_blocks` batch API (v0.7 W1), which transparently picks up
  the v0.6 SIMD fanout when `sm4-bitsliced-simd` is also enabled.
- **Zero default-build impact.** XTS is opt-in behind a new `sm4-xts` feature.
  The `#![no_std]` default-features build is byte-for-byte unaffected; no new
  always-on dependency; no change to any existing module.
- **Constant-time discipline on the tweak.** The XTS tweak `T = SM4_E(Key2,
  tweak)` is **secret-derived** (Key2 is secret). The Œ±-doubling's carry
  reduction must be a **masked** XOR, never an `if` on `T`. Branching on the
  *length* of the data unit is fine ‚Äî length is public input, like every other
  mode. A new dudect target `ct_sm4_xts_decrypt` gates this.

## 1. Context

The cipher-modes arc has shipped, one mode-family per cycle, on the v0.7 W1
batch API (`Sm4Cipher::encrypt_blocks` / `decrypt_blocks`):

- **v0.7** ‚Äî SM4-CTR (single-shot + streaming).
- **v0.8** ‚Äî AEAD core: SM4-GCM + SM4-CCM single-shot (+ the GHASH primitive
  in `gmcrypto-simd`).
- **v0.9 / v0.10** ‚Äî AEAD ergonomics + streaming AEAD FFI.
- **v0.11** ‚Äî RustCrypto `digest 0.11` / `cipher 0.5` trait-fit modernization
  (no new mode).

SM4-XTS is the natural next mode: tweakable, length-preserving (no expansion,
no tag), built for random-access sector/disk encryption. It is the GM-T
counterpart to the ubiquitous AES-XTS used by dm-crypt / BitLocker / FileVault.
Unlike the AEAD modes, XTS provides **confidentiality only** ‚Äî it is *not*
authenticated. That is by design for the disk-sector threat model (no room for
a tag in a fixed-size sector), and the module docstring will state the
non-authentication contract loudly.

The building blocks already exist: `Sm4Cipher` block ops + the batch API, the
`subtle` constant-time toolkit, the `Option`-return failure-mode discipline,
and the `mode_ctr` / `mode_ccm` module templates. The one genuinely new
primitive is the GF(2¬π¬≤‚Å∏) Œ±-doubling of the tweak ‚Äî small, in-core, and
constant-time by construction.

### KAT sourcing reality (resolved at W0)

The project's KAT north star is "byte-identical to gmssl 3.1.1." For XTS that
star is unavailable, exactly as it was for SM4-CCM in v0.8:

- **gmssl 3.1.1 has no XTS** (`sm4` supports only `-cbc|-ctr|-gcm|
  -cbc_sm3_hmac|-ctr_sm3_hmac`). So there is **no gmssl interop test** for XTS.
- **OpenSSL 3.x registers SM4-XTS** ‚Äî verified on OpenSSL 3.6.2:
  `{ 1.2.156.10197.1.104.10, SM4-XTS } @ default`. The `openssl enc` CLI
  blocks XTS ("enc XTS ciphers not supported"), but the cipher is reachable via
  the **EVP API**. So KAT vectors are sourced via a small C oracle harness
  (`tests/data/sm4_xts_oracle.c`, mirroring v0.8's `sm4_ccm_oracle.c`), and the
  validation surface is "byte-identical to OpenSSL 3.x EVP `SM4-XTS`."

This mirrors the v0.8 W0 CCM sourcing decision precisely; `docs/v0.8-ccm-kat-
sourcing.md` is the template for `docs/v0.12-xts-kat-sourcing.md` (W1).

## 2. Hard contract

- **Byte-identical to OpenSSL 3.x EVP `SM4-XTS`.** Every KAT vector ‚Äî whole-block
  and ciphertext-stealing ‚Äî produced by the W1 oracle harness must match our
  output byte-for-byte. A mismatch is a bug, full stop.
- **Confidentiality only ‚Äî no authentication.** XTS provides no integrity tag.
  `decrypt` returning `Some` does **not** mean the ciphertext is authentic. The
  module docstring states this prominently; callers needing integrity use an
  AEAD mode (GCM/CCM) instead. This is not a failure-mode the API hides ‚Äî it is
  the defining property of the mode.
- **Tweak-uniqueness contract (caller-owned).** The caller MUST supply a
  **unique** 16-byte tweak per data unit under a given key ‚Äî in disk use, the
  tweak is the data-unit (sector) number. Reusing a `(key, tweak)` pair across
  different plaintexts leaks equality structure (XTS is deterministic). The
  encoding of a sector number into the 16-byte tweak (endianness, width) is the
  caller's responsibility; this module takes the raw 16 bytes as-is. The module
  docstring states the uniqueness contract loudly (same posture as the CTR/GCM
  nonce contracts).
- **Full ciphertext stealing.** Data units of any length ‚â• 16 bytes encrypt and
  decrypt correctly, round-trip to identity, and match OpenSSL.
- **Data-unit length bounds.** `16 ‚â§ data_unit.len() ‚â§ 2¬≤‚Å∞ ¬∑ 16` (16 MiB) ‚Äî the
  NIST SP 800-38E ceiling of 2¬≤‚Å∞ blocks per data unit, which
  OpenSSL also enforces. Lengths outside this range ‚Üí `None` (input
  validation).
- **Single `None` failure mode.** `encrypt` / `decrypt` return
  `Option<Vec<u8>>`; `None` is returned only for input-validation failures
  (`data_unit.len() < 16` or `> 16 MiB`, or weak key `Key1 == Key2`). No
  distinguished error variants ‚Äî per the project's failure-mode invariant. XTS
  has no tag, so there is no MAC-failure path at all.
- **Constant-time discipline.** The Œ±-doubling carry is a masked XOR, never a
  branch on the secret-derived tweak `T`. The `Key1 == Key2` weak-key check
  uses `subtle::ConstantTimeEq`. Branching on the public data-unit *length* is
  permitted. The dudect target `ct_sm4_xts_decrypt` gates the discipline.
- **Default-features build unchanged.** XTS lives behind the new opt-in
  `sm4-xts` feature; `cargo build -p gmcrypto-core --no-default-features` and
  the wasm32 target build are byte-for-byte unaffected. No new always-on
  dependency; no `gmcrypto-simd` dependency added by this feature.
- **MSRV stays 1.85.** No new dependency; nothing moves the MSRV.
- **Additive only.** No existing public API changes; no existing module is
  touched except the `pub mod` line in `sm4/mod.rs` and the feature/version
  bookkeeping. v0.11's other invariants (`unsafe_code = "forbid"` on
  `gmcrypto-core`, `Cargo.lock` gitignored) are untouched.

## 3. Q-list (Q12.1‚ÄìQ12.13) with provisional answers

### Q12.1 ‚Äî Scope: anchor only

SM4-XTS in `gmcrypto-core` is the sole feature of v0.12. The `aead` trait fit
(`v0.11-Q12.1`) stays deferred ‚Äî `aead` re-verified `0.6.0-rc.10` (latest
stable `0.5.2`) on 2026-05-23, unchanged since 2026-02-04. Infra/perf items
(pinned dudect runner [`v0.11-Q12.2`], AVX-512 `sbox_x64` [`v0.11-Q12.3`]) and
CCM incremental input [`v0.11-Q12.5`] each warrant their own cycle. **C FFI for
XTS is explicitly deferred to v0.13** (Q12.11).

**Provisional decision: anchor only ‚Äî SM4-XTS Rust core, full stop.**

### Q12.2 ‚Äî Ciphertext stealing vs. whole-block-only

Support arbitrary data-unit lengths ‚â• 16 bytes (full CTS), or restrict to
multiples of 16? Real-world sectors (512 / 4096 bytes) are block-multiples, so
whole-block-only covers the common case ‚Äî but it is non-standard-complete,
surprises callers who expect standard XTS, and cannot validate the non-aligned
OpenSSL vectors. A correctness-first library ships the full standard.

**Provisional decision: full ciphertext stealing. Data units of any length
in `[16, 2¬≤‚Å∞¬∑16]` bytes (16 B .. 16 MiB); the last partial block is handled by
the standard XTS last-two-block steal. Lengths < 16 (XTS undefined for sub-block
data units) or > 16 MiB (NIST SP 800-38E 2¬≤‚Å∞-block ceiling, also
enforced by OpenSSL) ‚Üí `None`.**

### Q12.3 ‚Äî Public API shape

`mode_ctr` takes `key: &[u8; KEY_SIZE]` + a 16-byte counter; `mode_ccm` returns
`Option<Vec<u8>>`. XTS needs 32 bytes of key (`Key1 ‚Äñ Key2`) and a 16-byte
tweak (the data-unit number). The 32-byte combined key + raw 16-byte tweak maps
1:1 to OpenSSL EVP's `(key=32, IV=16)` inputs, so KAT vectors validate without
a translation layer. Single-shot per data unit ‚Äî XTS is inherently
random-access; streaming is not a natural fit and is not in scope.

```rust
pub const XTS_KEY_SIZE: usize = 32;

pub fn encrypt(
    key: &[u8; XTS_KEY_SIZE],
    tweak: &[u8; BLOCK_SIZE],   // BLOCK_SIZE = 16; the data-unit number
    data_unit: &[u8],           // len >= 16
) -> Option<Vec<u8>>;

pub fn decrypt(
    key: &[u8; XTS_KEY_SIZE],
    tweak: &[u8; BLOCK_SIZE],
    data_unit: &[u8],
) -> Option<Vec<u8>>;
```

The tweak is consumed as **raw 16 bytes** ‚Äî the caller owns any sector-number‚Üí
tweak encoding (endianness/width) and the per-data-unit uniqueness contract
(¬ß2). `data_unit.len()` is validated to `[16, 16 MiB]` (¬ß2 / Q12.5).

**Provisional decision: 32-byte combined key + raw 16-byte tweak + single-shot
`Option<Vec<u8>>`. `XTS_KEY_SIZE = 32`; reuse `BLOCK_SIZE = 16` for the tweak.
Tweak encoding + uniqueness are caller-owned; length bounded `[16, 16 MiB]`.**

### Q12.4 ‚Äî KAT sourcing

gmssl 3.1.1 lacks XTS; OpenSSL 3.x EVP has `SM4-XTS` (OID
`1.2.156.10197.1.104.10`) reachable only via the EVP API (the `enc` CLI blocks
XTS). Source vectors via a C oracle harness (`tests/data/sm4_xts_oracle.c`,
`EVP_CIPHER_fetch("SM4-XTS")`), mirroring v0.8's `sm4_ccm_oracle.c`. KAT
vectors are embedded **inline as `const`** in `tests/sm4_xts_kat.rs` (the v0.8
W3 CCM precedent ‚Äî auditor reads the vectors next to the assertions), with the
reproducibility harness kept at `tests/data/sm4_xts_oracle.c`. The W1
deliverable `docs/v0.12-xts-kat-sourcing.md` records provenance (OpenSSL
version, OID, C source, command lines, expected outputs) and a reproducibility
recipe. **No gmssl interop test** (gmssl lacks the mode).

Oracle pinning (codex review + verified against OpenSSL 3.6.2 during W0): the
harness fetches `SM4-XTS` from OpenSSL's **default (non-FIPS) provider** (a
`fips=yes` context may refuse to fetch XTS). Each vector is a **single-shot**
`EVP_EncryptUpdate` with `EVP_CIPHER_CTX_set_padding(ctx, 0)` (XTS is
length-preserving ‚Äî no block padding; `EVP_EncryptFinal_ex` adds 0 bytes). The
OpenSSL IV (16 bytes) is the **raw tweak**, matching our API 1:1; the key is 32
bytes (`Key1 ‚Äñ Key2`). Verified during W0: whole-block, min-16, and CTS lengths
(17, 20) all round-trip; the W0 probe's 20-byte CTS output matched the Q12.5
construction exactly. The rejection cases are **local** assertions (our `None`
contract), **not** oracle KATs, with differing oracle behavior noted:
- **`len < 16`** ‚Äî OpenSSL *also* rejects (`EVP_EncryptUpdate` fails); our
  `None` matches.
- **`Key1 == Key2`** ‚Äî OpenSSL's **default provider does NOT reject** (it emits
  ciphertext; only `fips=yes` rejects). Our `None` is a deliberate
  **stricter-than-OpenSSL-default** policy (GB/T 17964 / FIPS-aligned). So this
  case has no oracle KAT and is asserted purely as our contract.
- **`len > 16 MiB`** ‚Äî our NIST SP 800-38E policy (OpenSSL's exact cap
  is version-dependent and not relied upon); asserted locally.

**Provisional decision: OpenSSL 3.x EVP `SM4-XTS` pinned to `xts_standard=GB`
(default/non-FIPS provider, single-shot, padding off, IV = raw tweak) via a C
oracle harness; KAT vectors embedded inline in `sm4_xts_kat.rs` (v0.8 W3
precedent); weak-key/length rejections tested locally; no
gmssl interop.**

### Q12.5 ‚Äî Algorithm + ciphertext-stealing semantics

Per GB/T 17964-2021 with SM4 as the block cipher:

1. Split `key` into `Key1 = key[..16]` (data), `Key2 = key[16..]` (tweak).
2. `T = SM4_E(Key2, tweak)` ‚Äî the initial tweak for block 0.
3. For each 16-byte block `j`: `C_j = SM4_E(Key1, P_j ‚äï T_j) ‚äï T_j`, then
   `T_{j+1} = T_j ¬∑ Œ±`.
4. **Œ±-doubling** (`T ¬∑ Œ±`): a GF(2¬π¬≤‚Å∏) multiply by the primitive element in the
   **bit-reflected** (GHASH-style) representation that GB/T 17964 uses. Treat
   the 16 bytes with byte 0 leading; **right-shift** the 128-bit value by one
   bit (carry each byte's LSB into the next byte's MSB); if the bit shifted off
   the end (LSB of byte 15) is 1, XOR `0xE1` into byte 0. (`0xE1` is the
   bit-reversed `0x87`. This is the same bit convention as GHASH but only a
   multiply-by-x, **not** GHASH's full carryless multiply ‚Äî so no
   `gmcrypto-simd` dependency. IEEE 1619 instead uses the little-endian
   `<<1` / `0x87` convention, which produces *different* ciphertext ‚Äî see the
   W1 finding in the header and ¬ß7.)
5. **Ciphertext stealing (encrypt)** for `len = 16¬∑q + r`, `r > 0`: encrypt the
   first `q-1` whole blocks normally; for the final full+partial pair ‚Äî
   `CC = SM4_E(Key1, P_{q-1} ‚äï T_{q-1}) ‚äï T_{q-1}`; the last (partial)
   ciphertext is `CC[..r]`; `PP = P_partial ‚Äñ CC[r..]`;
   `C_{q-1} = SM4_E(Key1, PP ‚äï T_q) ‚äï T_q`. Output order is
   `C_0 ‚Ä¶ C_{q-2} ‚Äñ C_{q-1} ‚Äñ CC[..r]`.
   **Ciphertext stealing (decrypt)** ‚Äî explicit steps (not just "swap tweaks"):
   decrypt the first `q-1` whole blocks normally; then
   `PP = SM4_D(Key1, C_{q-1} ‚äï T_q) ‚äï T_q` (the second-to-last *ciphertext*
   block, under `T_q`); the last (partial) plaintext is `PP[..r]`;
   `CC = C_partial ‚Äñ PP[r..]`; `P_{q-1} = SM4_D(Key1, CC ‚äï T_{q-1}) ‚äï T_{q-1}`
   (under `T_{q-1}`).
6. **Weak-key rejection:** GB/T 17964 (and FIPS) mandate `Key1 ‚âÝ Key2`; reject
   equal halves with `None` (constant-time compare ‚Äî Q12.6). Length bounds
   `16 ‚â§ len ‚â§ 2¬≤‚Å∞¬∑16` (16 MiB) else `None` (¬ß2).

The full algorithm above was **verified during W1** (a standalone C reference
using the bit-reflected doubling + this CTS structure) against OpenSSL
`xts_standard=GB` byte-for-byte across 16, 32, 48 (whole-block) and 17, 20
(CTS) byte lengths. The W2 KAT test re-confirms it.

**Provisional decision: GB/T 17964-2021 XTS-SM4 with the bit-reflected
Œ±-doubling (right-shift, reduce constant `0xE1` into byte 0) and standard
last-two-block CTS (decrypt steps in step 5); `Key1 ‚âÝ Key2` enforced; length
`[16, 16 MiB]`.**

### Q12.6 ‚Äî Constant-time posture

- **Œ±-doubling carry.** `T = SM4_E(Key2, tweak)` is secret-derived (Key2 is
  secret), so the carry reduction is a **masked** XOR, never an `if`:
  `t[0] ^= 0xE1 & carry.wrapping_neg()` (carry ‚àà {0,1} ‚Üí mask 0x00/0xFF). No
  branch on any byte of `T`.
- **Weak-key check.** `Key1 == Key2` via `subtle::ConstantTimeEq` over the two
  16-byte halves. The boolean *outcome* gates the reject (a usage error,
  acceptable to reveal); the comparison itself leaks no byte positions.
- **Length branching is public.** `q`, `r`, and the CTS branch derive from
  `data_unit.len()`, which is public input ‚Äî branching on it is fine, exactly
  as in every existing mode.
- **Zeroize.** Intermediate tweak/working buffers on the secret path are
  zeroized where the existing modes do (consistent with `mode_ccm` /
  `decrypt.rs` discipline).
- **Dudect.** A new target `ct_sm4_xts_decrypt` (Q12.9), class-split by key,
  gates the discipline at `|tau| < 0.20`.

**Provisional decision: masked Œ±-doubling carry; `subtle` weak-key compare;
length branching public; `ct_sm4_xts_decrypt` added.**

### Q12.7 ‚Äî Block processing: batched, no `gmcrypto-simd` dependency

The whole-block bulk goes through `Sm4Cipher::encrypt_blocks` /
`decrypt_blocks` (v0.7 W1): precompute the tweak sequence (cheap sequential
doublings), XOR into all plaintext blocks, one batch cipher call (which rides
the v0.6 SIMD fanout under `sm4-bitsliced-simd`), then XOR the tweaks back. The
CTS tail (‚â§ 2 blocks) is handled with single-block calls. The Œ±-doubling is a
trivial √ó2, so XTS does **not** depend on `gmcrypto-simd`'s GHASH ‚Äî unlike
`sm4-aead`, the `sm4-xts` feature pulls in **no** sibling crate.

**Provisional decision: batched whole-block path via the existing batch API;
CTS tail single-block; Œ±-doubling in-core; no `gmcrypto-simd` dependency.**

### Q12.8 ‚Äî Feature flag

A new opt-in `sm4-xts = []` feature (pure-core, no `dep:` entries), default-off,
gating `pub mod mode_xts;` in `sm4/mod.rs`. Mirrors the `sm4-bitsliced` /
`sm4-aead` opt-in posture; default-features build unchanged.

**Provisional decision: `sm4-xts = []`, default-off; `#[cfg(feature =
"sm4-xts")] pub mod mode_xts;` in `sm4/mod.rs`.**

### Q12.9 ‚Äî Testing + dudect target

- **`tests/sm4_xts_kat.rs`** (cfg `sm4-xts`): byte-identical to the W1 OpenSSL
  oracle across whole-block lengths 16/32/48/64 B and CTS lengths 17/20/31 B
  (`r = 1/4/15`), plus a min-block-prefix consistency check. (The actual
  landed vectors; the rejection cases live in the `mode_xts` inline unit tests
  since they reference the module-private `MAX_LEN`.)
- **Round-trip property test** (inline unit test) exhaustively over
  `len ‚àà 16..=80`, asserting `decrypt(encrypt(x)) == x`. The rejection cases
  (`len < 16`, `len > 16 MiB`, `Key1 == Key2 ‚Üí None`) are inline unit tests too.
- **Dudect `ct_sm4_xts_decrypt`** cfg-gated on `sm4-xts`, class-split by master
  key (both classes' data units are valid encrypts under their own key, so both
  decrypt via identical control flow ‚Äî the `ct_sm4_ctr` / `ct_sm4_gcm` shape).
  Fixed tweak + a fixed **CTS (non-block-multiple) data-unit length**
  (e.g. 100 bytes = 6 whole blocks + 4) ‚Äî **mandatory, not budget-optional**
  (codex review): a whole-block-only length never executes the final-pair
  `T_{q-1}/T_q` CTS path, so the partial-block branch (the riskiest tweak
  arithmetic) would go ungated. Gate `|tau| < 0.20` at both 10K (PR-smoke) and
  100K (nightly).
- **CI dudect matrix** gains an `sm4-xts`-bearing slot so the target gates; the
  `required_low` dict adds `ct_sm4_xts_decrypt = 0.20` under an `"sm4-xts"`
  `MATRIX_FEATURES` substring match.

**Provisional decision: `sm4_xts_kat.rs` (OpenSSL-equal scenarios, weak-key +
length-bound rejections local) + round-trip property test + `ct_sm4_xts_decrypt`
dudect target at a **CTS (non-block-multiple) length** + a CI matrix slot
carrying `sm4-xts`.**

### Q12.10 ‚Äî Supply chain + MSRV

XTS adds **no new dependency** (pure-core; no `gmcrypto-simd`, no new
transitive crate). `deny.toml`'s allowlist is unchanged; only the `cargo deny
--features ‚Ä¶` invocation gains `gmcrypto-core/sm4-xts` so the feature is
exercised. MSRV stays 1.85 (nothing new to raise it). The `cargo +1.85 build`
gate adds `sm4-xts` to its feature list.

**Provisional decision: no allowlist change; `sm4-xts` added to the `cargo
deny` and `cargo +1.85 build` feature lists; MSRV 1.85 unchanged.**

### Q12.11 ‚Äî C FFI deferred to v0.13

Following the v0.8 precedent (AEAD core shipped with no FFI; the C ABI landed
in v0.9/v0.10 once the Rust API soaked), the ~2 XTS C ABI symbols
(`gmcrypto_sm4_xts_encrypt` / `_decrypt`) are **deferred to v0.13**. Bundling
them would freeze a C ABI before the Rust API has any real-world use.
`gmcrypto-c` gets only its path-dep version bump in W4 (no new symbols, no
header regen).

**Provisional decision: no XTS C FFI in v0.12; deferred to v0.13.**

### Q12.12 ‚Äî Workstream ordering

XTS needs its KAT fixtures **before** the impl (TDD), exactly as CCM did in
v0.8. So sourcing is its own workstream ahead of the implementation.

**Provisional decision: W0 (this doc) ‚Üí W1 (KAT sourcing: oracle harness +
inline KAT vectors + `docs/v0.12-xts-kat-sourcing.md`) ‚Üí W2 (`mode_xts`
encrypt/decrypt + full CTS + constant-time Œ±-doubling + `sm4-xts` feature +
`sm4_xts_kat.rs`, TDD against W1 fixtures) ‚Üí W3 (`ct_sm4_xts_decrypt` dudect +

## 7. Decision history

- **2026-05-23**: scope doc lands as v0.12 W0. Anchor confirmed as **SM4-XTS**
  (the former candidate `v0.11-Q12.4` / `v0.10-Q11.5`). Full ciphertext
  stealing chosen over whole-block-only (standard-completeness +
  OpenSSL-vector validation). API: 32-byte combined key + raw 16-byte tweak +
  single-shot `Option<Vec<u8>>` (1:1 with OpenSSL EVP inputs). KAT sourcing
  resolved: gmssl 3.1.1 lacks XTS (no interop test), OpenSSL 3.6.2 EVP
  `SM4-XTS` (OID `1.2.156.10197.1.104.10`) reachable via the EVP API ‚Üí C oracle
  harness, mirroring v0.8 W0 CCM. Pure-core feature `sm4-xts` (no
  `gmcrypto-simd` dep ‚Äî Œ±-doubling is a trivial √ó2, not GHASH). C FFI deferred
  to v0.13 (v0.8 AEAD-core precedent). `aead` trait fit re-verified
  `0.6.0-rc.10` (stalled) and re-deferred. Re-open via a "Q12.x amendment"
  section appended below if a workstream surfaces a blocker (e.g. the OpenSSL
  oracle proving unable to emit deterministic CTS vectors, in which case W1
  falls back to hand-derived vectors cross-checked against a second independent
  implementation ‚Äî the v0.8 W0 fallback posture).
- **2026-05-23 (codex scope review)**: `codex exec` reviewed this doc. **No
  BLOCKER** ‚Äî the last-two-block CTS construction, the secret-derived-tweak
  constant-time posture, and the min-length-16 / key-as-secret treatment were
  confirmed correct. (The Œ±-doubling *convention* was stated as little-endian /
  `0x87` here; that was **superseded** by the W1 codex review below, which found
  GB/T 17964 uses the bit-reflected / `0xE1` convention.) Four SHOULDs + one NIT
  folded in: (1) explicit per-data-unit **tweak-uniqueness contract**,
  caller-owned encoding (Q12.3 / ¬ß2); (2) the **2¬≤‚Å∞-block (16 MiB) data-unit
  ceiling** (NIST SP 800-38E), `> 16 MiB ‚Üí None`, matching OpenSSL (¬ß2 / Q12.3 /
  Q12.5); (3) **oracle pinning** ‚Äî OpenSSL default/non-FIPS provider, single-shot
  EVP, padding off, IV = raw tweak, weak-key/length rejections tested locally not
  as oracle KATs (Q12.4); (4) the **CTS-length dudect case is mandatory**, not
  budget-optional ‚Äî whole-block-only never exercises the final-pair CTS path
  (Q12.9); (5) NIT ‚Äî spelled out the explicit CTS *decrypt* steps in place of
  "swap the final two tweaks" (Q12.5).
- **2026-05-23 (W0 oracle verification)**: built a throwaway OpenSSL 3.6.2 EVP
  `SM4-XTS` probe to ground the design. Confirmed: EVP path works (key=32,
  iv=16, single-shot, `set_padding(0)`); whole-block / min-16 / CTS (17, 20 B)
  all round-trip; the **20-byte CTS output matched the Q12.5 construction
  byte-for-byte** (output `C_0 ‚Äñ CC[..r]`, last r bytes = first r of the
  whole-block block-0 ct). OpenSSL rejects `len < 16` (matches our `None`) but
  its **default provider does *not* reject `Key1 == Key2`** ‚Äî so the weak-key
  `None` is documented as a deliberate stricter-than-OpenSSL policy (Q12.4
  corrected; the earlier "OpenSSL rejects those inputs" was wrong for the
  weak-key case). Real verified KAT vectors captured for the W2 acceptance gate.
- **2026-05-23 (W1 codex review ‚Äî standard correction)**: the codex review of
  the W1 oracle harness raised a **BLOCKER**: OpenSSL 3.6 SM4-XTS has an
  `xts_standard` parameter (`{GB, IEEE}`) and the W0 default vectors are **GB**,
  not IEEE 1619. Investigated empirically: GB and IEEE produce **identical**
  whole-block-0 output but **differ from block 1 onward** ‚Äî the GF(2¬π¬≤‚Å∏)
  Œ±-doubling differs. Cracked the GB convention: it is the **bit-reflected**
  (GHASH-style) representation ‚Äî a right-shift with reduce constant `0xE1` into
  byte 0 (vs IEEE's `<<1` / `0x87`). A standalone GB reference (this doubling +
  the Q12.5 CTS structure) matched OpenSSL `xts_standard=GB` byte-for-byte across
  16/32/48 (whole) and 17/20 (CTS) lengths. **Decision: target GB/T 17964-2021**
  (the SM4 national standard + OpenSSL default for SM4-XTS), not IEEE 1619.
  Corrections folded throughout: the standard name (header / Q12.5 / ¬ß2), the
  Œ±-doubling description and masked-carry constant (`0x87` ‚Üí `0xE1`, Q12.5 /
  Q12.6), and the oracle harness now **pins `xts_standard=GB`** explicitly + adds
  input-length validation (codex SHOULDs). The user-facing contract
  ("byte-identical to OpenSSL EVP SM4-XTS") is unchanged; only the internal
  standard characterization was corrected.
