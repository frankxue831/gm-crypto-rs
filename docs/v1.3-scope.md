# v1.3 scope ‚Äî X.509-with-SM2: leaf certificate parse + signature verify

Status: scope locked 2026-06-11 (maintainer sign-off Q3.1‚ÄìQ3.2 via brainstorm;
Codex direction consult ranked **A1 > C > A3 > B > A2** ‚Äî "path validation is
where small auditable cycles go to die"). Companion design:
`docs/v1.3-x509-sm2-design.md`.

## 1. Sign-off decisions

- **Q3.1 Direction** ‚Äî **X.509-with-SM2** (maintainer-locked). The v1.1 market
  scan's second TLCP prerequisite (SM2-KX was the first, shipped v1.1/v1.2);
  deferred from both v1.1 and v1.2 precisely so it could get its own bounded
  cycle.
- **Q3.2 Depth** ‚Äî **A1: leaf parse + SM2-with-SM3 signature verify ONLY.**
  Explicitly **no trust decisions**: no chain building, no hostname / time /
  EKU / KU / basicConstraints / revocation / CA logic. The API name is
  `verify_signature` ‚Äî never "validate" ‚Äî and the docs say bluntly what it
  does NOT do (the Codex "misread as trust validation" trap).
- **Q3.3 Dependency** ‚Äî **in-repo strict `asn1` reader; NO `x509-cert`/`der`
  dependency.** The house pattern (hand-rolled strict DER since v0.3 W1) and
  the conservative dep policy (the pre-1.0 crypto-bigint decouple). Every
  primitive needed already exists: `read_sequence` / `read_tlv` /
  `read_integer` / `read_oid` / `read_bit_string` /
  `read_context_tagged_explicit`.
- **Q3.4 Packaging** ‚Äî opt-in feature **`x509 = []`** on `gmcrypto-core`
  (pure-core, default-off, no new dependency; default build byte-identical).
  The `sm4-xts` / `sm2-key-exchange` precedent.
- **Q3.5 API shape** ‚Äî `x509::Certificate::from_der(&[u8]) -> Option<...>`
  (single `None` for every malformed input ‚Äî the failure-mode invariant;
  DER decode returns `Option`, never error variants) +
  `verify_signature(&self, issuer: &Sm2PublicKey) -> bool` /
  `verify_signature_with_id(...) -> bool` (the `verify_with_id` bool
  precedent) + raw accessors. Self-signed = verify against the cert's own
  SPKI; no special API.
- **Q3.6 AlgorithmIdentifier parameters** ‚Äî signature AlgId (`sm2-sign-with-
  sm3`, 1.2.156.10197.1.501): parameters **absent or NULL both accepted**
  (wild GM certs vary; documented). SPKI: **strict** ‚Äî delegated to the
  existing `spki::decode`, which already enforces `id-ecPublicKey` +
  `sm2p256v1` (1.2.156.10197.1.301) named curve + on-curve point.
  The outer `signatureAlgorithm` and the inner `tbsCertificate.signature`
  AlgIds MUST match (RFC 5280 ¬ß4.1.1.2 + ¬ß4.1.2.3).
- **Q3.7 Raw-not-parsed surface** ‚Äî `subject` / `issuer` exposed as **raw DER
  `Name` bytes** (no string-y Name/RDN parsing in v1.3 ‚Äî minimal parser
  surface; byte-equality is sufficient for issuer/subject matching and for
  the TLCP follow-on). `validity` parsed only as far as a plain
  `(year, month, day, hour, minute, second)` struct per time (UTCTime +
  GeneralizedTime, Zulu only ‚Äî strict-DER requires it); **no time
  comparison** ‚Äî a "not expired" check is a trust decision the caller owns.
- **Q3.8 Verification ID** ‚Äî default `1234567812345678` for the Z-hash in
  certificate signature verification, with a `_with_id` override (RFC 8998
  ¬ß3.2.1: the default is specified but CA practice can vary).
- **Q3.9 Assurance** ‚Äî KAT fixtures: **gmssl 3.1.1-generated** SM2
  certificates (the v0.3 W2 committed-binary-fixture + regen-recipe pattern):
  self-signed CA + CA-issued leaf; positive verifies + negatives
  (tampered tbs byte, wrong issuer key, swapped signature OID, truncations).
  Cross-checked against the RFC 8998 example cert where usable. New fuzz
  target **`fuzz_x509`** (decode path, no-panic/OOM/hang) ‚Äî fuzz census
  26 ‚Üí **27**: `FUZZ_TARGETS` in fuzz-nightly.yml + fuzz/README + CLAUDE.md
  censuses ALL updated in the same PR (the #102 must-match-`[[bin]]`
  invariant). **NO new dudect target** ‚Äî certificate verification consumes
  only public inputs (issuer public key, public cert bytes); there is no
  secret-dependent path. First cycle since v0.11 where that holds by
  construction.
- **Q3.10 Versioning** ‚Äî additive minor ‚Üí **1.3.0** lockstep (`=1.3.0`
  sibling pins). C FFI: **deferred** (v1.4 candidate per the core-in-vN /
  FFI-in-vN+1 cadence ‚Äî and TLCP needs the Rust surface, not the C one).
  Publish + tag remain the maintainer's per-release call.
- **Q3.11 Version field + extensions posture** ‚Äî require **v3** certificates
  (`[0] EXPLICIT INTEGER 2`): GM/T 0015 profile certs are v3 and v1/v2 GM
  certs effectively don't exist; rejecting them keeps one code path.
  Extensions are **structurally parsed (skipped) but NOT interpreted** ‚Äî
  including `critical` flags. That is part of "no trust decisions" and is
  documented loudly: a caller who needs extension semantics (basicConstraints,
  keyUsage) must wait for the chain-validation cycle or parse `extensions_raw`
  themselves.

## 2. Out of scope (deliberate)

- Chain / path validation (the twice-deferred creep ‚Äî its own future cycle).
- Certificate / CSR **generation** (no emit path: an emitted format choice
  propagates forever ‚Äî the `encode_c1c2c3_legacy` lesson).
- Name/RDN string parsing, hostname matching, time validity *decisions*,
  extension interpretation, revocation (CRL/OCSP).
- TLCP (the v1.4+ direction this cycle feeds).
- RSA / non-SM2 certificates (single `None`; this is a GM library).
- C FFI (v1.4 candidate).
