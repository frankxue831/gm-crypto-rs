# v1.3 design — X.509-with-SM2 leaf certificate parse + signature verify

Companion to `docs/v1.3-scope.md` (Q3.1–Q3.11, locked 2026-06-11). This doc is
the implementable spec: structures, parsing rules, verification semantics,
reused assets, and assurance plan.

## 1. Context

v1.1/v1.2 completed the SM2 family (sign / encrypt / key exchange, Rust + C).
The v1.1 market scan identified TLCP (GB/T 38636) as the headline pure-Rust
gap; its two prerequisites are SM2 key exchange (shipped) and SM2 certificate
handling (this cycle). A Codex direction consult ranked leaf-only parse+verify
first by a wide margin: it converts the deferred roadmap item into a real
substrate for TLCP / generation / chains **without accepting their scope**,
and it fits the house style — bounded parser, exact-byte verification, single
failure mode, fuzzable.

**The one-sentence contract:** given DER bytes and an issuer public key, this
module tells you (a) what the certificate *says* and (b) whether its
SM2-with-SM3 signature over those exact bytes verifies — and nothing more.

## 2. Profile

Accepted input: a DER-encoded RFC 5280 `Certificate` whose:

```text
Certificate  ::=  SEQUENCE  {
    tbsCertificate       TBSCertificate,
    signatureAlgorithm   AlgorithmIdentifier,   -- sm2-sign-with-sm3
    signatureValue       BIT STRING }           -- DER SEQUENCE { r, s }

TBSCertificate  ::=  SEQUENCE  {
    version         [0]  EXPLICIT INTEGER (2),  -- v3 REQUIRED (Q3.11)
    serialNumber         INTEGER,               -- ≤ 20 content bytes (RFC 5280 §4.1.2.2)
    signature            AlgorithmIdentifier,   -- MUST equal outer (§4.1.2.3)
    issuer               Name,                  -- kept as raw DER (Q3.7)
    validity             SEQUENCE { notBefore, notAfter },  -- UTCTime | GeneralizedTime, Zulu
    subject              Name,                  -- kept as raw DER (Q3.7)
    subjectPublicKeyInfo SubjectPublicKeyInfo,  -- delegated to spki::decode
    issuerUniqueID  [1]  IMPLICIT OPTIONAL,     -- skipped if present
    subjectUniqueID [2]  IMPLICIT OPTIONAL,     -- skipped if present
    extensions      [3]  EXPLICIT OPTIONAL }    -- skipped, kept raw, NOT interpreted
```

OID requirements (all three constants already exist in `asn1::oid`):

| Field | OID | Constant |
|---|---|---|
| signatureAlgorithm / tbs.signature | 1.2.156.10197.1.501 (SM2-with-SM3) | `SM2_SIGN_WITH_SM3` |
| SPKI algorithm | 1.2.840.10045.2.1 (id-ecPublicKey) | `ID_EC_PUBLIC_KEY` (inside `spki::decode`) |
| SPKI parameters | 1.2.156.10197.1.301 (sm2p256v1) | `SM2P256V1` (inside `spki::decode`) |

AlgorithmIdentifier parameters for `sm2-sign-with-sm3`: **absent or NULL both
accepted** (Q3.6); anything else → `None`. Everything else is strict DER via
the existing reader (canonical lengths, no trailing bytes at any level — the
established `asn1::reader` discipline).

## 3. Reused assets (the whole point — near-zero new primitive surface)

| Need | Existing asset | Notes |
|---|---|---|
| SEQUENCE / TLV walking | `asn1::reader::{read_sequence, read_tlv, read_length}` | strict-canonical, no_std |
| version `[0]` / extensions `[3]` | `read_context_tagged_explicit` | already used by sec1/pkcs8 |
| uniqueID `[1]`/`[2]` skip | `read_context_tagged_implicit` | skip-only |
| serial / version INTEGER | `read_integer` | minimality enforced by reader |
| OIDs | `read_oid` + `asn1::oid` constants | `SM2_SIGN_WITH_SM3` already defined |
| signatureValue BIT STRING | `read_bit_string` | unused-bits MUST be 0 |
| SPKI → public key | `spki::decode(&[u8]) -> Option<Sm2PublicKey>` | already enforces both SPKI OIDs + on-curve |
| (r,s) decode + verify | `sm2::verify_with_id(pub, id, msg, der_sig) -> bool` | parses DER sig internally via the strict `decode_sig` (INTEGER minimality, range) |
| default ID | `sm2::DEFAULT_SIGNER_ID` | `"1234567812345678"` |

The signature check is therefore: extract `tbs_raw` + check OIDs + call
`verify_with_id(issuer, id, tbs_raw, signature_der)`. No new cryptographic
code is written in this cycle — only structural parsing.

## 4. API (new module `x509`, cfg-gated on the opt-in feature `x509 = []`)

```rust
/// A parsed X.509 v3 certificate. Holds BORROWED or OWNED raw spans of the
/// input (implementation detail; owned Vec<u8> keeps lifetimes simple and
/// matches pem/pkcs8 house style) — parsing makes NO trust decisions.
pub struct Certificate { /* private */ }

impl Certificate {
    /// Strict-DER parse. Single `None` for EVERY malformed input.
    pub fn from_der(der: &[u8]) -> Option<Certificate>;

    /// Verify the certificate's SM2-with-SM3 signature against `issuer`,
    /// over the EXACT tbsCertificate bytes as they appeared on the wire,
    /// using the GM/RFC 8998 default ID "1234567812345678".
    /// THIS IS NOT CERTIFICATE VALIDATION: no chain, no time, no
    /// extension/keyUsage/basicConstraints evaluation, no revocation.
    pub fn verify_signature(&self, issuer: &Sm2PublicKey) -> bool;

    /// As `verify_signature`, with a caller-supplied ID (CA practice varies;
    /// RFC 8998 §3.2.1).
    pub fn verify_signature_with_id(&self, issuer: &Sm2PublicKey, id: &[u8]) -> bool;

    /// The subject's SM2 public key (already validated on-curve by parse —
    /// a Certificate cannot be constructed with an invalid SPKI).
    pub fn subject_public_key(&self) -> Sm2PublicKey;

    /// Raw DER spans / values, no interpretation:
    pub fn tbs_raw(&self) -> &[u8];
    pub fn serial_raw(&self) -> &[u8];          // INTEGER content bytes, ≤ 20
    pub fn issuer_raw(&self) -> &[u8];          // full DER Name TLV
    pub fn subject_raw(&self) -> &[u8];         // full DER Name TLV
    pub fn extensions_raw(&self) -> Option<&[u8]>; // [3] content if present
    pub fn not_before(&self) -> X509Time;
    pub fn not_after(&self) -> X509Time;
    pub fn is_self_issued(&self) -> bool;       // subject_raw == issuer_raw (byte equality)
}

/// A calendar timestamp parsed from UTCTime/GeneralizedTime (Zulu only).
/// Comparison helpers only (`PartialOrd`); the caller owns any "is it
/// currently valid" decision — this library has no clock.
pub struct X509Time { pub year: u16, pub month: u8, pub day: u8,
                      pub hour: u8, pub minute: u8, pub second: u8 }
```

Notes:
- `subject_public_key()` is infallible by construction: `from_der` already ran
  `spki::decode`, so a non-SM2 / off-curve SPKI never becomes a `Certificate`.
- UTCTime year pivot: RFC 5280 §4.1.2.5.1 — `YY ≥ 50` → 19YY, else 20YY.
  GeneralizedTime must be 4-digit-year Zulu, no fractional seconds (strict DER).
- No `Display`/string rendering of Names in v1.3 (Q3.7).
- The PEM armor (`CERTIFICATE`) rides the existing `pem::decode` — callers
  compose `pem::decode` + `Certificate::from_der` (document the recipe; no
  new wrapper needed).

## 5. Parsing rules (strictness checklist)

1. Whole-input exactness: the outer SEQUENCE must consume the entire input;
   trailing bytes → `None` (reader discipline).
2. `tbs_raw` is captured as the **full TLV byte span** of `TBSCertificate`
   exactly as read — verification never re-encodes (the Codex landmine).
3. version: `[0] EXPLICIT INTEGER` with value exactly 2 (v3). Absent (v1) or
   other values → `None` (Q3.11).
4. serialNumber: `read_integer`, content length 1..=20 bytes.
5. Both AlgorithmIdentifiers parsed; OID must be `SM2_SIGN_WITH_SM3`;
   parameters absent or NULL; **outer == inner byte equality** on the OID +
   the same params-tolerance applied to each independently.
6. validity: SEQUENCE of exactly two times; each UTCTime (tag 0x17, 13 bytes
   `YYMMDDHHMMSSZ`) or GeneralizedTime (tag 0x18, 15 bytes
   `YYYYMMDDHHMMSSZ`); digits validated, ranges checked (month 1–12, day
   1–31, hour ≤ 23, minute/second ≤ 59 — calendar-exact day-per-month is NOT
   checked: that's interpretation, and CAs emit what they emit).
7. SPKI: the full TLV span handed to `spki::decode`; `None` bubbles.
8. `[1]`/`[2]` uniqueIDs: skipped if present (order enforced).
9. `[3]` extensions: if present, must be the last element; content kept as
   `extensions_raw`; the inner Extensions SEQUENCE is shape-checked
   (`read_sequence` + per-extension TLV walk, ensuring it is well-formed DER)
   but NOT interpreted — `critical` flags included (documented loudly).
10. signatureValue: at parse time, ONLY the BIT STRING shape is enforced
    (tag, canonical length, 0 unused bits). The content's `SEQUENCE
    { INTEGER r, INTEGER s }` structure and the r/s semantic checks (range,
    minimality) are checked exclusively by `decode_sig` inside
    `verify_with_id` at verify time — one source of truth, no duplicated
    signature parsing. A cert with garbage in the BIT STRING therefore
    PARSES but never VERIFIES.

## 6. Verification semantics

```text
verify_signature_with_id(issuer, id):
    verify_with_id(issuer, id, tbs_raw, signature_bitstring_content)
```

That is the entire function — `verify_with_id` already rejects over-long IDs
and identity public keys, so no pre-guard is duplicated here. All rejects (bad sig DER, r/s out of range, math
mismatch) collapse into `verify_with_id`'s existing `false` — no new failure
modes, no oracle surface. `verify_signature` = the same with
`DEFAULT_SIGNER_ID`.

Threat notes:
- Public inputs only ⇒ **no constant-time obligations and no dudect target**
  (Q3.9). The underlying `verify_with_id` is already the hardened verify path
  (identity-pubkey + over-long-ID rejects).
- The "two certificates, same tbs bytes" confusion attack (signature
  algorithm substitution) is closed by the outer==inner AlgId equality rule.
- Parser bombs: lengths are bounded by the input slice by reader
  construction; extensions are walked, not recursed into (no nested
  interpretation) — `from_der` is O(input length), no allocation
  proportional to anything but the input.

## 7. Assurance

- **KAT fixtures** (`crates/gmcrypto-core/tests/data/`): gmssl 3.1.1-generated
  self-signed SM2 CA + CA-issued leaf (the v0.3 W2 committed-fixture +
  regen-recipe pattern; gotcha: `gmssl certgen`/`certparse` CLI verbs).
  Tests (`tests/x509_kat.rs`, cfg-gated on `x509`):
  - parse fields byte-exact (serial, issuer/subject raw, validity, SPKI key);
  - CA cert self-verifies; leaf verifies against CA key;
  - leaf does NOT verify against its own key; CA does NOT verify the leaf
    with a wrong ID;
  - negatives: every tampered-tbs byte class (version/serial/spki/validity),
    swapped signature OID, params ≠ absent/NULL, non-zero BIT STRING unused
    bits, truncation sweep (every prefix of a valid cert must parse to
    `None`, never panic);
  - RFC 8998 example certificate cross-check (parse + verify) if its DER is
    reproducible from the RFC text.
- **Fuzz**: `fuzz_x509` — `Certificate::from_der` + (on successful parse)
  both verify calls against a fixed key; no-panic/OOM/hang. Census 26 → 27
  with `FUZZ_TARGETS` / fuzz README / CLAUDE.md updated in the same PR
  (the #102 invariant). Seeds: the two KAT certs + a minimal hand-built v3
  cert.
- **CI legs**: `x509` joins the clippy / test / deny / MSRV-1.85 / wasm32
  feature matrices (the sm2-key-exchange pattern); docs.rs already builds
  `--all-features`.
- **Docs**: README scope section + feature list + lib.rs crate-docs feature
  bullet + SECURITY.md one-liner (public-input rationale for no CT target) —
  the v1.2 public-web audit checklist applied at authoring time, not
  post-hoc.

maintainer's per-release call.
