# gmcrypto-core v0.4 scope

**Crate:** `gmcrypto-core` (plus new `gmcrypto-c` workspace member in W4).
**Status:** v0.3.0 published to crates.io 2026-05-11.
**Document scope:** the maintainer signs off on this scope before any
implementation work begins. Implementation plans for each workstream are
drafted *after* sign-off, the same chunked-PR cadence v0.2 / v0.3 used.
**Hard constraints:** all carry-overs from `CLAUDE.md` apply unchanged
(`unsafe_code = "forbid"` workspace-wide on `gmcrypto-core`, `#![no_std]`
+ `alloc`, MSRV 1.85, edition 2024, constant-time discipline on secrets,
failure-mode invariant, fixed-K masked-select sign retry, `Cargo.lock`
gitignored, `sign_raw_with_id` doc-hidden / not-SemVer). Don't restate
them; defer to `CLAUDE.md`. **One exception applies to W4 only**: the new
`gmcrypto-c` FFI crate cannot use `unsafe_code = "forbid"` because raw-
pointer FFI fundamentally requires `unsafe` blocks for slice
reconstruction and `Box::from_raw`. Q4.7 pins the discipline: a single
audited `unsafe` boundary per entry point with a `// SAFETY:` comment;
`gmcrypto-core` itself stays `unsafe_code = "forbid"`.

---

## 1. Context

v0.3.0 shipped 2026-05-11: reusable strict-canonical DER reader/writer
subset, PEM/PKCS#8/SPKI/SEC1 codecs (encrypted PKCS#8 via PBES2 +
PBKDF2-HMAC-SM3 + SM4-CBC), full bidirectional gmssl 3.1.1 interop
including SM2 sign/verify + SM2 encrypt/decrypt with PEM-wrapped keys +
SM4-CBC, raw byte-concat `C1||C3||C2` ciphertext helpers (`C1||C2||C3`
legacy decrypt-only), streaming `HmacSm3` / `Sm4CbcEncryptor` /
`Sm4CbcDecryptor` over in-crate `Hash`/`Mac`/`BlockCipher` traits, and
comb-table `mul_g` (~5× sign-side speedup) on top of `spin::Once`
runtime init. Dudect harness expanded to **12 targets** all gating at
`|tau| < 0.20`.

What v0.4 inherits as constraint surface:

- `gmcrypto-core` API surface as of `v0.3.0` is **stable for additive
  change only**. Removing or renaming a public item is a v1.0-track
  change; v0.4 may not break any v0.3 caller. Notably:
  - `traits::{Hash, Mac, BlockCipher}` (v0.3 W5) keep exact shape. The
    v0.4 W2 RustCrypto-trait fit adds *third-party* trait impls on the
    same types — `Sm3`, `HmacSm3`, `Sm4Cipher` — behind opt-in feature
    flags. v0.3's in-crate traits stay unbounded on public API surface
    (per the W5 W-risk in the v0.3 scope doc) and are not removed.
  - `Sm4Cipher::{encrypt_block, decrypt_block}` and the streaming
    `Sm4CbcEncryptor` / `Sm4CbcDecryptor` keep byte-identical output
    when v0.4 W3's bitsliced S-box is selected via the
    `sm4-bitsliced` feature. The feature is a *throughput-trade*
    swap, not a behavioural change.
  - `sm2::sign_with_id`, `sm2::verify_with_id`, `sm2::encrypt`,
    `sm2::decrypt`, `hmac_sm3`, `kdf::pbkdf2_hmac_sm3`, `pem::*`,
    `pkcs8::*`, `spki::*`, `sec1::*`, `asn1::*`, and
    `sm2::raw_ciphertext::*` — all preserved verbatim.
- Failure-mode invariant carries forward verbatim. The C ABI's int-
  return convention (Q4.8) is the new-shaped mirror: `0` = success,
  non-zero = single-`Failed`-equivalent; do not enumerate.
- Dudect targets: 12 today; v0.4 adds at most 1 (Q4.10: bitsliced
  SM4 gets its own target gated under `cfg(feature = "sm4-bitsliced")`).
  All new targets gate at `|tau| < 0.20`.
- Dep landscape today: `crypto-bigint = 0.7.3`, `subtle = 2.6.1`,
  `zeroize = 1.8.2`, `rand_core = 0.10.1`, `getrandom = 0.4.2`
  (dev-only, `sys_rng`), `spin = 0.10` (`once` feature only). Dev-deps
  `dudect-bencher = 0.7.0`, `hex-literal = 1.1.0`. `std` feature is
  opted-out by default; reserved for future file-I/O wire-format
  helpers and now also for v0.4 W4's `gmcrypto-c` FFI shim.
- v0.4 introduces **opt-in** runtime deps (W2's `digest` and `cipher`
  RustCrypto crates, plus the W4 `cbindgen` build-script dep on the
  `gmcrypto-c` crate). Per Q4.5, these are feature-gated and only
  activated when the caller asks; `cargo deny --exclude-dev` runs in
  the default (no-feature) configuration to preserve the zero-runtime-
  deps stance on the core build path. A second deny pass with the
  feature flags on runs in CI for visibility.

The bridge: v0.2 was the *primitive-complete* release; v0.3 was the
*interop-complete* release (wire formats + gmssl interop); v0.4 is the
*ecosystem-complete* release. Same primitives, same dudect-gated
constant-time posture, with three new doors into the crate:

1. **Other languages.** `gmcrypto-c` exposes a cdylib + staticlib + a
   cbindgen-generated header so C / C++ / Python / Go / Zig callers
   can drive SM2 / SM3 / SM4 / HMAC / PBKDF2 through opaque handles.
2. **Other targets.** `wasm32-unknown-unknown` becomes a supported
   build target — `gmcrypto-core` builds, links, and tests run on
   wasm32 (with caller-supplied RNG; the crate itself does not pull
   in `getrandom`'s `wasm_js` backend).
3. **RustCrypto fit.** `Sm3`, `HmacSm3`, `Sm4Cipher` get optional
   third-party trait impls (`digest::Digest`, `digest::Mac`,
   `cipher::BlockEncrypt`/`BlockDecrypt`) behind opt-in feature
   flags. v0.3's in-crate traits stay; v0.4 adds the third-party
   trait layer on top.

And one performance row:

4. **SM4 throughput.** v0.2's linear-scan S-box ships ~1-2M blocks/s.
   v0.4 W3 adds a Boyar-Peralta-style constant-time bitsliced S-box
   under `sm4-bitsliced`; target ~20-50× speedup while preserving the
   `ct_sm4_*` dudect gate.

Scope boundary: v0.4 does NOT change any cryptographic primitive's
on-wire output. SM2 / SM3 / SM4 byte paths are preserved bit-for-bit;
the bitsliced S-box is a constant-time-equivalence reimplementation
of the same S-box table. The dudect-gated constant-time properties
and the failure-mode invariant carry through unchanged.

## 2. In-scope workstreams

### W1 — `wasm32-unknown-unknown` build target

**Goal.** `gmcrypto-core` builds and tests run on
`wasm32-unknown-unknown` with **caller-supplied RNG only**. The crate
does NOT pull `getrandom`'s `wasm_js` backend or `js-sys`/`wasm-bindgen`
into its dependency graph; wasm callers wire their own
`rand_core::Rng` impl (typically via `getrandom = { features =
["wasm_js"] }` in the *caller's* `Cargo.toml`, not ours).

**Files touched.**

- New: `.github/workflows/ci-wasm.yml` (or a job inside the existing
  `ci.yml`) — `rustup target add wasm32-unknown-unknown` +
  `cargo build -p gmcrypto-core --target wasm32-unknown-unknown
  --no-default-features` + with `--features std` if `std` ever wires
  up wasm-compatible helpers (it doesn't today, so the `std` build is
  a smoke-only line).
- Modified: `Cargo.toml` workspace root — no functional changes;
  document wasm32 build target in a comment block. The `getrandom =
  "0.4.2"` dev-dep stays scoped to `[dev-dependencies]` so it does
  not infect the wasm build.
- Modified: `crates/gmcrypto-core/tests/interop_gmssl.rs` and any
  test that imports `getrandom::SysRng` — already gated behind
  `#[cfg(test)]` and only runs on platforms where `SysRng` works.
  No source change needed; verify wasm32 builds the test crate
  to `--target wasm32-unknown-unknown` *for the lib path only*
  (`cargo build --tests` would fail since `getrandom` `sys_rng` is
  not wasm32-portable; that's expected and documented).
- New: `README.md` "wasm32 support" subsection under "Build & test"
  with a 5-line caller recipe. Q4.2 decision pins this.

**Reuse.** All v0.3 code paths build under `no_std + alloc`. The
existing `spin = "0.10"` dep already supports wasm32 (`no_std` +
no platform-specific code). `crypto-bigint 0.7.3`, `subtle 2.6.1`,
`zeroize 1.8.2`, `rand_core 0.10.1` all already advertise wasm32
support in their respective release notes.

**KAT vectors.** No new KATs. The wasm32 path is *build-target* only;
KAT correctness is established on x86-64 and the wasm32 binary is
not different code, just a different target triple.

If the maintainer ever wants per-target KAT cross-validation
(`wasm-bindgen-test` driving the unit tests under Node or browser
WASM runtime), that's a *post-v0.4* deliverable. Q4.1 pins this:
build-target gate only for v0.4.

**Dudect implications.** None. `cargo bench` doesn't run on
wasm32 (no `criterion` / `dudect-bencher` wasm support, and the
harness needs OS-level RDTSC / `clock_gettime` access). Dudect
stays x86-only; gates unchanged.

**SemVer surface.** None. wasm32 is a target-triple feature, not
an API surface.

**Risks / failure-mode traps.**

- **Accidental `getrandom` re-export trap.** If any source file in
  `crates/gmcrypto-core/src/` ever `use`-imports `getrandom` outside
  `#[cfg(test)]`, the wasm32 build fails because `getrandom`'s
  `sys_rng` feature doesn't compile on wasm32 without the
  `wasm_js` backend. Verify with `grep -nR 'getrandom' crates/gmcrypto-core/src/`
  before merging W1. **Current state confirmed clean at v0.3.0**:
  all `getrandom` imports are inside `#[cfg(test)]` modules
  (`sm2::sign::tests`, `sm2::verify::tests`, `sm2::encrypt::tests`,
  `sm2::decrypt::tests`, `sm2::raw_ciphertext::tests`).
- **`std::sync::*` import trap.** Already enforced by
  `#![no_std]` on the lib root; the v0.3 W6 `spin::Once` decision
  is the precedent for "no `std::sync::LazyLock`".
- **`Vec<u8>` allocation under wasm.** `alloc::vec::Vec` Just Works
  under wasm32 (the wasm32 runtime / Wasmtime / browser provides
  the allocator); v0.4 W1 only confirms it compiles + links.
- **CI matrix expansion.** The wasm32 build adds ~30s to the
  CI build matrix. Acceptable; well under the 15-min CI cap.
  Workflow timeout stays at 15 minutes.
- **Don't enable `getrandom`'s `wasm_js` backend in `gmcrypto-core`.**
  Per Q4.2, that would force every wasm caller to use the JS-bound
  RNG even if they have their own (e.g. a wasi-runtime-supplied
  `wasi_random_get`). Caller-supplied RNG is the architectural
  invariant we keep.

### W2 — RustCrypto-trait fit (`digest::Digest`, `digest::Mac`, `cipher::BlockEncrypt`/`BlockDecrypt`)

**Goal.** Make `gmcrypto-core` a drop-in fit for any Rust pipeline
that expects RustCrypto's de-facto trait surface. Pure additive: v0.3's
in-crate `Hash` / `Mac` / `BlockCipher` traits stay; v0.4 wraps them
in RustCrypto's traits behind opt-in feature flags so callers using
`pbkdf2` / `hkdf` / `cipher::cbc::Encryptor` / `aead`-combinators
can drop in our primitives without an adapter.

**Files touched.**

- Modified: `crates/gmcrypto-core/Cargo.toml` — add optional deps
  `digest = { version = "0.10", default-features = false, optional =
  true }` and `cipher = { version = "0.4", default-features = false,
  optional = true }`. Add features:
  - `digest-traits` → enables `digest::Digest` impl on `Sm3` and
    `digest::Mac` (and `digest::MacMarker`, `digest::KeyInit`) impl
    on `HmacSm3`.
  - `cipher-traits` → enables `cipher::BlockEncrypt` /
    `cipher::BlockDecrypt` / `cipher::KeyInit` /
    `cipher::BlockSizeUser` / `cipher::KeySizeUser` impls on
    `Sm4Cipher`.

  Q4.3 / Q4.4 pin the feature naming and the version lines.
- Modified: `crates/gmcrypto-core/src/sm3.rs` — add a
  `#[cfg(feature = "digest-traits")]` `impl digest::FixedOutput` /
  `impl digest::Update` / `impl digest::Reset` /
  `impl digest::OutputSizeUser` block for `Sm3`. `digest::Digest`
  is the supertrait; the blanket impl is automatic once the four
  subtraits are satisfied.
- Modified: `crates/gmcrypto-core/src/hmac.rs` — add a
  `#[cfg(feature = "digest-traits")]` impl block for `HmacSm3`
  satisfying `digest::Mac` (via `KeyInit + Update + FixedOutput +
  MacMarker`).
- Modified: `crates/gmcrypto-core/src/sm4/cipher.rs` — add a
  `#[cfg(feature = "cipher-traits")]` impl block for `Sm4Cipher`
  satisfying `cipher::BlockEncrypt + BlockDecrypt + KeyInit +
  BlockSizeUser + KeySizeUser`. Block size is the existing
  `BLOCK_SIZE = 16`, key size is `KEY_SIZE = 16`.
- Modified: `crates/gmcrypto-core/src/traits.rs` — add a module-level
  doc note explaining how the in-crate traits relate to the
  feature-gated RustCrypto traits: same operations, two surfaces.
- New: `crates/gmcrypto-core/tests/rustcrypto_traits.rs` —
  feature-gated integration test that pulls in
  `pbkdf2 = { version = "0.12", features = ["hmac"] }` (or similar
  RustCrypto consumer crate) and feeds `HmacSm3` to it. Verifies
  cross-ecosystem composition produces the same bytes as our
  in-crate `kdf::pbkdf2_hmac_sm3`. **Test is dev-dep gated**, never
  affects the production build.
- Modified: `deny.toml` — `digest` and `cipher` (and their transitive
  deps: `block-buffer`, `crypto-common`, `inout`, etc.) added to
  the workspace allow-list, **only under the feature-on configuration**.
  Default `cargo deny check --exclude-dev` continues to pass with
  zero new runtime deps.

**Reuse.**

- `Sm3::{new, update, finalize}` shape already matches `digest`'s
  `Update + FixedOutput + Reset` triple. The trait impls are thin
  glue: `Update::update` delegates to `Sm3::update`,
  `FixedOutput::finalize_into` writes the finalized digest into
  the caller-supplied `GenericArray<u8, U32>`, `Reset::reset`
  re-initializes from `IV`.
- `HmacSm3::{new, update, finalize, verify}` shape matches
  `digest::Mac` exactly. The `verify` semantics already use
  `subtle::ConstantTimeEq`; `digest::Mac::verify_slice` uses the
  same primitive under the hood (RustCrypto's `Mac::verify_slice`
  documented contract).
- `Sm4Cipher::{new, encrypt_block, decrypt_block}` shape matches
  `cipher::BlockEncrypt + BlockDecrypt` exactly. The trait works
  on `inout::InOut<Block>`; our existing in-place `encrypt_block`
  uses `&mut [u8; 16]`. The trait impl converts.

**KAT vectors.** No new cryptographic KATs. The trait impls produce
the same bytes as the existing in-crate paths — that's the contract.
Regression tests:

- `tests/rustcrypto_traits.rs`: drive `pbkdf2::pbkdf2_hmac::<HmacSm3>(
  password, salt, iterations, &mut out)` and compare to
  `kdf::pbkdf2_hmac_sm3(password, salt, iterations, &mut out)`.
  All v0.2/v0.3 PBKDF2 KAT vectors round-trip.
- `tests/rustcrypto_traits.rs`: drive
  `<Sm3 as digest::Digest>::digest(b"abc")` and compare to
  `sm3::hash(b"abc")`. GB/T 32905 KATs (empty, "abc", 16×"abcd")
  round-trip.
- `tests/rustcrypto_traits.rs`: encrypt a 16-byte block via
  `<Sm4Cipher as cipher::BlockEncrypt>::encrypt_block` and compare
  to `Sm4Cipher::encrypt_block` direct. GB/T 32907 single-block
  KAT (`01 23 45 67 89 ab cd ef fe dc ba 98 76 54 32 10` →
  `68 1e df 34 d2 06 96 5e 86 b3 e9 4f 53 6e 42 46`) round-trips.

**Dudect implications.** None new. The RustCrypto trait impls are
thin wrappers over the existing constant-time-discipline paths
(`Sm3::update`, `HmacSm3::*`, `Sm4Cipher::encrypt_block`). v0.3's
`ct_hmac_sm3`, `ct_sm4_encrypt_block`, `ct_sm4_key_schedule` targets
already gate the underlying paths; the trait impls don't introduce
new branches or data-dependent control flow. **No new target.**

**SemVer surface.** Additive only, and feature-gated:

- New optional crate features: `digest-traits`, `cipher-traits`.
- New `impl` blocks on `Sm3`, `HmacSm3`, `Sm4Cipher` for the
  RustCrypto traits, behind `#[cfg(feature = "...")]`. Default-
  feature build is unchanged.
- New optional runtime deps (`digest`, `cipher`) appear in
  `Cargo.toml` but are not pulled unless the feature is on. Default
  `cargo deny check --exclude-dev` continues to gate only the
  default surface.

**Risks / constant-time / failure-mode traps.**

- **MSRV drift trap.** RustCrypto's `digest 0.10` and `cipher 0.4`
  current MSRV is 1.71 / 1.74 respectively (well below our 1.85),
  but a future minor bump could drag MSRV upward. **Pin major+minor
  with a comment in `Cargo.toml`** (`digest = "0.10"` accepts
  0.10.x; explicitly do not move to 0.11 in v0.4.x). Document the
  pin policy in CLAUDE.md after the W2 merge.
- **`block-buffer` transitive dep.** `digest 0.10`'s `Update` trait
  in its lower-level path uses `block-buffer` for the streaming
  framing; our `Sm3` already has its own internal buffer. Implement
  `Update` directly on `Sm3` without going through `block-buffer`
  — saves the transitive surface and uses our existing
  battle-tested buffering. Same for `HmacSm3`.
- **`GenericArray` vs `[u8; 32]` impedance.** `digest`'s output
  type is `digest::Output<Self>` = `GenericArray<u8, OutputSize>`.
  Convert from our `[u8; 32]` via `GenericArray::from(arr)` /
  `(*ga).into()`. No allocation; just type laundering.
- **`KeyInit` panic semantics.** `cipher::KeyInit::new` takes a
  `&Key<Self>` (i.e. `&GenericArray<u8, U16>`); panics on the wrong
  size are impossible because the type carries the length. Our
  existing `Sm4Cipher::new(&[u8; 16])` panic-on-wrong-length is
  preserved on the non-trait path; the trait path can't reach it.
- **Feature interaction with `std` feature.** RustCrypto's
  `digest 0.10` and `cipher 0.4` both default to `default-features =
  false` for `no_std + alloc`. Pin `default-features = false` in
  our `Cargo.toml`; opt back into `std` only if a downstream caller
  requests our `std` feature *and* RustCrypto's `std` feature
  expressly. v0.4 leaves the RustCrypto-`std` interaction
  unspecified; the default no-std build path is the supported one.
- **Don't bound public API on RustCrypto traits.** Same trap as
  v0.3 W5's in-crate `Hash`/`Mac`/`BlockCipher`. The trait impls
  are *implementations on our types*, not *bounds on our public
  fns*. `pub fn pbkdf2_hmac_sm3` stays concrete; do not
  generic-over-PRF it.

### W3 — Bitsliced SM4 S-box (`sm4-bitsliced` feature)

**Goal.** Replace the v0.2 W1 linear-scan-S-box's 256-iteration
`subtle::ConditionallySelectable` scan with a Boyar-Peralta-style
constant-time bitsliced S-box: ~140 boolean ops per byte, no
data-dependent memory access, ~20-50× throughput speedup over
linear-scan while preserving the `ct_sm4_*` dudect gate.

**Files touched.**

- New: `crates/gmcrypto-core/src/sm4/sbox_bitsliced.rs` — the
  Boyar-Peralta-style gate sequence implementing the SM4 S-box.
  ~140 `XOR` / `AND` / `OR` / `NOT` operations on 8 input bits
  produce 8 output bits. Reference: the SM4 S-box decomposition
  is structurally similar to AES's; published gate counts hover
  around 100-150 (see e.g. Yang et al., "An Energy-Efficient ASIC
  Implementation of SM4 with Optimized Substitution Box", and
  the RustCrypto `aes` crate's `bitslice` mod for AES analog).
  Module is `pub(super)`; only `sm4::cipher` consumes it.
- Modified: `crates/gmcrypto-core/src/sm4/cipher.rs` — gate the
  S-box call inside `tau` (the four parallel S-box invocations
  per round) on `cfg(feature = "sm4-bitsliced")`. Default build
  (feature off) uses the existing linear-scan path verbatim.
  Both paths produce byte-identical output for every input — the
  gate-equivalence-to-table is the cryptographic correctness
  contract.
- Modified: `crates/gmcrypto-core/Cargo.toml` — add `sm4-bitsliced`
  to `[features]`. Default off (Q4.9).
- Modified: `crates/gmcrypto-core/benches/timing_leaks.rs` — add
  `ct_sm4_encrypt_block_bitsliced` target gated under
  `cfg(feature = "sm4-bitsliced")` (Q4.10). Class-split by master
  key bytes, same shape as v0.2 W1's `ct_sm4_encrypt_block`;
  measures the bitsliced path directly. Gate `|tau| < 0.20`.
- Modified: `.github/workflows/dudect-pr.yml` and
  `.github/workflows/dudect-nightly.yml` — add a second pass with
  `--features sm4-bitsliced` so the new target gates in CI.
  First pass (default features) measures the 12 existing targets;
  second pass measures `ct_sm4_encrypt_block_bitsliced` (and
  re-measures `ct_sm4_*` under the bitsliced path for
  defense-in-depth equivalence). Combined runtime under the
  15-min CI cap.
- Modified: `crates/gmcrypto-core/tests/data/` — no new KATs;
  v0.2's GB/T 32907 single-block + 1M-round vectors are the
  byte-equivalence gate.

**Reuse.**

- v0.2's `Sm4Cipher` struct layout and round-key array. Bitsliced
  S-box is a *substitution under tau*, not a structural change;
  the linear `L`/`L'` permutations, the round-key schedule's
  `T'` function, the 32-round Feistel structure all stay.
- `subtle` is **not used in the bitsliced path**. Boyar-Peralta's
  output is bit-by-bit boolean composition — every gate operates
  on independent bit slots, no data-dependent branch. The
  resulting code is constant-time *by construction*; `subtle`'s
  conditional-select primitive is unnecessary (and would just be
  noise) in the inner loop. The cfg-gated implementation provides
  its own constant-time guarantee.
- `zeroize::Zeroize` / `ZeroizeOnDrop` posture on `Sm4Cipher`
  unchanged. Round keys still zeroize on drop; bitsliced S-box
  intermediates (8 booleans per byte transformed) are stack-only
  and ephemeral.

**KAT vectors.** Byte-identical-output equivalence:

- GB/T 32907 single-block KAT under both feature-off and
  feature-on builds:
  `key = pt = 01 23 45 67 89 ab cd ef fe dc ba 98 76 54 32 10`
  → `ct = 68 1e df 34 d2 06 96 5e 86 b3 e9 4f 53 6e 42 46`.
- GB/T 32907 1,000,000-round KAT — re-run under `--release
  --features sm4-bitsliced -- --ignored`. Expected: bitsliced
  finishes the 1M-round test in seconds (vs. minutes for linear-
  scan), and produces the same final ciphertext
  `59 52 98 c7 c6 fd 27 1f 04 02 f8 04 c3 3d 3f 66`.
- New `crates/gmcrypto-core/src/sm4/sbox_bitsliced.rs` unit test:
  for every byte `b ∈ 0..=255`, `sbox_bitsliced(b) == S_BOX[b]`.
  This is the byte-level cryptographic-correctness gate — if it
  fails on any input, the round-level KATs are moot.

**Dudect implications.**

- **One new target: `ct_sm4_encrypt_block_bitsliced`** (Q4.10
  decision — definite addition). Mirror of v0.2's
  `ct_sm4_encrypt_block`: class-split by master key bytes,
  "construct cipher + encrypt one block" timed under one
  window. Gates `|tau| < 0.20`. CI runs this target *only* under
  `--features sm4-bitsliced`; default-features-off CI continues
  to measure the 12 existing targets.
- **Don't drop `ct_sm4_encrypt_block`.** The default (linear-scan)
  path stays the supported v0.3 surface for callers who cannot
  enable the feature (e.g. compiled-out by a downstream `default-
  features = false` cascade). Both targets coexist.
- **No `ct_sm4_key_schedule_bitsliced`.** Key schedule uses the
  same `tau` substitution; if the per-byte bitsliced S-box is
  constant-time, the key schedule inherits it structurally. Same
  rationale v0.2 W1 used for skipping a separate `ct_sm4_decrypt`
  target. (Documented in the W3 implementation PR.)

Adds 1 new target (`ct_sm4_encrypt_block_bitsliced`); total dudect
targets reach **13 under `--features sm4-bitsliced`**, **12 under
default features**. Same `|tau| < 0.20` gate across both
configurations.

**SemVer surface.** Additive only:

- New crate feature `sm4-bitsliced` (default off).
- New module `sm4::sbox_bitsliced` is `pub(super)` only — not in
  the public API.
- `Sm4Cipher::{encrypt_block, decrypt_block}` signatures unchanged;
  the feature flag is a *throughput-trade implementation swap*,
  not a behavioural change.

**Risks / constant-time / failure-mode traps.**

- **Boyar-Peralta gate-count correctness.** Wrong gate sequence
  → wrong ciphertext on some inputs. The per-byte exhaustive test
  (`for b in 0..=255 { assert_eq!(sbox_bitsliced(b), S_BOX[b]) }`)
  is the gate — if it passes, the S-box is correct on every
  possible input; the full 32-round cipher then inherits
  correctness from the gate-level test.
- **Constant-time-by-construction trap.** Boolean ops on `u8` /
  `u32` are constant-time on every Rust target gmcrypto-core
  supports today (x86, x86-64, aarch64, wasm32). **No
  variable-time-multiplier CPU is in scope** (`CLAUDE.md` Don't).
  Document this assumption in the module-doc.
- **`#[inline]` discipline.** The bitsliced S-box is small enough
  to `#[inline(always)]` into `tau`. Verify on the W3 branch with
  `cargo asm` (or `cargo +nightly rustc -- --emit=asm`) that no
  table-lookup is inserted by the optimizer at any optimization
  level the supported builds use (`opt-level = 3` for release;
  the `RUSTFLAGS = "-D warnings"` on CI; profile-guided opt OFF).
- **Don't enable `sm4-bitsliced` by default in v0.4.** Q4.9
  decision: feature stays off-default. Callers opt in. Same posture
  as W2's `digest-traits` / `cipher-traits`.
- **Don't ship a SIMD-packed multi-block bitsliced variant.**
  Multi-block parallel bitsliced (e.g. 8-way for parallel CTR
  mode) is a separate workstream; deferred post-v0.4. Q4.11 pins
  this. The v0.4 W3 scope is *single-block bitsliced S-box*,
  which gives a substantial speedup on its own without the
  added complexity of packed-block layouts.
- **Don't replace the linear-scan S-box in the default build.**
  The v0.2 W1 decision ("subtle-style linear-scan is the
  throughput trade") stays the documented default. The bitsliced
  path is an opt-in fast path; both paths are supported.
- **`unsafe_code = "forbid"` on the bitsliced impl.** No unsafe
  needed — boolean gates compile fine to `u32` / `u8` operations
  in safe Rust. The Boyar-Peralta sequence is a sequence of XOR /
  AND / NOT calls on `u8`-typed locals.

**Reuse.**

- All v0.3 public API on `gmcrypto-core`. The FFI shim is pure
  glue: pointer-validate, slice-reconstruct, call `gmcrypto-core`,
  map result to int return code.
- `Sm2PrivateKey`, `Sm2PublicKey`, `Sm4Cipher`, `Sm3`, `HmacSm3`
  are all the opaque handle types. C callers get a `void*`-typed
  handle from `*_new` and pass it back to other entry points.
  Free with `*_free`. (Q4.6 pins this handle model.)
- `zeroize::ZeroizeOnDrop` on `Sm2PrivateKey` and `Sm4Cipher` —
  inherits via `Box::from_raw` in `*_free`: the Rust drop glue
  runs, zeroizing the inner state, before the heap slot is freed.

**KAT vectors.** No new cryptographic KATs. The C-side correctness
gate is *Rust-equivalence*:

- `tests/c_smoke.rs`: every entry point produces the same bytes as
  the equivalent direct call into `gmcrypto-core` from Rust. Covers

  SEC1 key import/export round-trip, PKCS#8 password-encrypted key

- The C example program `examples/sm2_sign.c` is documentation

  toolchain to CI if demand surfaces.

**Dudect implications.**

- **No new dudect target.** The FFI shim does not introduce new
  secret-touching code paths. Every cryptographic op runs in
  `gmcrypto-core`'s already-dudect-gated code; the C entry point
  is a thin null-check + slice-reconstruct + delegate.
- **Watch for null-check timing leak.** A null pointer check via
  `ptr.is_null()` is constant-time (single integer compare). A
  return-on-null-via-early-exit has a different timing signature
  than a successful call — but the *attacker who could measure
  this* is local-host, and at that point they have far more
  invasive options. Not in scope; document in the W4 PR.

**SemVer surface.** New (the `gmcrypto-c` crate is brand-new):

- The C header `gmcrypto.h` is the v0.4 ABI contract. Any
  signature change in v0.4.x is a breaking ABI change; v0.5 may
  break the C ABI.
- The Rust-side `gmcrypto-c` lib is `pub` but the recommended
  consumption mode is via the cdylib / staticlib + header.
- `gmcrypto-core` itself is unchanged by W4 — W4 is purely additive
  via a new workspace member.

**Risks / constant-time / failure-mode traps.**

- **Pointer-deref safety contract.** Every `unsafe` block in
  `gmcrypto-c/src/lib.rs` carries a `// SAFETY:` comment naming
  the caller-side preconditions (non-null, valid length, no
  aliasing-on-mutation). Reviewer gate.
- **`unsafe_code = "forbid"` exception.** `gmcrypto-c` cannot
  forbid unsafe — slice reconstruction requires it. Q4.7 pins
  `unsafe_code = "deny"` (default Rust posture). Document the
  scope of unsafe (only at FFI entry points; the entire inner
  call path is safe Rust through `gmcrypto-core`). The workspace-
  wide CLAUDE.md "forbid" rule applies to `gmcrypto-core`; the
  `gmcrypto-c` crate is a deliberate exception.
- **Failure-mode invariant via int return.** `0` = success; every
  non-zero return is equivalent failure. **Do not enumerate
  named error codes.** Q4.8 pin. C consumers who want failure-mode
  diagnostics get debug-mode log output (printed to stderr; off
  by default; gated on `GMCRYPTO_C_DEBUG=1`).
- **Free vs. drop ordering.** Every handle type has a paired
  `_free` entry point. Calling `_free` on a null pointer is a
  no-op (return without error). Calling `_free` twice on the same
  pointer is undefined behaviour (the C caller's contract; mirror
  `free()`'s standard semantics). Document in the header.
- **Output-buffer convention.** `(out, out_cap, out_actual_len)`.
  If `out_cap < out_actual_len`, return non-zero and write the
  required length into `*out_actual_len` so the C caller can
  re-allocate. Q4.13 pin.
- **Zeroize-on-free.** Handles wrapping `Sm2PrivateKey`,
  `Sm4Cipher`, `HmacSm3` zeroize their inner state on `_free` via
  the inherited `ZeroizeOnDrop` glue. Handles wrapping public
  types (`Sm2PublicKey`, `Sm3`) do not need to zeroize. Document
  the contract in the header per-type.
- **Don't leak Rust panics across the FFI boundary.** Wrap every
  entry point's body in `std::panic::catch_unwind`; on panic,
  return non-zero (failure code). UB if a Rust panic unwinds into
  C. Per the failure-mode invariant, the caller cannot distinguish
  panic from other failure; that's fine.
- **cbindgen drift.** The header is checked in. CI re-runs
  cbindgen and diffs to verify no drift. If a maintainer adds an
  entry point without regenerating the header, CI fails. Q4.12.
- **`#[unsafe(no_mangle)]` discipline.** Rust 2024 edition requires
  the explicit `#[unsafe(no_mangle)]` attribute (vs. the legacy
  bare `#[no_mangle]`). All FFI entry points use the explicit
  attribute; the bare attribute is a clippy warning in 2024.
- **Don't expose `sign_raw_with_id`.** The dudect-harness-only
  `#[doc(hidden)]` shim stays Rust-side. The C ABI gets the
  full DER-emitting `gmcrypto_sm2_sign` only.
- **`Sm2PrivateKey::to_sec1_be` exposure.** Q4.19: the C ABI does
  expose `gmcrypto_sm2_privkey_to_sec1_be` — C callers need the
  scalar bytes to wire into other Chinese-standard libraries.
  Document the caller-must-zeroize contract in the header. Same
  `#[doc(hidden)]` / not-SemVer posture as the Rust side.

## 3. Sequencing

W1 → W2 (independent foundation) → W3 (SM4 fast-path) → W4 (C ABI
on top of stabilized v0.4 surface). **W1 and W2 can land in parallel**;
W3 starts after W1 to avoid CI matrix collisions on the dudect
workflow path-allowlist; W4 lands last to absorb a stable v0.4
surface.

**Dependency graph.**

```text
W1 (wasm32 build target)         — independent of all
W2 (RustCrypto-trait fit)        — independent of all
  ↓ (only ordering pressure: shape stability for C ABI mirror)
W3 (bitsliced SM4 S-box)         — touches sm4 module, depends only on W1's CI matrix slot
  ↓
W4 (C ABI / gmcrypto-c crate)    — depends on a stable v0.4 Rust API surface
                                   (W2's optional trait impls don't appear in
                                   the C ABI; bitsliced S-box is opt-in via
                                   the feature, mirrored on the C side via
                                   the same feature flag.)
```

**Phase plan.**

- **Phase 1 (W1).** wasm32 build target. One PR. Codex review
  after. Gate: `cargo build -p gmcrypto-core --target
  wasm32-unknown-unknown --no-default-features` succeeds in CI;
  no source change beyond a workflow file and a README subsection.
- **Phase 2 (W2).** RustCrypto-trait fit. One PR, two commits
  (digest first, then cipher). Codex review per PR. Gate:
  `tests/rustcrypto_traits.rs` passes under `--features
  digest-traits,cipher-traits`; default-features build unchanged;
  `cargo deny --exclude-dev` continues to pass.
- **Phase 3 (W3).** Bitsliced SM4 S-box. One PR. Codex review.
  Gate: per-byte exhaustive equivalence test passes; GB/T 32907
  single-block + 1M-round KATs pass under `--features
  sm4-bitsliced`; `ct_sm4_encrypt_block_bitsliced` measures
  `|tau| < 0.20` at 100K samples.
- **Phase 4 (W4).** C ABI / `gmcrypto-c` crate. **Largest PR**;
  may split into 4a (crate skeleton + workspace member + cbindgen
  build), 4b (SM3 / HMAC / PBKDF2 / SM4 / SM4-CBC entry points),
  4c (SM2 entry points + key import/export), 4d (CI integration
  + header-drift gate). Codex review per chunk. Gate:
  `tests/c_smoke.rs` round-trips against every entry point;
  cbindgen header is committed; CI's cabi-build-and-header-drift
  job passes.

**Concurrency.** W1 and W2 can land in parallel — they touch
disjoint files (workflow yaml + manifest + a tests file vs.
trait impls in source modules). W3 should land *after* W1 because
W1's wasm32-build job is shared infrastructure; landing W3 first
means re-running W1 CI configuration on the W3 branch. W4 lands
last because its surface mirrors the Rust API; a stabilizing W2
shape reduces churn on the cbindgen header.

**Why this order.**

- W1 is the smallest, lowest-risk gain — banking it first
  validates the autonomous cadence and provides a clean CI matrix
  template for the remaining workstreams.
- W2 is independent of every other workstream and adds the
  largest ecosystem-fit value per LOC. Ship it second so callers
  who only care about the trait fit can pick up v0.4 from
  pre-release builds.
- W3 is medium-size and touches a tight subset of files (sm4
  module + benches). Ships behind a feature flag; no risk to
  default-build callers.
- W4 is the largest workstream by LOC and the highest-risk
  surface (FFI invariants, unsafe blocks, header generation,
  cdylib build). Land last so the v0.4 Rust API is stable and
  the FFI shim doesn't churn.

## 4. Out of scope for v0.4

- **API stabilization.** v1.0. v0.4 may add public surface; it
  does not lock down what's there.
- **TLS / TLCP / GM-TLS profiles.** Not on any roadmap row.
- **Certificate chain validation, CRL, OCSP, CSR, CMS.** Not on
  roadmap.
- **HKDF, scrypt, Argon2.** v0.4 ships PBKDF2-HMAC-SM3 (already
  v0.2); other KDFs are post-v0.4.
- **HSM / SDF / SKF integration.** Not on roadmap.
- **SM9 / ZUC / post-quantum.** Not on roadmap.
- **AEAD modes (SM4-GCM / SM4-CCM).** Not on roadmap for v0.4;
  candidate for v0.5+.
- **`std::io::Read`/`Write`-driven streaming.** v0.4's W2 traits
  work on `&[u8]` chunks; integration with `std::io` traits is a
  separate concern (would require the `std` feature) and is
  deferred.
- **`wasm-bindgen-test`-driven test runner on wasm32.** Q4.1
  decision: build-target gate only for v0.4. Test cross-validation
  on wasm32 is post-v0.4.
- **`getrandom`'s `wasm_js` backend wired in `gmcrypto-core`.**
  Q4.2 decision: caller-supplied RNG only. Pulling in `wasm-
  bindgen` / `js-sys` transitively into our default dep graph is
  off-the-table.
- **`digest 0.11` / `cipher 0.5` (or any future major bump) trait
  fit.** Q4.4: pinned at `digest = "0.10"` and `cipher = "0.4"`.
  v0.4.x stays on these lines; a future major-bump is a v0.5
  candidate.
- **Default-on `sm4-bitsliced` feature.** Q4.9: stays off-default.
  Callers opt in.
- **Multi-block parallel bitsliced SM4 (SIMD-packed).** Q4.11
  decision: single-block Boyar-Peralta-style bitsliced is the
  v0.4 W3 scope. Multi-block SIMD packing is post-v0.4.
- **C++ / Go / Python language bindings beyond the bare C ABI.**
  v0.4 ships C ABI + cbindgen header. Anyone can wrap that header
  in Go (cgo), Python (cffi / ctypes), Ruby (FFI), Zig
  (`@cInclude`), etc. — language-specific wrapper crates are
  post-v0.4.
- **C-side example builds in CI.** Q4.14: examples are docs-only
  for v0.4. Adding a C toolchain to CI is post-v0.4.
- **Caller-supplied RNG callback on the C ABI.** Q4.18: the C
  ABI uses `getrandom::SysRng` internally for v0.4. A "register
  custom RNG" entry point is post-v0.4.
- **A separate `ct_sm4_key_schedule_bitsliced` dudect target.**
  The W3 bitsliced S-box covers the key-schedule path
  structurally (same tau-substitution primitive). Same precedent
  as v0.2's `ct_sm4_encrypt_block` covering `_decrypt_block`.
- **A "raw byte-concat SM2 ciphertext" C-ABI entry point.** Q4.17:
  v0.3 W4's `encode_c1c3c2` / `decode_c1c3c2` /
  `decode_c1c2c3_legacy` are Rust-only for v0.4. C consumers
  asking for raw byte-concat ciphertext are most likely interop-
  bridging to a non-gmssl 3.1.1 Chinese-standard library; that
  interop bridge can use the Rust crate directly. A C-ABI wrap
  is a post-v0.4 chore.
- **Streaming CBC FFI entry points.** Q4.16: v0.4 C ABI ships
  single-shot CBC only. Streaming `Sm4CbcEncryptor` /

## 7. Decisions (autonomous-mode sign-off, 2026-05-11)

Each question below is now answered by the autonomous-execution
agent per the prompt's instruction to "make the calls and document
them" with each Q defaulted to its "Recommended:" option (matching
the v0.3 sign-off pattern). Pro/con bullets remain so future readers
can see the trade-offs that were considered.

### Q4.1 wasm32 testing — `wasm-bindgen-test` runner in v0.4?

`wasm-bindgen-test` lets `cargo test --target wasm32-unknown-unknown`
drive unit tests under Node or a headless browser. Adds
`wasm-bindgen-test` as a dev-dep and `wasm-bindgen-cli` as a CI
prereq.

- **Pro:** real KAT cross-validation on wasm32, not just build.
- **Con:** new dev-dep tree (`wasm-bindgen`, `js-sys`, etc.) — even
  dev-only, it shows up in `cargo metadata` and complicates the
  audit story. CI infrastructure overhead (Node install).
- **Alternative:** build-target gate only — confirm `cargo build
  --target wasm32-unknown-unknown` succeeds.

**Question:** ship `wasm-bindgen-test` driving the tests on wasm32,
or build-target gate only?

**Recommended:** build-target gate only.

**Decision:** **build-target gate only.** v0.4 W1 is a *build-target*
deliverable. KAT correctness is established on x86-64. wasm32 is a
target-triple invariant; the crate is `no_std` + `alloc` only and
no platform-specific code path differs by target. Cross-validation
test runner is a post-v0.4 chore.

### Q4.2 wasm32 RNG — pull `getrandom`'s `wasm_js` backend?

`getrandom 0.4.2` has a `wasm_js` feature that wires `crypto.getRandomValues`
on wasm32. Without it, wasm32 callers must supply their own RNG
(typically via `getrandom = { features = ["wasm_js"] }` in their own
manifest).

- **Pro:** OOTB wasm32 usability for callers who don't care about
  RNG sourcing.
- **Con:** drags `wasm-bindgen` + `js-sys` into the default dep
  graph. Not pure-Rust anymore on wasm32 targets. Forces the JS
  binding even on `wasi` targets that have their own random source.
- **Alternative:** caller-supplied RNG only — `gmcrypto-core` takes
  `&mut impl Rng` everywhere; callers can wire any RNG.

**Question:** wire `wasm_js` in v0.4, or caller-supplied only?

**Recommended:** caller-supplied only.

**Decision:** **caller-supplied only.** Architectural invariant: the
crate's `sign` / `encrypt` paths take `&mut impl rand_core::Rng`.
Callers on wasm32 wire their own RNG (typically via
`getrandom` with `wasm_js` in *their* manifest, or any other RNG
that satisfies `rand_core::Rng`). Document the caller recipe in
the README.

### Q4.3 RustCrypto trait feature naming — one flag or two?

Options:

- **One unified `rustcrypto-traits` flag.** Pulls in both `digest`
  and `cipher`; simpler API.
- **Two flags (`digest-traits`, `cipher-traits`).** Splits the
  dep tree; callers who only need one don't pay for the other.

**Recommended:** two flags.

**Decision:** **two flags (`digest-traits`, `cipher-traits`).**
Matches RustCrypto's own crate split (`digest` and `cipher` are
separate crates). Callers using `pbkdf2` need `digest-traits` only;
callers using `cbc::Encryptor<Sm4Cipher>` need `cipher-traits`
only. Minimal blast radius per opt-in.

### Q4.4 RustCrypto dep version pinning — 0.10 / 0.4 or follow latest?

`digest = "0.10"` and `cipher = "0.4"` are the current stable lines;
both have been stable for 2-3 years. RustCrypto is preparing 0.11 /
0.5 (post-2025) with breaking trait shape changes.

- **Pro of pinning major+minor:** stable MSRV; no surprise upgrades.
- **Con:** doesn't track upstream improvements; future v0.4.x can't
  silently upgrade.

**Recommended:** pin major+minor with a CLAUDE.md note.

**Decision:** **pin major+minor.** `digest = "0.10"`,
`cipher = "0.4"`. Both pulled with `default-features = false`. A
future major-bump (0.11 / 0.5) is a v0.5 candidate, not a v0.4.x
silent change.

### Q4.5 Feature-on `cargo deny` posture — block opt-in deps or allow?

W2 introduces opt-in `digest` / `cipher` deps; W4 introduces opt-in
`cbindgen` build-script dep. Default `cargo deny check --exclude-dev`
sees zero new deps; what about under `--all-features`?

- **Pro of allowing:** authoritative cross-check that opt-in features
  don't accidentally pull a banned dep.
- **Con of strict allow-list growth:** every new opt-in dep needs
  an allow-list entry with a rationale comment. More churn on
  `deny.toml`.

**Recommended:** dual posture — default deny stays zero-runtime-dep;
add an `--all-features` deny check as a CI-only signal job
(non-blocking on individual PRs but blocks release).

**Decision:** **dual posture.** Default `cargo deny check --exclude-dev`
continues to gate the default-features dep graph (zero new runtime
deps in v0.4 default builds). A second CI job runs
`cargo deny check --exclude-dev --all-features` and the opt-in
deps (`digest`, `cipher`, `cbindgen`, their transitives) are
allowlisted with rationale comments pointing back to Q4.5. Release
review gates on both.

### Q4.6 C ABI handle model — `Box::into_raw` or slab/index?

- **`Box::into_raw`:** heap-allocate, return `*mut T`; caller
  passes back to `_free`; `Box::from_raw` reclaims and drops.
  Standard idiom; one heap allocation per handle.
- **Slab/index:** internal slab indexed by `u64` / `u32`; handle is
  the index; `_free` releases the slot. More complex; lets us
  cap the total number of handles; supports fuzzing into invalid
  indices safely. Adds runtime state (the slab).

**Recommended:** `Box::into_raw`.

**Decision:** **`Box::into_raw`.** Standard Rust FFI idiom. Each
entry point's `// SAFETY:` block names the caller contract
(non-null, valid pointer, no double-free). Slab/index is over-
engineering for v0.4's audience; can revisit if FFI hardening
demand surfaces.

### Q4.7 `unsafe_code` posture on `gmcrypto-c`

CLAUDE.md says `unsafe_code = "forbid"` workspace-wide. The
`gmcrypto-c` crate cannot satisfy `forbid` — slice reconstruction
(`from_raw_parts`), `Box::from_raw`, `Box::into_raw` all require
`unsafe`. Options:

- **`unsafe_code = "deny"`** (default Rust posture; per-block
  opt-in with `// SAFETY:` comments).
- **`unsafe_code = "forbid"`** (impossible; would block compile).
- **Hide all FFI in a single audited module and `#[allow(unsafe_code)]`
  on just that module.** Practically equivalent to `deny`.

**Recommended:** `deny` with rigorous per-block `// SAFETY:` discipline.

**Decision:** **`unsafe_code = "deny"`** (default Rust posture).
Every `unsafe` block carries a `// SAFETY:` comment naming the
caller-side preconditions. CLAUDE.md is updated post-W4 to
document the workspace-wide `forbid` policy with an explicit
exception for `gmcrypto-c`. The exception is named, scoped, and
audited.

### Q4.8 C ABI error reporting — int return or named enum?

- **Int return** (0 = success; non-zero = failure). One int per
  entry point. Matches the failure-mode invariant verbatim.
- **Named enum** (`GMCRYPTO_OK`, `GMCRYPTO_INVALID_KEY`,
  `GMCRYPTO_MAC_FAIL`, ...). Helpful to C callers; violates
  failure-mode invariant.

**Recommended:** int return.

**Decision:** **int return.** `0` = success; every non-zero value
is a single-`Failed`-equivalent. The header documents `0` and
"any non-zero" as the only two cases. C consumers who want
debug-mode diagnostics use a debug-build path (`GMCRYPTO_C_DEBUG=1`
env var prints to stderr; off by default in release builds).

### Q4.9 `sm4-bitsliced` default-on?

- **Default-on:** OOTB throughput; users opting *out* with
  `default-features = false` get the linear-scan slower path.
- **Default-off:** smaller default surface; users opt *in*.

**Recommended:** default-off.

**Decision:** **default-off.** Same posture as `digest-traits`,
`cipher-traits`. Callers opt in. README's quick-start recipe
demonstrates the feature flag. Future v0.5+ may flip default-on
after Codex-review on the bitsliced-S-box gate sequence has
matured.

### Q4.10 New dudect target for bitsliced SM4?

PBKDF2-via-`ct_hmac_sm3` is the v0.2 precedent for "skip the
separate target; existing target covers the path structurally."
Could apply the same logic here: bitsliced SM4 reuses the same
`tau` substitution shape; if `ct_sm4_encrypt_block` (linear-scan)
gates `|tau| < 0.20`, the bitsliced path inherits because the gate
sequence is constant-time by construction.

- **Pro of separate target:** direct measurement on the new code
  path; defense-in-depth.
- **Con:** doubles CI bench runtime under feature-on; extra
  allowlist entry.

**Recommended:** ship a separate target.

**Decision:** **ship `ct_sm4_encrypt_block_bitsliced`.** Defense-
in-depth on the bitsliced gate sequence; cost is one new bench
function + CI matrix slot under `--features sm4-bitsliced`.
Different rationale than v0.2's "skip ct_pbkdf2_hmac_sm3":
PBKDF2's inner loop reused `hmac_sm3` byte-for-byte; the bitsliced
S-box is a *new* implementation, not a wrapper. New impl gets its
own gate.

### Q4.11 Multi-block SIMD-packed bitsliced SM4 in v0.4?

8-way parallel bitsliced (e.g. for CTR mode or parallel CBC
decrypt) would push throughput to ~50-100M blocks/s. Significant
implementation complexity (block packing / unpacking, SIMD
intrinsics gating, target-feature detection).

- **Pro:** real per-core throughput on modern hardware.
- **Con:** large surface; per-target-CPU code paths; v0.4 W3
  budget already covers the most-bang-for-buck single-block
  bitsliced.

**Recommended:** single-block only for v0.4.

**Decision:** **single-block Boyar-Peralta S-box only for v0.4.**
Multi-block SIMD-packed bitsliced is a v0.5+ candidate. The v0.4
W3 scope is bounded at "constant-time S-box reimplementation;
~20-50× over linear-scan via fixed-gate sequence on `u8`/`u32`
operands."

### Q4.12 cbindgen header — committed or build-time-generated?

- **Committed:** header is in-tree at
  `crates/gmcrypto-c/include/gmcrypto.h`. Builds don't run
  cbindgen by default. Drift gate via CI: regen + git-diff.
- **Build-time-generated:** every `cargo build` runs cbindgen.
  Always-fresh header; build-script dep on cbindgen unconditional.

**Recommended:** committed + drift gate.

**Decision:** **committed + drift gate.** Header at
`crates/gmcrypto-c/include/gmcrypto.h`; cbindgen runs only under
`--features regen-header` (or `GMCRYPTO_C_REGEN_HEADER=1`).
Default `cargo build -p gmcrypto-c` does NOT run cbindgen — saves
~5s on every build. CI's `cabi-build-and-header-drift` job runs
cbindgen and `git diff --exit-code` to verify in-tree header
matches generator output.

### Q4.13 C ABI output buffer convention

- **`(out, out_cap, out_actual_len)`:** caller provides buffer
  and capacity; lib writes actual length. If too-small, returns
  non-zero and writes required length so caller can re-allocate.
- **`(out, out_len_inout)`:** single param doubling as cap-in /
  actual-out. POSIX-style.
- **Library-allocates + `_free`:** lib mallocs, returns pointer;
  caller calls `_free`. Hides the allocation contract; complicates
  C callers' memory plans.

**Recommended:** `(out, out_cap, out_actual_len)`.

**Decision:** **`(out, out_cap, out_actual_len)`.** Three params,
explicit. C callers manage their own buffers (often stack-
allocated for known-bounded outputs like signatures). On too-small
buffer: return non-zero, write required length to
`*out_actual_len`, do not write to `out`. Document the contract
on every entry point in the header.

### Q4.14 C example builds in CI

The `gmcrypto-c/examples/*.c` files demonstrate usage but
require a C compiler in CI to verify they build.

- **Pro:** keeps examples in working order.
- **Con:** adds C toolchain to CI (already implicit on
  ubuntu-latest, but adds compile time). Conflicts with the
  policy of keeping CI lean.

**Recommended:** docs-only for v0.4.

**Decision:** **docs-only for v0.4.** `examples/README.md`
documents how to compile them locally. A post-v0.4 chore can
add a CI job (`gcc -I include -L target/release -lgmcrypto
examples/sm2_sign.c`) once demand surfaces.

### Q4.15 Workspace `default-members` posture

Adding `gmcrypto-c` to the workspace: should `default-members`
include it?

- **Yes:** `cargo test` runs both crates' tests; `cargo build`
  builds both.
- **No:** `default-members = ["crates/gmcrypto-core"]`. Building
  `gmcrypto-c` requires explicit `-p gmcrypto-c`.

**Recommended:** `default-members` stays `["crates/gmcrypto-core"]`.

**Decision:** **`default-members = ["crates/gmcrypto-core"]`.** The
C ABI is a niche surface; most contributors work on
`gmcrypto-core`. Building the cdylib on every `cargo build` of
the workspace default would be surprising. CI explicitly
`-p gmcrypto-c` for the FFI build / header-drift jobs.
`cargo test --workspace` still picks up `gmcrypto-c`'s tests.

### Q4.16 C ABI entry-point surface scope

Sub-question of W4. Section 2's W4 entry-point list defines the
v0.4 C ABI baseline.

- **All of `gmcrypto-core`'s public Rust surface mirrored?**
  Including raw byte-concat ciphertext, streaming SM4-CBC,
  asn1::reader/writer primitives? Large surface; large maintenance
  cost.
- **Sign/verify, encrypt/decrypt, SM3, HMAC, PBKDF2, SM4, SM4-CBC,
  SEC1, PKCS#8 only?** Tighter surface; covers 95% of FFI use
  cases.

**Recommended:** tighter surface (per W4 section 2 enumeration).

**Decision:** **tighter surface** (per W4 entry-point list in
Section 2). Streaming SM4-CBC FFI deferred to v0.5; raw byte-concat
SM2 ciphertext deferred (Q4.17). ASN.1 primitives never exposed to
C — they're a Rust implementation detail of the higher-level
codecs.

### Q4.17 Raw byte-concat SM2 ciphertext (`C1||C3||C2`) on the C ABI?

v0.3 W4 shipped `encode_c1c3c2` / `decode_c1c3c2` /
`decode_c1c2c3_legacy` in Rust. Mirror to C?

- **Pro:** completeness; some C consumers may need the raw byte
  ordering for interop with legacy Chinese-standard libs.
- **Con:** adds three more entry points; legacy decrypt-only adds
  the contract awkwardness ("don't auto-detect; caller knows the
  format") that's harder to communicate cleanly in C headers.

**Recommended:** defer to v0.5.

**Decision:** **defer to v0.5.** v0.4 C ABI ships DER-only on the
SM2 ciphertext path (`gmcrypto_sm2_encrypt` / `gmcrypto_sm2_decrypt`
produce GM/T 0009 DER). C consumers needing raw byte-concat can
use the Rust crate directly via cargo's path-dependency mechanism.

### Q4.18 RNG sourcing on the C ABI

The `gmcrypto_sm2_sign` and `gmcrypto_sm2_encrypt` entry points
need an RNG.

- **`getrandom::SysRng` internally:** simple; `gmcrypto-c` is
  `std`-OK; system random source on every supported platform.
- **Caller-supplied via callback function pointer:** flexible
  but tricky FFI semantics (callback ABI, error propagation).

**Recommended:** `getrandom::SysRng` for v0.4; callback callback
deferred to v0.5.

**Decision:** **`getrandom::SysRng` internally for v0.4.**
`gmcrypto-c` already drops `gmcrypto-core`'s no-std posture; system
RNG is the simplest and most portable choice for the C ABI's
first revision. A "register custom RNG" entry point is a post-v0.4
chore if a use case surfaces.

### Q4.19 `Sm2PrivateKey::to_sec1_be` on the C ABI

v0.3 W2 shipped `Sm2PrivateKey::to_sec1_be` in Rust as
`#[doc(hidden)] / not-SemVer-stable`. Mirror to C?

- **Pro:** C callers may need raw scalar bytes for interop with
  other Chinese-standard libraries (e.g. an HSM driver that
  takes raw scalar input).
- **Con:** exposes secret bytes to a callee that can't enforce
  zeroization. Increases blast radius of misuse.

**Recommended:** ship with documented caller-must-zeroize contract.
