# v0.18 — Scope doc

**dudect-gate hardening** — make the two dudect CI workflows robust against
GitHub-hosted runner noise by **pinning both drift axes** (`ubuntu-24.04` image +
an exact `dtolnay/rust-toolchain@1.XX.0`) and computing a **multi-run median**
`|tau|` at the CI level, then — contingent on empirical CI calibration —
re-promoting `ct_fn_invert` / `ct_fp_invert` from telemetry/sentinel back to an
authoritative gate. v0.18 is a **repository / infra-assurance milestone, not a
crates.io release**: no crate code changes (the bench harness `timing_leaks.rs`
stays byte-unchanged), the workspace stays at `0.16.0`, crates.io skips
`0.18.0` (the v0.14 assurance-cycle / v0.17 public-flip precedent). The
self-calibrating relative gate + fuzzing expansion are deferred to **v0.19**.

> **Q-numbering note:** this doc's `Q18.x` are the *design decisions* for v0.18.
> They **supersede** the forward-looking hardening menu in `docs/v0.16-scope.md`
> §5/§6 (`v0.16-Q17.x`) and the v0.17 scope doc §5 — that menu bundled dudect-
> gate hardening with two fuzzing expansions; v0.18 takes **only the dudect-gate

## 1. Context

The two dudect timing-leak workflows (`.github/workflows/dudect-pr.yml`,
`dudect-nightly.yml`) enforce the project's constant-time discipline by gating on
the scale-free dudect statistic `|tau|`. Today that gate is **single-run** (one
`cargo bench` invocation per matrix leg), **absolute** (a fixed `|tau| < 0.20`),
and runs on a **floating** `dtolnay/rust-toolchain@stable` over `ubuntu-latest`.

On **2026-05-12** a GitHub `actions/runner-images` `ubuntu-24.04` update (image
`20260413.86.1` → `20260512.134.1`, kernel `6.17.0-1010-azure` →
`6.17.0-1013-azure`, **Rust toolchain `1.94.1` → `1.95.0`**) shifted the noise
floor on the two direct-invert diagnostics `ct_fn_invert` / `ct_fp_invert` from
~0.006 to intermittent **[0.29–0.40]** — *same commit passing and failing across
consecutive nightly runs* (full data in `docs/v0.5-dudect-recalibration.md`). The
noise is **run-to-run variance, not a cryptographic leak**: the SM2 invert code
is byte-identical to its v0.2 baseline. As a stopgap, those two targets were
demoted to **telemetry-only** at the PR (10K) budget and to a **0.55 gross-
regression sentinel** at the nightly (100K) budget, and `ct_sign_k_class` to
telemetry (PR) / 0.25 (nightly).

The recalibration doc named the **authoritative fix** as a *pinned /
noise-isolated self-hosted dudect runner* (CPU isolation, no SMT, pinned
affinity). That option is now **off the table**: **v0.17 decommissioned the
self-hosted runner** because a self-hosted runner on a public repo is remote code
execution. So v0.18 must recover gate robustness **purely in software, on
GitHub-hosted ubuntu** — and CLAUDE.md mandates the dudect jobs stay on the
`ubuntu-24.04` noise floor they were calibrated against ("don't move dudect").

Two software levers address the two failure axes directly:
1. **Pinning the Rust toolchain to an exact version** freezes the variable whose
   silent drift caused the 2026-05-12 shift — the **load-bearing** pin. Pinning
   `runs-on` to the `ubuntu-24.04` **OS label** (vs the `ubuntu-latest` alias)
   additionally prevents a future jump to a new OS major. (GitHub still
   periodically *refreshes* the 24.04 image/kernel/toolcache, so the OS-label pin
   is a cheap secondary guard, not an exact-image freeze; lever 2 absorbs the
   residual.)
2. A **multi-run median** `|tau|` (loop the bench N times, gate the per-target
   median) is robust to single-run noise spikes — the residual run-to-run
   variance that pinning cannot remove (incl. the periodic 24.04 image refresh).

Together they are the right pre-v1.0 hardening: a dudect gate that fires on a
real leak and does **not** flake on runner noise — and, if the calibration data
supports it, the re-promotion of `ct_fn_invert` / `ct_fp_invert` to authoritative
gates again.

### Building blocks already in place (grounded 2026-05-30)

- **Harness** `crates/gmcrypto-core/benches/timing_leaks.rs` (`harness = false`,
  `required-features = ["crypto-bigint-scalar"]`, `dudect-bencher 0.7.0`). Emits
  one `max tau` per target per process; **no multi-run logic** (the loop +
  median live entirely in CI). Stays byte-unchanged this cycle.
- **Gate parsing** — an inline `python3 - <<'PY'` step in each workflow parses
  the bench log with `^bench\s+(\S+)\s*\.+\s*:.*?max tau\s*=\s*([+\-]?\d+\.\d+)`
  (`re.M`) and applies `required_high` (negative_control > 1.0) / `required_low`
  (< 0.20) / feature-conditional / telemetry / sentinel dicts. `statistics.median`
  is available on the runner's py3.
- **Latent-bug invariant** (v0.12 W3 fix): `MATRIX_FEATURES: ${{ matrix.features }}`
  is re-declared as step-scoped `env:` on **both** the bench step and the parse
  step in each file. Must be preserved on every step that reads it.
- **PR-smoke path allowlist already includes `.github/workflows/dudect-pr.yml`**
  → a PR editing it self-tests. It does **not** include `dudect-nightly.yml`
  (which has no `pull_request` trigger) → nightly changes are validated via
  `gh workflow run dudect-nightly.yml --ref <branch>`.
- **`ubuntu-latest` currently resolves to `ubuntu-24.04`** — pinning to
  `ubuntu-24.04` is the *same image today*; it only freezes the axis against the
  future silent roll to 26.04 (so it is **not** "moving dudect").

## 2. Hard contract

- **`crates/gmcrypto-core/benches/timing_leaks.rs` is byte-unchanged.** The
  harness has no multi-run logic; the CI loops it N times and the Python takes
  the per-target median. No new bench target, no target removed.
- **No published-crate code change.** The diff touches only
  `.github/workflows/dudect-{pr,nightly}.yml` + `docs/` + `CLAUDE.md` +
  `README.md`. **No `crates/`, no `Cargo.toml`, no `Cargo.lock`, no new dep.**
- **Assurance cycle — no crates.io release.** No version bump (workspace stays
  `0.16.0`), **no `cargo publish`, no signed `v0.18.0` tag**. crates.io skips
  `0.18.0` per the v0.14 (Q14.11) / v0.17 (Q17.2) precedent; the milestone lives
  in `docs/` + `CLAUDE.md` (+ optionally Notion).
- **The `|tau|` gate stays on `ubuntu-24.04`** (the calibrated noise floor) — the
  dudect jobs do **not** move to another image or to a self-hosted runner.
- **MATRIX_FEATURES re-declaration preserved** on every step that reads it.
- **No re-promotion without evidence.** Re-promoting `ct_fn_invert` /
  `ct_fp_invert` to an authoritative gate is gated on dispatched-CI median data
  meeting concrete criteria (Q18.7) — never a hard-coded gate that then flakes.

## 3. Q-list (Q18.1–Q18.7) with answers

### Q18.1 — Anchor (user-chosen)
**dudect-gate hardening, nothing else.** Rejected for this cycle: the round-trip
/ differential and streaming-decryptor fuzzing items (→ v0.19), and the
self-calibrating relative gate (→ v0.19; a larger change). Single-theme cadence,
matching v0.7→v0.16.

### Q18.2 — Technique (user-chosen)
**Pin toolchain + image + multi-run median.** The self-calibrating relative gate
(a threshold derived from the run's own noise floor) is **explicitly deferred** —
the median over a pinned toolchain is sufficient to kill the observed run-to-run
variance, and the relative gate is a materially larger, more experimental change.

### Q18.3 — OS-label pin (not an exact image pin)
`runs-on: ubuntu-latest` → **`runs-on: ubuntu-24.04`** in both workflows. This is
an **OS-label** pin, not an exact-image pin: GitHub still periodically refreshes
the `ubuntu-24.04` image/kernel/toolcache (the 2026-05-12 event was itself a
24.04 *refresh* that carried the Rust 1.94→1.95 bump), so the label only prevents
the future silent `ubuntu-latest` → `ubuntu-26.04` alias jump to a new OS major.
The **load-bearing pin for the axis that actually drifted is the exact toolchain
pin (Q18.4)**; the OS label is a cheap secondary guard, and the multi-run median
absorbs the residual image-refresh noise. The noise floor stays on 24.04, so
CLAUDE.md's "don't move dudect" mandate is satisfied.

### Q18.4 — Toolchain pin (the load-bearing axis fix)
`dtolnay/rust-toolchain@stable` → **`dtolnay/rust-toolchain@1.XX.0`** (exact),
**captured empirically** from a real ubuntu run — not invented. Must be **≥ MSRV
1.85**. To make a future noise shift attributable, the workflows' "Capture runner
environment" step logs **`rustc -Vv` + the runner `ImageVersion` (`$ImageOS` /
`$ImageVersion`) + kernel (`uname -r`)** — the full rebaseline evidence. A future
**deliberate** toolchain bump is a **reviewed re-baseline**, never a casual
`@stable`: bump the pin → dispatch the nightly several times → confirm the
per-target medians are stable → record the new floor (with the ImageVersion +
kernel of the run) in `docs/v0.5-dudect-recalibration.md`.

### Q18.5 — N (run count)
**PR N=3, nightly N=5.** PR wall-clock ≈ 3 × ~7.5 s (10K) ≈ 23 s of bench +
build/cache — well within the existing 10-min PR timeout (unchanged). Nightly ≈
5 × ~75 s (100K) ≈ 6.25 min of bench per matrix job + cold-cache build → raise
the nightly `timeout-minutes` **15 → 25** for margin. Both N are **odd**, so the
median is an actual observed run value, not an interpolated average.

### Q18.6 — Median vs min, per gate class
`negative_control` gates on the **MIN** across the N runs (it MUST fire — be
`> 1.0` — on *every* run; a median could mask one silently-broken run, and a
liveness/wiring check is correctly a floor-across-runs). All `required_low`
upper-bound gates (and the nightly sentinel) gate on the **MEDIAN** — the
variance-killing purpose of the cycle. Per-run values + median/min/max print as
telemetry on every line for trend-tracking.

### Q18.7 — Re-promotion (data-driven, contingent)
After pin + median land, re-promote `ct_fn_invert` / `ct_fp_invert` from
telemetry (PR) / sentinel (nightly) to authoritative `required_low` **@0.20**
**iff**, across **≥ 3 dispatched nightly runs** (every matrix leg they appear in):
the per-run **median |tau| < 0.15** in every run, **at most one of the N raw
passes per run exceeds 0.20** (an instability guard — re-promotion must prove
rare *isolated* spikes, not a recurring e.g. 2-of-5 pattern the median quietly
absorbs; the raw passes are visible in the `runs=[…]` telemetry), and **no single
raw pass exceeds 0.40** (still inside the historically observed noise envelope).
If all hold → re-promote
(and re-dispatch the nightly 2× to prove the new gate does not flake). If any
fails → **honest fallback**: keep telemetry/sentinel and document the measured
medians + the explicit decision-not-to-promote (the pin + median still reduce the
variance — that is the cycle's value regardless). `ct_sign_k_class` at the PR
(10K) budget re-promotes to `required_low` @0.25 only if its 3-run median stays
< 0.18 across ≥ 3 PR runs (historical spread [0.21–0.37] at 10K → expected to
**stay telemetry** at PR; it already gates at 0.25 nightly). Decide on data.

## 4. Workstreams

Each workstream is its own commit / milestone; W0–W3 each get a `codex exec`
read-only milestone review, matching the v0.10–v0.16 cadence.

### W0 — Scope doc (this PR)
`docs/v0.18-scope.md` + codex review of the scope. No code. **Hard gate before
any workflow edit.**

### W1 — Pin + multi-run median in both workflows
- `dudect-pr.yml`: pin `ubuntu-24.04` + `dtolnay/rust-toolchain@<pin>`; replace
  the single-run bench step with an **N=3** loop (numbered `dudect-$i.log`);
  rewrite the parse step to read all logs, gate the per-target **median**
  (`required_low`) / **min** (`negative_control`); artifact path → `dudect-*.log`.
- `dudect-nightly.yml`: same pin; **N=5** loop (`dudect-nightly-$i.log`); raise
  `timeout-minutes` 15 → 25; rewrite the parse step (glob `dudect-nightly-*.log`,
  keep `ct_sign_k_class: 0.25` in `required_low`, **retain** the median-gated
  `gross_regression_sentinel`); artifact path → `dudect-nightly-*.log`.
- Preserve `MATRIX_FEATURES` on every step. `actionlint` both. **Milestone codex
  review** of the two workflow diffs.

### W1 verify — push + PR-smoke green + reconcile the pin
Push the branch, open a draft PR, confirm the PR-smoke is green on all 4 matrix
legs (the run log shows the 3 numbered runs + `median=` telemetry). **Capture the
real toolchain pin** from the green run's `rustc -Vv`; if it differs from the
placeholder, correct the pin in both workflows (≥ 1.85) and re-confirm green.

### W2 — Empirical calibration + re-promotion decision
Dispatch the nightly on the branch ≥ 3× (`gh workflow run dudect-nightly.yml
--ref <branch>`); collect per-target median/min/max for `ct_fn_invert` /
`ct_fp_invert` (and `ct_sign_k_class` for the PR question); apply the Q18.7
criteria; re-promote (+ 2× anti-flake re-dispatch) **or** keep telemetry/sentinel
(honest fallback). Record the data table + decision in
`docs/v0.5-dudect-recalibration.md` (new "## v0.18 resolution" section).
**Milestone codex review** of the calibration + any gate change.

### W3 — Docs + verification + PR + merge (no publish)
`CLAUDE.md` (dudect-gate section, the invert table rows per the W2 decision, the
latent-bug/gotcha bullet extended with the pin + median + dispatch-only-nightly
gotchas, the `v0.18 = …` retrospective line + top header, the v0.18+ → v0.19+
candidate-menu rewrite); `README.md` (v0.18 shipped/assurance row; fuzzing items
→ "Deferred to v0.19+"). Full verification matrix (diff touches only workflows +
docs; local bench still runs; `actionlint` clean; MATRIX_FEATURES count = 2 per
file; dispatched CI green). **Codex review of the full diff.** Mark PR ready,
squash-merge. **No publish, no tag, version stays 0.16.0.**

## 5. Out of scope / deferred to v0.19+

- **Self-calibrating relative dudect gate** — a threshold derived from the run's
  own noise floor (e.g. relative to `negative_control` or a per-target rolling
  baseline) rather than a fixed 0.20. The larger, more experimental robustness
  step; v0.18's pin + median is the pragmatic first pass.
- **Round-trip / differential parser fuzzing** (`decode∘encode` stability) —
  `docs/v0.16-scope.md` §5 (`v0.16-Q17.1`).
- **Streaming-decryptor fuzzing** (`Sm4CbcDecryptor` / `Sm4GcmDecryptor` under
  arbitrary update/finalize sequences) — `v0.16-Q17.2`.
- **`cargo fuzz coverage` reporting** in CI; **RustCrypto `aead` trait fit**
  (still `aead 0.6.0-rc.10`); **AVX-512 16-way `sbox_x64`**; **CCM incremental
  input** — unchanged from `docs/v0.16-scope.md` §5.
- **v1.0 readiness pass** — API-stability / SemVer review + the above hardening.
