# v0.14 — Scope doc

**Parser fuzzing** — add `cargo-fuzz` (libFuzzer) coverage over the entire
untrusted-input **decode / decrypt** boundary of `gmcrypto-core`, proving the
project's failure-mode invariant on adversarial bytes: **no panic, no OOM, no
hang** — every malformed input collapses to the single safe `None` / `Failed`
return. The recommended pre-v1.0 hardening cycle (Q14.1 in `docs/v0.13-scope.md`
§6). A new **workspace-excluded** `fuzz/` dev crate plus a capped nightly CI job;
**no published-crate code changes unless fuzzing surfaces a bug** (in which case
the fix is the release's value).

> **Q-numbering note:** this doc's `Q14.x` are the *design decisions* for v0.14.
> They **supersede** the provisional `Q14.x` *candidate* list in
> `docs/v0.13-scope.md` §6 (a forward-looking menu). Where this doc references
> that older list it disambiguates with the `v0.13-Q14.x` prefix — the same
> convention prior cycles used (`v0.12-Q13.x` in the v0.13 doc). v0.14 anchors
> on the former candidate `v0.13-Q14.1` (parser fuzzing).

> **Decision provenance:** the three load-bearing forks (anchor, fuzz breadth,
> CI integration) were delegated to a `codex exec` review on 2026-05-24, which
> chose **parser fuzzing / full decode surface / nightly cron on the
> self-hosted macOS aarch64 runner** — adopted verbatim below.

## 1. Context

`gmcrypto-core` is `#![no_std]` + `forbid(unsafe_code)`, but it carries a large
**hand-rolled** untrusted-input decode surface: a strict-canonical DER reader,
PEM, PKCS#8 / PBES2, SPKI, SEC1, the SM2 DER + raw ciphertext codecs, and the
SM4 mode decrypts (CBC / GCM / CCM / XTS). Every one of these is governed by the
project's **failure-mode invariant**: a single `None` / `Error::Failed` on *any*
malformed input, never a panic, never a distinguishing error (oracle defense).

That invariant is currently enforced by KAT + interop + unit tests over
*curated* inputs. It has never been exercised against *adversarial* inputs at
scale. Fuzzing is the standard assurance for exactly this: hostile,
coverage-guided byte streams driven into the parsers to find the input that
panics, allocates unboundedly, or hangs. It is the natural pre-v1.0 gate — the
one piece of assurance a "commercial-grade" crypto SDK is expected to carry and
this one does not yet have.

`safe`-Rust parsers don't have memory-corruption bugs, so the realistic finds
are: **(a)** a `panic!` / `unwrap` / `expect` / slice-index / integer-overflow
(debug) / `unreachable!` reachable from attacker bytes; **(b)** an **unbounded
allocation** where a length field is trusted before it is bounded against the
remaining input (OOM); **(c)** a **super-linear / runaway** parse (hang). Each is
a real bug under the invariant; each fix lands in `gmcrypto-core` and *is* the
user-facing value of the release.

### Building blocks already in place (grounded 2026-05-24)

- **Public untrusted-input decode/decrypt surface** (all `pub`, externally
  reachable from a consumer crate — no `pub(crate)` plumbing needed). The
  high-level decoders, plus the low-level DER reader primitives they're built on
  and the verify path that parses a DER signature:

  | 2 | `pkcs8::decode(input: &[u8])` | — | DER | `Result<Sm2PrivateKey, Error>` |
  | 3 | `pkcs8::decrypt(input: &[u8], password: &[u8])` | — | DER + pw | `Result<Sm2PrivateKey, Error>` |
  | 4 | `spki::decode(input: &[u8])` | — | DER | `Option<ProjectivePoint>` |
  | 5 | `sec1::decode(input: &[u8])` | — | DER | `Option<EcPrivateKey>` |
  | 6 | `asn1::sig::decode_sig(input: &[u8])` | — | DER | `Option<(U256, U256)>` |
  | 7 | `asn1::ciphertext::decode(input: &[u8])` | — | DER | `Option<Sm2Ciphertext>` |
  | 8 | `sm2::raw_ciphertext::decode_c1c3c2(input: &[u8])` | — | raw | `Option<Sm2Ciphertext>` |
  | 9 | `sm2::raw_ciphertext::decode_c1c2c3_legacy(input: &[u8])` | — | raw | `Option<Sm2Ciphertext>` |

  | 17 | `sm4::mode_xts::decrypt(key, tweak, data_unit)` | `sm4-xts` | data | `Option<Vec<u8>>` |

  **Out of the boundary, by design (Q14.4):** `Sm2PrivateKey::from_bytes_be`
  (fixed-width constant-time range check, not a parser); `sm4::mode_ctr::{encrypt,
  decrypt}` (infallible, length-preserving, `-> Vec<u8>`, no malformed-input /
  `None` invariant to violate); the encode/encrypt paths (don't take untrusted
  bytes); and the **streaming** decryptors (`cbc_streaming::Sm4CbcDecryptor`,
  `gcm_streaming::Sm4GcmDecryptor`) — stateful chunked APIs, deferred to v0.15
  (Q14.4 rationale).

- **`cargo-fuzz` is the standard.** libFuzzer in-process, coverage-guided,
  `cargo fuzz` driver; nightly-only (sanitizer instrumentation). Works on
  `aarch64-apple-darwin` (the self-hosted runner) and `x86_64`.

- **Seed material exists.** `crates/gmcrypto-core/tests/data/` binary KAT
  fixtures + the inline `hex_literal` vectors give valid PEM / PKCS#8 / SPKI /
  SEC1 / DER-sig / SM2-ciphertext / SM4-AEAD encodings to seed each corpus.

- **Self-hosted CI runner** (`[self-hosted, macos, arm64, gmcrypto]`) already runs
  `ci.yml`; adding a separate nightly fuzz workflow on the same runner is the
  established pattern. The `ubuntu-latest` dudect jobs are independent (and
  currently blocked on GitHub billing — out of scope here).

## 2. Hard contract

- **New `fuzz/` crate at repo root**, created by `cargo fuzz init`:
  - Its own `Cargo.toml` with `[package] name = "gmcrypto-fuzz"`,
    `[package.metadata] cargo-fuzz = true`, and an **empty `[workspace]` table**
    so it is its *own* workspace (cargo-fuzz convention). The parent
    `Cargo.toml` *also* gains `exclude = ["fuzz"]` (belt-and-suspenders).
  - Dependencies: `libfuzzer-sys`, `arbitrary` (derive), and
    `gmcrypto-core = { path = "../crates/gmcrypto-core", features = ["sm4-aead", "sm4-xts"] }`
    — the AEAD + XTS features are **always on** in the fuzz crate so every target
    compiles with no per-target feature juggling.
  - **MSRV 1.85 does NOT bind the fuzz crate** — it is unpublished, nightly-only,
    not a workspace member, not in `Cargo.lock` policy, not in `cargo deny`.
- **Fuzz targets** (`fuzz/fuzz_targets/*.rs`), each a `fuzz_target!` harness that
  drives one entry point and asserts only that it **returns without panicking**.
  The full list is Q14.3.
- **Invariant under test:** no panic, no OOM (libFuzzer `-rss_limit_mb` /
  `-malloc_limit_mb`), no hang (libFuzzer `-timeout`). **Nothing else** — no
  round-trip, no correctness oracle, no constant-time assertion (those are KAT /
  interop / dudect territory). Per codex: "focused negative-input fuzzing, not
  exhaustive crypto validation."
- **Default-features build of all three published crates is byte-unchanged.**
  `gmcrypto-core` / `-c` / `-simd` get **no code changes** from the fuzzing
  scaffolding itself. The *only* way a published crate changes is a **bug fix**
  for something fuzzing finds (Q14.10).
- `gmcrypto-core` stays `forbid(unsafe_code)`. The fuzz crate inherits
  `libfuzzer-sys`'s `unsafe` (the harness macro) — it is a dev tool, not a
  published surface, and is exempt from the workspace `unsafe` posture.

## 3. Q-list (Q14.1–Q14.12) with answers

### Q14.1 — Anchor (codex-chosen)
**Parser fuzzing, nothing else.** A hardening release on the v1.0 path. Rejected:
`aead` trait fit (blocked — `aead 0.6.0-rc.10`), AVX-512 `sbox_x64` (needs an
x86_64 AVX-512 env; we're aarch64), pinned dudect runner (narrower; entangled
with the billing block). Single-anchor cadence, matching v0.7→v0.13.

### Q14.2 — Fuzz breadth (codex-chosen)
**Full decode surface.** All 15 entry points in §1, including the SM4
GCM/CCM/XTS/CBC decrypts. The invariant is "*all* hostile decode/decrypt inputs
collapse to one safe failure," so leaving the symmetric decrypts out would be a
visible gap immediately after v0.13 shipped the SM4-XTS C FFI. Rejected:
"wire-format parsers only" and "ASN.1 + PKCS#8 core only" (both leave the AEAD
path unfuzzed).

### Q14.3 — Target list
16 targets (the two `raw_ciphertext` orderings fold into one target that runs
both; `fuzz_sm4_gcm_decrypt` exercises **both** `decrypt` and
`decrypt_with_tag_len`):

*Encoding / wire-format parsers (always reachable):*
1. `fuzz_pem` — `pem::decode`
2. `fuzz_pkcs8_decode` — `pkcs8::decode`
3. `fuzz_pkcs8_decrypt` — `pkcs8::decrypt` (fixed test password)
4. `fuzz_spki` — `spki::decode`
5. `fuzz_sec1` — `sec1::decode`
6. `fuzz_sig` — `asn1::sig::decode_sig`
7. `fuzz_asn1_reader` — the low-level DER reader primitives `read_tlv` /
   `read_integer` / `read_sequence` / `read_oid` / `read_bit_string` /
   `read_length` / `read_context_tagged_explicit` (the foundation every
   higher decoder is built on; direct fuzzing reaches malformed length/tag
   shapes the higher grammars constrain away)
8. `fuzz_sm2_ciphertext_der` — `asn1::ciphertext::decode`
9. `fuzz_sm2_raw_ciphertext` — `decode_c1c3c2` **and** `decode_c1c2c3_legacy`
10. `fuzz_sm2_pubkey_sec1` — `Sm2PublicKey::from_sec1_bytes`
11. `fuzz_sm2_decrypt` — `sm2::decrypt` with a fixed test private key (composite:
    DER parse + KDF + MAC; exercises the allocation-from-parsed-length path)
12. `fuzz_sm2_verify` — `verify_with_id(pub, id, message, sig_der)`; the
    `sig_der` (and `id`, `message`) are attacker bytes, so this fuzzes the DER
    signature parse + the verify path (returns `bool`, must never panic)

*Symmetric decrypt (negative-input):*
13. `fuzz_sm4_cbc_decrypt` — `mode_cbc::decrypt`
14. `fuzz_sm4_gcm_decrypt` — `mode_gcm::decrypt` (fixed 16-byte tag) **and**
    `mode_gcm::decrypt_with_tag_len` (arbitrary `GcmTagLen` + variable tag slice)
15. `fuzz_sm4_ccm_decrypt` — `mode_ccm::decrypt`
16. `fuzz_sm4_xts_decrypt` — `mode_xts::decrypt`

### Q14.4 — What is *not* a fuzz target (and why)
- `Sm2PrivateKey::from_bytes_be(&[u8; 32])` — fixed-width, branch-free
  constant-time range check; no parsing, no allocation; zero fuzz value.
- `sm4::mode_ctr::{encrypt, decrypt}` — infallible, length-preserving,
  `-> Vec<u8>`; accepts *any* bytes and has **no** malformed-input / `None`
  invariant to violate. Nothing to find.
- **Streaming decryptors** (`cbc_streaming::Sm4CbcDecryptor`,
  `gcm_streaming::Sm4GcmDecryptor`) — stateful, chunked-feed APIs. Fuzzing them
  well needs an `arbitrary`-driven *sequence* of update/finalize calls, a
  materially larger harness. Their per-call crypto rides the same single-shot
  primitives this cycle already fuzzes, and v0.9's differential KAT proved them
  byte-equal to single-shot across arbitrary chunking on *valid* input.
  Adversarial-input + adversarial-chunking fuzzing of the state machine is a
  genuine but separable follow-on → **deferred to v0.15** (§6).
- Encode / encrypt paths — they don't consume untrusted bytes.

`decrypt_with_tag_len` is **not** a separate target but **is** exercised: it has
a distinct signature from `decrypt` (variable-length tag + `GcmTagLen`), so
`fuzz_sm4_gcm_decrypt` calls *both* (the W0 codex review flagged the earlier
"transitively covered" claim as wrong — the fixed-`[u8; 16]` `decrypt` does not
reach the variable-tag path).

### Q14.5 — Structured inputs (`arbitrary`)
Targets 1–12 take raw `&[u8]` (or `&str` via `from_utf8`) directly — the
function's natural input is bytes. The symmetric-decrypt targets 13–16 need
*several* sized inputs (key / iv / nonce / aad / ct / tag / tag_len), so they
use `arbitrary` to carve the fuzz buffer (`arbitrary` is a fuzz-only dev-dep).
Note (per the W0 codex review): `#[derive(Arbitrary)]` consumes the buffer
field-by-field; it does **not** literally make a trailing `Vec` "take the
remainder," and bounded `Vec`s / a controlled CCM `tag_len ∈ {4,6,8,10,12,14,16}`
distribution want a **hand-written `Arbitrary` impl over `Unstructured`**
(`u.bytes(n)`, `u.int_in_range`, `u.arbitrary_take_rest` for the final field).
v0.14 uses the hand-written `Unstructured` form for the SM4 targets to get
tight, intentional field splitting. The key/tag being attacker-controlled is
intentional — we fuzz the *parse/validate* path, not key secrecy.

### Q14.6 — Seed corpus
Curated valid encodings committed under `fuzz/seeds/<target>/` (one or more
known-good inputs per target, lifted from `tests/data/` + the inline KAT
vectors). `cargo fuzz run <target> fuzz/seeds/<target>` bootstraps coverage off
real structure rather than starting from zero. Seeds are **committed**; the
runtime-grown corpus is not (Q14.7).

### Q14.7 — `.gitignore` (lockfile trap — W0 codex finding)
Add `fuzz/target/`, `fuzz/corpus/`, `fuzz/artifacts/` (build output, grown
corpus, crash repros — all derived/large). `fuzz/Cargo.toml`,
`fuzz/fuzz_targets/`, and `fuzz/seeds/` are committed. **`fuzz/Cargo.lock` IS
committed** — cargo-fuzz binary workspaces pin their lockfile for reproducible
fuzzing. **Trap:** the root `.gitignore` line 5 is a bare `Cargo.lock`, which
matches `fuzz/Cargo.lock` too. Fix at W1: anchor it to `/Cargo.lock` (root only)
**or** add an explicit `!fuzz/Cargo.lock` negation. Without this, the fuzz
lockfile is silently un-committable.

### Q14.8 — CI integration (codex-chosen)
**Nightly cron + `workflow_dispatch`**, new `.github/workflows/fuzz-nightly.yml`,
`runs-on: [self-hosted, macos, arm64, gmcrypto]`. **Time-capped** — each target
runs `-max_total_time=<N>` (default 120 s) with `-rss_limit_mb` set, so total
wall-clock is bounded and cheap on the runner. Steps: install nightly +
**a pinned `cargo-fuzz`** (`cargo install cargo-fuzz --version <pinned>` — W0
codex finding: pin for reproducibility, same posture as the cbindgen-0.29 pin),
cached; `cargo +nightly fuzz build`, then per-target `fuzz run` seeded from
`fuzz/seeds/`. **Crash artifacts** (`fuzz/artifacts/`) uploaded on failure with
retention. **PR workflows untouched** (`ci.yml` / `dudect-pr.yml`) — fuzzing
does not gate PRs. `concurrency: cancel-in-progress=false` (a partial fuzz run
is still useful signal). The runner-setup runbook in `CLAUDE.md` gains a
one-time `rustup toolchain install nightly` + pinned `cargo install cargo-fuzz`
note. **Runner-exposure caveat** (W0 codex): even on a private repo, a
`workflow_dispatch` or a future collaborator runs arbitrary native fuzz code on
the personal Mac — acceptable now, but revisit runner isolation before the
public flip.

### Q14.9 — Run budget / limits
`-max_len=16384` (keep inputs small — the parsers' own 16 MiB ceilings are not
the fuzzer's job to brute-force), `-rss_limit_mb=2048` (W0 codex: 2 GiB is the
libFuzzer default and plenty — a target exceeding it is itself a finding),
`-timeout=25`. Nightly `-max_total_time=120` per target (≈32 min wall-clock for
16 targets; tunable via a `workflow_dispatch` input). Local runbook in
`CLAUDE.md` / `fuzz/README.md`.

### Q14.10 — If fuzzing finds a bug
A reproducible crash/OOM/hang is a **genuine failure-mode-invariant violation**.
The fix lands in `gmcrypto-core` (or wherever), with the crashing input
minimized (`cargo fuzz cmin` / `tmin`) and **committed as a regression seed**
under `fuzz/seeds/<target>/` (a crash that becomes a permanent corpus entry).
The fix is then a normal TDD change with a unit test asserting `None`/`Failed`
on the minimized input. **This is the path that turns v0.14 into a published
bugfix release.**

### Q14.11 — Publish decision (revised per W0 codex)
v0.14 is primarily an **assurance** cycle, so it does **not** auto-publish.
The W0 codex review was decisive: *do not publish on a clean fuzz run.* Two
sub-cases:
- **Fuzzing finds a fix** in a published crate → cut **v0.14.0** as a normal
  bugfix release (version bump + SSH-signed tag + `cargo publish` triplet
  `simd → core → c`), the fix being the user-facing value.
- **Fuzzing is clean** → **no crates.io release, no version bump.** The fuzz
  harness + nightly CI + `SECURITY.md`/`README.md` posture notes merge to `main`
  as an **infra/assurance PR**; the assurance milestone is recorded by the merge
  (optionally a lightweight annotated git tag, *not* a crates.io publish).
  Publishing byte-identical crypto as a new version is release noise; the next
  *code* change carries the next version number. crates.io README staleness
  until then is acceptable.

Either way, the **publish step is flagged for explicit user confirmation** at
W4 (irreversible per `CLAUDE.md`). The default per codex is: clean ⇒ merge
without publishing.

### Q14.12 — `cargo deny` / supply chain
The fuzz crate is **not** a workspace member and is **excluded** from
`cargo deny check --exclude-dev` (which scans the workspace). `libfuzzer-sys` /
`arbitrary` are dev-only fuzz deps in a separate workspace; they do not enter the
published dependency graph of any of the three crates. No `deny.toml` change.

Five workstreams, each its own commit / milestone; W1–W3 each get a `codex exec`

## 5. Out of scope / deferred to v0.15+

- **Round-trip / differential fuzzing** — `decode(encode(x)) == x`, or
  cross-checking against an independent implementation. Higher-value but needs
  total encoders and an oracle; a follow-on once the no-panic baseline is in.
- **Streaming-decryptor fuzzing** — `Sm4CbcDecryptor` / `Sm4GcmDecryptor` under
  an `arbitrary`-driven sequence of update/finalize calls (adversarial input +
  adversarial chunking of the state machine). Deferred from Q14.4.
- **Structure-aware fuzzing** of the DER grammar (e.g. `arbitrary`-derived
  ASN.1 ASTs) — deeper coverage than the byte-level `fuzz_asn1_reader`; defer.
- **`cargo fuzz coverage` reporting** in CI (coverage-over-time dashboards).
- **RustCrypto `aead` trait fit** — still `aead 0.6.0-rc.10`.
- **Pinned / noise-isolated dudect runner**, **AVX-512 `sbox_x64`**, **SM4-XTS
  streaming / per-sector batch helper**, **CCM incremental input**,
  **Argon2-with-SM3**, **`wasm-bindgen-test` KAT runner** — unchanged from
  `docs/v0.13-scope.md` §5.
- **GitHub billing** for the `ubuntu-latest` dudect-pr jobs — account-level, not
  a code task.
