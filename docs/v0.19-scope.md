# v0.19 — Scope doc

**Self-calibrating relative dudect gate** — close the one assurance compromise v0.18 left
open: re-promote the two direct-invert diagnostics `ct_fn_invert` / `ct_fp_invert` from
telemetry/sentinel back to an authoritative gate, by gating each against a **matched-sensitivity
fix-vs-fix noise probe measured in the same process** instead of a fixed absolute threshold. The
gate becomes `median(target) ≤ max(ABS_FLOOR, K · median(probe))` — a threshold derived from the
run's own noise floor. v0.19 is a **repository / infra-assurance milestone, not a crates.io
release**: the only `crates/` file touched is the **dev-only** bench harness
(`benches/timing_leaks.rs` — a `[[bench]]` target, not compiled into the library, not part of the
crate's API or dependency build graph), the published library behaviour is byte-unchanged, the
workspace stays at `0.16.0`, and crates.io skips `0.19.0` (the v0.14 assurance-cycle / v0.17
public-flip / v0.18 dudect-hardening precedent). Fuzzing expansion and the v1.0 readiness pass are
deferred to **v0.20+**.

## 1. Context

The two dudect timing-leak workflows (`.github/workflows/dudect-pr.yml`, `dudect-nightly.yml`)
enforce the project's constant-time discipline by gating on the scale-free dudect statistic
`|tau|`. **v0.18** pinned both drift axes (`runs-on: ubuntu-24.04` OS label +
`dtolnay/rust-toolchain@1.95.0`) and added a **CI-level multi-run median** (PR N=3 / nightly
N=5) so the gate no longer flakes on single-run runner noise.

But v0.18 **deliberately did not re-promote** `ct_fn_invert` / `ct_fp_invert` to the
authoritative `|tau| < 0.20` gate. They remain **telemetry-only at PR** and on a
**median-gated gross-regression sentinel `@0.55` at nightly**. The reason (recorded in
`docs/v0.5-dudect-recalibration.md`): the 2026-05-12 `ubuntu-24.04` runner-image refresh (which
carried a Rust `1.94.1 → 1.95.0` bump) shifted the noise floor on these two targets from ~0.006
to intermittent **[0.29–0.40]**, *same commit passing and failing across consecutive runs*. That
noise has two properties a fixed-threshold + median gate cannot absorb:

1. It is **sensitivity-specific.** Only the two short *single-inversion* measurements were hit;
   every *composite* target (`ct_sign`, `ct_sm2_decrypt`, the SM4 surface, …) stayed `< 0.01`
   through the same incident. A single field inversion is the smallest-signal, most
   measurement-noise-sensitive operation in the harness; longer operations average the per-call
   jitter away. So the floor these two targets sit on is genuinely higher and more volatile than
   the rest of the cohort's.
2. It is **intermittent.** A 5-run *median* can itself land above 0.20 if a majority of the runs
   spike (e.g. 3 of 5) — re-flaking the gate, the exact failure v0.18 exists to prevent.

The v0.18 100K×5 calibration on the pinned runner measured both back near the ~0.006 baseline
(medians 0.006–0.028) — but v0.18 **kept them on telemetry/sentinel anyway** (robustness-first,
Q18.7), because the pin froze `1.95.0` (the *version present during the spike*), what is quiet
today is the *current* image (the OS-label pin does not freeze the image build), and a future
image refresh can re-introduce the noise. `docs/v0.5-dudect-recalibration.md` and `CLAUDE.md`
both name a **self-calibrating relative gate** — a threshold derived from the run's own noise
floor — as the change that **could** safely re-promote these two, because it adapts to the
runner floor instead of betting on a fixed number. That is v0.19.

### The mechanism

A real timing leak makes **one** target's `|tau|` diverge from the run's noise floor; image-wide
measurement noise lifts a whole **operation-class** together. So instead of comparing the suspect
to a fixed 0.20, compare it to a **noise reference of matched sensitivity, measured in the same
process**:

- Add a **fix-vs-fix probe** for each suspect — a bench target that runs the *exact same
  inversion* (`Fn::invert` for `ct_fn_invert`, `Fp::invert` for `ct_fp_invert`) but feeds
  **both dudect classes the one same precomputed value** (not two equal-but-distinct objects —
  a single value/reference, so there is no address- or layout-dependent difference either). With
  no class split, the probe **cannot carry an *input-dependent* class signal, by construction**:
  any residual `|tau|` is purely statistical measurement noise of that operation at the current
  image/budget (sampling false positives, temporal drift, random class-count imbalance) — never a
  cryptographic leak. It tracks the very sensitivity that flaked (same operation, same duration)
  but can never encode a class-correlated input difference.
- Gate each suspect `T` against its probe `P`:
  `threshold(T) = max(ABS_FLOOR, K · median(P))`, `PASS iff median(T) ≤ threshold(T)`.

When the image is quiet, `median(P) ≈ 0.01`, `K·P` is small, and the gate is `max(0.20, small) =
0.20` — **identical to the historical absolute gate**. When a 2026-05-12-style image refresh
lifts the inversion-noise floor, `median(P)` rises *with the suspect* (matched sensitivity), the
threshold rises with it, and a constant-time target that merely tracks its own noise floor still
passes — the anti-flake property. When a *real* `Fn::invert`/`Fp::invert` leak appears, the
suspect (which **does** split small-d vs large-d) rises while the fix-vs-fix probe (identical
inputs) stays pinned to the floor, so the ratio diverges and the gate fires.

### Why re-promotion is safe (the load-bearing argument)

Because `threshold = max(0.20, K · floor)`, the relative gate is **never stricter than today's
fixed 0.20 gate** — it only *loosens* when the matched probe proves the operation-class is noisy.
The two comparisons to draw, kept distinct (an over-broad "strictly better than everything" claim
would be dishonest):

- **vs re-promoting to a fixed `0.20` gate** (the option v0.18 declined purely on flake risk):
  the relative gate is a **strict (Pareto) improvement**. Leak detection is identical in the
  quiet regime (both gate at 0.20) and strictly better in a high-floor regime (the relative gate
  loosens, the fixed gate would false-positive); flake risk is strictly lower (the loosening is
  exactly the anti-flake mechanism).
- **vs the current v0.18 posture** (telemetry PR / `@0.55` sentinel-only nightly): the relative
  gate **plus** the retained sentinel fires on a *superset* of the sentinel's cases
  (`median(T) > min(max(0.20, K·floor), 0.55)`), so it catches **more** real leaks — notably the
  `[0.20, 0.55]` band the bare sentinel misses entirely, whenever the floor is quiet. The honest
  trade is **flake risk, not leak detection**: the bare `0.55` sentinel essentially never
  false-positives (observed noise maxed at ~0.40), whereas the relative gate *can* false-positive
  **iff the probe fails to track a noise return** — but only down to the `0.20` floor, i.e. it can
  never flake worse than a fixed-`0.20` gate would have. So re-promotion buys materially better
  leak detection for a residual flake risk that is bounded by the structural floor.

The re-promotion decision therefore rests on **(a)** this structural floor argument plus **(b)**
quiet-regime CI validation that the gate does not false-positive today — **not** on reproducing
the now-gone noisy image.

**Honest residual** (stated up front, mirroring v0.18's posture): the 2026-05-12 image is gone,
so the W3 calibration can only validate the *quiet* regime — it cannot *prove* the probe tracks a
future noise return until one happens. If the probe ever fails to track (the noise turns out to
be specific to the class-split structure rather than the operation's duration), the relative gate
degenerates to a fixed `0.20` gate — *no worse than re-promoting to `0.20` would have been*,
though it does reintroduce the flake exposure the bare `0.55` sentinel did not have (that is the
residual being weighed). The `@0.55` sentinel is **not** a catch-all: it backstops **gross
regressions** (a real `ConstMontyForm::invert`-class leak at ~0.70) and **probe mis-measurement**,
not a moderate `[0.20, 0.55]` leak that happens to coincide with a high noise floor — in that
narrow regime neither term catches it, exactly as the v0.18 sentinel-only posture also would not.
That bounded downside is acceptable and pre-committed.

### Building blocks already in place (grounded 2026-05-30)

- **Harness** `crates/gmcrypto-core/benches/timing_leaks.rs` (`harness = false`,
  `required-features = ["crypto-bigint-scalar"]`, `dudect-bencher 0.7.0`). Emits one `max tau`
  per target per process. `ct_fn_invert` (line ~354) measures `(1+d).invert()` on `Fn`,
  `ct_fp_invert` (line ~381) measures `Z.invert()` on `Fp`; both call `x.invert()` inside
  `runner.run_one(class, …)` with no `black_box`. The probes mirror this exactly.
- **Gate parsing** — an inline `python3 - <<'PY'` step in each workflow parses the bench logs
  with `^bench\s+(\S+)\s*\.+\s*:.*?max tau\s*=\s*([+\-]?\d+\.\d+)` (`re.M`), accumulates
  `findings[name] = [|tau|, …]` across the N numbered logs, and applies `required_high`
  (`negative_control` on MIN > 1.0) / `required_low` (on MEDIAN < 0.20) / feature-conditional /
  telemetry / (nightly) sentinel. `statistics.median` is available on the runner's py3. The
  relative gate slots in as an additional block consuming the same `findings` + `med()`.
- **Completeness gate** (v0.18) — every required target must be measured in **every** run, else
  FAIL (never gate over a reduced sample set). v0.19 extends this to the probes + relatively-
  gated targets.
- **Latent-bug invariant** (v0.12 W3 fix): `MATRIX_FEATURES: ${{ matrix.features }}` is
  re-declared as step-scoped `env:` on **both** the bench step and the parse step in each file.
  Preserved on every step that reads it.
- **PR-smoke path allowlist** includes `.github/workflows/dudect-pr.yml`, `Cargo.toml`,
  `Cargo.lock`, and `crates/gmcrypto-core/benches/**` → a PR editing the harness *or* the PR
  workflow **self-tests**. It does **not** include `dudect-nightly.yml` (no `pull_request`
  trigger) → nightly changes are validated via `gh workflow run dudect-nightly.yml --ref
  <branch>`.

## 2. Hard contract

- **The v0.18 "byte-unchanged harness" contract is LIFTED this cycle.** That was a
  *v0.18-specific* constraint (`docs/v0.18-scope.md` §2), correct there because v0.18's technique
  (a CI-level multi-run median) needed *zero* harness change — the loop + median lived entirely
  in CI. It was **never** a permanent prohibition. v0.19's technique *requires* the matched probe
  to run the inversion **in-process** at the same budget on the same image; there is no way to
  measure the noise floor of the exact `Fn::invert` operation without running `Fn::invert`
  fix-vs-fix in the bench. So v0.19 deliberately adds two harness targets. (Restated here so a
  future reader does not mistake the change for a contract violation.)
- **No published-crate behaviour change.** The only `crates/` file touched is
  `crates/gmcrypto-core/benches/timing_leaks.rs` — a `[[bench]]` target (`harness = false`),
  **dev-only**: not compiled into the library, not part of the crate's public API, and not in its
  dependency build graph. (The bench *source* is still part of the package tarball — there is no
  `include`/`exclude` in `Cargo.toml` — but editing it changes no shipped behaviour.) The library
  (`crates/gmcrypto-core/src/**`) is byte-unchanged. The full diff touches only:
  `benches/timing_leaks.rs` + `.github/workflows/dudect-{pr,nightly}.yml` + `docs/` + `CLAUDE.md`
  + `README.md`. **No `src/`, no `Cargo.toml`, no `Cargo.lock`, no new dep, no new feature flag.**
- **Assurance cycle — no crates.io release.** No version bump (workspace stays `0.16.0`), **no
  `cargo publish`, no signed `v0.19.0` tag**. crates.io skips `0.19.0` per the v0.14 (Q14.11) /
  v0.17 (Q17.2) / v0.18 precedent; the milestone lives in `docs/` + `CLAUDE.md` (+ Notion).
- **All v0.18 hardening retained:** `ubuntu-24.04` pin, `@1.95.0` toolchain pin, the N=3/N=5
  multi-run median, `negative_control` on MIN / `required_low` on MEDIAN, the completeness gate,
  the comma-free `rust-cache` `shared-key` (job-index), and `MATRIX_FEATURES` re-declared on
  every step that reads it. The dudect jobs stay on `ubuntu-24.04` (the calibrated noise floor).
- **No re-promotion without evidence.** Re-promoting `ct_fn_invert` / `ct_fp_invert` to the
  authoritative (relative) gate is gated on dispatched-CI data meeting the concrete Q19.5
  criteria — never a hard-coded change that then flakes. Honest fallback documented if the
  criteria are not met.

## 3. Q-list (Q19.1–Q19.7) with answers

### Q19.1 — Anchor (theme)
**Self-calibrating relative dudect gate, scoped tightly to `ct_fn_invert` / `ct_fp_invert`.**
Rejected for this cycle: a *generalized* relative gate over all targets (YAGNI — no other target
ever flaked on runner noise), and (deferred to v0.20+) the fuzzing items + v1.0 readiness.
Single-theme cadence.

### Q19.2 — Probe design
**Two matched-sensitivity fix-vs-fix probes**: `noise_floor_fn_invert` (runs `Fn::invert` on a
single fixed value fed to both classes) and `noise_floor_fp_invert` (same over `Fp`). Each wraps
the *exact* operation of its suspect at the *same* duration. Rejected alternatives:
- **A single generic probe** — the noise was sensitivity-specific, so a generic probe (e.g. a
  fix-vs-fix `sign`) would measure a *different, longer* operation's floor that stayed quiet
  during the incident; it would not track the spike, so it could not calibrate against it.
- **A cohort-derived floor (no harness change)** — a robust statistic over the existing
  `required_low` medians. Rejected as primary: the `required_low` cohort is all *composite*
  targets that stayed `< 0.01` during the incident, so `K · cohort_floor` would still flake when
  the invert-specific noise lifts only the invert targets. (A cohort-floor *telemetry* line is a
  useful cross-check — see Q19.3.)

Two probes (not one shared) because `Fn` and `Fp` are distinct moduli that the recalibration data
shows spiking *independently* (several incident runs were `ct_fp_invert`-only). **The probes
cannot leak by construction** (identical class inputs), which is the decisive safety property: a
real leak lifts the class-split suspect but never its fix-vs-fix probe. Probes are **not**
cfg-gated — they go in the base `BenchMetadata` block and appear in every matrix leg, exactly
like the targets they calibrate.

### Q19.3 — Gate formula
**Hybrid `threshold = max(ABS_FLOOR, K · median(probe))`, median-each-then-combine** (matches
v0.18's per-target-median model: take each target's and each probe's median across the N runs,
then compose). Starting (calibration-pending, tuned in W3) values **`K = 4.0`, `ABS_FLOOR =
0.20`**. Rationale for hybrid over the pure forms:
- *Pure multiplicative* `K · floor` blows up when the probe is tiny (`4 × 0.005 = 0.02` is far
  too tight — a benign 0.025 target run would fail).
- *Pure additive* `floor + δ` does not scale proportionally when the floor genuinely lifts.
- *Hybrid* gives an absolute safety floor (`ABS_FLOOR` dominates in the quiet regime, so the gate
  is **identical to today's 0.20** when `K · floor < 0.20`, i.e. `floor < 0.05`) **and** a
  proportional term that takes over only once the probe proves `floor > 0.05` — exactly the
  regime where the fixed 0.20 was flaking. The `max` is the correct combinator: the gate is the
  *more permissive* of "a sane absolute bound" and "K times the measured floor."

A telemetry-only **ratio** `median(T)/median(P)` prints on every line, so a reviewer can see at a
glance whether a failure is "target diverged from its own probe" (real-leak-shaped) vs "whole
operation-class noisy" (probe lifted too). Never gates. The ratio is guarded: when `median(P) ≤ 0`
(degenerate — a probe that measured exactly zero noise) it is reported as `inf` and the gate
falls back to the `ABS_FLOOR` term (which makes the threshold safe at `0.20` regardless), so there
is no division-by-zero in either the gate or the telemetry.

### Q19.4 — Scope of relativization
**Only `ct_fn_invert` / `ct_fp_invert` gate relatively; all other targets keep the fixed 0.20.**
The relative gate's *entire justification* is the sensitivity-specific noise that hit exactly
these two; no other target has ever needed it (the v0.18 100K×5 table and the recalibration data
show every other `required_low` target `< 0.03`). Relativizing a target that never flaked adds a
probe, a calibration burden, and a new failure mode for zero benefit — a YAGNI violation. The
machinery is a reusable `relative_gate = {target: probe}` dict if a future target ever needs it,
but v0.19 instantiates it for exactly the two pairs. `ct_sign_k_class` is **not** relativized — it
is a *composite* (`sign_raw_with_id` class-split by nonce magnitude), not a matched single
inversion; a fix-vs-fix `sign` probe would measure a different, much longer operation. It keeps
its v0.18 posture (telemetry PR / `@0.25` nightly); relativizing it (a third matched probe) is a
v0.20+ candidate.

### Q19.5 — Re-promotion + honest fallback + sentinel
**Re-promote both targets from telemetry (PR) / sentinel (nightly) to the relative `required`
gate at nightly iff** the W3 dispatched-CI data meets the criteria below; **honest fallback**
keeps the v0.18 posture *and leaves the two probes in as telemetry* (a permanent longitudinal
noise-floor record — the diagnostic value the harness gains regardless of the gate outcome). The
**`@0.55` gross-regression sentinel stays in both states** as defense-in-depth against probe
mis-measurement: if a bug made the probe report 0.5, `K × 0.5 = 2.0` would permit a real leak,
but the independent absolute sentinel still fires on it. Belt and suspenders; it costs nothing.

**Calibration acceptance criteria** (mirror v0.18 Q18.7 — dispatched-nightly, anti-flake
re-dispatch — adapted to the relative gate): across **≥ 3 dispatched nightly runs** on the branch,
in **every** matrix leg the two targets appear in —
1. the **relative gate PASSES every run** (`median(T) ≤ max(0.20, K · median(P))` for both);
2. **probe stability**: each `median(P) < 0.05` in every run, **and** stable run-to-run defined
   numerically as the per-leg spread `max(median(P)) − min(median(P)) < 0.02` across the ≥ 3
   dispatched runs (the probe is a trustworthy floor, not itself flaking);
3. **ratio sanity**: with `median(P) > 0`, `median(T)/median(P) < 0.75·K` (= 3.0 at `K = 4`)
   every run (the target tracks its probe — no class-correlated divergence). If `median(P) ≤ 0`
   in any run the ratio is undefined → treat criterion (3) as **not met** for that run (do not
   re-promote on a degenerate floor);
4. **no single raw pass** of either target exceeds **0.40** (the v0.18 historical-envelope
   instability guard — re-promotion must prove rare isolated spikes, not a recurring pattern the
   median absorbs; visible in the `runs=[…]` telemetry).

If all hold → re-promote, then **re-dispatch the nightly 2× to prove the new gate does not
flake**. If any fails → honest fallback. `K`/`ABS_FLOOR` may be tuned against the real measured
floor in W3 (the starting values are not final).

### Q19.6 — PR (10K) vs nightly (100K)
**Probes run at both budgets; the relative gate is AUTHORITATIVE at nightly (100K) and
TELEMETRY-ONLY at PR (10K).** Rationale: 10K is materially noisier ("the effect is even stronger
at 10K" — `docs/v0.5-dudect-recalibration.md`), so the prudent, robustness-first posture —
consistent with v0.18 keeping PR on telemetry — is to prove the relative gate at the cleaner
100K budget first and keep PR as a non-blocking `REL-TELEMETRY` verdict this cycle.
`ct_fn_invert` / `ct_fp_invert` therefore **stay in `telemetry_only` at PR** (unchanged from
v0.18); the PR job additionally prints `REL-TELEMETRY: … would-PASS/would-FAIL` for trend
tracking. A future cycle can promote the PR relative gate once the nightly relative gate has a
track record (v0.20+). The two probes are added to the PR **completeness** set (a missing probe
is caught) even though they do not gate at PR.

### Q19.7 — Negative-control non-interaction + completeness
The two probes form a **new `noise_probe` reference set** — their medians are *inputs* to the
relative threshold, never gated themselves. They are **not** in `required_high` / `required_low`
/ `negative_control`, so they cannot be confused with the negative control. `negative_control`
is **unchanged**: `required_high`, gated on the **MIN** across N runs, must exceed 1.0 every run;
the probes sit near 0.0 (the opposite extreme) and never touch that logic. The **completeness
gate extends** to FAIL if either probe **or** either relatively-gated target is measured `< N`
runs — otherwise the relative threshold would be computed over a reduced sample set (the exact
silent degradation the completeness gate exists to prevent). Min/median split is preserved
(`negative_control` MIN; everything upper-bound on MEDIAN, including the relative inputs).
