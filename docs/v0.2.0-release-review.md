# gmcrypto-core v0.2.0 release review checklist

**Crate:** `gmcrypto-core` 0.2.0
**Repository:** <https://github.com/frankxue831/gm-crypto-rs>
**Release candidate:** record the exact commit you intend to tag with
`git rev-parse HEAD`.
**Purpose:** decide whether that exact commit is safe to publish to
<https://crates.io/crates/gmcrypto-core> as v0.2.0. Do not review "latest main"
by habit; review a frozen commit.

v0.1.0 published 2026-05-10 on `crypto-bigint = 0.6`. v0.2.0 is the first
post-publish minor: `crypto-bigint 0.7.3`, edition 2024, MSRV 1.85, plus the
W0 / W1 / W2 / W3 / W4 workstreams (SM4 + CBC, HMAC-SM3, PBKDF2-HMAC-SM3,
SM2 encrypt/decrypt, dudect harness expansion to 11 targets).

Publishing to crates.io is permanent. A yanked version is hidden from normal
resolution, but the `.crate` archive remains downloadable.

Use `BLOCK`, `WARN`, and `INFO`:

- `BLOCK`: do not publish until fixed or explicitly deferred.
- `WARN`: publishable, but track for v0.2.1 or v0.3.
- `INFO`: useful note, no release action required.

## 5. Security-critical source review

### 5.1 Signing

File: `crates/gmcrypto-core/src/sm2/sign.rs`

Check:

- `compute_z` uses `ENTL || ID || a || b || Gx || Gy || Px || Py`, where
  `ENTL` is a 16-bit big-endian bit length;
- over-long IDs are rejected before RNG use;
- `SIGN_RETRY_BUDGET` iterations always run; no early exit on first success;
- `ct_or_else` and `RsPair::conditional_select` do not branch on secret choices;
- `SignError` has exactly one variant;
- `sign_raw_with_id` is `#[doc(hidden)] pub` and documented as not SemVer-stable;
- nonce sampling is uniform over `[1, n-1]` with rejection on `0` or `≥ n`
  (no modulo-bias reduction).

### 5.8 HMAC-SM3 (NEW — W3)

File: `crates/gmcrypto-core/src/hmac.rs`

Check:

- `hmac_sm3(key, message) -> [u8; 32]` is single-shot — no streaming
  `HmacSm3::new`/`update`/`finalize` shape (deferred to v0.3 with the
  trait generalization);
- RFC 2104 hash-first long-key path (`key.len() > 64` → `K' = SM3(key)`);
- the long-key SM3 digest stack buffer is zeroized after the copy into `K'`
  (since for long keys the digest is the *effective* HMAC key per
  `HMAC(K, m) == HMAC(SM3(K), m)`);
- `K'`, `K' XOR ipad`, and `K' XOR opad` are all zeroized before return;
- 4 KAT vectors cross-validated against `gmssl sm3hmac` v3.1.1 (the test 6
  long-key KAT was hand-validated by running `gmssl sm3` over the 131-byte
  key first, since gmssl's CLI rejects keys > 32 bytes — RFC 2104 hash-first
  equivalence makes this rigorous).

```bash
cargo test -p gmcrypto-core hmac::
GMCRYPTO_GMSSL=1 cargo test --test interop_gmssl hmac_sm3_matches_gmssl
```

### 5.9 PBKDF2-HMAC-SM3 (NEW — W4)

### 5.10 SM2 encrypt + decrypt + KDF + on-curve check (NEW — Phase 3 / W2)

Files:

- `crates/gmcrypto-core/src/sm2/encrypt.rs`
- `crates/gmcrypto-core/src/sm2/decrypt.rs`

Check encrypt:

- rejects identity recipient public key at the API boundary;
- `ENCRYPT_RETRY_BUDGET = 64` (raised from 4 in pre-publish review — for
  short plaintexts the per-iteration KDF-zero probability is `2^(-8·|M|)`,
  not the asymptotic `2^-256`; budget=4 fails ~`2^-32` per call for 1-byte
  plaintexts; budget=64 makes cumulative-failure `≤ 2^-512` for any length);
- `EncryptError` has exactly one variant;
- `kdf` (counter-mode SM3 per GB/T 32918.4 §5.4.3) is `pub(super)`, NOT
  in the public API — top-level `kdf.rs` is reserved for PBKDF2;
- `point_on_curve(x, y)` is `pub(super)`, NOT in the public API;
- `Z = x2 || y2` buffer is zeroized before return.

Check decrypt:

- on-curve check on `C1` before `mul_var(d_B, C1)` (invalid-curve attack
  defense — without this an attacker can leak `d_B` bits via a small-order
  rogue curve subgroup);
- identity `C1` rejected;
- `DecryptError::Failed` is the single variant — every failure mode
  (malformed DER, off-curve `C1`, identity `C1`, all-zero KDF, MAC mismatch)
  collapses to one shape;
- **KDF-zero detection is non-branching**: folded into a `subtle::Choice`
  via `ct_all_zero(&t)`, the SM3 MAC compute runs unconditionally, and
  `valid = mac_ok & !kdf_zero` combines both checks. Pre-publish review
  flagged the early-return as a chosen-ciphertext timing oracle for short
  `C2` (1-byte `C2` hits all-zero KDF ~1/256 of the time, and an
  early-return that skips XOR/SM3/MAC is observably faster than a normal
  MAC failure);
- empty-`C2` vacuous suppression preserved via a `nonempty` Choice mask;
- MAC compare uses `subtle::ConstantTimeEq` on the 32-byte digest;
- on MAC failure the already-XOR'd plaintext buffer is zeroized before
  return;
- `Z = x2 || y2` buffer zeroized regardless of outcome;
- module-doc documents the constant-time stance and the invalid-curve
  defense rationale.

```bash
cargo test -p gmcrypto-core sm2::encrypt::
cargo test -p gmcrypto-core sm2::decrypt::
```

Note: SM2 encrypt/decrypt cross-validation against gmssl is **explicitly
out of scope** for v0.2 — gmssl's CLI requires PEM/PKCS#8/SPKI key wrapping,
which is v0.3 work. v0.2 SM2 envelope encryption is KAT-validated via
internal round-trip + a fixed-`k` smoke test only.
