# v1.9 scope — TLCP toolkit C FFI (cadence cycle)

Status: v1.9 W1 scope doc. The **FFI cadence cycle of the TLCP arc** —
`docs/tlcp-decomposition.md` §7: *"v1.9 — TLCP FFI (cadence cycle): the
accumulated toolkit surface through `gmcrypto-c`, shape decided then."* The
decomposition deferred the one-cycle-vs-three question to "a maintainer call
at v1.9 time"; that call is now made (**Q9.1 = one cycle**). The four
cycle-shaping forks were **maintainer-locked to the recommended shape** (the
v1.7/v1.8 fork pattern); a fifth (always-on + reader exposure) was
agent-folded to the recommendation. This cycle inherits the FFI-shape
constraints each core cycle's scope doc recorded (**Q6.9, Q7.10, Q8.15** — the
"no ABI frozen early" payoff) rather than discovering conflicts.

This is the cycle that **closes the TLCP arc's core/FFI cadence**: the
in-core toolkit (v1.6 key schedule + no-confirmation SM2-KX, v1.7 record
protection, v1.8 chain/pair verification) becomes callable from C. A C TLCP
client can run a full handshake end-to-end: agree the pre-master via the
no-confirmation KX completers, derive `master_secret`/`key_block`/`Finished`,
protect/deprotect the record wire, and verify a peer's `[sign, enc]`
certificate pair.

## 0. The one thing a reader must not misread

**This is a *thin shim*. It crosses no new trust, failure-mode, or
constant-time invariant — it only re-exposes the in-core toolkit's existing
ones across the ABI.** Three corollaries the implementation must hold sacred:

1. **`deprotect_cbc` is the arc's Lucky13 path. The shim adds NO logic** — no
   early return, no length-distinguishing branch, no plaintext on failure. The
   constant-time work lives entirely in `tlcp::record::deprotect_cbc` (guarded
   by `ct_tlcp_cbc_deprotect`); the C entry point is a pure `Option<Vec<u8>>`
   → copy-out-or-single-`GMCRYPTO_ERR` translation. The **post-strip length is
   NEVER an input param** (it is the secret) — Q7.10's hard rule survives the
   ABI verbatim. The CT-preserving shim shape (Codex consult #1, confirmed):
   validate the non-secret ABI pointers, call core **once**, then `Some(pt)`
   writes `out_actual_len` and copies exactly `pt` while `None` touches **no**
   output and returns the single `GMCRYPTO_ERR` — one return point, no pad/MAC-
   specific cleanup (both are just `None`), no length-dependent early branch.
2. **`verify_pair` through C is STILL NOT endpoint authentication.** A `0`
   (success) return means "each chain links to a caller-trusted anchor and
   each cert is usable for its TLCP role" — it does **not** mean "the peer I
   dialed". **Endpoint identity binding stays the caller's, permanently.** The
   committed C header is the *only* place a non-Rust consumer reads this, so
   the `gmcrypto.h` doc comment carries the disclaimer in the same words as
   the `tlcp::chain` module doc.
3. **Every failure is one `GMCRYPTO_ERR`.** Never bad-pad vs bad-MAC vs
   wrong-length (the record deprotect oracle); never a chain-rejection reason
   (the chain oracle). The single-failure-mode invariant is the whole point of
   the core API and it does not get "more helpful" at the ABI.

## 1. Sign-off decisions

### The cycle-shaping forks (Q9.1–Q9.4 maintainer-locked; Q9.5 agent-folded)

- **Q9.1 Scope breadth — ONE CYCLE, full toolkit.** v1.9 exposes the entire
  accumulated surface: key schedule (3 fns) + no-confirmation KX completers
  (the 2 the v1.2 confirmed-flow FFI left out) + record protect/deprotect
  (CBC + GCM, + key carriers) + chain/pair verify (+ the KeyUsage/
  BasicConstraints readers). Rationale: a C client cannot complete a handshake
  with any subset — it needs KX → key schedule → record end-to-end — and the
  net-new surface is small (most is stateless free-fns reusing the v1.2 KX
  handles and v1.4 cert/time types; ≤2 genuinely new opaque types). The
  maintainer's recorded fallback if one PR proves too heavy: split at the
  Q9.1 seam (v1.9 = handshake-completion [KX + key schedule + record];
  v1.10 = chain/pair verify — the one self-contained, public-inputs
  sub-surface that strands no handshake).

- **Q9.2 Record key carriers — OPAQUE HANDLES.** `RecordKeysCbc` /
  `RecordKeysGcm` (both `ZeroizeOnDrop`) cross as two independent immutable
  opaque handles `gmcrypto_tlcp_record_keys_cbc_t` /
  `gmcrypto_tlcp_record_keys_gcm_t`, constructed once from a role + key block
  and torn down by `_free` (which triggers the `ZeroizeOnDrop` wipe — the
  `gmcrypto_sm4_gcm_encryptor_free` precedent). Const-`*`, **no consume-on-use**
  (protect/deprotect borrow the keys). **This is a deliberate REVISION of a
  recorded v1.7 constraint, not a reconciliation** (Codex consult must-fix #2):
  Q7.10 literally recorded *"caller-held `seq` / IV / keys by value — NO opaque
  handle"*. v1.9 overrides the **key-carrier** half of that — the carriers are
  `ZeroizeOnDrop` secret material and a managed `_free` (the
  `gmcrypto_sm4_gcm_encryptor_free` precedent) is a materially better home for
  the wipe than asking every C caller to zero a raw key-block buffer it
  re-passes on every call — while honoring the **operation** half verbatim
  (`seq`/IV by value, no streaming/connection handle, copy-out, single error).
  **If the maintainer prefers to honor Q7.10 to the letter, the fallback is
  by-value key bytes** (the raw 128-/40-byte block + role on every
  protect/deprotect call, no handle, no `_free`) — flagged for re-confirmation.

- **Q9.3 Cert-array idiom + verdict — HANDLE-PTR ARRAY + OUT-PARAM BOOL +
  STATUS** (reconciles the two mappings Q8.15 left open). `verify_chain` /
  `verify_pair` take `&[Certificate]` as `(const gmcrypto_x509_certificate_t*
  const*, size_t n)` — arrays of the **existing v1.4 immutable cert handle**
  (no new opaque type, no consume-on-use); `Option<X509Time>` as `const
  gmcrypto_x509_time_t*` (NULL = None, the v1.4 struct reused). The `bool`
  verdict is an **out-param** (`int* out_verified`) with the int return
  carrying `GMCRYPTO_OK`/`GMCRYPTO_ERR` status. This keeps the header's
  universal "every int return is `0` on success" banner true and separates the
  two axes cleanly: `ERR` = NULL/structural misuse; `verified = 0` =
  bad-chain/over-depth (over-depth stays a *verdict*, not an error — the C
  boundary must **not** add an `n > MAX_CHAIN_DEPTH` reject). A bare
  predicate-or-`OK=false` return would invert in C `if()` — the v1.4
  `is_self_issued` lesson. **Two adapter must-fixes (Codex consult #3):**
  (a) a C `*const *const Certificate` array is NOT a contiguous
  `&[Certificate]`, and `Certificate` is not `Clone`, so the shim must **not**
  fabricate an unsafe slice — core gains a thin **additive** refs-taking
  verifier (`&[&Certificate]`, the `sign_raw_with_id` `#[doc(hidden)] pub`
  precedent) that the existing `verify_chain`/`verify_pair` delegate to, and the
  shim builds a `Vec<&Certificate>` from the handle pointers. This makes v1.9 a
  shim **plus a thin additive core adapter** (no new crypto), so the api-baseline
  IS regenerated additively. (b) NULL semantics: `(NULL, 0)` = an empty slice;
  `(NULL, n>0)` or any NULL element pointer = ABI misuse → `GMCRYPTO_ERR` (not a
  `verified = 0` verdict).

- **Q9.4 RNG crossing — SysRng DEFAULT + `_with_rng` CallbackRng** (settles
  v1.7 Q7.10's one named-open CBC-IV fork). The two RNG-needing symbols
  (`protect_cbc`'s 16-byte IV; the no-confirmation responder's `R_B`) each get
  a default form (internal `getrandom::SysRng`) **and** a `_with_rng(callback,
  ctx)` form reusing the v0.5/v1.2 `CallbackRng` (`TryCryptoRng`; `rc != 0` →
  single error, never a panic). The `_with_rng` form is **required**: it is how
  `c_smoke` replays the record KATs (fixed IV) and the GM/T 0003.5 KX vector
  (fixed ephemeral) **byte-for-byte through the ABI** (the
  `fixed32_rng_callback` precedent); the callback also serves SDF/SKF/HSM IV
  sources. CallbackRng-only was rejected (forces every caller to wire an RNG);
  explicit-IV-for-`protect_cbc` was rejected (a second divergent shape).
  **Output staging (Codex consult #4):** a callback `rc != 0` becomes one
  `TryRng::Err` → one `GMCRYPTO_ERR`; the shim writes **no** `R_B`/key/IV output
  until the core call returns `Ok` (no partial state, no handle corruption), and
  the no-confirmation responder spends/frees its handle exactly as the confirmed
  flow does. `ffi_guard`'s `catch_unwind` covers Rust panics — **not** a C
  callback that violates the ABI contract (the caller's responsibility,
  documented in the header).

- **Q9.5 Always-on + reader exposure — ALWAYS-ON, EXPOSE READERS**
  (agent-folded; maintainer may override to verifiers-only). The TLCP symbols
  are **always-on** on the C shim: `gmcrypto-c`'s core dep gains **`tlcp`** in
  its always-on feature list (currently `sm4-aead, sm4-xts, sm2-key-exchange,
  x509` — `x509` + `sm4-aead` already present; Codex consult #5) unconditionally
  — no forwarding C-shim feature — so the committed `gmcrypto.h` == a default `cargo build -p
  gmcrypto-c` (the locked v0.23/v1.2/v1.4 posture). This also **neutralizes**
  the cbindgen cfg-gated-opaque-typedef-drop for the GCM key carrier (the
  v0.10 gotcha): with `sm4-aead` always-on, `gmcrypto_tlcp_record_keys_gcm_t`
  is never dropped, so **no `regen-header` IMPLY hack is needed**. The
  `KeyUsage`/`BasicConstraints` readers are **mirrored** as copy-out accessors
  on the existing const cert handle (KeyUsage → a `u16` bitmask + present-flag;
  BasicConstraints → `is_ca` + `has_path_len`/`path_len` + present-flag, the
  `extensions_raw`-absent-flag precedent) — cheap, pure structural reads, no
  trust decision, the v1.4 mirror-everything pattern; a C caller building
  custom role logic needs them.

### Inherited FFI-shape constraints (the "no ABI frozen early" payoff)

- **Q6.9 (key schedule + no-confirm KX).** Key schedule = **stateless
  byte-in/byte-out free fns**, fixed-size buffers, **no handle, no
  `write_output` two-call discovery** (the inputs/outputs are fixed lengths).
  `TlcpRole` crosses as a plain `int` param (`0` = Client, `1` = Server) — an
  *input* int has no return-value contract to falsify (the `is_self_issued`
  lesson again). The no-confirmation KX completers **consume the existing v1.2
  born-waiting handles** (`Box::from_raw` even on the error path — the
  `_finalize*` precedent); the responder's `_respond_unconfirmed` is **one
  call** (there is no `S_A` to verify ⇒ no Waiting state ⇒ simpler than the
  confirmed two-step respond/finish).

- **Q7.10 (record).** Plain **copy-out** (`out` / `out_capacity` /
  `out_actual_len` two-call discovery), single error code, caller-held
  `seq`/IV/keys, **no separate strip-pad / verify-MAC entry points**, the
  equalization constant is NEVER a param. `deprotect_*` take **no
  length-in** param.

- **Q8.15 (chain/pair).** Slices of the v1.4 const cert handle + anchor slice
  + optional time struct → single `int` status (resolved by Q9.3 to
  out-param-bool + status); no new opaque type; no consume-on-use; readers may
  surface as copy-out accessors (resolved by Q9.5 to "yes").

### Census, version, posture

- **Q9.6 Symbol census.** ~19 new symbols + **2** new opaque types
  (`record_keys_{cbc,gcm}_t`) + ~7 new always-on `#define` consts
  (`GMCRYPTO_TLCP_MASTER_SECRET_LEN` = 48, `_FINISHED_VERIFY_DATA_LEN` = 12,
  `GMCRYPTO_TLCP_CBC_KEY_BLOCK_LEN` = 128, `GMCRYPTO_TLCP_GCM_KEY_BLOCK_LEN` = 40
  [Codex consult #5 — name the key-block sizes, no magic numbers],
  `GMCRYPTO_TLCP_RECORD_VERSION` bytes, `GMCRYPTO_X509_MAX_CHAIN_DEPTH` = 8;
  cbindgen emits free consts unconditionally — the `GMCRYPTO_SM4_XTS_KEY_SIZE`
  precedent). **85 → ~104 entry points** (exact count locked at W3). Names:
  `gmcrypto_tlcp_*` for key schedule + record + `verify_pair`;
  `gmcrypto_x509_verify_chain` + `gmcrypto_x509_certificate_{key_usage,
  basic_constraints}` for the x509 surface; the no-confirm KX completers as
  `gmcrypto_sm2_kx_{initiator,responder}_*_unconfirmed[_with_rng]` siblings to
  the v1.2 confirmed symbols.

- **Q9.7 Header & drift gate.** Regenerate + commit `include/gmcrypto.h` via
  `cargo build -p gmcrypto-c --features regen-header`; the `ci.yml` cabi job's
  `git diff --exit-code` is the gate. With Q9.5 always-on, `regen-header` need
  **not** imply any feature (all symbols + the opaque GCM typedef emit from the
  default build). Any convenience disclaimer text rides cbindgen.toml
  documentation, not a macro.

- **Q9.8 Version + posture — PUBLISHING cycle.** New FFI surface + the thin
  additive `&[&Certificate]` refs verifier in core (Q9.3) ⇒ a real crates.io
  release. Workspace **1.8.0 → 1.9.0**; sibling pins `=1.9.0`; lockstep publish
  simd → core → c is the maintainer's authenticated per-release call.
  `cargo-semver-checks`: **additive** (new C symbols + a new `#[doc(hidden)]
  pub` core fn — additive, non-breaking). The api-baseline
  (`docs/api-baseline/gmcrypto-core.txt`) **is** regenerated for the refs
  variant, as the **last pre-PR step** (the bit-v1.7 lesson — the drift-check
  isn't in the local gate).

## 2. Out of scope (this cycle)

- **The O2 sans-I/O connection engine** — no streaming/connection handle, no
  stateful auto-incrementing `seq`, no 5-byte record-header framing, no hidden
  global/policy state. `(key, seq)` uniqueness stays the **caller's** contract
  across the ABI exactly as in-core (v1.7 Q7.5). The sans-I/O engine is the
  explicitly-uncommitted O2 later decision (`docs/v1.5-scope.md` Q5.3 /
  `tlcp-decomposition.md` §7 v2.x).
- **Endpoint identity binding** through a convenience predicate — out (the
  v1.8 Codex consult #1 stance carries: the SDK cannot parse the Name behind a
  baked-in identity check, so it would invite false assurance). The caller
  composes identity against the cert handle.
- **Private/internal items** — `p_sm3`, the record CT helpers
  (`mac_equalized`/`check_tls_padding_ct`/`extract_mac_ct`/`gcm_nonce`/
  `record_header`), the x509 chain helpers (`find_extension`/
  `has_unknown_critical`/`is_ca_issuer`/`within_window`/`*::parse`/
  `subject_is_empty`). The shim must **not** re-implement any of them
  (re-implementing the CT helpers would break the Lucky13 equalization); they
  are encapsulated by the public verifiers/protectors.
- **Re-freezing any existing ABI** — the v1.4 cert handle + time struct, the
  v1.2 KX handles + `CallbackRng`, are reused verbatim.
- **In-place transforms** — record protect/deprotect are **copy-out** (Q7.10),
  not the v0.16 in-place `xts_*_sectors` shape, so the key-copy-before-`&mut`
  aliasing gotcha does not arise.
- **New core crypto, new dudect target, new dep** — none. The one core
  addition is a thin **non-crypto** `&[&Certificate]` refs adapter (Q9.3)
  delegating to the existing v1.8 chain walk; the shim otherwise rides
  public-inputs / already-CT-guarded core paths.

## 4. Assurance plan

- **`c_smoke`** (the v1.2/v1.4 pattern): Rust-equivalence per symbol +
  **KAT-through-the-ABI** — the OpenSSL/GmSSL record vectors via
  `_protect_cbc_with_rng` (fixed IV), the GM/T 0003.5 KX vector via
  `_respond_unconfirmed_with_rng` (fixed ephemeral), the key-schedule vectors
  vs core. Negatives: NULL sweeps, buffer-too-small two-call discovery,
  **`deprotect_cbc` bad-pad and bad-MAC indistinguishable through the ABI**
  (single ERR), the GmSSL 3-level chain verifying + minted-cert pair-binding
  negatives through `verify_pair`. No spent-handle machinery for record/verify
  (stateless); the no-confirm KX completers consume-on-use like the existing
  confirmed flow (spent-handle negative reused).
- **Fuzz** — extend `fuzz_c_abi`'s op-dispatch modulus (bump `% N`; **audit
  every committed seed's op byte against the new modulus** — the v1.4 `%8→%9`
  precedent), **not** a new fuzz binary. Census unchanged (30). The FUZZ_TARGETS
  list already names `fuzz_c_abi`.
- **Dudect** — **NO new target.** The record path rides core's
  `ct_tlcp_cbc_deprotect`; chain/pair is public-inputs; key schedule rides
  `ct_hmac_sm3`; KX rides `ct_sm2_key_exchange`. The thin-shim rationale
  (v1.2/v1.4) doubles. Count stays 23.
- **api-baseline** — **regenerated** (the additive `&[&Certificate]` refs
  verifier of Q9.3 widens the `gmcrypto-core` surface, even as `#[doc(hidden)]
  pub`). Pinned `cargo-public-api 0.52.0` + `nightly-2026-05-23`, the **last
  pre-PR step** (the bit-v1.7 lesson — the drift-check isn't in the local gate).
- **Build wiring (Codex consult #5)** — `gmcrypto-c`'s `gmcrypto-core` dep
  feature list gains `"tlcp"` (→ `sm4-aead, sm4-xts, sm2-key-exchange, x509,
  tlcp`); `tlcp::chain` needs `all(tlcp, x509)`, both now present. The
  `cargo deny` opt-in-features pass already lists `gmcrypto-core/tlcp`.
- **Full local gate** — fmt; clippy on every feature leg (default + the
  always-on C shim); `cargo test -p gmcrypto-c`; the `gmcrypto.h` drift
  `git diff --exit-code`; MSRV 1.85; the C example compiled + run locally.
