# v0.20 — Scope doc

**Fuzzing expansion + codify the v1.0 constant-time posture.** Two new
**streaming-decryptor differential fuzz targets** (`Sm4CbcDecryptor` /
`Sm4GcmDecryptor` fed in arbitrary chunk boundaries, byte-compared against the
single-shot `mode_{cbc,gcm}::decrypt` oracle) + a **`cargo fuzz coverage`**
reporting job in nightly CI, plus a small docs deliverable that **codifies the
settled v1.0 dudect CT baseline (with narrow revisit criteria)**. v0.20 is a **repository / infra-assurance milestone, publish-contingent**:
it touches only the workspace-excluded `fuzz/` crate + `fuzz-nightly.yml` + docs
(incl. `SECURITY.md`); the published 3-crate library is byte-unchanged. Per the
v0.14 (Q14.11) precedent, a clean fuzz run ⇒ **no version bump, no `cargo publish`,
no tag** (workspace stays `0.16.0`, crates.io skips `0.20.0`). The class-split
noise-twin, AVX-512, CCM-buffered, `aead 0.6`, and the full v1.0 API audit are
deferred to **v0.21+**.

## 1. Context

After v0.18 (pin + multi-run median) and v0.19 (self-calibrating relative gate,
**falsified**), the two short single-field-inversion dudect diagnostics
(`ct_fn_invert`, `ct_fp_invert`) remain un-gateable at `|tau| < 0.20` on shared
GitHub runners. v0.19's 100K calibration localized *why*: the runner noise lives
in the **two-input class-split difference** (`z_small` vs `z_large` timed against
each other), not the operation duration a same-input fix-vs-fix probe can observe —
so the probe stayed quiet (~0.005) while `ct_fp_invert`'s median hit 0.2606 (ratio
50). The fix-vs-fix probe approach is structurally blind to the relevant noise.

Two dedicated dudect cycles is enough. A strategy consultation with Codex and Grok
(provenance above) converged on: **stop chasing these two targets on shared CI**,
do **one more assurance cycle** whose artifacts also serve v1.0 readiness, and
**declare the terminal posture**. The chosen cycle continues the v0.14 fuzzing arc:
v0.14 deferred round-trip/differential parser fuzzing, **streaming-decryptor
fuzzing**, and **`cargo fuzz coverage`** (its `§6` candidate list). v0.20 takes the
streaming-decryptor + coverage subset because:

- The **streaming decryptors are shipped, security-relevant, stateful code**
  (`Sm4CbcDecryptor` buffer-back-by-one PKCS#7 boundary; `Sm4GcmDecryptor`
  commit-on-verify) whose chunk-boundary behavior is currently covered only by
  **fixed-chunk** in-crate tests. A differential fuzz target (streaming-fed-in-
  arbitrary-chunks vs single-shot) generalizes that to adversarial inputs +
  arbitrary chunking — exactly the gap fuzzing closes.
- The harnesses + a coverage report are **direct v1.0 readiness evidence**, earned
  **without** the irreversible 1.0 commitment (which a v0.21 readiness audit makes).
- It reuses the existing `fuzz/` infra (16 targets, nightly sweep, pinned
  `cargo-fuzz 0.13.1`, front-consuming `arbitrary` carve idiom) — low-risk, known.

### Building blocks already in place (grounded 2026-05-31)

- **`fuzz/` is its own workspace** (parent `Cargo.toml` `exclude = ["fuzz"]` + an
  empty `[workspace]` in `fuzz/Cargo.toml`); `libfuzzer-sys`/`arbitrary` never enter
  the published graph. `fuzz/Cargo.lock` is committed (pins `arbitrary 1.4.2`).
- **Existing SM4 targets** `fuzz_sm4_{cbc,gcm}_decrypt.rs` establish the
  **front-consuming carve** (key/iv/nonce/aad/tag via `arbitrary::<[u8;N]>()` /
  `bytes` / `take_rest`, all front; pinned to `arbitrary 1.4.2` read order). New
  targets mirror this + add a `chunk_len` byte.
- **Streaming APIs** — `Sm4CbcDecryptor::{new(&[u8;16],&[u8;16]), update(&[u8]),
  finalize() -> Option<Vec<u8>>}` (`cbc_streaming.rs`); `Sm4GcmDecryptor::{new(&[u8;16],
  &[u8],&[u8]), update(&[u8]), finalize_verify(&[u8]) -> Option<Vec<u8>>}`
  (`gcm_streaming.rs`).
- **Single-shot oracles** — `mode_cbc::decrypt(&[u8;16],&[u8;16],&[u8]) ->
  Option<Vec<u8>>`; `mode_gcm::decrypt(&[u8;16],&[u8],&[u8],&[u8],&[u8;16]) ->
  Option<Vec<u8>>`. Existing differential tests
  (`gcm_streaming.rs::decryptor_chunked_matches_single_shot` at fixed chunk sizes;
  CBC has only encrypt-side chunk tests) are the baselines the fuzz targets
  generalize.
- **`fuzz-nightly.yml`** — `ubuntu-latest`, nightly + pinned `cargo-fuzz 0.13.1` per
  run, a `TARGETS` sweep loop with `-max_len=16384 -rss_limit_mb=2048 -timeout=25
  -max_total_time=$BUDGET`, crash-artifact upload on failure. **No coverage job yet.**

## 2. Hard contract

- **No published-crate code change.** The diff touches only `fuzz/` (workspace-
  excluded, unpublished), `.github/workflows/fuzz-nightly.yml`, and docs (`fuzz/
  README.md`, `SECURITY.md`, `CLAUDE.md`, `README.md`). **No `crates/`, no root
  `Cargo.toml`/`Cargo.lock`, no new published dep.** The 3 published crates'
  `cargo build/test --workspace`, `cargo deny`, and MSRV-1.85 are unaffected by
  `fuzz/`.
- **Assurance / publish-contingent.** A clean fuzz run ⇒ **no version bump, no
  publish, no tag** (workspace stays `0.16.0`; crates.io skips `0.20.0`; the
  v0.14/v0.17/v0.18/v0.19 precedent). The milestone lives in `docs/` + `CLAUDE.md`
  (+ Notion). **Only** a real divergence/crash in **shipped streaming code** would
  warrant a fix + a possible patch release (maintainer decision at that point).
- **The two v0.19 noise probes are UNCHANGED** — permanent telemetry controls, not
  gated, not promoted, not evolved. A class-split twin is a *named revisit-door
  candidate* (Q20.4), not v0.20 work.
- **dudect untouched.** No change to the dudect workflows, the gate logic, or the
  invert-target posture beyond the docs codification (Q20.4). The dudect jobs stay
  on `ubuntu-24.04`; `MATRIX_FEATURES` invariants untouched (we don't edit those
  files).
- **Seed discipline** preserved: `cargo fuzz run` writes to the **corpus** dir
  (first arg), never `seeds/` (second arg) — committed seeds stay curated.

## 3. Q-list (Q20.1–Q20.5) with answers

### Q20.1 — Anchor (theme)
**Fuzzing expansion: streaming-decryptor differential fuzzing + `cargo fuzz
coverage`.** Single focused theme (two facets of one deliverable, a combination
both models endorsed). Maintainer-chosen over the v1.0 readiness audit (deferred to
v0.21) after the Codex+Grok strategy discussion. Rejected for this cycle: new
features (AVX-512 / CCM-buffered — both enlarge the v1.0 audit), and a full
noise-twin cycle (15–30% feasibility, "good money after bad").

### Q20.2 — Targets (the differential oracle)
Two new targets: **`fuzz_sm4_cbc_streaming_decrypt`** and
**`fuzz_sm4_gcm_streaming_decrypt`**. Each carves inputs **front-consuming** (the
established `arbitrary 1.4.2` read order; seeds are plain byte concatenations in that
order), then feeds `ct` to the **streaming** decryptor in fixed-size chunks of
`max(1, chunk_len)` (a `chunk_len` of 0 ⇒ 1-byte chunks) and asserts **byte-equality**
against the **single-shot** API fed all-at-once (`Some==Some` plaintext, or
`None==None`), plus the failure-mode invariant (no panic / OOM / hang). **Exact
pinned layouts** (W1 confirms against the existing `fuzz_sm4_{cbc,gcm}_decrypt.rs`):

- **CBC** — `[key:16][iv:16][chunk_len:1][ct:take_rest]`. Oracle:
  `Sm4CbcDecryptor::new(&key,&iv)` + chunked `update` + `finalize()` **vs**
  `mode_cbc::decrypt(&key,&iv,ct)`.
- **GCM** — `[key:16][tag:16][nonce_len:1][nonce:nonce_len][aad_len:1][aad:aad_len][chunk_len:1][ct:take_rest]`.
  Oracle: `Sm4GcmDecryptor::new(&key,&nonce,&aad)` + chunked `update` +
  `finalize_verify(&tag)` **vs** `mode_gcm::decrypt(&key,&nonce,&aad,ct,&tag)`. A
  **fixed 16-byte tag** is used deliberately (the `mode_gcm::decrypt` path) — this is
  *not* a literal copy of `fuzz_sm4_gcm_decrypt`'s variable-tag carve; truncated-tag
  (`decrypt_with_tag_len` vs `finalize_verify(tag_var)`) is a deferred extension.

This generalizes the fixed-chunk in-crate differential tests
(`gcm_streaming.rs::decryptor_chunked_matches_single_shot`) to arbitrary chunking +
adversarial inputs; the CBC target closes a real gap (no decrypt-vs-single-shot test
exists today). Both `nonce`/`aad` are variable-length (matching the single-shot
domain), so streaming==single-shot is expected for *all* carved inputs — any
divergence is a genuine bug.

### Q20.3 — Coverage reporting
Add a **`cargo fuzz coverage`** job to `fuzz-nightly.yml` (`rustup +nightly
component add llvm-tools-preview`; `cargo +nightly fuzz coverage <t>` over
corpus+seeds → `.profdata`; `llvm-cov report` summary → uploaded artifact, 30-day
retention). **Gate only on "job completes + uploads," never a numeric % threshold**
(both models: the report is the deliverable, not a vanity gate). Scheduled only (not
a PR gate); a separate job / guarded step so a coverage failure can't mask a fuzz
crash. **Corpus semantics (explicit):** `cargo fuzz coverage` needs existing input
files; a fresh CI job sees only the **committed `seeds/`** (the gitignored `corpus/`
is empty on a clean checkout unless plumbed via cache/artifact). v0.20 scopes the
coverage report as **seed-corpus coverage** — that is acceptable (the report is the
deliverable; growing-corpus coverage is a later refinement). W2 pins the concrete
paths: the coverage-built target binary under
`fuzz/target/<host-triple>/coverage/<host-triple>/release/<target>` and the merged
`fuzz/coverage/<target>/coverage.profdata`, rendered via the `llvm-tools-preview`
`llvm-cov report`.

### Q20.4 — v1.0 CT posture = Option 2 (codify + narrow revisit door)
Codify the current posture as the v1.0 constant-time baseline in `SECURITY.md`
(the user-facing posture doc), cross-referenced from `CLAUDE.md` + the recalibration
doc. Decision reached via a maintainer-delegated Codex discussion. Rejected:
**Option 1** ("permanent/final") — overclaims; no clean replacement experiment
proves the diagnostics can *never* be gated. **Option 3** ("keep open") — signals an
active CT hardening gap to v1.0 reviewers and invites repeating v0.19's failed line.
**Framing** (Codex's words): composite dudect targets remain release-gated
`|tau| < 0.20`; the two single-field-inversion micro-diagnostics are telemetry (PR)
+ a nightly gross-regression sentinel `@0.55` because shared-runner measurements
show an empirically unstable **two-input class-split** floor after the 2026-05-12
image refresh; v0.19 falsified same-input probe normalization as a re-promotion
path. **Revisit door, narrow:** only if a *materially different measurement design*
exists — a **class-split-twin** diagnostic that reproduces the dudect split geometry
**without** the inversion operation, **or** an **offline / dedicated / manual
hardware** experiment. **Not** PR-executing public self-hosted CI (RCE; rejected at
v0.17).

### Q20.5 — Release posture
Assurance / publish-contingent; default **no publish** (Q20.2 / Q14.11 precedent).
Publish only if a fuzz target surfaces a real fix to shipped code that warrants it.

## 5. Out of scope / deferred to v0.21+

- **Round-trip / differential *parser* fuzzing** (`decode∘encode` stability) — an
  optional W2 stretch at most; otherwise v0.21.
- **Class-split noise-twin** — the Q20.4 revisit-door *candidate* (a split that
  reproduces the dudect two-input geometry through a known-CT op, not the
  inversion). NOT built this cycle; revisited only if/when designed rigorously.
- **AVX-512 16-way `sbox_x64`**; **SM4-CCM incremental/buffered input** — feature
  cycles; both enlarge the v1.0 audit, so post-readiness.
- **`aead 0.6` trait fit** — still `0.6.0-rc.10` upstream (blocked).
- **v1.0 readiness audit** — API/SemVer review + docs freeze; the likely **v0.21**
  theme, with v0.20's fuzz harnesses + coverage report as input evidence.
