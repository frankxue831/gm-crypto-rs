# v0.5 dudect recalibration — runner-image drift on `ubuntu-24.04`

**Status:** landed 2026-05-13 (this branch).
**Targets recalibrated:** `ct_fn_invert`, `ct_fp_invert`.
**Authoritative fix deferred:** pinned / noise-isolated dudect runner.

## Summary

The 10K PR-smoke (`dudect-pr.yml`) and 100K nightly (`dudect-nightly.yml`)
gates for `ct_fn_invert` and `ct_fp_invert` began producing flaky
failures on `main` starting **2026-05-11 ~ 16:00 UTC**. The same commit
passes one nightly and fails the next, with `|tau|` values spanning
[0.20–0.40] at the 100K budget — orders of magnitude above the
historical noise floor of ~0.006 documented in `CLAUDE.md` ("v0.1
timing-leak narrative" section). The failures span both `sm4-bitsliced`
and `sm4-bitsliced-simd` matrix entries; no source change in v0.5
intersects the SM2 `Fp::invert` / `Fn::invert` paths these targets
measure.

The most likely root cause is the
[GitHub `actions/runner-images` `ubuntu-24.04` update](https://github.com/actions/runner-images/releases)
that landed during the same window: image `20260413.86.1` →
`20260512.134.1`, kernel `6.17.0-1010-azure` → `6.17.0-1013-azure`,
Rust toolchain `1.94.1` → `1.95.0`. A kernel or toolchain change can
shift binary layout (hot-loop alignment, branch predictor state) and
scheduler noise enough to perturb a sensitive `|tau|` measurement
without touching crypto code.

This document records the empirical data, the recalibration applied
to both workflows, and the deferred authoritative fix.

## Recalibration applied

### `dudect-pr.yml` (10K PR-smoke)

- `ct_fn_invert` and `ct_fp_invert` removed from the blocking
  `required_low` allowlist.
- Both targets added to a `telemetry_only` set: the bench still runs,
  the `|tau|` value still lands in the log (prefixed `TELEMETRY:`),
  but the value no longer gates the PR.
- `ct_sign_k_class` was already non-blocking at PR-smoke; added
  explicitly to the telemetry set for consistency.

### `dudect-nightly.yml` (100K nightly)

- `ct_fn_invert` and `ct_fp_invert` removed from the blocking
  `required_low` allowlist.
- Both targets added to a new `gross_regression_sentinel` dict at
  `|tau| ≥ 0.55`. This keeps a coarse protection against a real
  cryptographic regression (the v0.1 `crypto-bigint 0.6`
  `ConstMontyForm::invert` measured `|tau| ≈ 0.70` in isolation;
  see `CLAUDE.md`'s "v0.1 timing-leak narrative" section) while
  tolerating the observed noise envelope of [0.29–0.40].
- `ct_sign_k_class` retains its existing 0.25 gate at nightly.

### Both workflows

- New "Capture runner environment" step prints `ImageVersion`,
  kernel (`uname -a`), CPU model (`lscpu`), CPU governor / turbo
  state, and `rustc -Vv` ahead of the bench. Lets a future
  noise-floor shift be correlated to a specific image / kernel /
  toolchain update without re-deriving the timeline from artifact
  inspection.

## What this is NOT

- **Not a code regression in v0.5**: the SM2 `Fp::invert` /
  `Fn::invert` code paths are byte-identical to v0.4 published.
  Compiled binaries at 100K samples on the older runner image
  measured `|tau| ≈ 0.006` consistently.
- **Not specific to a feature**: failures appear under both
  `sm4-bitsliced` and `sm4-bitsliced-simd` matrix entries (and
  occasionally `default` on the W5 PR). The matrix asymmetry
  observed on individual runs is noise, not a feature interaction.
- **Not unique to invert diagnostics**: `ct_sign_k_class` showed
  the same shift on 2026-05-11 nightlies. The invert pair is more
  sensitive (smaller real signal-to-noise margin) so it crossed
  the boundary first.

## Deferred: authoritative fix

A pinned / noise-isolated dudect runner — ideally a dedicated
self-hosted machine running on a controlled kernel with CPU
isolation (`isolcpus`), no SMT, fixed governor, and pinned bench
process affinity — would lift the noise floor back below the
historical 0.006 baseline at the 100K budget and re-enable the
original `|tau| < 0.20` gate on all targets. This is the same
posture as `ci.yml`'s self-hosted aarch64 macOS runner setup but
targeting noise-floor stability rather than minute-cap economics.

Not in scope for this PR. Tracked separately; recommended for a
v0.5.x or v0.6 scope-doc workstream.

## v0.18 resolution — pin + multi-run median (landed 2026-05-30)

**The "authoritative fix" above (a noise-isolated self-hosted runner) is OFF THE
TABLE.** v0.17 decommissioned the self-hosted runner — a self-hosted runner on a
public repo is remote code execution. v0.18 instead hardens the gate purely in
software on GitHub-hosted `ubuntu-24.04`, per `docs/v0.18-scope.md`:

1. **Pinned both drift axes** — `runs-on: ubuntu-24.04` (an **OS-label** pin,
   freezing the OS-major axis against the future `ubuntu-latest`→26.04 roll) plus
   an exact `dtolnay/rust-toolchain@1.95.0` (the **load-bearing** pin — the
   2026-05-12 drift rode a 1.94→1.95 bump). The OS label is *not* an exact-image
   freeze (GitHub still refreshes the 24.04 image/kernel); the median (below)
   absorbs that residual.
2. **CI-level multi-run median** — the bench loops N times (PR N=3 / nightly N=5)
   into numbered logs; the parse step gates the per-target **median** `|tau|`
   (`required_low` + the sentinel) and the **min** (`negative_control`, must fire
   every run). `timing_leaks.rs` is byte-unchanged; the loop + median live
   entirely in CI. The median is robust to the *intermittent* single-run spikes
   the runner noise produces.

### Empirical calibration (100K×5, pinned branch, 2026-05-30)

A dispatched nightly on the pinned `ubuntu-24.04` / Rust 1.95.0 runner measured
the two direct-invert diagnostics an **order of magnitude below the 0.20 gate** —
back near the pre-incident ~0.006 baseline:

| leg | target | per-run tau (×5) | median |
|---|---|---|---|
| sm4-bitsliced      | ct_fn_invert | 0.0052 0.0048 0.0112 0.0067 0.0063 | **0.0063** |
| sm4-bitsliced      | ct_fp_invert | 0.0158 0.0140 0.0067 0.0177 0.0082 | **0.0140** |
| sm4-bitsliced-simd | ct_fn_invert | 0.0109 0.0111 0.0097 0.0040 0.0105 | **0.0105** |
| sm4-bitsliced-simd | ct_fp_invert | 0.0303 0.0314 0.0231 0.0278 0.0178 | **0.0278** |

(All 20 individual 100K measurements ≤ 0.0314. `ct_sign_k_class` medians on the
same run: 0.2116 / 0.1244 — within the 0.25 nightly gate.)

### Re-promotion decision: KEPT telemetry / sentinel (NOT re-promoted)

Although those medians clear the `docs/v0.18-scope.md` Q18.7 re-promotion
criteria by a wide margin, the two invert targets **remain telemetry-only (PR) /
median-gated gross-regression sentinel @0.55 (nightly)** — *not* re-promoted to
an authoritative `|tau| < 0.20` gate. Rationale (robustness-first — the cycle's
whole purpose):

- **The pin froze 1.95.0 — the version present during the spike.** What is quiet
  today is the *current* 24.04 image, not the pin; a future image refresh can
  re-introduce the noise (the OS-label pin does not freeze the image build).
- **A noise return is one the median cannot robustly tame.** The 2026-05-12
  pattern was *intermittent* [0.29–0.40] with same-commit pass/fail across
  consecutive runs. If it returns, a 5-run *median* can itself land above 0.20
  (e.g. 3 of 5 spiky runs) — re-flaking the gate, the exact failure v0.18 exists
  to prevent.
- **The sentinel survives a noise return AND catches a real leak.** The observed
  noise maxed at ~0.40; the sentinel @0.55 tolerates it, while the v0.1
  `ConstMontyForm::invert` leak (~0.70) still fires. The median-of-5 telemetry
  now records the per-run trend, so a creeping real leak stays visible.

The v0.18 outcome is therefore **pin + median** (robustness), with the invert
pair kept on the sentinel as the deliberate, robustness-aligned choice. A
**self-calibrating relative gate** — which could safely re-promote by deriving
the threshold from the run's own noise floor — is the v0.19 candidate that would
revisit re-promotion.

## v0.19 resolution — self-calibrating relative gate TESTED and FALSIFIED (2026-05-30)

v0.19 implemented the self-calibrating relative gate the v0.18 resolution named
as the path to re-promotion: each invert target `T` gated against a
**matched-sensitivity fix-vs-fix probe** `P` (a new bench target running the
*same* inversion — `Fn::invert` / `Fp::invert` — but feeding **both dudect
classes one identical input**, so any `|tau|` is pure measurement noise of that
operation). Threshold = `max(ABS_FLOOR=0.20, K=4·median(P))`. The hypothesis: a
2026-05-12-style image-noise return would lift the probe *and* the target
together (matched sensitivity), so the gate would absorb it; a real leak would
lift only the class-split target, so the ratio would diverge. Per `docs/v0.19-scope.md`
this was merged to `main` as authoritative-at-nightly **contingent on** a
dispatched-CI calibration meeting the Q19.5 criteria, with an honest-fallback
path if the probe failed to track.

### Empirical calibration (100K×5, dispatched on `main`, run 26686570785, 2026-05-30)

| leg | target | per-run `|tau|` (×5) | **median** | probe median | ratio | relative verdict |
|---|---|---|---|---|---|---|
| sm4-bitsliced      | ct_fn_invert | 0.0059 0.2550 0.0100 0.0752 0.0862 | 0.0752 | 0.0053 | 14.1 | would-PASS |
| sm4-bitsliced      | ct_fp_invert | 0.0671 0.2922 0.0719 0.0432 0.3157 | 0.0719 | 0.0064 | 11.3 | would-PASS |
| sm4-bitsliced-simd | ct_fn_invert | 0.1267 0.0075 0.0081 0.0044 0.0569 | 0.0081 | 0.0073 | 1.1  | would-PASS |
| sm4-bitsliced-simd | **ct_fp_invert** | **0.2606 0.2870 0.1359 0.0390 0.2983** | **0.2606** | **0.0052** | **50.0** | **would-FAIL** |

(The `default` and `sm4-bitsliced-simd,sm4-aead,sm4-xts` legs were cancelled
during the dispatch wave — branch-deletion-at-merge + the one-pending-slot
`concurrency` config — so they carry no data for this run. The two legs above
are sufficient: the pattern is unambiguous.)

### Finding: the matched-sensitivity hypothesis is FALSIFIED

The **fix-vs-fix probes stay uniformly quiet** (`noise_floor_*` medians
0.0052–0.0073 across *both* legs, every individual run ≤ 0.0096), while the
**real class-split targets spike intermittently into [0.26–0.32]** — the
2026-05-12 noise envelope, present *now* on the current pinned image. On the
`sm4-bitsliced-simd` leg `ct_fp_invert`'s **median itself reached 0.2606**
(3 of 5 runs spiked), exceeding even a plain `0.20` gate, while its probe sat at
0.0052 (ratio **50**).

So the runner noise lives in the **two-input class-split difference**
(`z_small` vs `z_large` timed against each other), **not** in the operation's
duration that a same-input probe can see. A fix-vs-fix probe is structurally
blind to it. The matched probe therefore does **not** track the noise it was
built to calibrate against, and the relative threshold stays pinned at
`ABS_FLOOR=0.20` (probe quiet) — i.e. the relative gate is **no better than the
fixed 0.20 gate the noise already breaks**. This is exactly the residual risk
flagged in `docs/v0.19-scope.md` §1 ("if the noise turns out to be specific to
the class-split structure rather than the operation's duration, the relative
gate degenerates to a fixed 0.20 gate").

### Decision: honest fallback (Q19.5) — keep telemetry/sentinel, keep the probes

Re-promotion criteria FAIL decisively (ratio 50 ≫ the 3.0 bar; the relative gate
FAILs on the simd leg). So:

- **The relative gate is demoted to non-blocking `REL-TELEMETRY`** in both
  workflows (it never sets `fail`).
- **`ct_fn_invert` / `ct_fp_invert` revert to the v0.18 posture** — telemetry at
  PR, the median-gated gross-regression **sentinel @0.55 at nightly** (the sole
  authoritative gate for these two again). `ct_fp_invert`'s 0.26 median sits well
  under 0.55, so the nightly is green; a real `ConstMontyForm::invert`-class leak
  (~0.70) still fires.
- **The two fix-vs-fix probes are KEPT** in the harness as permanent telemetry.
  They are now the *evidence* that the noise is class-split-specific (a uniformly
  ~0.005 floor against which the target spikes stand out), which is the key input
  to the v0.20 redesign. (Per Q19.5 the harness keeps them even on fallback.)

### Implication for v0.20

A working self-calibrating reference must **capture the class-split structure**,
not a same-input operation. The candidate is a **"noise-twin" target**: a
two-different-input split through an operation *known* to be constant-time (so
its `|tau|` measures the runner's *two-input comparison* noise floor at the
current image), against which the real invert targets are gated *relatively*.
Designing and validating that twin (which operation; how to argue it is a sound
noise reference without itself becoming a leak surrogate) is materially more
subtle than the fix-vs-fix probe and is the v0.20 candidate. Until then the
v0.18 pin + median + sentinel posture stands — robustness-first, unchanged.

## 2026-06-07 resolution — `ct_sign_k_class` demoted to the sentinel + nightly timeout fix (post-1.0)

A post-1.0 CI-health audit (after the `1.0.1` publish) found the
`dudect-nightly` regression watch had been **mostly red for ~2 weeks**
(32 failure / 17 cancelled / 11 success of the last 60 runs) — i.e. no longer
watching. Two independent bugs, both unrelated to the crypto; Codex-reviewed
(read-only consult, 2026-06-07).

### Bug A — `ct_sign_k_class` trips its 0.25 nightly gate on image-noise

`ct_sign_k_class` (nightly-only composite, class-split by nonce `k` magnitude
with `d` fixed) was the last two-input class-split target still on a **hard**
gate (`median |tau| < 0.25`, retained by v0.4 release-prep on a "SNR is
meaningful at 100K" rationale). Recent nightlies:

| run | `ct_sign_k_class` median | `ct_fn_invert` | `ct_fp_invert` |
|---|---|---|---|
| [26999546427](https://github.com/frankxue831/gm-crypto-rs/actions/runs/26999546427) | **0.2768** | 0.0053 | 0.0115 |
| 26935244818 | **0.2570** | quiet | quiet |
| 26868150469 | **0.2582** | quiet | quiet |

The composite trips 0.25 while the **authoritative direct invert diagnostics
stay at ~0.005–0.012** — a **false alarm, not a leak**. This is the v0.19
finding applied to one more target: the runner noise lives in the **two-input
class-split difference**, which a composite like `ct_sign_k_class` sees but a
real leak need not drive. The v0.4 "0.25 at 100K" rationale was overtaken by
the 2026-05-12 image drift (the same drift that demoted the invert pair).

**Decision (Codex-reviewed):** demote `ct_sign_k_class` from `required_low`
(0.25) to the median-gated **gross-regression sentinel @0.55** — the same
posture as `ct_fn_invert` / `ct_fp_invert`, for the same reason. The direct
invert diagnostics remain the authoritative invert-leak guards; the sentinel
still fires on a gross nonce-path regression (a `ConstMontyForm::invert`-class
leak measures ~0.70). **Honest trade:** a *medium* `k`-only full-sign leak in
the band [0.25, 0.55] is no longer authoritatively gated at nightly until a
**class-split-aware "noise-twin"** reference exists (the v0.20+ candidate,
"Implication for v0.20" above) — the same residual the invert pair already
carries.

**Safety fix (Codex-flagged):** the completeness check covered only
`required_high + required_low`, and the sentinel loop prints-without-failing on
a `None` (not-measured) value — so a demoted sentinel target that went
missing/renamed could pass green. The **sentinel dict is now completeness-gated**
too (every sentinel target must be measured in all `n_runs`), closing that hole
for `ct_sign_k_class` *and* the invert pair.

### Bug B — `default` + `full` legs time out (the "leg-cancellation" item)

The recurring "`default`+`full` leg-cancellation" noted since v0.19 was a **job
timeout**, not a `concurrency` effect: the two slowest legs (`default` =
linear-scan S-box; the full `sm4-bitsliced-simd,sm4-aead,sm4-xts` feature set)
ran 5×100K samples + cold build to **~25m15s**, hitting `timeout-minutes: 25` —
GitHub renders the timed-out leg as "cancelled". They never reached the gate, so
half the matrix went red on a clock, not a measurement.

**Fix:** `timeout-minutes: 25 → 40`. The 100K×5 budget is the calibrated
measurement contract and is **not** reduced; 40 min adds headroom while staying
bounded. (Both fixes are needed for green: the timeout bump alone still leaves
all four legs failing the `ct_sign_k_class` gate; the demotion alone still
leaves `default`/`full` timing out.)

### Validation

`timing_leaks.rs` is **byte-unchanged** (the gate + loop live in the workflow
YAML); the embedded gate Python was extracted and `py_compile`-checked clean.
Validated by dispatching the nightly on the fix branch
(`gh workflow run dudect-nightly.yml --ref fix/dudect-nightly-recalibration`)
and confirming all four legs complete within 40 min and the run is green —
`ct_sign_k_class` now reports under a non-blocking `SENTINEL:` line, and the
`required_low` targets + the three `@0.55` sentinels all pass.

## 2026-06-17 resolution — `ct_hmac_sm3` demoted to the sentinel (the 4th two-input class-split target)

`ct_hmac_sm3` is the **fourth** target to be image-noise-dominated on
GitHub-hosted `ubuntu-24.04`, following `ct_fn_invert` / `ct_fp_invert` (v0.5)
and `ct_sign_k_class` (2026-06-07). Codex + Grok-reviewed (independent read-only
consults, both `A now / B-as-a-separate-cycle / C wrong`).

### Why it is the same phenomenon (constant-time by construction)

`ct_hmac_sm3` (`benches/timing_leaks.rs`) is class-split by **two fixed 32-byte
keys** (`0x42…` vs `0xa5…`) over a **fixed 64-byte message**. HMAC-SM3 over a
fixed-length input does an **identical number of SM3 compression calls for any
key** — there is no key-length branch and no secret-dependent loop bound. So the
target is *constant-time by construction*: it can only ever measure the runner's
**two-input class-split noise floor**, the exact quantity the 2026-05-12 image
drift inflates (v0.19 finding: the noise lives in the two-input difference, not
the operation's duration). It stayed under 0.20 until the image noise reached it.

### Empirical data (100K nightly)

The **2026-06-14 nightly FAILED** on `ct_hmac_sm3` — the 5-run **median itself
crossed** 0.20 (3 of 5 runs spiked), co-spiking with the already-demoted invert
pair on the same run; the very next nightly (06-16) was quiet:

| run (UTC) | leg | `ct_hmac_sm3` runs (×5) | median | same-run `ct_fp_invert` median |
|---|---|---|---|---|
| 2026-06-14 ([27490799728](https://github.com/frankxue831/gm-crypto-rs/actions/runs/27490799728)) | one leg | 0.0140 0.2375 0.0159 0.2196 0.2135 | **0.2135 (FAIL)** | 0.4737 / 0.3304 |
| 2026-06-14 | other legs | — | 0.0063 / 0.0091 / 0.0155 | — |
| 2026-06-16 ([27599740914](https://github.com/frankxue831/gm-crypto-rs/actions/runs/27599740914)) | all legs | all ≤ 0.0314 | 0.0086–0.0134 | quiet |

Because the **median** (not a single run) crossed — exactly the failure mode the
v0.18 resolution predicted ("3 of 5 spiky runs") — a partial fix that drops only
the 10K PR gate while keeping the 100K nightly at `0.20` is **falsified**.

### Decision

`ct_hmac_sm3` → **telemetry-only at the 10K PR gate** + **median-gated
gross-regression sentinel @0.55 at the 100K nightly** — the same posture as the
three prior two-input targets, completeness-gated (it must be measured in all
`n_runs`; the 2026-06-07 sentinel-completeness fix already covers it). The
sentinel still fires on a gross HMAC-key leak (a real `ConstMontyForm::invert`-
class leak measured ~0.70). `timing_leaks.rs` is **byte-unchanged** — the gate
dicts + comments live entirely in the workflow YAML.

### Honest residual (louder than the invert pair's)

The invert pair retained `ct_sign` as a direct-ish sign-path backstop.
**`ct_hmac_sm3` has NO direct non-composite backstop after this demotion** — and
the consults sharpened the residual coverage claim:

- ✅ `ct_pkcs8_decrypt` (gated 0.20) *is* a real HMAC-key backstop — PBKDF2 calls
  `hmac_sm3(password, …)` with secret bytes — but **iterated (1024×) + composite
  + different message lengths**, so it *dilutes* a primitive leak.
- ❌ `ct_tlcp_cbc_deprotect` (gated 0.20) is **not** a meaningful key-byte HMAC
  backstop — fixed key, class-split on recovered-fragment *length*, routed
  through the deliberately-different `mac_equalized` / `black_box` Lucky13 wrapper
  (it guards equalization, not ordinary keyed-HMAC timing).

**Worst concrete unguarded leak:** a key-byte-dependent timing effect in
`sm3::compress` ipad/opad first-block absorption, visible in *ordinary
single-shot / streaming `hmac_sm3`* at the 32-byte-key / 64-byte-message shape,
`|tau|` ∈ [0.20, 0.55], that PBKDF2's iteration + composite work keeps below 0.20
in `ct_pkcs8_decrypt`. The **TLCP key schedule (`p_sm3`, v1.6)** explicitly rides
`ct_hmac_sm3` per `SECURITY.md`, so it loses its *direct* guard and becomes an
indirect claim resting on the iterated/composite paths. This residual is
acceptable (the prior `0.20` gate was already non-authoritative on the noisy
shared runner — pretending otherwise was the worse option) but is documented
loudly here and in `SECURITY.md`.

### The noise-twin (deferred to its own cycle — B was NOT taken)

`ct_hmac_sm3` is a *constant-time-by-construction two-input split that
demonstrably co-spikes with the demoted family* — i.e. a candidate for the
**class-split-aware "noise-twin"** the v0.19/v0.20 baseline named as the only
design that could re-promote these targets (the same-input fix-vs-fix probes
failed because they stay quiet). The 06-14 co-spike is the first evidence it
could work. But both consults flagged the **leak-surrogate hazard**: using
`ct_hmac_sm3` as a relative-gate *denominator* means a real HMAC leak would
inflate the "noise" reference and mask the targets it gates — and it cannot
re-promote *itself*. The explicit **re-promotion bar** for a future cycle:
(1) multi-run / multi-image co-spike correlation (one night is not calibration),
(2) **injected-leak tests** proving a real invert / sign / HMAC leak still FAILs
under the relative rule, and (3) a written soundness argument that the twin
cannot absorb a real HMAC leak. Until then the demotion stands.
