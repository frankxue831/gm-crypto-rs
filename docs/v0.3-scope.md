# gmcrypto-core v0.3 scope

**Crate:** `gmcrypto-core`
**Status:** v0.2.0 published to crates.io 2026-05-10 (this morning).
**Document scope:** the maintainer signs off on this scope before any
implementation work begins. Implementation plans for each workstream are
drafted *after* sign-off, the same chunked-PR cadence v0.2 used.
**Hard constraints:** all carry-overs from `CLAUDE.md` apply unchanged
(`unsafe_code = "forbid"`, `#![no_std]` + `alloc`, MSRV 1.85, edition 2024,
constant-time discipline on secrets, failure-mode invariant, fixed-K masked-
select sign retry, `Cargo.lock` gitignored, `sign_raw_with_id` doc-hidden /
not-SemVer). Don't restate them; defer to `CLAUDE.md`.

---

## 1. Context

v0.2.0 shipped this morning: SM4 + SM4-CBC, HMAC-SM3 (single-shot),
PBKDF2-HMAC-SM3, SM2 envelope encryption with GM/T 0009-2012 ciphertext
DER, dudect harness expansion to **11 targets** all gating at
`|tau| < 0.20`. The `crypto-bigint 0.6 ‚Üí 0.7.3` upgrade resolved the
v0.1-era `ConstMontyForm::invert` leak directly (`ct_fn_invert` /
`ct_fp_invert` both at `|tau| ‚âà 0.006` at 100K samples ‚Äî two orders of
magnitude below the gate). The v0.2 Fermat-invert workstream was
dropped on that evidence.

What v0.3 inherits as constraint surface:

- `gmcrypto-core` API surface as of `v0.2.0` is **stable for additive change
  only**. Removing or renaming a public item is a v1.0-track change; v0.3
  may not break any v0.2 caller.
- `asn1::sig::{encode_sig, decode_sig}` and `asn1::ciphertext::{Sm2Ciphertext,
  encode, decode}` are public re-exports today; their *behaviour* must be
  preserved bit-for-bit during the W1 reader/writer generalization (i.e.
  the new shared subset must produce byte-identical output and accept/reject
  exactly the same byte sequences).
- `hmac_sm3` and `Sm4Cipher::new`/`encrypt_block`/`decrypt_block` stay
  shipped as single-shot. v0.3 adds streaming surfaces *alongside* them
  (W5).
- Failure-mode invariant carries forward verbatim. New error types in W2
  (PEM / PKCS#8 / SPKI) get one variant. DER decode returns `Option`. The
  CLAUDE.md "Don't make errors more helpful" rule applies to every wire-
  format module W1 produces.
- Dudect targets: 11 today; v0.3 adds at most 2 (see W2 / W5 below). All
  new targets gate at `|tau| < 0.20`.
- Dep landscape today: `crypto-bigint = 0.7.3`, `subtle = 2.6.1`,
  `zeroize = 1.8.2`, `rand_core = 0.10.1`, `getrandom = 0.4.2` (with
  `sys_rng`). Dev-deps `dudect-bencher = 0.7.0`, `hex-literal = 1.1.0`.
  No-std + alloc only; `std` feature reserved for v0.3+ wire-format I/O ‚Äî
  v0.3 finally exercises it (PEM file-loading helpers behind `std`).

The bridge: v0.2 is the *primitive-complete* release (every algorithm the
target audience needs). v0.3 is the *interop-complete* release ‚Äî same
primitives, with the wire-format and trait-shape glue that lets callers
plug `gmcrypto-core` into existing Rust crypto pipelines (in-crate
streaming `Mac` / `Hash` / `BlockCipher` traits in `pub mod traits`;
RustCrypto-trait fit deferred to v0.4 per Q7.3) and exchange artifacts
with the broader SM-crypto ecosystem (gmssl, OpenSSL with GM patchset).

Scope boundary: v0.3 does NOT change any cryptographic primitive's
internal byte path. SM2/SM3/SM4 implementations, the dudect-gated
constant-time properties, and the failure-mode invariant are preserved;
the work is wrappers, traits, and a precomputed-table optimization that
sits *underneath* `mul_g` without changing its public signature.

## 2. In-scope workstreams

### W1 ‚Äî Reusable ASN.1 reader/writer subset

**Goal.** Replace the ad-hoc DER helpers duplicated across `asn1::sig` and
`asn1::ciphertext` with a single reusable subset, in preparation for W2's
SPKI/PKCS#8/PEM work (which needs OBJECT IDENTIFIER, BIT STRING, NULL,
explicit/implicit context-tagged fields, and nested SEQUENCE ‚Äî none of which
the current ad-hoc helpers cover).

**Files touched.**

- New: `crates/gmcrypto-core/src/asn1/reader.rs` (TLV parser, length
  decoder, primitive readers ‚Äî INTEGER, OCTET STRING, BIT STRING,
  OBJECT IDENTIFIER, NULL, context-tagged \[n\] EXPLICIT/IMPLICIT,
  SEQUENCE/SET-OF iterators).
- New: `crates/gmcrypto-core/src/asn1/writer.rs` (length encoder,
  primitive writers, in-order SEQUENCE-body composer that computes
  bodies bottom-up so outer length is exact on first pass).
- New: `crates/gmcrypto-core/src/asn1/oid.rs` (compile-time OID
  literal-from-dotted-decimal helper; const-fn DER encoding of a fixed
  set of well-known OIDs needed for W2 ‚Äî see table below).
- Modified: `crates/gmcrypto-core/src/asn1/sig.rs` (re-implement
  `encode_sig` / `decode_sig` on top of the new reader/writer; preserve
  byte-identical output and identical accept/reject behaviour).
- Modified: `crates/gmcrypto-core/src/asn1/ciphertext.rs` (same ‚Äî port
  on top of reader/writer; preserve the field-bound `< p` check, the
  canonical-zero `02 01 00` accept, the strict canonical-INTEGER rules,
  and the 16 MB length cap with panic above).
- Modified: `crates/gmcrypto-core/src/asn1/mod.rs` (re-export the new
  modules; keep the existing `pub use sig::{decode_sig, encode_sig}`).

**Reuse.** The strict canonical-INTEGER discipline in
`asn1::sig::read_integer` (rejects empty content, sign-bit-set first
byte, redundant `00`-pad, lengths `> 32`) is the template for the new
`reader::read_integer`. The two SM2-specific deltas in
`asn1::ciphertext::read_integer` (accept `02 01 00`, reject `‚â• p`) become
parametrized variants of the same reader. The 1- / 2- / 3-byte length
encoder in `ciphertext.rs::push_length` is the model for `writer::write_length`.

**KAT vectors.** No new cryptographic KATs needed ‚Äî W1 is byte-format
plumbing. Regression tests:

- All v0.2 `asn1::sig::tests` (round-trip, malformed rejection, non-
  canonical leading-zero, negative-integer, empty-INTEGER) pass byte-
  identically post-port.
- All v0.2 `asn1::ciphertext::tests` (round-trip variants, malformed
  rejection, hash-length mismatch, non-canonical-x, negative-y,
  trailing-bytes, x-zero, x-‚â•-p, x-(p-1), 65,536-byte ciphertext) pass
  byte-identically post-port.
- New `asn1::reader::tests` and `asn1::writer::tests` cover each
  primitive shape (INTEGER edge cases, BIT STRING zero-unused-bits
  rule, OBJECT IDENTIFIER known-vector cross-checks against published
  hex from RFC 5912 / X.690 examples).

**OIDs needed at-rest** (encoded into `asn1::oid` as compile-time const
arrays of bytes for inclusion in SPKI / PKCS#8 wrappers):

| OID | Dotted form | Used by |
|---|---|---|
| `id-ecPublicKey` | 1.2.840.10045.2.1 | SPKI algorithm |
| `sm2p256v1` (GM/T 0006-2012) | 1.2.156.10197.1.301 | SPKI namedCurve param |
| `sm2-sign-with-sm3` | 1.2.156.10197.1.501 | future use; not v0.3 critical |
| `id-PBKDF2` | 1.2.840.113549.1.5.12 | PKCS#8 PBES2 KDF |
| `pbes2` | 1.2.840.113549.1.5.13 | encrypted PKCS#8 outer |
| `id-hmacWithSM3` | 1.2.156.10197.1.401.2 | PBKDF2 PRF (W2-implementation detail; gmssl 3.1.1 default is the operational anchor) |
| `sm4-cbc` | 1.2.156.10197.1.104.2 | PBES2 EncryptionScheme |

**Dudect implications.** None. W1 is wire-format code over public bytes
only ‚Äî no secret material flows through `reader.rs`/`writer.rs`. Existing
`asn1::ciphertext` decode runs on attacker-supplied bytes but the strict-
canonical rules + `Option`-returning shape are the side-channel posture;
W1 must preserve that shape exactly. **Don't add a dudect target for the
ASN.1 layer.**

**SemVer surface.** Additive only:

- `pub mod asn1::reader` + `pub mod asn1::writer` + `pub mod asn1::oid`
  appear in the public API.
- Existing `asn1::sig::{encode_sig, decode_sig}` and
  `asn1::ciphertext::{Sm2Ciphertext, encode, decode}` keep their exact
  signatures and byte behaviour. They become thin wrappers over the new
  modules.
- New `pub fn`s in `reader` / `writer`: specific function set is a
  W1-PR implementation detail; signatures must accept `&[u8]` slices
  and return `Option<(T, &[u8])>` to fit the existing zero-allocation,
  no-error-variant style.

**Risks / constant-time / failure-mode traps.**

- **Failure-mode invariant trap.** The new reader's primitive helpers
  must return `Option`, never a structured `Asn1Error` enum. A "more
  helpful" error type (distinguishing tag-mismatch from length-overflow
  from non-minimal-length) is exactly the surface CLAUDE.md rejects on
  sight. Single shape, every failure folds to `None`.
- **Length-encoder panic boundary.** `asn1::ciphertext::push_length`
  panics above 16 MB. The new `writer::write_length` must keep the
  same boundary (or document and consciously raise it) ‚Äî the boundary
  is a documented limit, not an arbitrary one.
- **Strict-canonical preservation.** Any deviation from the v0.2 canonical-
  INTEGER rules introduces signature/ciphertext malleability. The byte-
  identical-output regression tests (existing `asn1::sig::tests` and
  `asn1::ciphertext::tests`) are the gate: if W1 changes any output byte
  or any accept/reject decision, the PR fails CI.
- **No-std / alloc trap.** W1 is pure parsing + byte-vector emission;
  must not depend on `std::io::Read`/`Write`. All readers take `&[u8]`,
  all writers emit into `Vec<u8>` (or, optionally, into a caller-provided
  `&mut Vec<u8>` for composability).
- **Const-fn OID encoding.** The `asn1::oid` compile-time encoder must
  use the same `const fn`-with-while-loop pattern that `sm4::cipher::CK`
  uses ‚Äî no `unsafe`, no proc macros, no build script. Edition 2024 +
  MSRV 1.85 supports this.

### W2 ‚Äî PEM, encrypted PKCS#8, X.509 SPKI extractor

**Goal.** Convert SM2 keys (private and public) to and from the on-disk
formats gmssl and OpenSSL use, so v0.3 callers can hand a PEM-armored
PKCS#8 blob from gmssl into `gmcrypto-core` (and vice versa). This
unblocks W3 (full bidirectional gmssl interop including SM2 encrypt/
decrypt ‚Äî what was deferred from v0.2).

**Files touched.**

- New: `crates/gmcrypto-core/src/pem.rs` (PEM `-----BEGIN X-----` ...
  `-----END X-----` decoder + base64 codec; **hand-rolled** per Q7.1 to
  preserve no_std + zero-runtime-deps; ~150 LOC).
- New: `crates/gmcrypto-core/src/pkcs8.rs` (RFC 5958 OneAsymmetricKey
  reader/writer for SM2 keys; RFC 8018 PBES2 + PBKDF2-HMAC-SM3 + SM4-CBC
  encryption scheme for the encrypted PKCS#8 variant).
- New: `crates/gmcrypto-core/src/spki.rs` (RFC 5280 SubjectPublicKeyInfo
  for SM2 public keys: `id-ecPublicKey` + `sm2p256v1` namedCurve, BIT
  STRING wrapping uncompressed point `04 || X || Y`).
- New: `crates/gmcrypto-core/src/sec1.rs` (RFC 5915 ECPrivateKey inner
  structure; SEC1 uncompressed point encoding `04 || X || Y` and the
  optional compressed forms `02`/`03 || X` for round-trip
  compatibility).
- Modified: `crates/gmcrypto-core/src/sm2/public_key.rs` ‚Äî add
  `Sm2PublicKey::from_sec1_bytes(&[u8]) -> Option<Self>` and
  `Sm2PublicKey::to_sec1_uncompressed(&self) -> [u8; 65]` accessors.
  Reject the identity point and any off-curve `(x, y)` at the boundary,
  same as `verify_with_id`.
- Modified: `crates/gmcrypto-core/src/sm2/private_key.rs` ‚Äî add
  `Sm2PrivateKey::to_sec1_be(&self) -> [u8; 32]` (zeroizing intermediate)
  and `Sm2PrivateKey::from_sec1_be(&[u8; 32]) -> CtOption<Self>`.
- Modified: `crates/gmcrypto-core/src/lib.rs` ‚Äî add `pub mod pem`,
  `pub mod pkcs8`, `pub mod spki`, `pub mod sec1`.
- Modified: `crates/gmcrypto-core/Cargo.toml` ‚Äî `std` feature gates
  newtype helpers that read from `&Path` if we add them. The core
  PEM/PKCS#8 codec stays `no_std` + alloc; `std`-only file-loading
  helpers, if added, sit in a `pem::std` submodule per the risk-trap
  below.

**Reuse.**

- Strict canonical INTEGER + the `02 01 00` accept rule from v0.2's
  `asn1::ciphertext`: SEC1 ECPrivateKey embeds the scalar as INTEGER,
  and the field-bound check (range `[1, n-2]`) reuses
  `Sm2PrivateKey::new`'s constant-time range gate.
- `point_on_curve` from `sm2::encrypt`'s `pub(super)` helper. **Surface
  it through `pub(crate)` for `spki.rs` / `sec1.rs` use** (still not
  exposed as public API per CLAUDE.md).
- `pbkdf2_hmac_sm3` (existing public API) for the encrypted-PKCS#8
  PBES2 KDF step. SM4-CBC (existing public API) for the encryption.
  HMAC-SM3 (existing) for any MAC-bound key derivation.
- The const-fn OID encoder from W1's `asn1::oid` for embedding
  algorithm-identifier OIDs in SPKI / PKCS#8 wrappers.
- W1's `asn1::reader` / `writer` for all DER framing ‚Äî W2 cannot land
  before W1.

**KAT vectors.** Round-trip tests against PEM/DER blobs **emitted by
gmssl 3.1.1** at commit time. Vectors:

- An SM2 key generated by `gmssl sm2keygen -pass <password>
  -out priv.pem` ‚Üí PKCS#8-encrypted PEM. Capture the file as a
  **binary blob** at
  `crates/gmcrypto-core/tests/data/v0.3-sm2-pkcs8-encrypted.pem` (a new
  `tests/data/` directory; regen recipe in
  `crates/gmcrypto-core/tests/data/README.md`). Decrypt in
  `gmcrypto-core`, verify the recovered private key. Standards: RFC 8018
  PBES2 ¬ß6.2.1, RFC 5958 OneAsymmetricKey, RFC 5915 ECPrivateKey.
- The corresponding public key extracted via gmssl into
  `pub.pem` (SPKI). RFC 5280 ¬ß4.1.2.7. Check round-trip.
- An unencrypted PKCS#8 SM2 key (RFC 5958 PrivateKeyInfo without the
  EncryptedPrivateKeyInfo wrap) for completeness.
- Cross-validation goes both ways: emit a key from `gmcrypto-core`,
  verify gmssl can load it (`gmssl sm2sign -key our.pem ...`). Hooks
  into W3.

Specs / standards:

- RFC 7468 (PEM); RFC 5958 (PKCS#8 OneAsymmetricKey); RFC 5915
  (ECPrivateKey); RFC 5280 ¬ß4.1 (SPKI); RFC 8018 (PBES2 / PBKDF2);
  GB/T 35276-2017 / GM/T 0010-2012 (SM2 key transport, defines the
  Chinese-flavor OID conventions in use for `sm2p256v1`); RFC 5639
  (the ASN.1 OID registry source where namedCurve OIDs are anchored).

**Dudect implications.**

- **One new target: `ct_pkcs8_decrypt`** (Q7.7 decision ‚Äî definite
  addition). The encrypted-PKCS#8 path runs PBKDF2-HMAC-SM3 over a
  caller-supplied password, derives an SM4-CBC key, decrypts, and
  parses the inner ECPrivateKey. The PBKDF2 path is **already covered
  structurally** by `ct_hmac_sm3` (W3 / v0.2). The SM4-CBC decrypt +
  PKCS#7 strip is already covered by the constant-time PKCS#7 strip in
  `sm4::mode_cbc::decrypt`. The *new* surface is the inner ECPrivateKey
  parse ‚Äî but parsing is over derived plaintext bytes, which is
  attacker-influenceable only via the password. Class-split by
  password bytes; "load and recover scalar" timed under one window.
  Gate `|tau| < 0.20`.
  - Rationale: defense-in-depth on the new ECPrivateKey-parse-over-
    derived-plaintext surface; cost is one new bench function. The
    PBKDF2 + SM4-CBC sub-paths' structural coverage is acknowledged but
    the maintainer chose to ship the dedicated target anyway.
- No new target on the unencrypted PKCS#8 / SPKI / SEC1 paths. They
  operate on public bytes only (SPKI = public key; unencrypted
  PKCS#8 wraps the scalar but exposes no oracle to an attacker who
  could time the import ‚Äî the attacker by hypothesis has the bytes).

**SemVer surface.** Additive only:

- `pub mod pem`, `pub mod pkcs8`, `pub mod spki`, `pub mod sec1`.
- New `Sm2PublicKey::{from_sec1_bytes, to_sec1_uncompressed}`.
- New `Sm2PrivateKey::{from_sec1_be, to_sec1_be}`. `to_sec1_be` ships
  with `#[doc(hidden)]` and is **not SemVer-stable** (Q7.2 decision ‚Äî
  same posture as v0.2's `sign_raw_with_id`). The caller-must-zeroize
  contract is documented on the method. PKCS#8 emit is the intended
  consumer.
- Each new module exposes one or two top-level fns plus an error type
  with a single `Failed` variant (e.g. `pem::Error::Failed`,
  `pkcs8::Error::Failed`).

**Risks / constant-time / failure-mode traps.**

- **Failure-mode invariant on encrypted PKCS#8.** Distinguishing "wrong
  password" from "valid password but corrupt PEM" from "valid PEM but
  inner ECPrivateKey is malformed" is **forbidden** ‚Äî collapse all
  three into a single `pkcs8::Error::Failed`. This is harder than v0.2's
  surfaces because users want a "wrong password" hint; CLAUDE.md says
  no.
- **Padding-oracle in PBES2.** PBES2 wraps PKCS#7-padded ciphertext.
  The decrypt path must reuse `sm4::mode_cbc::decrypt`'s constant-time
  PKCS#7 strip exactly; do not write a parallel padding strip. After
  strip, the inner ASN.1 parse must fold any failure into the same
  `Failed` outcome ‚Äî no early-return on "padding looks bad" before the
  parse path runs. Same posture as v0.2's SM2 decrypt KDF-zero
  branchless rewrite.
- **Identity / off-curve trap on SPKI import.** `Sm2PublicKey::from_sec1_bytes`
  must reject the identity point and any (x, y) not on the curve. v0.2's
  `verify_with_id` already does this at the verify boundary; W2 surfaces
  the check at the **import** boundary so callers cannot construct a
  bad `Sm2PublicKey` and have it pass through to encrypt / verify and
  panic later.
- **Scalar range trap on SEC1 import.** `Sm2PrivateKey::from_sec1_be`
  must reuse `Sm2PrivateKey::new`'s `[1, n-2]` constant-time range
  check. Out-of-range scalars produce `CtOption::none()`.
- **PEM whitespace / line-length canonicalization.** RFC 7468 prescribes
  64-char lines. The decoder must accept the relaxed RFC 7468 production
  *and* the strict RFC 1421 production; emit the strict form. Liberal
  decoder, conservative encoder.
- **Encrypted-PKCS#8 KDF parameter validation.** The PBKDF2 `iterations`
  field is attacker-controllable in a malformed blob. Reject `iterations == 0`
  (matches `pbkdf2_hmac_sm3`'s contract) and reject DoS-grade upper
  bounds (e.g. `> 10_000_000`). Document the bound in the module-doc.
- **`std`-feature creep.** The PEM/PKCS#8/SPKI/SEC1 modules must remain
  no_std + alloc only. Any `std::fs`-based helpers (e.g. "load from
  path") sit behind `#[cfg(feature = "std")]` and live in a separate
  `pem::std` submodule ‚Äî never in the no_std core path. v0.2's
  `Cargo.toml` already reserves the `std` feature; v0.3 finally
  uses it.

### W3 ‚Äî Full bidirectional gmssl interop, including SM2 encrypt/decrypt

**Goal.** Close the v0.2 deferral. Cross-validate every public surface
against gmssl 3.1.1 in both directions where the gmssl CLI supports it.

**Files touched.**

- Modified: `crates/gmcrypto-core/tests/interop_gmssl.rs` (add
  bidirectional SM2 sign/verify, SM2 encrypt/decrypt, SM4-CBC if the
  gmssl CLI supports the same IV contract, and key-format round-trip
  tests using PEM blobs from W2).
- New: `crates/gmcrypto-core/tests/data/` (directory holding the
  fixture PEM/DER blobs generated by gmssl, **committed as binary
  files**; regen recipe in `tests/data/README.md` so an auditor can
  reproduce them. CI without gmssl still sees the committed fixtures.
  Cross-validation against live gmssl runs gated on `GMCRYPTO_GMSSL=1`
  as today.) Q7.9 decision.

**Reuse.**

- The existing `enabled()` / `gmssl_present()` / `hex` helpers in
  `interop_gmssl.rs`. Same env-var gate, same skip-silently behaviour.
- The existing `hmac_sm3_matches_gmssl` and `pbkdf2_hmac_sm3_matches_gmssl`
  patterns (subprocess invocation, hex round-trip).
- W2's PEM/PKCS#8/SPKI helpers ‚Äî W3 cannot land before W2.
- v0.2's `sm2::sign_with_id`, `sm2::verify_with_id`, `sm2::encrypt`,
  `sm2::decrypt`, `sm4::mode_cbc::{encrypt, decrypt}` ‚Äî all reused
  unchanged.

**KAT vectors / cross-validation matrix.**

| Direction | gmssl input | gmcrypto-core output | What it proves |
|---|---|---|---|
| **SM2 sign**: gmssl ‚Üí us | gmssl signs; we verify | n/a (verify-only) | `sm2::verify_with_id` accepts gmssl-emitted `(r, s)` |
| **SM2 sign**: us ‚Üí gmssl | n/a (sign-only) | we sign; gmssl verifies | `sm2::sign_with_id` produces gmssl-acceptable signatures with default ID `1234567812345678` |
| **SM2 encrypt**: gmssl ‚Üí us | gmssl encrypts to our pubkey | n/a (decrypt-only) | `sm2::decrypt` recovers plaintext from gmssl-emitted GM/T 0009 ciphertext |
| **SM2 encrypt**: us ‚Üí gmssl | n/a (encrypt-only) | we encrypt; gmssl decrypts | `sm2::encrypt` produces gmssl-acceptable ciphertext |
| **SM4-CBC**: gmssl ‚Üí us | gmssl encrypts | we decrypt | byte-level CBC compat (caller supplies same IV) |
| **SM4-CBC**: us ‚Üí gmssl | we encrypt | gmssl decrypts | same |
| **PEM/PKCS#8 round-trip** | gmssl-generated PEM | we parse + emit | wire-format compat for keys |

Specs/standards anchoring each: GB/T 32918.2 (SM2 sign), GB/T 32918.4
(SM2 encrypt), GM/T 0009-2012 (SM2 ciphertext DER), GB/T 32907 + RFC 5652
(SM4-CBC + PKCS#7), RFC 5958 (PKCS#8), RFC 5280 (SPKI).

**Dudect implications.** None directly ‚Äî these are interop tests, not
constant-time benches. Indirect: if a gmssl-emitted SM2 ciphertext
exposes a decrypt path our v0.2 dudect harness didn't cover, that's a
**finding to feed back** into the harness; v0.2's `ct_sm2_decrypt`
target was deliberately designed to fail at MAC for both classes via
identical control flow, and W3 should not regress that.

**SemVer surface.** None. W3 is tests-only.

**Risks / failure-mode traps.**

- **Padding-oracle on gmssl-emitted ciphertext.** If an attacker can
  feed us a CBC ciphertext that gmssl emitted, we already have a
  padding-oracle defense in `mode_cbc::decrypt`. The interop test does
  NOT add a new oracle.
- **Default signer ID mismatch.** gmssl uses `1234567812345678` by
  default per GM/T 0009. v0.2's `DEFAULT_SIGNER_ID` matches. Confirm at
  cross-validation time; if a gmssl version drifts, the test fails
  loud.
- **gmssl CLI version pinning.** CI runs against gmssl 3.1.1. If the
  GitHub Actions runner upgrades gmssl, the test may break. Pin the
  version in the workflow and document in `interop_gmssl.rs` module-
  doc, the same way v0.2 does.
- **Fixture stability.** The PEM blobs in `tests/data/` are
  deterministic-once-generated; commit them, do not regenerate per
  test run. (Live cross-validation against gmssl runs separately,
  gated on `GMCRYPTO_GMSSL=1`.)

### W4 ‚Äî Raw byte-concat ciphertext helpers (`C1||C3||C2` modern + `C1||C2||C3` legacy)

**Goal.** Interop fallback for callers (older gmssl, OpenSSL with the
GM patchset, third-party libraries) that don't speak GM/T 0009 DER.
Modern emit + decode (additive); legacy decrypt-only path.

**Files touched.**

- New: `crates/gmcrypto-core/src/sm2/raw_ciphertext.rs` (Q7.4 decision ‚Äî
  split from `asn1::ciphertext`; raw helpers are explicitly NOT DER) with:
  - `pub fn encode_c1c3c2(ct: &Sm2Ciphertext) -> Vec<u8>` (modern raw,
    matches gmssl's default for non-DER output paths since around 2018).
  - `pub fn decode_c1c3c2(input: &[u8]) -> Option<Sm2Ciphertext>` (modern
    raw decode).
  - `pub fn decode_c1c2c3_legacy(input: &[u8]) -> Option<Sm2Ciphertext>`
    (legacy gmssl pre-2018 byte order; **decrypt-only** ‚Äî we do not
    emit this format; emitting it would propagate the legacy ordering
    forever).
- Modified: `crates/gmcrypto-core/src/lib.rs` ‚Äî `pub mod sm2::raw_ciphertext`
  re-export so callers reach the helpers as
  `gmcrypto_core::sm2::raw_ciphertext::encode_c1c3c2` etc.
- `asn1::ciphertext` is **not** modified by W4 ‚Äî it stays the DER-only
  module.

**Reuse.** The `Sm2Ciphertext` struct from v0.2's `asn1::ciphertext`. The
SM2-curve field-bound check (`< p`) on `x`/`y`. The 32-byte fixed-length
`hash` field. **No new wire format primitives** ‚Äî just byte
concatenation + decode-side range checks.

**Wire format.**

```text
C1 (uncompressed point): 65 bytes = 0x04 || X (32) || Y (32)
                         (Q7.5 decision ‚Äî pinned 65-byte; matches
                         SEC1 / SPKI / gmssl 3.1.1 emit)
C2:                      |M| bytes (variable)
C3:                      32 bytes (SM3 digest)

modern (C1||C3||C2): [C1][C3][C2]   length = 65 + 32 + |M|
legacy (C1||C2||C3): [C1][C2][C3]   length = 65 + |M| + 32
```

The decoder must reject:

- inputs shorter than `|C1| + 32`;
- `C1` not on the curve (invalid-curve attack defense ‚Äî same rule as
  v0.2 `sm2::decrypt`);
- coordinates `‚â• p` (same field-bound rule as v0.2 `asn1::ciphertext`);
- the identity point as `C1`.

`C1` is **65 bytes** (`0x04 || X || Y`) per Q7.5 ‚Äî matches SEC1 /
SPKI / gmssl 3.1.1's `sm2encrypt -binary` output. Decode side MAY
tolerantly accept the 64-byte form behind a feature gate if a
real-world fixture from W3 cross-validation demands it; emit side
never produces 64-byte.

**KAT vectors.**

- gmssl-emitted modern raw ciphertext (gmssl 3.x command:
  `gmssl sm2encrypt -pubkey ... -in plaintext -out ciphertext.bin
  -binary`); decode with `decode_c1c3c2`; verify recovered plaintext
  matches.
- gmssl-emitted legacy raw ciphertext (gmssl pre-3.0 / OpenSSL GM-patch
  output where available); decode with `decode_c1c2c3_legacy`; verify
  recovery. **If we cannot produce a legacy fixture from gmssl 3.1.1,
  hand-construct one as a regression vector** and document in the
  module-doc that we cannot cross-validate against a live gmssl on this
  format; round-trip equivalence (re-encoding the modern form, swapping
  bytes to legacy order, decoding) is the next-best regression.
- Round-trip: `decode_c1c3c2(encode_c1c3c2(ct)) == ct` for the same
  fixture set v0.2's `asn1::ciphertext::tests` uses (short, 300-byte,
  empty, 65,536-byte, x-zero, x-(p-1), x-high-bit).

**Dudect implications.** None new. The decoder is wire-format only;
the actual `decrypt` call still goes through `sm2::decrypt` which is
already covered by `ct_sm2_decrypt`. **Important constant-time trap**:
the new decoders must not branch on length-mismatch in a way that
distinguishes "too short" from "wrong byte at position N" ‚Äî same
single-`None` posture as v0.2's DER decoder.

**SemVer surface.** Additive: 3 new public fns. No struct changes.

**Risks / failure-mode traps.**

- **Legacy decrypt-only is non-negotiable.** Don't add `encode_c1c2c3_legacy`.
  Per CLAUDE.md, the legacy format is a backwards-compat decode helper,
  not a new emit format. Adding emit propagates the legacy ordering.
- **Module placement pinned to `sm2::raw_ciphertext`** (Q7.4 decision).
  The raw helpers are explicitly *not* DER and live in their own SM2
  submodule; `asn1::ciphertext` stays DER-only.
- **Failure-mode trap on legacy path.** Callers who pass a modern blob
  to `decode_c1c2c3_legacy` will get nonsense back (or `None` if the
  embedded hash check fails downstream in `sm2::decrypt`). Document the
  caller-must-know-which-format requirement. Don't try to auto-detect ‚Äî
  auto-detect is a malleability/timing surface, the caller knows the
  source.
- **C1 is 65 bytes** (`0x04 || X || Y`) per Q7.5. Decode side MAY
  tolerate the 64-byte form behind a feature gate if a real-world
  fixture from W3 cross-validation demands it; emit side never
  produces 64-byte.

### W5 ‚Äî Streaming traits for `Mac` / `Hash` / `BlockCipher`

**Goal.** Bring v0.2's single-shot `hmac_sm3` and `Sm4Cipher::encrypt_block`
up to a streaming shape (an **in-crate** `Mac` / `Hash` / `BlockCipher`
trait surface ‚Äî RustCrypto trait fit is deferred to v0.4 per Q7.3 / Q7.10).
SM3 already has `Sm3::new`/`update`/`finalize` (v0.1); v0.3 adds the
analog for HMAC and SM4, plus an internal trait surface that both SM3 and
new HMAC implement uniformly.

**Files touched.**

- Modified: `crates/gmcrypto-core/src/hmac.rs` ‚Äî add streaming
  `pub struct HmacSm3 { ... }` with `new(&[u8]) -> Self`,
  `update(&mut self, &[u8])`, `finalize(self) -> [u8; 32]`, and
  `verify(self, expected: &[u8; 32]) -> bool` (constant-time compare via
  `subtle::ConstantTimeEq`). Keep `hmac_sm3` as a single-shot
  thin-wrapper. Per CLAUDE.md "Don't make `hmac_sm3` a streaming
  `HmacSm3::new`/`update`/`finalize` shape **in v0.2**" ‚Äî that
  prohibition is v0.2-scoped; v0.3 lifts it.
- Modified: `crates/gmcrypto-core/src/sm4/cipher.rs` and
  `crates/gmcrypto-core/src/sm4/mode_cbc.rs` ‚Äî add a small internal
  `BlockEncrypt` / `BlockDecrypt` trait (or use the existing
  `encrypt_block` / `decrypt_block` directly) and expose an
  `Sm4CbcEncryptor::new(key, iv) -> Self` / `update(&mut self, &[u8])` /
  `finalize(self) -> Vec<u8>` streaming surface that PKCS#7-pads the
  trailing partial block on `finalize`. Symmetric `Sm4CbcDecryptor`.
- New: `crates/gmcrypto-core/src/traits.rs` ‚Äî small in-crate trait
  module defining `Hash`, `Mac`, `BlockCipher` (all internally-public
  so SM3, HMAC-SM3, and SM4 implement them). Third-party trait
  re-export deferred to v0.4 per Q7.3; v0.3 ships in-crate traits only.
- Modified: `crates/gmcrypto-core/src/lib.rs` ‚Äî add `pub mod traits`.
- Modified: `crates/gmcrypto-core/src/sm3.rs` ‚Äî implement the new
  in-crate `Hash` trait on `Sm3`. The existing `Sm3::new`/`update`/
  `finalize` API is already streaming and stays unchanged byte-for-byte;
  the trait impl is purely additive.

**Reuse.**

- `Sm3::{new, update, finalize}` (v0.1 API) is the **exact template**
  for `HmacSm3::{new, update, finalize}`. The implementation
  composes two `Sm3` instances: outer keyed by `K' XOR opad`, inner
  keyed by `K' XOR ipad`. `update` forwards to the inner;
  `finalize` finalizes inner, feeds digest to outer, finalizes outer.
- `Sm4Cipher::{encrypt_block, decrypt_block}` from v0.2 is the
  building block. The streaming CBC encryptor maintains a `prev_block`
  state, processes full blocks as they arrive in `update`, buffers
  partial blocks, applies PKCS#7 on `finalize`. The streaming CBC
  decryptor is **trickier** because PKCS#7 strip can only happen after
  the last block is known ‚Äî the implementation must hold one block
  back from emission until `finalize` confirms it's the last. Same
  constant-time strip as v0.2's single-shot decrypt.
- HMAC zeroization discipline (K', ipad_key, opad_key) ‚Äî same as v0.2
  single-shot, extended to the streaming intermediate state.

**KAT vectors.**

- `HmacSm3` streaming-vs-one-shot equivalence: feeding any
  partition of `m` to `update`/`finalize` produces the same digest as
  `hmac_sm3(key, m)` for v0.2's 4 gmssl-cross-validated KAT vectors.
- SM4-CBC streaming-vs-one-shot equivalence: any partition of plaintext
  through the streaming encryptor produces byte-identical ciphertext as
  the v0.2 single-shot `mode_cbc::encrypt`. Same KAT inputs as v0.2.
- New gmssl cross-validation gated on `GMCRYPTO_GMSSL=1`: run
  `Sm4CbcEncryptor` over chunked input, compare to gmssl's full-input
  output.

**Dudect implications.**

- **Decision (Q7.6): skip `ct_hmac_sm3_streaming`; rely on
  `ct_hmac_sm3` structural coverage.** The streaming path reuses the
  same `Sm3` compress function, ipad/opad key derivation, and outer-
  hash composition; there's no new secret-touching path, just a
  different framing of the same `update` calls. Same precedent as
  v0.2's `ct_hmac_sm3` covering PBKDF2's inner PRF without a separate
  target. Rationale documented in the W5 implementation PR.
- `Sm4CbcEncryptor` / `Sm4CbcDecryptor`: the per-block path is identical
  to v0.2's `mode_cbc::{encrypt, decrypt}`. Already covered by
  `ct_sm4_encrypt_block` (W1 of v0.2). No new dudect target.
- The streaming PKCS#7 strip in `Sm4CbcDecryptor::finalize` must use
  the **same constant-time scan** v0.2 ships ‚Äî re-use, do not re-
  implement.

**SemVer surface.** Additive only:

- `pub struct HmacSm3` + impl block.
- `pub struct Sm4CbcEncryptor` + `pub struct Sm4CbcDecryptor` + impl
  blocks.
- `pub mod traits` with `Hash` / `Mac` / `BlockCipher` trait
  definitions.
- v0.2's `hmac_sm3`, `mode_cbc::encrypt`, `mode_cbc::decrypt`,
  `Sm4Cipher::*`, `Sm3::*` keep exact signatures and byte behaviour.

**Risks / constant-time / failure-mode traps.**

- **Streaming CBC decrypt + padding oracle.** `Sm4CbcDecryptor::update`
  must not emit anything that would let an attacker distinguish "this
  block is the last, padding ok" from "this block is the last, padding
  bad" before `finalize` is called. The straightforward design holds
  the *current* last-block emit back by one block, processing it on
  the next `update` call or stripping its padding constant-time on
  `finalize`. Document this contract on the type; the PKCS#7 strip
  must remain the v0.2 constant-time scan.
- **HMAC streaming finalize wipe.** `HmacSm3::finalize` consumes
  `self`, but the inner-hash state and the outer-hash state both
  contain key-XOR'd material. Both must zeroize on finalize. The drop
  glue alone is insufficient because `finalize` consumes the type.
  Use `Zeroize::zeroize` explicitly on the consumed self, or wrap in
  `ZeroizeOnDrop` and let drop run as the consumed value falls out of
  scope.
- **Trait surface scope creep.** Don't ship the in-crate `Hash`/`Mac`/
  `BlockCipher` traits as bound-on-public-API constraints ‚Äî they're
  for ergonomic wiring of internal sites and to give downstream callers
  generic-over-our-types options. Adding `where T: Hash` constraints
  on public functions would be a breaking change in v1.0 if we ever
  remove them. Keep traits in `pub mod traits` but unbound on public
  fns.
- **RustCrypto trait re-export ‚Äî deferred (Q7.3 decision).** Wiring
  RustCrypto's `digest::Digest` and `digest::Mac` impls onto `Sm3` /
  `HmacSm3` is **desirable for ecosystem fit** but adds a new
  dependency. Deferred to v0.4 behind an opt-in feature flag, alongside
  the C ABI / wasm work. Preserves no_std + zero-runtime-deps in v0.3.

### W6 ‚Äî Comb-table `mul_g` optimization

**Goal.** v0.1/v0.2 ships `mul_g(k) = mul_var(k, &generator())` ‚Äî
constant-time but ~10-20√ó slower than necessary on the fixed-base case.
v0.3 adds a precomputed comb table that speeds up signing by an order
of magnitude while keeping the linear-scan constant-time-designed
selection.

**Files touched.**

- Modified: `crates/gmcrypto-core/src/sm2/scalar_mul.rs` ‚Äî replace
  `mul_g`'s body with a comb-table walk over a precomputed
  `[ProjectivePoint; 2^w]` table (typical `w = 4` window-width ‚Üí
  16-entry table; potentially split into `n_subtables = 64`
  sub-windows for the standard signed-comb structure). Public
  signature `pub fn mul_g(k: &Fn) -> ProjectivePoint` is **unchanged**.
- New: `crates/gmcrypto-core/src/sm2/comb_table.rs` ‚Äî generation logic.
  Q7.8 decision: **runtime init at first `mul_g` call** (no const-eval;
  `crypto-bigint 0.7.3` arithmetic is not const-eval-friendly).
  Specific `no_std`-compatible primitive (e.g. `once_cell::race::OnceBox`,
  `spin::Once`, or a hand-rolled atomic-guarded eager init) is a
  W6-PR implementation detail. **`std::sync::LazyLock` is forbidden**
  ‚Äî it violates the `no_std` constraint in `CLAUDE.md`. New runtime
  deps require explicit maintainer approval per Q7.1's zero-runtime-
  deps stance. Cross-validated against `mul_var(k, &generator())` for
  a battery of scalars.

**Reuse.**

- The fixed-window pattern from v0.1 `mul_var` (4-bit nibbles, linear-
  scan selection via `subtle::ConditionallySelectable`).
- The `subtle::ConstantTimeEq` + `ConditionallySelectable::conditional_select`
  idiom ‚Äî every comb-table read MUST go through this, **never** a
  direct `table[idx]` index. The whole point of the constant-time
  posture is unaffected by the precomputation; only the *what* and
  *how often* of the lookups change.

**KAT vectors.**

- Equivalence test: for a battery of 100+ scalars (small, large,
  pathological ‚Äî `n-1`, `1`, `n/2`, fresh-random),
  `mul_g(k) == mul_var(k, &ProjectivePoint::generator())`.
- All v0.1/v0.2 sign tests pass byte-identically (sign output depends
  on `mul_g` output for `kG`).
- GB/T 32918.2 Appendix A.2 fixed-`k` sign vector continues to produce
  the documented `r=0x88348A09‚Ä¶` and `s=0x0AD2CE55‚Ä¶`.

**Dudect implications.** **Critical** ‚Äî `ct_mul_g`'s
`|tau| < 0.20` gate is the contract this workstream must preserve.
v0.1/v0.2's `ct_mul_g` measures within noise because `mul_g` delegates
to `mul_var`. v0.3's standalone comb-table implementation must measure
the same. **No new target needed** ‚Äî `ct_mul_g` exists; it just stops
being a structural-equivalence-of-mul_var measurement and becomes a
direct measurement of the new path. Class-split by `k` magnitude.

**SemVer surface.** None. `mul_g`'s public signature is unchanged. The
new `sm2::comb_table` module is `pub(crate)` or `pub(super)` only ‚Äî
implementation detail.

**Risks / constant-time / failure-mode traps.**

- **Comb-table linear scan must cover the full table on every nibble.**
  The constant-time invariant is that each scalar bit (or window of
  bits) selects a table entry without a data-dependent branch. The
  textbook comb-table optimization uses a *signed* representation
  (entries `¬±1¬∑[2^j]¬∑G`) to halve table size; the sign flip must be a
  constant-time `ConditionallySelectable` negation, not a branch.
- **Sign-flip via `subtle`.** The signed-comb negation step on a
  `ProjectivePoint` is `(X, Y, Z) ‚Üí (X, -Y mod p, Z)`. Implement via
  `Fp::negate` (pure arithmetic, constant-time) wrapped in
  `ConditionallySelectable::conditional_select` against the original
  point. Do **not** branch on the sign bit.
- **Precomputed-table genesis.** The 16-entry (or larger) precomputed
  table holds public points (`G`, `2G`, `3G`, ...). Generation is
  **public-input** ‚Äî no constant-time required during table build;
  only *use* of the table at sign-time is constant-time. Q7.8 pinned
  the build mechanism to runtime init at first `mul_g` call (specific
  `no_std`-compatible primitive selected during W6 implementation; see
  Files-touched bullet above for the constraint set).
- **`ct_mul_g` regression.** If the new implementation measures
  `|tau| > 0.10` (close to the 0.20 gate), the workstream fails on
  Branch B. Validate at PR time on the W6 branch with 100K samples
  before merging.
- **Table size vs. binary bloat.** A 16-entry (256-bit-coordinate)
  table is ~1 KB; reasonable. A 256-entry table is ~16 KB and starts
  affecting `no_std` embedded targets. Pin window width at `w = 4` or
  `w = 5` and document the trade.

## 3. Sequencing

W1 ‚Üí W2 ‚Üí W3 ‚Üí W4. W5 and W6 are independent of the W1‚ÄìW4 chain and
can run in parallel.

**Dependency graph.**

```text
W1 (ASN.1 reader/writer)
  ‚Üì
W2 (PEM / PKCS#8 / SPKI / SEC1)
  ‚Üì
W3 (gmssl bidirectional interop, depends on PEM keys for SM2 encrypt/decrypt)
  ‚Üì
W4 (raw ciphertext helpers ‚Äî depends on stable Sm2Ciphertext shape after W1's port)

W5 (streaming traits)             ‚Äî independent of W1‚ÄìW4
W6 (comb-table mul_g)             ‚Äî independent of W1‚ÄìW4
```

**Phase plan.**

- **Phase 1 (W1).** ASN.1 reader/writer + OID. One PR, ~1 reviewable
  chunk. Codex review pass after. Gate: byte-identical regression
  on all v0.2 `asn1::sig::tests` and `asn1::ciphertext::tests`.
- **Phase 2 (W2).** PEM + SPKI + SEC1 (read + write) in PR 2a. Encrypted
  PKCS#8 (PBES2 + PBKDF2-HMAC-SM3 + SM4-CBC) in PR 2b. Codex review
  pass on each. Two PRs because PBES2's KDF + cipher composition is
  large enough to deserve its own review window.
- **Phase 3 (W3).** Test-only PR. Adds gmssl bidirectional fixtures and
  cross-validation. Gates on `GMCRYPTO_GMSSL=1` like v0.2's
  interop tests; CI matrix slot enables it.
- **Phase 4 (W4).** Raw ciphertext helpers. One PR.
- **Phase 5 (W5).** Streaming traits. Single PR for `HmacSm3` +
  streaming SM4-CBC + the in-crate trait module. Codex review pass.
- **Phase 6 (W6).** Comb-table `mul_g`. Standalone PR. Gate: 100K
  `ct_mul_g` measurement + all SM2 sign KATs unchanged.

**Concurrency.** W5 and W6 can land at any time during the W1‚ÄìW4 chain.
The maintainer's preference (chunked PRs, codex-review per chunk) means
no parallel merging unless review bandwidth allows.

**Why this order.**

- W1 is the foundation. The new readers/writers underlie W2; W2
  cannot land first without re-implementing parsing primitives that
  W1 then displaces.
- W2 is the foundation for W3's SM2 encrypt/decrypt cross-validation
  (gmssl's `sm2encrypt`/`sm2decrypt` CLI requires a PEM-wrapped key).
  W3 cannot land first.
- W4 needs the post-W1 `Sm2Ciphertext` shape; landing W4 before W1
  ports `asn1::ciphertext` would require porting twice.
- W5 has no dependencies on the wire-format chain; it can land first
  if review bandwidth allows. The CLAUDE.md prohibition on streaming
  HMAC ("not in v0.2") explicitly contemplates v0.3 lift.
- W6 has no dependencies. Lands whenever; `ct_mul_g` is the gate.

## 4. Out of scope for v0.3

- **C ABI / `gmcrypto-c` crate.** v0.4 (per README roadmap).
- **`wasm32-unknown-unknown` build target.** v0.4.
- **Bitsliced SM4 S-box.** v0.4. v0.3 keeps the v0.2 linear-scan
  S-box throughput trade.
- **API stabilization.** v1.0. v0.3 may add public surface; it does
  not lock down what's there.
- **TLS / TLCP / GM-TLS profiles.** Not on any roadmap row.
- **Certificate chain validation, CRL, OCSP, CSR, CMS.** Not on
  roadmap. SPKI extraction in W2 stops at "load a single public key
  from a SPKI blob"; chain walking is out of scope.
- **HKDF, scrypt, Argon2.** v0.3 ships PBKDF2-HMAC-SM3 only (v0.2
  shipped this primitive); other KDFs are post-v0.3.
- **HSM / SDF / SKF integration.** Not on roadmap.
- **SM9 / ZUC / post-quantum.** Not on roadmap.
- **`std::io::Read`/`Write`-driven streaming.** v0.3's W5 streaming
  traits work on `&[u8]` chunks; integration with `std::io` traits is
  a separate concern (would require the `std` feature) and is
  deferred.
- **RustCrypto `digest::Digest` / `digest::Mac` / `cipher::BlockMode`
  trait impls.** Deferred to v0.4 alongside the C ABI / wasm work,
  behind opt-in feature flags. (Q7.3 / Q7.10.)
- **`ct_hmac_sm3_streaming` separate dudect target.** Streaming
  HMAC is structurally covered by `ct_hmac_sm3` (same rationale v0.2
  used for PBKDF2-via-`ct_hmac_sm3`). (Q7.6.)
- **Compressed SEC1 point encoding (`02`/`03 || X`).** v0.3 supports
  *decoding* compressed forms for round-trip with gmssl-emitted blobs;
  emit-side ships uncompressed (`04 || X || Y`) only. The compressed-
  emit decision waits on user demand.
- **A `ct_mul_g_comb` separate dudect target.** v0.3's W6 reuses the
  existing `ct_mul_g` target ‚Äî no new gate.
- **A separate `ct_sm2_encrypt` dudect target.** v0.2 deferred this on
  the rationale that encrypt operates on a public key + freshly-sampled
  `k`. v0.3 W3 cross-validates encrypt against gmssl but does not add
  the dudect target ‚Äî same rationale.

What readers might *expect* in v0.3 but won't get:

- A "v0.2.1 fix-only" release line. v0.2 is treated as stable; bug
  fixes go to `main` and ride into v0.3.
- Streaming `Sm4Ctr` mode (CTR is unauthenticated like CBC; the
  encrypt-then-MAC story already covers the integrity-needing case
  via SM4-CBC + HMAC-SM3). Not in scope.
- AEAD modes (SM4-GCM / SM4-CCM). The Chinese standards include
  these (GB/T 32907-2016 references modes); v0.3 doesn't ship them.
  Future-version candidate.

  - `sm2_spki_round_trips_with_gmssl` (read SPKI from gmssl; emit one
    gmssl reads).
- `tests/data/` (new) **committed as binary files** under
  `crates/gmcrypto-core/tests/data/` (regen recipe in
  `tests/data/README.md`) so cross-validation has byte-stable
  regression vectors. (Q7.9.)

### Dudect target additions

- `ct_pkcs8_decrypt` (W2; class-split by password bytes; PBES2 +
  inner-key parse; gate `|tau| < 0.20`). Q7.7 decision ‚Äî definite
  addition.

Adds 1 new target (`ct_pkcs8_decrypt`); total dudect targets reach
**12** (11 v0.2 + 1).

`dudect-pr.yml` and `dudect-nightly.yml` `required_low` allowlists
extended for the new target. Same `|tau| < 0.20` gate.

### Failure-mode invariants for new error types

- `pem::Error::Failed` ‚Äî single variant.
- `pkcs8::Error::Failed` ‚Äî single variant.
- `spki::Error::Failed` ‚Äî single variant.
- `sec1::Error::Failed` ‚Äî single variant.
- All new `decode_*` fns return `Option<T>`, never `Result<T, E>`.

### Zeroize coverage additions

- PEM-decode intermediate `Vec<u8>` holding the inner DER (zeroize
  before drop on the encrypted-PKCS#8 path).
- PBES2-decryption intermediate plaintext (post-CBC-decrypt, pre-
  ASN.1-parse) zeroized in-place after the inner ECPrivateKey is
  recovered.
- `HmacSm3` streaming state (inner Sm3, `K' XOR opad` buffer)
  zeroized on `finalize` and on drop.
- `Sm2PrivateKey::to_sec1_be`'s output buffer is the caller's
  responsibility to zeroize; document this.
- Comb-table state on `mul_g` is purely public (the precomputed
  table is `G`, `2G`, ..., `15G`); no zeroization needed at
  build, only on `mul_g`-internal scratch (running accumulator).

## 7. Decisions (rev-1 sign-off, 2026-05-10)

Each question below is now answered; the rest of the doc has been updated
to reflect these decisions. Preserved here for posterity ‚Äî implementation
plans reference these decisions, not the original questions. Pro/con
bullets remain as written so future readers can see the trade-offs that
were considered.

### 7.1 PEM codec ‚Äî `pem` crate or hand-rolled?

The `pem` crate (currently 3.x on crates.io) is small (~few hundred
lines), well-maintained, has no transitive deps, and supports no_std +
alloc.

- **Pro:** zero implementation work; battle-tested base64.
- **Con:** runtime dep ‚Äî moves us from the v0.2 "zero-runtime-deps"
  stance to "one runtime dep." `cargo deny --exclude-dev` will
  newly check it. The crate is permissively licensed (MIT/Apache-2.0).
- **Alternative:** hand-rolled base64 + line-wrap codec inside
  `crates/gmcrypto-core/src/pem.rs`. ~150 LOC. No new dep.

**Question:** zero-runtime-deps preference, or accept `pem` as
the first runtime dep?

**Decision:** hand-roll. Preserves the v0.2 zero-runtime-deps stance
and the codec is simple enough to audit fully.

### 7.2 SEC1 private-key emit ‚Äî ship `to_sec1_be`?

`Sm2PrivateKey::to_sec1_be(&self) -> [u8; 32]` produces secret bytes.
The caller must zeroize. Shipping the inverse direction
(`from_sec1_be`) is uncontroversial ‚Äî callers who already have the
bytes need a way to wrap them. Shipping `to_sec1_be` enables PKCS#8
emit (W2's encrypt-private-key path), but it also exposes a way for
careless callers to leak the scalar.

**Question:** ship `to_sec1_be` for W2's PKCS#8 emit path, or keep
PKCS#8 emit as the only public path that ever emits the scalar
(forcing callers through the wrapped form)?

**Decision:** ship `to_sec1_be` but mark `#[doc(hidden)]` and "not
SemVer-stable", same posture as v0.2's `sign_raw_with_id`. The
caller-must-zeroize contract is documented on the method.

### 7.3 RustCrypto trait re-export ‚Äî `digest`/`mac` features?

`HmacSm3` could implement `digest::Mac` and `Sm3` could implement
`digest::Digest` from RustCrypto's `digest` crate. This would make
`gmcrypto-core` usable in any pipeline expecting RustCrypto's traits.

- **Pro:** drop-in fit with RustCrypto's `pbkdf2`, `hkdf`, `scrypt`,
  authenticated-encryption combinators, etc.
- **Con:** new dep (`digest` brings `block-buffer`, `crypto-common`,
  ...). Even behind a feature flag, MSRV could shift; the trait
  versions evolve and could force a v0.3.x bump.
- **Alternative:** ship in-crate traits in `pub mod traits` only;
  defer RustCrypto trait fit to v0.4 alongside the C ABI / wasm
  work, behind opt-in features.

**Question:** RustCrypto trait fit in v0.3, or in-crate traits only
(deferring third-party traits to v0.4)?

**Decision:** in-crate only for v0.3; defer RustCrypto trait fit to
v0.4 behind opt-in feature flags. Preserves no_std + zero-runtime-deps;
the v0.3 README row promises "streaming `Mac`/`Hash`/`BlockCipher`
traits" without committing to RustCrypto's specific traits.

### 7.4 W4 module placement ‚Äî `asn1::ciphertext` or `sm2::raw_ciphertext`?

The raw byte-concat helpers (`C1||C3||C2`, `C1||C2||C3`) are explicitly
NOT ASN.1. Putting them in `asn1::ciphertext` is misleading; putting
them in a new `sm2::raw_ciphertext` module is more honest but adds a
new top-level module under `sm2`.

**Question:** keep `asn1::ciphertext` as the catch-all SM2-ciphertext
module (DER + raw helpers under one roof, despite the misleading
name), or split into `asn1::ciphertext` (DER) + `sm2::raw_ciphertext`
(byte-concat)?

**Decision:** split. `sm2::raw_ciphertext` is the clearer name ‚Äî
the raw helpers are explicitly NOT DER, so co-locating them with
`asn1::ciphertext` would mislead readers. v0.2's CLAUDE.md "Don't
ship raw `C1||C3||C2`" rule already contemplated raw helpers as a
separable concern.

### 7.5 W4 C1 byte-length ‚Äî 64 bytes or 65 (with `0x04` prefix)?

The "uncompressed point" wire convention used in OpenSSL/SEC1 is
`0x04 || X || Y` (65 bytes); some Chinese-standard implementations
emit `X || Y` (64 bytes) instead, dropping the `0x04` lead. gmssl's
binary output for SM2 raw ciphertext (per the 3.x `sm2encrypt -binary`
path) reportedly uses the 65-byte form.

**Question:** which C1 byte length do `encode_c1c3c2` / `decode_c1c3c2`
use as the canonical form? Pin one; document that the other is not
auto-detected.

**Decision:** 65-byte (`0x04 || X || Y`) ‚Äî matches SEC1, W2's SPKI
BIT STRING convention, and gmssl 3.1.1's `sm2encrypt -binary` emit.
Decode side MAY tolerantly accept the 64-byte form behind a feature
gate if a real-world fixture from W3 cross-validation demands it; emit
side never produces 64-byte.

### 7.6 New dudect target for streaming HMAC?

The streaming `HmacSm3` reuses the same `Sm3` compress + ipad/opad
key XOR as v0.2's single-shot `hmac_sm3`. v0.2's `ct_hmac_sm3` covers
the structural surface; arguably `ct_hmac_sm3_streaming` is redundant
because the only new code is iterator framing.

**Question:** ship `ct_hmac_sm3_streaming`, or rely on `ct_hmac_sm3`
structural coverage and document why?

**Decision:** **skip the new target.** Rely on `ct_hmac_sm3`
structural coverage; document in the W5 implementation PR, same
precedent as v0.2's `ct_hmac_sm3` covering PBKDF2's inner PRF without
a separate `ct_pbkdf2_hmac_sm3` target.

### 7.7 New dudect target for encrypted-PKCS#8 decrypt?

PBES2 = PBKDF2 (covered by `ct_hmac_sm3`) + SM4-CBC (covered by
`ct_sm4_encrypt_block`). The new code path is the inner ECPrivateKey
parse. Class-split by password bytes is achievable but the post-
decrypt-but-pre-parse plaintext is *the password-derived plaintext*,
not a fresh secret.

**Question:** ship `ct_pkcs8_decrypt`, or rely on the PBKDF2 +
SM4-CBC structural coverage?

**Decision:** **ship `ct_pkcs8_decrypt`.** Defense-in-depth on the
new ECPrivateKey-parse-over-derived-plaintext surface; cost is one new
bench function. The class split is by password; both classes succeed
at PBES2 unwrap and parse, so the work-done is identical. Differs from
`ct_sm2_decrypt`'s "both classes fail at MAC" posture but the
principle is the same. Total dudect targets reach 12.

### 7.8 Comb-table build mechanism ‚Äî `const fn` or `OnceCell`?

A 16-entry `[ProjectivePoint; 16]` precomputed table at compile-time
is plausible via `const fn` if `ProjectivePoint::add` and
`ProjectivePoint::generator` are `const fn`. They aren't today
(`ConstMontyForm` arithmetic isn't const-eval-friendly in
crypto-bigint 0.7.3).

**Question:** force `const`-eval (which means the table is in
`.rodata`, no init cost) by working around the const-fn limitation
with a static-bytes-to-points init, or use `OnceCell` / `LazyLock`
(which means runtime init on first call, ~100 ¬µs)?

**Decision:** **runtime init at first `mul_g` call.** Init cost
(~100 ¬µs) is amortized across the process lifetime; const-eval of
`crypto-bigint` arithmetic is expected to land sooner via the
upstream roadmap than via our workarounds. **`std::sync::LazyLock` is
forbidden** ‚Äî it violates the no_std constraint in `CLAUDE.md`. The
specific no_std-compatible primitive (e.g. `once_cell::race::OnceBox`,
`spin::Once`, or a hand-rolled atomic-guarded eager init) is a W6-PR
implementation detail, subject to the Q7.1 zero-runtime-deps stance:
new runtime deps require explicit maintainer approval.

### 7.9 Tests-data fixture provenance ‚Äî committed, regenerable, or both?

W3's `tests/data/` directory holds gmssl-emitted PEM/DER blobs. They
need to be regenerable (an auditor running `gmssl sm2keygen ...`
must be able to reproduce them) but also stable across runs (CI
without gmssl must still see the fixtures).

**Question:** commit the fixtures as hex-encoded `&[u8]` literals in
the test source, or commit them as binary files under `tests/data/`?

**Decision:** commit binary files under `tests/data/`; document the
regeneration recipe in `tests/data/README.md`. Hex-encoded literals
are unreadable for the encrypted-PKCS#8 case (~600 bytes / blob).

### 7.10 v0.3 scope of W5 ‚Äî also bring `mode_cbc` to `cipher::BlockMode`?

v0.5 mentions streaming `BlockCipher` traits. The natural Rust
ecosystem trait for CBC is `cipher::BlockMode` (from RustCrypto's
`cipher` crate). Question 7.3 already covers RustCrypto trait fit in
general; this sub-question asks specifically whether the SM4-CBC
streaming surface gets the trait fit even if the rest doesn't.

**Question:** out of scope per Q7.3, or split ‚Äî in-crate trait fit
for HmacSm3 / SM3, RustCrypto trait fit for SM4-CBC?

**Decision:** uniform ‚Äî in-crate only for v0.3 (consistent with Q7.3).
SM4-CBC streaming uses our own `BlockCipher` trait; RustCrypto
`cipher::BlockMode` fit deferred to v0.4 alongside the rest.
