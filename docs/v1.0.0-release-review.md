# v1.0.0 release review checklist (three-crate workspace)

**Crates:** `gmcrypto-simd` 1.0.0 → `gmcrypto-core` 1.0.0 → `gmcrypto-c` 1.0.0
(published in that order; a crate must be live on crates.io before its dependents).
**Repository:** <https://github.com/frankxue831/gm-crypto-rs>
**Release candidate:** record the exact commit you intend to tag with
`git rev-parse HEAD` (the merge commit of the `feat/v1.0-release` PR on `main`).
Review a frozen commit, not "latest main".
**Purpose:** decide whether that exact commit is safe to publish — the **first
stable, irreversible** publish of all three crates. crates.io history goes
**0.16.0 → 1.0.0** (0.17.0–0.23.0 were non-publishing assurance cycles).

Publishing to crates.io is permanent. A yanked version is hidden from normal
resolution, but the `.crate` archive remains downloadable. From 1.0, SemVer is a
binding contract: a breaking change to the covered surface requires 2.0.

Use `BLOCK` (do not publish until fixed/deferred), `WARN` (publishable, track for
1.0.x/1.1), `INFO` (note only).

## 2. Exact sibling pins (the 1.0 lockstep contract)

The three crates must only resolve as a matched `=1.0.0` set (`docs/v1.0-readiness.md`
§3.D). Verify the path-dep version reqs are **exact**:

```bash
grep -n 'gmcrypto-simd' crates/gmcrypto-core/Cargo.toml   # version = "=1.0.0"
grep -n 'gmcrypto-core' crates/gmcrypto-c/Cargo.toml      # version = "=1.0.0"
```

Expected: both read `version = "=1.0.0"` (not `"1.0"`/`"^1.0"`). A non-exact req
would let a dependent resolve against a mismatched sibling 1.x.

## 4. MSRV reproducibility

```bash
rm -f Cargo.lock
cargo +1.85 build -p gmcrypto-core
cargo +1.85 build -p gmcrypto-core --features digest-traits,cipher-traits,sm4-bitsliced,sm4-bitsliced-simd,sm4-aead,sm4-xts,crypto-bigint-scalar

```

Expected: all exit 0. Red flag: any dependency whose own `rust-version` exceeds
1.85 breaks the MSRV claim.

## 5. API/ABI freeze — drift checks (the 1.0 contract surface)

```bash
# Rust public API (pinned tool + nightly; must be drift-free)
cargo +nightly-2026-05-23 public-api -p gmcrypto-core --all-features \
  --omit blanket-impls --omit auto-trait-impls --omit auto-derived-impls \
  | diff - docs/api-baseline/gmcrypto-core.txt
cargo +nightly-2026-05-23 public-api -p gmcrypto-simd \
  --omit blanket-impls --omit auto-trait-impls --omit auto-derived-impls \
  | diff - docs/api-baseline/gmcrypto-simd.txt
# C ABI header
cargo build -p gmcrypto-c --features regen-header
git diff --exit-code crates/gmcrypto-c/include/gmcrypto.h
```

Expected: both `diff`s empty; header drift-check exits 0. Confirm the
`gmcrypto-core` baseline names **no** `crypto-bigint` type except the
`crypto-bigint-scalar`-gated `from_scalar(U256)` residual.

## 6. Runtime correctness — byte-identical to 0.16.0

```bash
cargo test -p gmcrypto-core           # full KAT + unit
GMCRYPTO_GMSSL=1 cargo test --test interop_gmssl   # 11/11
cargo test -p gmcrypto-c              # 65 c_smoke FFI-equivalence
```

Expected: all pass; gmssl interop **11/11**. The SM2 DER signature + ciphertext
wire bytes, SM4 mode outputs, and PKCS#8/SPKI fixtures are unchanged from 0.16.0
(the 1.0 break is API *shape* only).

## 7. Constant-time harness

```bash
DUDECT_SAMPLES=10000 cargo bench --bench timing_leaks --features crypto-bigint-scalar
```

Expected: `negative_control` `|tau| > 1.0`; the composite `ct_*` targets
`|tau| < 0.20`; `ct_fn_invert`/`ct_fp_invert` are telemetry-only at the PR budget
(nightly sentinel @0.55) per `SECURITY.md` / `docs/v0.5-dudect-recalibration.md`.

## 8. Publish dry-runs + package contents (in publish order)

```bash
# 1) gmcrypto-simd — full verify (no sibling dep)
cargo publish -p gmcrypto-simd --dry-run
# 2) gmcrypto-core — package only (sibling not yet on registry at review time)
cargo publish -p gmcrypto-core --dry-run --no-verify
cargo package -p gmcrypto-core --list | sort
# 3) gmcrypto-c — package only
cargo publish -p gmcrypto-c --dry-run --no-verify
cargo package -p gmcrypto-c --list | sort
```

Expected:
- each packaging line names the crate at `v1.0.0`;
- packages contain `src/`, `Cargo.toml`, `Cargo.toml.orig`, `Cargo.lock`,
  `README.md`, `.cargo_vcs_info.json` (+ `include/`, `examples/` for `gmcrypto-c`;
  + `benches/`, `tests/` for `gmcrypto-core`);
- no `.git/`, `target/`, IDE folders, the `fuzz/` crate, or local secrets;
- `gmcrypto-c`'s package does **not** vendor `gmcrypto-core` source (path dep is
  rewritten to the `=1.0.0` registry dep on publish).

Note: `core`/`c` use `--no-verify` at *review* time only because their `=1.0.0`
sibling isn't on crates.io yet. At **real** publish time each is published in
order and the next crate's full verify resolves the now-live sibling.

## 10. The publish sequence (only after GO)

```bash
cargo publish -p gmcrypto-simd
# wait for the index to update, then:
cargo publish -p gmcrypto-core
cargo publish -p gmcrypto-c
git tag -s v1.0.0 -m "gm-crypto-rs v1.0.0"   # SSH-signed (gpg.format=ssh)
git push origin v1.0.0
gh release create v1.0.0 --title "v1.0.0" --notes-from-tag   # or curated notes
```

Then **flip `cargo-semver-checks` to enforced** in `.github/workflows/api-stability.yml`
(drop `continue-on-error`; baseline = the published `1.0.0`) — the forward
breaking-change gate (`docs/v1.0-readiness.md` §4 step 7). This is a follow-up PR.
