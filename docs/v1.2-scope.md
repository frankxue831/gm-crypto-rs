# v1.2 scope — C FFI for SM2 key exchange (GM/T 0003.3)

Status: scope locked 2026-06-11 (maintainer sign-off on Q2.1–Q2.3; the rest
follows settled precedent). Completes the core-in-vN / FFI-in-vN+1 cadence for
the v1.1 `sm2::key_exchange` core (the v0.12→v0.13 and v0.15→v0.16 pattern).

## 1. Sign-off decisions

- **Q2.1 Scope** — **FFI-only**. X.509-with-SM2 (the other v1.2 candidate
  named in `docs/v1.1-sm2-key-exchange-design.md` §9) moves to the **v1.3
  direction candidate** with its own brainstorm/scope cycle; its
  path-validation scope creep is exactly why it was deferred from v1.1 and
  bundling it into a thin-shim cycle would repeat the mistake.
- **Q2.2 Handle shape** — **two opaque handles, ephemeral folded into the
  initiator constructor**. C cannot express the Rust 4-type consume-on-
  transition typestate, so:
  - `gmcrypto_sm2_kx_initiator_t` is born *waiting*: `_new` samples the
    ephemeral internally and writes `R_A` (SEC1 65 B) to an out-param;
    `_confirm` consumes + frees (the v0.10 `_finalize*` precedent), writing
    `K` (klen B) + `S_A` (32 B).
  - `gmcrypto_sm2_kx_responder_t`: `_new` → `_respond` (takes `R_A`, writes
    `R_B` + `S_B`, transitions internally; a second `_respond` →
    `GMCRYPTO_ERR`) → `_finish` (takes `S_A`, writes `K`, consumes + frees).
  - `_free` on both for abandonment. Misuse-ordering collapses to the single
    `GMCRYPTO_ERR`; single-use is enforced structurally by consume-on-final.
- **Q2.3 RNG** — default constructors use `getrandom::SysRng` (the v0.4 Q4.18
  posture) **plus `_with_rng` variants** (`gmcrypto_sm2_kx_initiator_new_with_rng`,
  `gmcrypto_sm2_kx_responder_respond_with_rng`) taking the existing v0.5
  `gmcrypto_rng_callback` + context, bridged by the existing `CallbackRng`.
  This buys the headline assurance item: the c_smoke suite reproduces the
  **GM/T 0003.5 recommended-curve KAT byte-for-byte through the C ABI**
  (fixed-ephemeral callbacks → assert `R_A`/`R_B`/`K`/`S_A`/`S_B`).
- **Q2.4 Feature posture** — **always-on** in `gmcrypto-c` (the v0.23 W3
  posture): `sm2-key-exchange` is added unconditionally to the C shim's
  `gmcrypto-core` dep features (alongside `sm4-aead`/`sm4-xts`), so the
  committed `gmcrypto.h` == a default `cargo build -p gmcrypto-c` and the
  cbindgen cfg-gated-opaque-type trap (the v0.10 lesson) cannot fire.
  `gmcrypto-core`'s own `sm2-key-exchange` feature stays opt-in/default-off.
- **Q2.5 Surface census** — **9 new symbols + 2 opaque types + 2 consts**
  (63 → 72 FFI entry points):
  `gmcrypto_sm2_kx_initiator_{new,new_with_rng,confirm,free}`,
  `gmcrypto_sm2_kx_responder_{new,respond,respond_with_rng,finish,free}`;
  `GMCRYPTO_SM2_KX_POINT_SIZE = 65`, `GMCRYPTO_SM2_KX_CONFIRM_SIZE = 32`.
  Keys ride the existing `gmcrypto_sm2_privkey_t` / `gmcrypto_sm2_pubkey_t`
  handles (the sign/verify/encrypt convention). `id_len == 0` defaults to
  `DEFAULT_SIGNER_ID` (`"1234567812345678"`) per the sign FFI — which is also
  exactly the KAT's ID, so the C example stays minimal.
- **Q2.6 Output convention** — fixed-size bare out-pointers with documented
  sizes (`R`/`S` like the GCM `tag_out` 16-B precedent); `key_out` must be
  ≥ the `klen` given at `_new` (caller chose `klen`, so no
  `out_capacity`/`out_actual_len` triple — the length is never discovered).
  **The caller owns wiping `key_out`** (the `to_bytes_be` contract); the
  handles' internal `Sm2SharedKey`/state zeroize on drop/consume.
- **Q2.7 Failure modes** — single `GMCRYPTO_ERR` for everything (null ptrs,
  null callback, RNG failure, invalid peer `R`, tag mismatch, wrong state
  order, bad `klen`/`id`) — the workspace failure-mode invariant; the FFI
  must not let C callers distinguish what the Rust core deliberately blinds.
- **Q2.8 Assurance** — **no new dudect target** (thin shim; core's
  `ct_sm2_key_exchange` covers the secret-dependent path — the v0.13/v0.16
  precedent). c_smoke additions: FFI↔FFI handshake, FFI↔Rust cross-handshake
  (both directions), the KAT-through-FFI (Q2.3), and negative paths
  (corrupt `R`, tampered `S_B`/`S_A`, double-`respond`, nulls, bad `klen`).
  **fuzz**: extend the existing `fuzz_c_abi` target to drive the KX entry
  points with adversarial `R`/`S` bytes (no new fuzz binary — the census
  stays 26; the FUZZ_TARGETS must-match-[[bin]] invariant from #102 holds).
- **Q2.9 Versioning** — additive minor → **1.2.0** lockstep (`=1.2.0`
  sibling pins). Publish + SSH-signed tag + GitHub release remain the
  maintainer's authenticated call (the v1.1.0 agent-publish was a recorded
  one-off delegation, not a precedent). Agent path: branch + PR.
- **Q2.10 Docs** — doc-only C example
  `crates/gmcrypto-c/examples/sm2_key_exchange.c` (full two-party handshake);
  README/CLAUDE.md/CHANGELOG narrative; `gmcrypto-c/README.md` symbol table.

## 2. Out of scope (deliberate)

- X.509-with-SM2 (v1.3 direction candidate; own cycle).
- TLCP protocol binding (future).
- An elision/`session`-style multi-shot KX API; streaming `id`s; any
  resumable-handshake surface — GM/T 0003.3 is a one-shot AKE.
- Any change to `gmcrypto-core` (the v1.1 core API is sufficient — the FFI
  is consume-compatible with the typestate as shipped).
