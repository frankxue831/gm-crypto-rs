# v0.7-aead — Scope doc for v0.8 SM4-GCM + SM4-CCM

**Status:** Lands in v0.7 W4 as the design cycle for v0.8 implementation. No code in v0.7; this doc is the v0.8 Q-list-with-sign-off bar.

**Why v0.7 ships the scope doc and v0.8 ships the code:** AEAD opens a net-new cryptographic surface (constant-time GHASH or fixed-iteration CMAC, fresh KAT vector sourcing, two distinct modes with non-overlapping internals, dedicated dudect targets, and a GHASH primitive that ideally rides hardware CLMUL on `x86_64` via the existing `gmcrypto-simd` sibling crate). Squeezing it into v0.7 alongside CTR + the batch-API promotion would either dilute review surface or push v0.7 out by weeks. Splitting along the natural scope-doc / implementation boundary keeps each version reviewable.

This doc mirrors the structure of `docs/v0.6-scope.md` — Q-list with provisional answers, hard contract, posture, deferred items.

---

## 1. Context

v0.6.0 closed the W4 throughput-win milestone (multi-block SIMD fanout for CBC-decrypt). v0.7.0 ships the cipher-mode surface that v0.6's machinery was designed to enable:

- **W1**: public batch API on `Sm4Cipher` (`encrypt_blocks` / `decrypt_blocks`, length-flexible).
- **W2**: SM4-CTR single-shot mode (`sm4::mode_ctr`).
- **W3**: SM4-CTR streaming (`sm4::ctr_streaming::Sm4CtrCipher`) + `ct_sm4_ctr_encrypt` dudect target.

CTR is **unauthenticated**. Every callsite that needs integrity has two options:

1. Encrypt-then-MAC manually: pair `mode_ctr::encrypt` with `hmac::hmac_sm3` over `counter || ciphertext`.
2. Wait for AEAD.

v0.8 ships option 2 — SM4-GCM and SM4-CCM, both with constant-time tag handling and timing-side-channel-resistant GHASH / CMAC paths. Plus the dedicated dudect targets that defend the property.

GB/T 0009 does not currently publish SM4-GCM / SM4-CCM in a standalone test-vector annex; the cipher modes themselves are referenced indirectly (GM/T 0024 TLS 1.3 ShangMi profile, RFC 8998). v0.8 will source KAT vectors from gmssl 3.1.1's `sm4 -gcm` and `sm4 -ccm` CLI subcommands (the same provenance route that v0.3 W3 used for the cross-validation suite); RFC 8998's TLS-level test vectors give the AEAD identifier (`TLS_SM4_GCM_SM3`) but not raw key/nonce/aad/plaintext/ciphertext/tag quintuples at the AEAD-primitive layer.

## 2. Hard contract

The same byte-identical invariant that v0.6 and v0.5 W4 hold to:

- `SM4-GCM(key, nonce, aad, plaintext)` produces byte-identical `(ciphertext, tag)` to **gmssl 3.1.1 `sm4 -gcm`** under matching inputs (subject to gmssl's nonce-length conventions).
- `SM4-CCM(key, nonce, aad, plaintext)` produces byte-identical `(ciphertext, tag)` to gmssl 3.1.1 `sm4 -ccm` (CCM is not in gmssl's CLI as of 3.1.1 — see Q8.4).
- Constant-time on key bytes and tag-comparison. The GHASH multiplication path is constant-time over `H` (the GCM hash subkey, derived from `SM4(key, 0^128)` — secret-touching).

The hard-contract bar is the same shape as v0.6's CBC-fanout work: a public reference (gmssl) plus a dudect gate. No "more helpful" failure variants (single `Failed` per the project invariant); no padding-oracle / tag-oracle distinguishing surface.

## 3. Q-list (Q8.1–Q8.8) with provisional answers

### Q8.1 — GHASH constant-time strategy

GHASH multiplies in `GF(2^128)` over the secret-derived hash subkey `H`. The standard fast software path uses Shoup's lookup tables (e.g. 4-bit `H · X` lookup with 16 precomputed multiples) — but those tables are **secret-indexed**, breaking constant-time discipline. Three viable strategies:

A. **CLMUL on `x86_64`** via `gmcrypto-simd` sibling crate (mirrors the v0.5 W4 phase 2 AVX2 pattern). `pclmulqdq` provides single-cycle carryless multiplication on essentially all `x86_64` CPUs from 2010+ (Intel Westmere onward, AMD Bulldozer onward). Hardware-CLMUL GHASH is constant-time by construction.

B. **NEON `pmull` on `aarch64`**. ARMv8.0 `PMULL` and `PMULL2` (Crypto extensions) provide the equivalent. Compile-time baseline on modern aarch64 chips when ARMv8.0 Crypto extensions are present; runtime feature detection via `cpufeatures` otherwise.

C. **Software Karatsuba fallback** on non-CLMUL targets. Reduced-degree polynomial multiplication over `GF(2)` via Karatsuba; constant-time. Slower than hardware CLMUL (~4-5× per multiplication) but still acceptable for v0.8 since CTR's throughput dominates anyway.

**Provisional decision A+B+C, dispatched at runtime via `gmcrypto-simd`:**

- Add `gmcrypto-simd::ghash::ghash_mul_avx2(h, x)` + `ghash_mul_neon(h, x)` + `ghash_mul_software(h, x)` (the three impls).
- `gmcrypto-simd` exposes a public `pub fn ghash_mul(h: &[u8;16], x: &[u8;16]) -> [u8;16]` that dispatches via `cpufeatures` (mirrors the existing AVX2 SM4 S-box path).
- `gmcrypto-core::sm4::mode_gcm` calls `gmcrypto_simd::ghash::ghash_mul` directly; no `cpufeatures` checks bleed into `gmcrypto-core`.

### Q8.2 — CCM constant-time path

SM4-CCM uses **CBC-MAC** over the auth blocks (no GHASH). CCM's internals reduce to:

1. CBC-MAC the (formatted nonce || aad || plaintext) under the encryption key, in the obvious encrypt-then-XOR-into-state loop.
2. CTR-encrypt the plaintext under the same key, using the formatted nonce as the initial counter.
3. CTR-encrypt the CBC-MAC tag under a tweaked counter and append.

All three sub-operations route through existing `Sm4Cipher::encrypt_block` and `Sm4Cipher::encrypt_blocks` (W1's public batch API). **CBC-MAC** itself is just `Sm4Cipher::encrypt_block` in a loop over XOR'd state; constant-time as long as the underlying cipher is (which is gated by `ct_sm4_encrypt_block`).

Tag comparison: constant-time via `subtle::ConstantTimeEq` (see Q8.3).

**Provisional decision: CCM lands as pure-Rust over the existing Sm4Cipher primitive. No new SIMD code needed; the SIMD-acceleration story is already captured by W1's batched path. New module `sm4::mode_ccm` mirroring `sm4::mode_gcm`.**

### Q8.3 — Tag comparison

Every AEAD decrypt path must compare the recomputed tag against the wire tag in constant time. Standard `==` on `[u8; 16]` leaks the first-differing-byte position via short-circuit. Use:

```rust
use subtle::{Choice, ConstantTimeEq};
let ok: Choice = computed_tag.ct_eq(&wire_tag);
if ok.into() {
    Some(plaintext)
} else {
    plaintext.zeroize();  // wipe attacker-controllable bytes
    None
}
```

This is the same pattern already in `sm2::decrypt` (v0.2 Phase 3); reuse the discipline.

**Provisional decision: `subtle::ConstantTimeEq::ct_eq` over `[u8; 16]`, with plaintext-on-failure zeroize. Single `None` per the failure-mode invariant.**

### Q8.4 — KAT vector sourcing

GB/T does not publish SM4-GCM/CCM-specific test vectors at this writing. Options:

- **gmssl 3.1.1**: `sm4 -gcm` is shipped (verified during W2 — `gmssl sm4 -help` lists it). Use its CLI to generate KAT triples `(key, nonce, aad, plaintext) → (ciphertext, tag)` and embed as binary fixtures under `crates/gmcrypto-core/tests/data/` (the v0.3 W2 fixture-regen pattern).
- **NIST SP 800-38D**: AES-GCM vectors derived; useful for GHASH algorithmic cross-check, NOT for byte-identical SM4-GCM ciphertext since the underlying block cipher differs.
- **RFC 8998 TLS 1.3 ShangMi**: TLS-level vectors; useful for end-to-end interop testing of a future TLS-bound consumer, NOT for the AEAD primitive directly.
- **Self-consistency**: encrypt + decrypt = identity, byte-equivalence between our software-fallback and CLMUL paths under feature flags. Necessary but not sufficient.

**Provisional decision: gmssl 3.1.1 is the primary KAT source for SM4-GCM. Document the provenance (gmssl-CLI-generated, not GB/T published) in the module docstring per the v0.3 W2 fixture-comment discipline. For SM4-CCM: pending — gmssl 3.1.1's `sm4 -help` does not list `-ccm`, so KAT sourcing for CCM is a v0.8 W0 (pre-work) deliverable. Possible alternative: GB/T 0009 SM4-CCM is referenced in some draft GM/T documents; investigate during v0.8 W0.**

**v0.8 W0 resolution (2026-05-15):** OpenSSL 3.x libcrypto EVP `SM4-CCM` cipher (OID `1.2.156.10197.1.104.9`) is the chosen reference oracle for SM4-CCM. See [`docs/v0.8-ccm-kat-sourcing.md`](v0.8-ccm-kat-sourcing.md) for the sourcing rationale, the ~80-line C harness source, and the parametric-coverage matrix for W3 fixtures.

### Q8.5 — Failure-mode invariant

Tag mismatch is the only AEAD failure mode that distinguishes legitimate-but-corrupt ciphertext from an attacker-supplied forgery. Per the workspace failure-mode invariant (`CLAUDE.md`'s "Hard constraints"), all failure modes collapse into one:

- `sm4::mode_gcm::decrypt` returns `Option<Vec<u8>>`. `None` covers: tag mismatch, malformed input length, nonce-length-out-of-range. No distinguishing variants.
- Same shape for `sm4::mode_ccm::decrypt`.

**Provisional decision: `Option<Vec<u8>>`, single `None`. Plaintext-on-failure zeroize before drop. PRs that "make errors more helpful" get rejected on sight per the project posture.**

### Q8.6 — Streaming variants

v0.7 ships streaming CTR (`Sm4CtrCipher::update`) for callsites that handle one chunk at a time. AEAD is awkward to stream because the tag isn't known until after the full ciphertext is processed (encrypt) or the full ciphertext is presented (decrypt — can't release plaintext bytes until the tag is verified, else you've leaked a chosen-plaintext distinguisher).

The standard streaming AEAD trait (RustCrypto `aead::AeadInOut`) does provide a hold-back-final-block / commit-on-tag-verify shape, but it's substantially more complex than the v0.7 CTR streaming.

**Provisional decision: single-shot only in v0.8 W1 (one-shot `encrypt` returning `(ciphertext, tag)`; one-shot `decrypt` returning `Option<Vec<u8>>`). Streaming `Sm4GcmEncryptor` / `Sm4GcmDecryptor` deferred to v0.9 if there's a concrete consumer. Reason: streaming AEAD doubles the dudect surface (decrypt has a "tag-verified-yet?" state machine that gets its own class-split) without the v0.7-W3 CTR streaming pattern as a direct template.**

### Q8.7 — Dudect targets

Mirror the v0.6 W6 / v0.7 W3 pattern:

- `ct_sm4_gcm_decrypt` — class-split by master key. Both classes' ciphertexts are valid encrypts under their own keys (a 16-block payload + valid tag), so both paths reach the same control flow (tag-check passes; payload decrypts). Mirrors `ct_sm4_cbc_decrypt_fanout`'s "both classes share control flow" discipline. Gate: `|tau| < 0.20`.
- `ct_sm4_ccm_decrypt` — same shape, class-split by master key. Same gate.
- **Tag-mismatch path is NOT separately gated** — it's the failure-mode side, and per the failure-mode invariant we shouldn't be optimizing for distinguishability. The `ct_sm4_gcm_decrypt` target as specified already exercises the constant-time-tag-compare via the same code path on both classes.

Both targets cfg-gated on a new `sm4-aead` feature flag (Q8.8). Workflow matrix adds the feature combination.

**Provisional decision: 2 new dudect targets (`ct_sm4_gcm_decrypt`, `ct_sm4_ccm_decrypt`), both cfg-gated on `sm4-aead`. PR-smoke + nightly workflows pick up the feature in a new matrix entry.**

### Q8.8 — Feature flag shape

The existing `sm4-bitsliced-simd` feature gates SIMD-accelerated S-box (used by the W1 batched path, transitively by W2/W3 CTR). AEAD adds:

- GHASH constant-time multiplication — needs CLMUL on `x86_64` (in `gmcrypto-simd`) for performance; software fallback on other targets.
- New top-level surface (`sm4::mode_gcm`, `sm4::mode_ccm`).
- New deps? — `subtle` is already in the workspace; nothing else strictly needed.

Two reasonable shapes:

- **A: separate `sm4-aead` feature** — additive with `sm4-bitsliced-simd`. Pulls in the GHASH primitive on activation. Conservative — keeps the AEAD surface out of the default build.
- **B: AEAD becomes default-on** — no flag, GHASH ships in default-features. Simpler API; larger default footprint (the GHASH software fallback adds ~1 KB).

**Provisional decision A: `sm4-aead` feature, additive with `sm4-bitsliced-simd` for CLMUL acceleration on `x86_64`. Rationale: matches the v0.4 W3 pattern (`sm4-bitsliced` as opt-in even though it's pure-Rust), preserves the no-extra-deps zero-runtime-deps default posture documented in `SECURITY.md`. Promote to default-on in v0.9+ once a stable Rust release ships AEAD trait fit (`aead 0.6` or `aead 1.0` — currently 0.5).**
