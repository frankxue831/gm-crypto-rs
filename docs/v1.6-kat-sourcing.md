# v1.6 KAT sourcing — TLCP key schedule + no-confirmation SM2-KX

## 1. Key schedule: OpenSSL 3.x `TLS1-PRF` with `digest:SM3`

gmssl 3.1.1 exposes no PRF CLI, so the oracle is OpenSSL's generic
TLS-1.2 PRF KDF instantiated with SM3 — the v0.8 (CCM) / v0.12 (XTS)
EVP-oracle pattern. Verified on **OpenSSL 3.6.2** (Homebrew,
2026-06-13), default (non-FIPS) provider.

Recipe pins (per the `EVP_KDF-TLS1_PRF` documentation, Codex-confirmed
in the v1.6 scope consult):

- `TLS1-PRF` has **no label parameter** — the ASCII label bytes are
  the leading part of the seed: `seed = label ‖ seed-part-1 ‖ …`.
- Binary material must ride `hexsecret:` / `hexseed:` (`secret:` /
  `seed:` are ASCII-literal forms).
- `digest:SM3` selects the TLS-1.2-style single-`P_hash` mode; OpenSSL
  special-cases only `MD5-SHA1` into the TLS 1.0/1.1 split PRF.
- Use the default provider (FIPS providers may reject SM3 or the
  TLS1-PRF usage).

### Inputs

```
PMS = ab × 48
CR  = 01 02 … 20   (32 bytes, sequential)
SR  = 21 22 … 40   (32 bytes, sequential)
TH  = cd × 32      (stand-in SM3 transcript hash)
labels (hex): "master secret"   = 6d617374657220736563726574
              "key expansion"   = 6b657920657870616e73696f6e
              "client finished" = 636c69656e742066696e6973686564
              "server finished" = 7365727665722066696e6973686564
```

### Commands

```bash
prf() { openssl kdf -keylen "$1" -kdfopt digest:SM3 \
        -kdfopt hexsecret:"$2" -kdfopt hexseed:"$3" -binary TLS1-PRF \
        | xxd -p -c 200; }

# master secret (GB/T 38636 §6.5.1): seed = "master secret" ‖ CR ‖ SR
MS=$(prf 48 "$PMS" "${MS_LABEL}${CR}${SR}")
# key block (§6.5.2): seed = "key expansion" ‖ SR ‖ CR   ← order flips
prf 128 "$MS" "${KE_LABEL}${SR}${CR}"   # CBC carve
prf 40  "$MS" "${KE_LABEL}${SR}${CR}"   # GCM carve (prefix property)
prf 200 "$MS" "${KE_LABEL}${SR}${CR}"   # multi-iteration case
# Finished (§6.4.5.10): seed = label ‖ TH, 12 bytes
prf 12 "$MS" "${CF_LABEL}${TH}"
prf 12 "$MS" "${SF_LABEL}${TH}"
```

The resulting vectors are embedded as `hex!` constants in
`crates/gmcrypto-core/tests/tlcp_key_schedule_kat.rs`. **Deliberate
deviation from scope Q6.6's "+ `tests/data/`"** (review amendment A5):
short vectors live inline in the test file — the `sm2_kx_kat.rs`
precedent — and nothing lands under `tests/data/`.

Cross-check posture: our implementation is written from the RFC 5246
§5 / GB/T 38636 §6.5 structure, independently of OpenSSL's; the KAT
match of two independent implementations is the assurance. A
full-handshake transcript replay (gotlcp or gmssl `tlcp_server`)
transitively re-validates the schedule when v1.7 record protection
makes it testable end-to-end.

## 2. No-confirmation SM2-KX: the GM/T 0003.5 worked example

The GM/T 0003.5-2012 recommended-curve worked example (already the
v1.1 KAT, `docs/v1.1-sm2kx-kat-sourcing.md`) derives
`K = 6C893473…` BEFORE the optional S_A/S_B confirmation tags, so the
same standard-sourced vector pins the v1.6 no-confirmation completers
byte-for-byte — no new sourcing needed. The worked example uses
klen = 16; the TLCP pre-master length (48) is covered by a
confirmed-vs-unconfirmed differential at both klens
(`sm2_kx_kat.rs::unconfirmed_matches_confirmed_at_klen_16_and_48`).
