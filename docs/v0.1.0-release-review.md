# gmcrypto-core v0.1.0 release review checklist

**Crate:** `gmcrypto-core` 0.1.0
**Repository:** <https://github.com/frankxue831/gm-crypto-rs>
**Release candidate:** record the exact commit you intend to tag with
`git rev-parse HEAD`.
**Purpose:** decide whether that exact commit is safe to publish to
<https://crates.io/crates/gmcrypto-core>. Do not review "latest main" by
habit; review a frozen commit.

Publishing to crates.io is permanent. A yanked version is hidden from normal
resolution, but the `.crate` archive remains downloadable.

Use `BLOCK`, `WARN`, and `INFO`:

- `BLOCK`: do not publish until fixed or explicitly deferred.
- `WARN`: publishable, but track for v0.1.1 or v0.2.
- `INFO`: useful note, no release action required.

git clone https://github.com/frankxue831/gm-crypto-rs.git

## 5. Security-critical source review

### 5.1 Signing

File: `crates/gmcrypto-core/src/sm2/sign.rs`

Check:

- `compute_z` uses `ENTL || ID || a || b || Gx || Gy || Px || Py`, where
  `ENTL` is a 16-bit big-endian bit length.
- over-long IDs are rejected before RNG use.
- `SIGN_RETRY_BUDGET` iterations always run; no early exit on first success.
- `ct_or_else` and `RsPair::conditional_select` do not branch on secret choices.
- `SignError` has exactly one variant.
- `sign_raw_with_id` is `#[doc(hidden)] pub` and documented as unstable.
- nonce sampling is uniform over `[1, n-1]`. Red flag: reducing a 256-bit RNG
  output with `Fn::new(&candidate)` before checking range creates modulo bias.
  Prefer rejection on `candidate != 0 && candidate < n`.

### 5.2 Verification

File: `crates/gmcrypto-core/src/sm2/verify.rs`

Check:

- identity public key returns `false`;
- over-long ID returns `false`;
- `r == 0`, `s == 0`, `r >= n`, `s >= n` return `false`;
- `t = (r + s) mod n == 0` returns `false`;
- function returns `bool`, not `Result`;
- no `panic!`, `expect`, or `unwrap` is reachable outside tests.

```bash
grep -nE 'panic|expect|unwrap' crates/gmcrypto-core/src/sm2/verify.rs
```

### 5.3 ASN.1 DER

File: `crates/gmcrypto-core/src/asn1/sig.rs`

Check:

- empty INTEGER content is rejected;
- sign-bit-set INTEGER content is rejected;
- redundant leading `00` padding is rejected;
- non-minimal long lengths are rejected;
- trailing bytes after `s` are rejected.

```bash
cargo test -p gmcrypto-core asn1::sig::tests
```

Expected: 6 tests pass.

### 5.4 Curve and scalar multiplication

Files:

- `crates/gmcrypto-core/src/sm2/curve.rs`
- `crates/gmcrypto-core/src/sm2/point.rs`
- `crates/gmcrypto-core/src/sm2/scalar_mul.rs`

Check:

- `PMod`, `NMod`, `b`, `Gx`, and `Gy` match GB/T 32918.5.
- point add/double use the documented RCB formulas and pass 2G/3G KATs.
- `ProjectivePoint::ct_eq` avoids affine conversion.
- `to_affine()` documents the `crypto-bigint::invert` caveat.
- scalar multiplication uses full-table linear scans for every nibble.
- `mul_g` documentation says it delegates to `mul_var` in v0.1; no stale
  references to unimplemented comb tables or internal task numbers.

### 5.5 Private and public keys

Files:

- `crates/gmcrypto-core/src/sm2/private_key.rs`
- `crates/gmcrypto-core/src/sm2/public_key.rs`

Check:

- private key range is `[1, n-2]`;
- validation uses `subtle` choices and returns `CtOption<Self>`;
- `ZeroizeOnDrop` is present;
- `Debug` redacts the scalar;
- public-key docs do not mention internal task numbers and do not imply
  validation exists where it does not.
