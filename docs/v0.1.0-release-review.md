## 3 · Constant-time / timing-leak harness review (~10 min)

This is the headline differentiator and most-likely-to-be-contested
public claim. Verify both the harness *works* and that its *coverage
gap* is honestly disclosed.

### 3.1 Harness self-check (negative control fires)

```bash
DUDECT_SAMPLES=10000 cargo bench --bench timing_leaks
```

| Bench | Expected `\|tau\|` | Status |
|---|---|---|
| `negative_control` | **> 1.0** (typically 25–80 — must be huge) | ☐ |
| `ct_mul_g` | < 0.20 | ☐ |
| `ct_mul_var` | < 0.20 | ☐ |
| `ct_sign` | < 0.20 | ☐ |

**🚩 If `negative_control` returns small `\|tau\|`**, the harness is
broken — it cannot detect the deliberately-leaky function it embeds,
which means a green real-target reading is meaningless. **Block release.**

### 3.2 The harness's documented blind spot

Read the section "Honest admission: this class-split is blind to
nonce-path leaks" in `crates/gmcrypto-core/benches/timing_leaks.rs`
(top-of-file module docs) and the matching section in `SECURITY.md`
("Honest admission about the dudect harness's coverage").

| Check | Status |
|---|---|
| 3.2.1 The blind-spot disclosure exists in *both* `SECURITY.md` and `benches/timing_leaks.rs` | ☐ |
| 3.2.2 The disclosure correctly names the affected site: `Fp::invert(Z)` inside `kg.to_affine()` after `mul_g(k)` | ☐ |
| 3.2.3 The disclosure says explicitly that a `ct_sign` pass is *not* evidence the nonce path is leak-free | ☐ |
| 3.2.4 The v0.2 plan names a fix (Fermat-invert via `pow_bounded_exp`) and a harness rework (class-split by `k`) | ☐ |
| 3.2.5 README's harness paragraph (around line 23 of `README.md`) mentions the blind spot above the fold and points to `SECURITY.md` | ☐ |

**🚩 If you (the reviewer) think the framing softens the limitation —
e.g. by hiding it in a sub-section, or hedging — flag it before
release. The whole point of v0.1 is the honest harness; misrepresenting
its coverage gap would invalidate the project's central claim.**

### 3.3 Sanity-check the reviewer's understanding

Open `crates/gmcrypto-core/src/sm2/sign.rs` lines 145–195 (`try_sign_once`).
Confirm by reading:

| Check | Status |
|---|---|
| 3.3.1 `mul_g(&k)` is called on the secret nonce `k` | ☐ |
| 3.3.2 `kg.to_affine()` is called on the result (line ~160) | ☐ |
| 3.3.3 `to_affine()` calls `Fp::invert(z)` (open `crates/gmcrypto-core/src/sm2/point.rs` around line 158 to confirm) | ☐ |
| 3.3.4 Therefore `Fp::invert` runs on a `z` derived from secret `k` — this is the leak the dudect harness can't see | ☐ |

If you reach 3.3.4 and *agree* this is a real (small but real) timing
side-channel in v0.1, the disclosure framing is honest. If you don't,
or if you think the leak is bigger than the docs claim, raise it.

## 10 · Escalation

If you find anything that should block release, write it up in a
GitHub issue under <https://github.com/frankxue831/gm-crypto-rs/issues>.
Use the format:

```
[BLOCK / WARN / INFO] <short title>
File: path/to/file.rs:LINE
Finding: <one paragraph>
Suggested action: <fix / document / discuss>
```

`BLOCK` = release should not happen until resolved.
`WARN` = release can happen, but should be tracked for v0.1.1 / v0.2.
`INFO` = nice-to-know; no action needed.

If you're unsure whether something is a `BLOCK` or `WARN`, default to
`BLOCK` and let the maintainer downgrade.
