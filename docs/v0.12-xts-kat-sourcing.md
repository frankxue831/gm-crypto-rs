# v0.12 W1 — SM4-XTS KAT vector sourcing

## Decision

gmssl 3.1.1 does **not** ship SM4-XTS (`sm4` supports only
`-cbc|-ctr|-gcm|-cbc_sm3_hmac|-ctr_sm3_hmac`). So, exactly as for SM4-CCM in
v0.8 (`docs/v0.8-ccm-kat-sourcing.md`), the KAT oracle is **OpenSSL 3.x EVP
`SM4-XTS`** (GM-T OID `1.2.156.10197.1.104.10`), vendor-neutral. The
`openssl enc` CLI blocks XTS ("enc XTS ciphers not supported"), so vectors are
sourced via the EVP API in `tests/data/sm4_xts_oracle.c`. **No gmssl interop
test** (the mode is unavailable there).

## Oracle pinning

- Cipher fetched from OpenSSL's **default (non-FIPS) provider**.
- Key is 32 bytes (`Key1 ‖ Key2`); the 16-byte tweak is the EVP IV (raw).
- `EVP_CIPHER_CTX_set_padding(ctx, 0)` — XTS is length-preserving.
- Single-shot `EVP_EncryptUpdate` / `EVP_DecryptUpdate`.

## Behavioral notes (verified 2026-05-23, OpenSSL 3.6.2 Homebrew openssl@3)

- `block_size` reported as 1 (stream-like); output length == input length.
- `len < 16` → OpenSSL `EVP_EncryptUpdate` fails (matches our `None`).
- `Key1 == Key2` → OpenSSL default provider **emits ciphertext** (does not
  reject; only `fips=yes` rejects). Our Rust API returns `None` (IEEE 1619 /
  FIPS-aligned). This case is asserted **locally** in `sm4_xts_kat.rs` /
  `mode_xts` unit tests, not as an oracle KAT.
- The 20-byte CTS output byte-for-byte matched the spec construction in
  `docs/v0.12-scope.md` Q12.5 (output `C_0 ‖ CC[..r]`; the trailing `r` bytes
  equal the first `r` bytes of the whole-block block-0 ciphertext).

## Reproduce (auditor-reproducible)

```text
OSSL=$(brew --prefix openssl@3)
cc -O2 -I$OSSL/include tests/data/sm4_xts_oracle.c \
   $OSSL/lib/libcrypto.dylib -o sm4_xts_oracle

KEY=0123456789abcdeffedcba9876543210000102030405060708090a0b0c0d0e0f
TW=11111111111111111111111111111111

./sm4_xts_oracle enc $KEY $TW 00112233445566778899aabbccddeeff
# -> b3fbef63165a03942ea2b4b7bc67af80

./sm4_xts_oracle enc $KEY $TW 00112233445566778899aabbccddeeff00112233445566778899aabbccddeeff
# -> b3fbef63165a03942ea2b4b7bc67af80455f6784df3b00cf4a388baf001da4c4

./sm4_xts_oracle enc $KEY $TW 0011223344556677889900112233445566
# -> 89956c7af6086bc701281ba668773d5852   (17-byte CTS, r=1)

./sm4_xts_oracle enc $KEY $TW 00112233445566778899aabbccddeeff00112233
# -> e42c1aff8629401515f2edac4eedbe69b3fbef63   (20-byte CTS, r=4)
```

## Coverage matrix (vectors embedded in tests/sm4_xts_kat.rs)

| len (B) | shape          | note                                       |
|---------|----------------|--------------------------------------------|
| 16      | whole-block    | min length; == first 16 B of the 32 B ct   |
| 32      | whole-block ×2 | two-block aligned                          |
| 17      | CTS r=1        | smallest steal (`normal == 0` path)        |
| 20      | CTS r=4        | matched the Q12.5 construction in W0        |

Round-trip property coverage (`encrypt` then `decrypt` == identity) over
`len ∈ 16..=80` plus the rejection cases (`len < 16`, `len > 16 MiB`,
`Key1 == Key2`) live in the `mode_xts` inline unit tests (the rejection
constants reference the module-private `MAX_LEN`).
