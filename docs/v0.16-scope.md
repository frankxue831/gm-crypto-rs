# v0.16 — Scope doc

**C FFI for the SM4-XTS multi-sector helper** — expose the v0.15
`sm4::mode_xts::{encrypt_sectors, decrypt_sectors}` in-place sector run helper
through the `gmcrypto-c` C ABI, behind the existing forwarding `sm4-xts`
feature. The deferred half of v0.15 (Q15.12), landed on the established
**core-in-vN, FFI-in-vN+1** cadence (XTS single-shot core in v0.12 → FFI in
v0.13; the sector helper core in v0.15 → FFI here). Additive only; default
builds unaffected.

> **Q-numbering note:** this doc's `Q16.x` are the *design decisions* for v0.16.
> They **supersede** the provisional `Q16.x` *candidate* list in
> `docs/v0.15-scope.md` §6 (a forward-looking menu). Where this doc references
> that older list it disambiguates with the `v0.15-Q16.x` prefix — the same
> convention prior cycles used (`v0.12-Q13.x` in the v0.13 doc). v0.16 anchors
> on the former candidate `v0.15-Q16.1` (C FFI for the sector helper).

## 1. Context

The C ABI (`gmcrypto-c`: cdylib + staticlib + cbindgen header) is the
workspace's single language-binding surface for C / C++ / Go / Zig / Python
callers. v0.13 brought the **single-shot** SM4-XTS core (`mode_xts::{encrypt,
decrypt}`) to that surface — one tweak, one data unit, out-of-place. v0.15
then added the **multi-sector** convenience to the Rust core
(`encrypt_sectors` / `decrypt_sectors`): encrypt/decrypt a contiguous run of
equal-size disk sectors **in place**, deriving sector `i`'s tweak as
little-endian-128(`start_sector + i`) — the standard disk-XTS data-unit
convention. v0.15 shipped that core-only (Q15.12 deferred the FFI one cycle,
mirroring v0.12→v0.13 and v0.8→v0.10). v0.16 closes the gap so non-Rust
callers get the same one-call, in-place, auto-incrementing-tweak ergonomics
the Rust core already offers — leaving the disk-encryption surface
FFI-complete ahead of an eventual v1.0.

The sector helper is **in-place, length-preserving, whole-block (no ciphertext
stealing), confidentiality-only**. That makes its FFI even simpler than the
v0.13 single-shot: there is no `out` buffer, no `out_capacity` /
`out_actual_len` dance (the transform is in place and length-preserving), no
opaque handle, no streaming state machine — just
`key + sector_size + start_sector + buf → buf transformed in place`.

### Building blocks already in place (grounded 2026-05-29)

- **Core API** (`gmcrypto_core::sm4::mode_xts`, v0.15):
  `encrypt_sectors(key: &[u8; 32], sector_size: usize, start_sector: u128,
  buf: &mut [u8]) -> Option<()>` and the matching `decrypt_sectors`. **All
  validation pre-flighted before `buf` is touched** → on any `None`, `buf` is
  untouched. Single `None` failure mode (`sector_size` ∉ `[16, 16 MiB]` or not
  a multiple of 16; `buf.len()` not a whole multiple of `sector_size`;
  `Key1 == Key2`; `start_sector + sector_count − 1` overflowing `u128`). Empty
  `buf` (zero sectors) → vacuous `Some(())`, **but the key is still validated**
  (empty + weak key → `None`). Already dudect-gated transitively
  (`ct_sm4_xts_decrypt` covers the per-sector secret path; the sector→tweak
  arithmetic is on **public** addresses — Q15.9).
- **FFI conventions** (`gmcrypto-c/src/lib.rs`): `ffi_guard` (catch_unwind →
  `GMCRYPTO_ERR`), `try_slice` (`(NULL,0)`→empty, null+len→`None`),
  **`try_slice_mut`** (the mutable-slice reconstruction already used by
  `write_output` and the streaming decryptor `_update`), `GMCRYPTO_OK = 0` /
  `GMCRYPTO_ERR = -1`, the always-on `GMCRYPTO_SM4_XTS_KEY_SIZE = 32`.
- **Single-shot XTS FFI** (`gmcrypto_sm4_xts_encrypt` / `_decrypt`,
  `lib.rs:1320`) — the immediate sibling shape; this cycle reuses its key
  reconstruction (`try_slice(key,32)` + `&[u8;32]` `try_into`) and its
  feature/header/doc posture verbatim.
- **Feature-forwarding precedent**: `sm4-xts = ["gmcrypto-core/sm4-xts"]`
  **already exists** on `gmcrypto-c` (added in v0.13). No new feature flag.
- **Test precedent**: `tests/c_smoke.rs` XTS block (`:1864+`) — import the
  symbol via `gmcrypto_c::{…}`, call through Rust's `extern "C"` interop,
  assert byte-equivalence with the core API + round-trip + error code.
- **Header-regen precedent**: `regen-header` runs cbindgen in `build.rs`; CI
  gates drift via `git diff --exit-code include/gmcrypto.h`.

## 2. Hard contract

- Two new symbols, behind `#[cfg(feature = "sm4-xts")]`:
  - `gmcrypto_sm4_xts_encrypt_sectors(key, sector_size, start_sector, buf, buf_len) -> c_int`
  - `gmcrypto_sm4_xts_decrypt_sectors(key, sector_size, start_sector, buf, buf_len) -> c_int`
- `key` = 32 bytes (`Key1 ‖ Key2`); `sector_size` = `size_t`; `start_sector` =
  `uint64_t` (the first sector's LBA; Q16.3); `buf` = `uint8_t *`
  (**read+write, transformed in place**); `buf_len` = `size_t`. Returns
  `GMCRYPTO_OK` / `GMCRYPTO_ERR`. **No `out` / `out_capacity` /
  `out_actual_len`** — in-place and length-preserving (Q16.2).
- **Single failure mode**: any of null `key`/`buf`, `sector_size` ∉
  `[16, 16 MiB]` or not a multiple of 16, `buf_len` not a whole multiple of
  `sector_size`, `Key1 == Key2` → `GMCRYPTO_ERR` (failure-mode invariant; no
  distinguishing — forwards the core `None`). **On `GMCRYPTO_ERR`, `buf` is
  untouched** (the core pre-flights all validation before any mutation).
  `buf_len == 0` → `GMCRYPTO_OK` (vacuous), **but a weak key still →
  `GMCRYPTO_ERR`** even at zero length.
- Byte-identical to `gmcrypto_core::sm4::mode_xts::{encrypt_sectors,
  decrypt_sectors}` (which is byte-identical to looping the single-shot
  `encrypt`/`decrypt` per sector with LE-128 tweaks — transitively OpenSSL
  `xts_standard=GB`-pinned). **Confidentiality only — no auth.**
- **No new dependency**, no new feature flag, no new `gmcrypto-core` API, no
  new dudect target, **no public API breakage** (additive, opt-in).
  Default-features build of both crates byte-unchanged.
- `unsafe_code = "warn"` stays; every new `unsafe` block carries `// SAFETY:`.
  `gmcrypto-core` remains `forbid`.

## 3. Q-list (Q16.1–Q16.12) with answers

### Q16.1 — Scope: anchor only
**C FFI for the SM4-XTS sector helper, nothing else.** No streaming, no
key-scheduled handle, no other `v0.15-Q16.x` candidate. Tightly-scoped,
matching the v0.7→v0.15 cadence.

### Q16.2 — API shape: in-place `buf`, NOT the out-of-place convention
The two functions are **in-place**: `buf: *mut u8` + `buf_len: size_t`, the
sector run transformed in situ, return `GMCRYPTO_OK` / `GMCRYPTO_ERR`. There is
**no** `(out, out_capacity, out_actual_len)` triple.

This is a **deliberate divergence** from the v0.13 single-shot XTS FFI (and
from every other single-shot symbol, which are uniformly out-of-place). The
justification:
- The Rust core (`encrypt_sectors`) is **in-place** (`&mut [u8]`) — chosen in
  v0.15 (Q15.3) precisely because disk callers must not double-allocate a
  whole-device buffer. Forcing the FFI through the out-of-place mold (allocate
  `out`, copy `buf → out`, transform) would defeat the helper's entire reason
  to exist.
- The transform is **length-preserving and whole-block**, so the
  out-of-place convention's only job (communicating a variable / larger output
  length via `out_actual_len`) is moot — output length == `buf_len`, known to
  the caller.
- `try_slice_mut(buf, buf_len)` already exists and is already used elsewhere in
  the shim; the in-place path is *simpler* than `write_output`, not a new
  unsafe shape.

Rejected: an out-of-place `(data, data_len, out, out_capacity,
out_actual_len)` shape for "convention uniformity." It contradicts the core
API, adds a mandatory full-buffer copy, and offers nothing — the single-shot
`gmcrypto_sm4_xts_encrypt` already covers any caller who wants out-of-place
one-data-unit-at-a-time.

### Q16.3 — `start_sector` as `uint64_t`
The Rust core takes `start_sector: u128`. C has **no portable 128-bit integer
type** (`__int128` is a non-standard GCC/Clang extension; cbindgen has no clean
mapping). The FFI takes `start_sector: uint64_t` and widens to `u128`
internally (`start_sector as u128`).

`u64` is the **idiomatic and sufficient** choice for the disk use case: it is
exactly the Linux block-layer `sector_t` width, and `2^64` sectors at the
512-byte minimum is `9.4 ZB` — beyond any addressable device. The Rust core
keeps the full `u128` range for Rust callers; the C narrowing loses nothing a
real disk caller can reach.

A welcome consequence: with `start_sector ≤ 2^64−1` and
`sector_count ≤ buf_len / 16 ≤ 2^60` (on 64-bit `size_t`), the highest sector
number `start + count − 1 < 2^65`, so the core's `u128` overflow check is
**unreachable** through this FFI — the C surface cannot hit the
sector-number-overflow `None`. (The core check stays; it just never fires from
here. The c_smoke suite therefore tests a **high** `start_sector` succeeds,
not an overflow error, since the latter is unreachable.)

Rejected: a 16-byte little-endian `start_sector` pointer (lossless, mirrors the
single-shot's raw-tweak param). The sector helper's *whole point* is to take a
**number** it increments per sector — a caller who wants to hand over raw
16-byte tweaks already has the single-shot API. Also rejected: two `uint64_t`
(lo/hi) — needless complexity for range no disk reaches.

### Q16.4 — Key-size constant
Reuse the existing always-on `GMCRYPTO_SM4_XTS_KEY_SIZE = 32` (added in v0.13).
**No new constant.**

### Q16.5 — Feature flag
Reuse the existing forwarding `sm4-xts = ["gmcrypto-core/sm4-xts"]` on
`gmcrypto-c` (added in v0.13; no new transitive dep — core's `sm4-xts` is
pure-core). The two new functions and the existing
`use gmcrypto_core::sm4::mode_xts;` import are `#[cfg(feature = "sm4-xts")]`.
**No new feature flag.**

### Q16.6 — Header regen (same posture as v0.13)
`regen-header` does **NOT** need to imply `sm4-xts`. cbindgen parses *source*
and does not evaluate `#[cfg(feature)]` for **free functions**, so the two new
sector prototypes appear in the regenerated header **unconditionally** — same
situation as the v0.13 single-shot XTS protos and the v0.9 single-shot AEAD
protos. The cycle adds **no new opaque struct types** (the only thing cbindgen
*drops* under inactive cfg, the v0.10 reason `regen-header` implies
`sm4-aead`). So the existing `regen-header` command regenerates a complete,
deterministic header and the drift gate stays green.

### Q16.7 — Failure modes
Single `GMCRYPTO_ERR` for every error: null `key`/`buf`, invalid `sector_size`,
`buf_len` not a whole multiple of `sector_size`, `Key1 == Key2`. No
distinguishing across the C boundary (failure-mode invariant;
weak-key-/length-oracle defense). The constant-time `Key1 == Key2` reject lives
in core; the FFI just forwards `None`. **`buf` is untouched on any
`GMCRYPTO_ERR`** (the core validates everything before the per-sector loop).
`key` (32 B) is fixed-size by ABI contract (no `key_len` param); the defensive
`try_into` match is kept to mirror the single-shot shape. `buf == NULL` with
`buf_len == 0` is the empty-run case → `try_slice_mut` returns `Some(&mut [])`
→ vacuous `GMCRYPTO_OK` (modulo the weak-key check, which still fires).

### Q16.8 — Testing + dudect
- `tests/c_smoke.rs`, cfg-gated on `sm4-xts`:
  (a) **whole-block multi-sector equivalence + round-trip** — FFI
  `encrypt_sectors` on `buf` == core `encrypt_sectors` on an identical clone;
  FFI `decrypt_sectors` restores the original (e.g. 3 sectors × 512 B,
  `start_sector` non-zero).
  (b) **single sector** (1 × 512) and a **non-zero high `start_sector`**
  crossing an LE-counter byte boundary, == core.
  (c) `sector_size` not a multiple of 16 → `GMCRYPTO_ERR`, **buf untouched**.
  (d) `buf_len` not a whole multiple of `sector_size` → `GMCRYPTO_ERR`, buf
  untouched.
  (e) weak key (`Key1 == Key2`) → `GMCRYPTO_ERR`, buf untouched.
  (f) empty `buf` (`buf_len == 0`) → `GMCRYPTO_OK`; empty `buf` + weak key →
  `GMCRYPTO_ERR`.
  (g) `buf == NULL` with `buf_len > 0` → `GMCRYPTO_ERR`.
- **No new dudect target.** The FFI is a thin shim with no new secret-touching
  logic; the per-sector secret path is already gated by `ct_sm4_xts_decrypt`
  (v0.12 W3, reused for the v0.15 sector helper). The only new logic — the
  sector-number→LE-128 tweak — is on **public** sector addresses.
- C example `examples/sm4_xts_multisector.c` (doc-only; CI does not build C
  examples) — an in-place multi-sector ("disk region") encrypt/decrypt
  round-trip, contrasting the single-shot `sm4_xts_sector.c`.

### Q16.9 — Supply chain + MSRV
No new dependency (`sm4-xts` already forwards to pure-core
`gmcrypto-core/sm4-xts`). MSRV stays **1.85**. The `cargo deny` opt-in pass
already includes `gmcrypto-core/sm4-xts` (no new crates in the graph).

### Q16.10 — No streaming / key-scheduled handle FFI
The in-place whole-run helper covers the disk use, same as the Rust core. A
per-sector streaming cursor or a key-scheduled `gmcrypto_sm4_xts_t` handle is a
future consumer-driven candidate, not v0.16 (matches Q13.10 / Q15 §5).

## 7. Decision history

- **2026-05-29**: scope doc lands as v0.16 W0. Anchor confirmed as **C FFI for
  the SM4-XTS sector helper** (the deferred v0.15 Q15.12 half), selected
  autonomously under the standing `/goal` directive ("complete the development
  of v0.16") on the established core→FFI cadence. Two FFI-shape decisions made
  and documented: (a) **in-place `buf`** (mirrors the core `&mut [u8]`; a
  deliberate divergence from the uniformly out-of-place single-shot convention,
  justified by the no-double-allocation disk use case + length-preservation),
  and (b) **`start_sector: uint64_t`** (C has no portable u128; `u64` is the
  block-layer `sector_t` width and covers all addressable devices; the Rust
  core keeps `u128`; a consequence is the overflow `None` is unreachable from
  the FFI). No new dep / feature / SIMD / dudect target. Real crates.io
  release: workspace `version` `0.15.0 → 0.16.0`. Direction unchanged: private
  feature cadence toward v1.0 (the self-hosted→hosted CI swap + public flip
  stay parked until just before publication). Auto-mode execution with codex
  milestone reviews; publish step (Q16.12) pending user confirmation at the W2
  gate.
- **2026-05-29 (W0 codex review)**: design assessed sound — both FFI-shape
  decisions endorsed ("in-place `buf` is the right divergence … uniform
  out-of-place would force the exact whole-buffer copy this helper exists to
  avoid"; the `uint64_t → u128` overflow-unreachable reasoning "checks out").
  No higher-severity findings. One **MEDIUM**, resolved in W1: the in-place path
  is the only FFI surface that holds a `&mut` over caller memory **alongside**
  the `&` key borrow, so a (caller-error) `key`/`buf` overlap would be `&`/`&mut`
  aliasing UB — fixed by **copying the 32-byte key into an owned array** before
  the `&mut [u8]` is constructed (the key borrow ends first; cheap, no
  double-allocation of the disk buffer). Two **LOW** test-coverage additions
  (also W1): a **null-`key`** error case (buf untouched) and a **decrypt-side**
  error case (guards encrypt/decrypt copy-paste asymmetry); the empty-`buf` test
  now uses the natural `(NULL, 0)` C call. One **LOW** doc carry-forward (W1):
  restate the **tweak-namespace uniqueness** caveat in the C README + example.
  Everything else (failure invariant, `regen-header` ∌ `sm4-xts`, no new dudect
  target, `0.15.0 → 0.16.0` bump + `simd → core → c` order) confirmed.
- **2026-05-29 (W1 codex review)**: implementation assessed sound — **no
  blocking concerns**. The key/buf aliasing fix verified correct ("`k.try_into()`
  copies into owned `[u8; 32]`; NLL ends the shared borrow before
  `try_slice_mut` creates `b`, so caller key/buf overlap is now just 'copy key
  first, then mutate buf' — no residual UB"); byte-equivalence/failure
  forwarding "exact"; `ffi_guard` unwind posture sound; tests free of test-side
  UB; header posture confirmed. One **LOW**: add a direct raw-pointer
  key/buf-**overlap** regression test to lock in the fix — added as
  `sm4_xts_sectors_key_buf_overlap_ok` (overlapping views into one backing
  buffer; result asserted == a non-overlapping reference captured before the
  call). c_smoke is now **11** sector tests.
